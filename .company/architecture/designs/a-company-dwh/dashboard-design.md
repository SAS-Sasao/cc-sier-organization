## 4. Power BI データセット設計

### 4.1 接続モード: Import（推奨）

| 項目 | 設定値 | 理由 |
|------|--------|------|
| **接続モード** | Import | DirectQuery と比較して応答速度が大幅に向上する。日次バッチで十分な鮮度 |
| **データソース** | Gold層テーブル（DWH）から JDBC/ODBC 接続 | マート層を活用することで Power BI 側の変換処理を最小化 |
| **スケジュール更新** | 毎朝 7:00（dbt バッチ完了後） | 1時間のバッファを設けてバッチ完了を待つ |
| **増分更新** | fact_estimates, fact_deals で設定 | ファクトテーブルの全件再取込を避けてパフォーマンスを維持 |

### 4.2 データモデル（Power BI 内）

```
Power BI データモデル（スタースキーマ）

  dim_date ──────────────────────── (date_key)
      │                                   │
      ├── created_date_key → fact_estimates ← customer_key ── dim_customer
      │                           │
      │                        staff_key ── dim_staff
      │
      └── start_date_key  → fact_deals ←── customer_key ── dim_customer
                                │
                             staff_key ── dim_staff

マートテーブル（単独配置、直接クロスフィルタしない）
  mart_sales_summary
  mart_conversion_funnel
  mart_phase_analysis
```

**リレーションシップ設定**:
- ファクト→ディメンションは多対1（`*:1`）、単方向フィルタ
- マートテーブルは他テーブルとのリレーションシップを持たず、独立したビジュアル用テーブルとして配置
- Power BI の DAX 計算列・メジャーは最小限に留め、集計はできる限り dbt マート層で処理する

### 4.3 主要 DAX メジャー

```dax
-- 受注率
受注率 =
DIVIDE(
    COUNTROWS(FILTER(fact_estimates, fact_estimates[is_won] = TRUE())),
    COUNTROWS(FILTER(fact_estimates, fact_estimates[is_won] = TRUE()))
    + COUNTROWS(FILTER(fact_estimates, fact_estimates[is_lost] = TRUE()))
)

-- 前年同期比（受注金額）
受注金額_前年比 =
VAR 当期金額 = SUM(mart_sales_summary[won_amount])
VAR 前年金額 =
    CALCULATE(
        SUM(mart_sales_summary[won_amount]),
        SAMEPERIODLASTYEAR(dim_date[full_date])
    )
RETURN
    DIVIDE(当期金額 - 前年金額, 前年金額)

-- 四半期達成率（目標マスタとの比較）
四半期達成率 =
DIVIDE(
    SUM(mart_sales_summary[won_amount]),
    RELATED(dim_targets[quarterly_target_amount])
)
```

### 4.4 増分更新設定（fact_estimates）

| 設定項目 | 値 |
|---------|-----|
| アーカイブデータ（更新しない） | 1 会計年度以前 |
| 増分更新（更新する） | 当会計年度 |
| リアルタイム更新 | 不要（日次バッチで十分） |
| 検出列 | `dbt_updated_at` |

## 5. RLS（Row Level Security）設計

### 5.1 RLS の設計方針

| 目的 | 設計内容 |
|------|---------|
| 担当者の個人情報保護 | 自部門の担当者データのみ閲覧可能（部門長は全員閲覧可能） |
| 担当者匿名化 | 管理職以外には担当者エイリアス（匿名）のみ表示 |
| 顧客情報保護 | 自担当顧客のみデータ閲覧可能（経営層は全顧客閲覧可能） |

### 5.2 ロール定義

| ロール名 | 対象ユーザー | アクセス範囲 |
|---------|------------|------------|
| `Executive` | 役員・経営企画 | 全データ・実名表示（担当者エイリアス解除） |
| `SalesManager` | 部門長・課長 | 自部門の全データ・実名表示 |
| `SalesRep` | 営業担当者 | 自担当の見積・案件のみ・担当者は匿名表示 |
| `ReadOnly` | 閲覧のみ（経理等） | 全データの読み取り・担当者は匿名表示 |

### 5.3 DAX フィルタ（RLS ルール定義）

```dax
-- ロール: SalesRep（自担当データのみ）
-- テーブル: fact_estimates
[staff_key] = LOOKUPVALUE(
    dim_staff[staff_key],
    dim_staff[employee_id], USERPRINCIPALNAME()
)

-- ロール: SalesManager（自部門のデータのみ）
-- テーブル: dim_staff
[department_code] = LOOKUPVALUE(
    dim_staff[department_code],
    dim_staff[employee_id], USERPRINCIPALNAME()
)

-- ロール: Executive（全データ閲覧）
-- フィルタなし（TRUE()）

-- 担当者表示名の切り替え（全ロール共通のメジャー）
担当者表示名 =
IF(
    OR(
        -- ログインユーザー本人のデータ
        MAX(dim_staff[employee_id]) = USERPRINCIPALNAME(),
        -- 管理職権限
        LOOKUPVALUE(dim_staff[is_manager], dim_staff[employee_id], USERPRINCIPALNAME()) = TRUE()
    ),
    MAX(dim_staff[staff_name_masked]),  -- ※Silver層から実名を別途参照する構成も可
    MAX(dim_staff[staff_alias])
)
```

### 5.4 RLS と匿名化の整合

```
匿名化レベルと表示内容:

[Executive]
  → dim_staff から Silver層の stg_staff に追加結合
  → 実名（staff_name_raw）を表示

[SalesManager]
  → dim_staff の staff_name_masked（部分マスク名）を表示
  → 例: 「山田○○」（苗字のみ表示）

[SalesRep / ReadOnly]
  → dim_staff の staff_alias を表示
  → 例: 「担当者001」
```

> **補足**: RLS は Power BI Service 上でのみ有効。Power BI Desktop では RLS のテスト機能（「ロールとして表示」）を使って検証すること。
