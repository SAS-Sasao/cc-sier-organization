# 13 LLM-as-Judge 品質評価

タスク完了時に成果物を3軸で自動評価し、スコアを継続学習に活用します。

---

## 概要

```
タスク完了（status: completed + docs/ に成果物あり）
  ↓
秘書が成果物を読み取り
  ↓
3軸で評価
  ├── completeness（網羅性）: 0-10
  ├── accuracy（正確性）:     0-10
  └── clarity（明瞭性）:      0-10
  ↓
total = (completeness + accuracy + clarity) / 30 → 0.0〜1.0
  ↓
.task-log/{task-id}.md の ## judge セクションに記録
  ↓
Case Bank に蓄積 → 次回ルーティングに反映
```

---

## 評価の3軸

### completeness（網羅性）

依頼内容に対する必須項目の網羅度を評価します。

| スコア | 基準 |
|--------|------|
| 10 | 必須セクションすべて完備。抜け漏れなし |
| 7-9 | 主要項目あり。軽微な抜け |
| 4-6 | 重要項目が欠落。そのままでは使いにくい |
| 0-3 | 依頼の主要部分が未対応 |

### accuracy（正確性）

技術的な正確さと矛盾のなさを評価します。

| スコア | 基準 |
|--------|------|
| 10 | 技術的に完璧。矛盾なし |
| 7-9 | 概ね正確。軽微な不正確さ |
| 4-6 | 技術的エラーまたは大きな矛盾あり |
| 0-3 | 基本的な技術誤りあり |

### clarity（明瞭性）

依頼の意図をどれだけ正確に理解し反映しているかを評価します。

| スコア | 基準 |
|--------|------|
| 10 | 依頼の意図を完璧に理解 |
| 7-9 | 概ね正しいが解釈のズレあり |
| 4-6 | 依頼の文脈を見落とし |
| 0-3 | 依頼を根本的に誤解 |

---

## トータルスコアと判定

```
total = (completeness + accuracy + clarity) / 30
```

| total | 判定 | Case Bankでの扱い |
|-------|------|-------------------|
| ≥ 0.7 | 高品質 | 高報酬ケースとして活用 |
| 0.4〜0.69 | 要改善 | 失敗パターンを抽出 |
| < 0.4 | 低品質 | 失敗パターンを重点記録 |

---

## タスクログへの記録

評価結果は `.task-log/{task-id}.md` に `## judge` セクションとして追記されます。

```yaml
## judge
completeness: 9
accuracy: 8
clarity: 9
total: 0.87
failure_reason: ""
judge_comment: "必須項目を網羅し技術的にも正確。依頼意図に沿った構成。"
judged_at: "2026-03-27T15:30:00+09:00"
```

`failure_reason` は最も改善すべき点を20字以内で記述します。高品質（total ≥ 0.7）の場合は空文字です。

## 実行タイミング

### 自動実行

秘書のGitワークフロー内で、コミット前に自動実行されます。

```
1. タスク完了（status: completed）
2. 成果物が docs/ 配下に存在するか確認
3. → 存在する場合: LLM-as-Judge 評価を実行
4. → task-log に ## judge セクションを追記
5. git add → commit → push → PR
```

### 評価をスキップするケース

- 成果物が `docs/` 配下にない（ログ・設定ファイルのみの変更）
- タスクが `abandoned` または `in_progress` で終了
- `.task-log/{task-id}.md` に既に `## judge` セクションがある

---

## 継続学習への活用

### Case Bank への蓄積

`/company-evolve` 実行時（または `rebuild-case-bank.sh`）が:

1. 全タスクログの `## judge` セクションをパース
2. reward スコアを `(completeness + accuracy + clarity) / 30` で算出
3. 失敗パターン（`failure_reason`）を集計
4. `.case-bank/index.json` に構造化して格納

### 失敗パターンの自動注入（Read フェーズ）

次のセッションで秘書がタスクを受け取ると、Case Bank から:

```
【{subagent} の既知の失敗パターン】
- ⚠️ {failure_reason}（{count}件発生）← 今回も注意すること
  → 対策: {how_to_apply}

【依頼文に関連する失敗教訓（feedbackメモリより）】
- ⚠️ {failure_reason}
  → 対策: {how_to_apply}
```

をルーティング判断に注入します。高スコアケースと同じアプローチを優先し、既知の失敗パターンを回避する判断が自動的に行われます。

## ダッシュボードでの可視化

`/company-dashboard` に以下の4つのウィジェットが追加されています:

| ウィジェット | 内容 |
|-------------|------|
| Judge Score Trend | 直近30件の評価スコア推移（折れ線グラフ） |
| Subagent Evaluation Radar | Subagentごとの3軸パフォーマンス（レーダーチャート） |
| Low-Score Case Analysis | total < 0.7 のケースと失敗理由（上位5件） |
| Improvement Insights | 軸ごとの弱点分析と改善アクション提案 |

### 改善インサイトの分析ロジック

各軸の平均スコアが 0.9 未満の場合、以下の改善アクションを提案します:

- **completeness が低い場合**: スコープ定義チェックリストの活用を推奨
- **accuracy が低い場合**: 公式ドキュメントURLの付記・事実と推測の区別を推奨
- **clarity が低い場合**: フォーマット標準化・読者ペルソナの明確化を推奨
