凡例

閲覧フロー（静的配信）— 閲覧ユーザー → Route 53 → CloudFront → S3（UI ビルド + レイアウト JSON）
リアルタイム購読・配信 — WebSocket API での接続管理と、イベント処理 Lambda からの PostToConnection ブロードキャスト
イベント収集・取り込み — Claude Code hooks → ローカル Relay → Ingest API → Lambda → DynamoDB（TTL 30日）
認証フロー — Cognito でのサインイン / JWT 取得と、WebSocket 接続時のオーソライザ検証

概要

AI Virtual Office（Claude Code のエージェント活動を CC-SIer 仮想組織のピクセルオフィスとして可視化するアプリ）を、Vercel ではなく AWS でホストする場合の参照構成。要件定義 v0.1 の制約 — ① hooks はローカルにしか POST できないためローカル Relay からクラウドへ転送する形は不変、② リアルタイム配信には常駐接続が必要、③ プロンプト本文等の機微情報はクラウドへ送らない — を AWS ネイティブに翻訳した。

Vercel 構成で外部 SaaS（Supabase Realtime）に頼っていたリアルタイム層を API Gateway WebSocket API + DynamoDB 接続テーブル + PostToConnection で AWS 内に完結させるのが最大の違い。UI は Next.js の静的ビルドを S3 + CloudFront（OAC）で配信し、閲覧認証は Cognito ユーザープールが担う。全コンポーネントがサーバーレスのため VPC・EC2 は一切不要で、アイドル時のコストはほぼゼロに収まる。

データフロー

取込 Claude Code (hooks)→ Relay（正規化・機微情報フィルタ）→ API Gateway HTTP API→ Lambda（イベント処理）→ DynamoDB イベントテーブル
配信 Lambda（イベント処理）→ 接続テーブル参照→ WebSocket API (PostToConnection)→ 閲覧ユーザーのブラウザ（オフィス描画）
閲覧 閲覧ユーザー→ Route 53→ CloudFront→ S3（UI + レイアウト JSON）
認証 閲覧ユーザー→ Cognito（JWT 取得）→ WebSocket $connect 時にオーソライザ検証

取り込み → 配信の一気通貫: Relay が Bearer トークン付きで POST /ingest に送った正規化イベントを、イベント処理 Lambda が ① イベントテーブルへ永続化（expireAt による TTL 30 日）、② 接続テーブルの全接続へ PostToConnection でブロードキャスト、の 2 系統に流す。ブラウザ側は WebSocket で受けたイベントをキャラ状態機械に適用する。リプレイはイベントテーブルの時刻範囲クエリで同じ OfficeEvent ストリームを再構成するだけで実現できる。

接続ライフサイクル: $connect/$disconnect ルートが接続管理 Lambda を起動し、接続 ID を接続テーブルに登録・削除する。接続にも TTL を付け、切断イベントの取りこぼしによるゴースト接続を自動掃除する（PostToConnection の 410 Gone でも削除）。

設計のポイント

1. リアルタイム層を AWS 内で完結（Supabase Realtime の置き換え） — Vercel 案では常駐 WebSocket を持てず外部 SaaS に配信を委ねていたが、AWS では API Gateway WebSocket API が接続維持をマネージドで担う。接続状態は DynamoDB に外出しするステートレス設計のため、Lambda はスケールしても配信先を見失わない。外部依存が 1 つ減り、認証（Cognito）とも同一エコシステムで完結する。

2. ingest と配信を 1 つの Lambda に集約 — イベント量（1 ユーザー数十件/分）では Stream 分離や SQS 挿入は過剰。保存と配信を同一関数で行い、遅延を 1 ホップに抑えた。スパイクや順序保証が必要になった段階で DynamoDB Streams → 配信 Lambda に分離できる進化パスを残している。

3. 機微情報は境界（Relay）で落とす — プロンプト本文・ファイル内容はローカルの Relay で破棄し、クラウドにはツール名・パス程度の正規化イベントしか送らない。ingest API は Bearer トークン + スロットリングで保護し、書き込み経路と閲覧経路（Cognito）を別系統の認証にしている。

4. VPC レス構成の割り切り — RDS も EC2 も無いため VPC・NAT Gateway を作らない。固定費が発生するのは事実上ゼロで、Dev 環境の月額は数十ドルに収まる。将来 RDS 等が必要になっても、この構成に VPC を後付けする必要はない（DynamoDB は IAM 認可のパブリックエンドポイント）。

5. TTL によるデータライフサイクル管理 — イベントは 30 日、接続 ID はセッション相当の TTL を expireAt 属性に持たせ、削除バッチを書かずにローテーションする。リプレイ用の長期保管が必要な日は S3 へ日次エクスポートする拡張余地を残す。

前提条件

Dev: 開発者 1 名・イベント数十万件/月・閲覧数名。Prod: 常時公開・閲覧同時 50 接続・イベント数百万件/月を想定。全サービスが従量課金のため、アイドル時（夜間・週末に誰も見ていない状態）のコストは Route 53 とストレージのみのほぼゼロに漸近する。

コスト最適化のポイント

WebSocket のメッセージ課金はブロードキャスト先接続数に比例するため、閲覧者が増えたら「同一イベントのバッチ集約（100ms ウィンドウ）」で送信回数を圧縮する。CloudWatch Logs は保持 30 日に設定（無期限が既定のため設定漏れが最頻の無駄コスト）。イベントテーブルは TTL に任せ、長期分析が必要になった時点で S3 エクスポート + Athena に移す。

学習ポイント

  • サーバーレスにおける WebSocket の定石パターン — 「API Gateway WebSocket + 接続テーブル（DynamoDB）+ PostToConnection」は、常駐プロセスを持てないサーバーレス環境でリアルタイム双方向通信を実現する AWS の定石。接続状態をデータストアに外出しして計算層をステートレスに保つ考え方は、チャット・通知・ライブダッシュボードなど広く応用できる。
  • 「ローカルの事実をクラウドに運ぶ」ときの境界設計 — ローカルでしか観測できないイベント（hooks）は、ローカル常駐の転送プロセス（Relay）で正規化・フィルタしてから送る。機微情報の destructive filtering を送信側の境界で行う原則は、テレメトリ収集・エッジ IoT・監査ログ転送すべてに共通する。
  • マネージドサービス選定は「置き換え可能な単位」で比較する — 本構成は Vercel + Supabase Realtime の各役割（ホスティング / 配信 / DB / 認証)を CloudFront+S3 / WebSocket API / DynamoDB / Cognito に 1:1 で写像した。プラットフォーム移行の検討は「サービス名」でなく「役割の単位」で対応表を作ると、差分とロックイン箇所が明確になる。
  • 従量課金アーキテクチャの固定費ゼロ設計 — VPC/NAT/EC2/RDS を避けて全コンポーネントを従量課金にすると、利用がない時間帯のコストがほぼゼロになる。個人開発・社内ツール・PoC では「アイドル時に何にいくら払うか」を最初に洗い出すことが、構成選定の実質的な決定打になる。
  • TTL 駆動のデータライフサイクル — 削除バッチを実装せず DynamoDB TTL に任せる設計は、「データの寿命を書き込み時に決める」イベントソーシング系システムの基本形。保持要件が伸びたら削除ではなく安価なストレージ（S3/Glacier）への段階移行で応える。
