凡例

本番トラフィック — 切替後の利用者アクセス。Route 53 → ALB → EC2 → RDS for Oracle / EFS
サーバー移行フロー — MGN が APP サーバーを Direct Connect 経由でブロックレプリケーションし EC2 へカットオーバー
DB 移行フロー — DMS が Oracle の REDO ログを読み取り、フルロード + CDC で RDS for Oracle へ継続同期
ファイル移行フロー — DataSync が NAS の共有ファイルを EFS へ、アーカイブを S3 へ転送
バックアップ経路（破線）— Direct Connect 障害時に Site-to-Site VPN で迂回する冗長経路

概要

オンプレミスの基幹システム（Oracle Database・業務 APP サーバー・ファイルストレージ（NAS））を AWS へ移行するための参照アーキテクチャ。移行方式は AWS の移行フレームワーク「7R」に基づき、APP サーバーは Rehost（AWS Application Migration Service / MGN によるリフト&シフト）、Oracle DB は Replatform（DMS によるマネージド RDS for Oracle への載せ替え）、ファイルは DataSync による EFS / S3 への転送を採用する。

オンプレと AWS 間は Direct Connect を主回線とし、移行レプリケーションと業務通信の帯域を確保する。障害時は Site-to-Site VPN が迂回路として機能する。移行期間中は MGN・DMS が継続レプリケーションで両環境を同期し続け、切替は Route 53 の加重ルーティングで段階的に実施することで、ビッグバン切替を避けてダウンタイムとリスクを最小化する。切替後の本番系は ALB + EC2（Private Subnet）+ RDS for Oracle（Multi-AZ）+ EFS で構成し、CloudWatch による監視と IAM による最小権限管理を土台に置く。

データフロー

移行 APP サーバー→ Direct Connect→ MGN→ EC2（カットオーバー起動）
移行 Oracle Database→ Direct Connect→ DMS→ RDS for Oracle（フルロード + CDC）
移行 ファイルストレージ (NAS)→ Direct Connect→ DataSync→ EFS（共有）/ S3（アーカイブ）
本番 利用者→ Route 53→ ALB→ EC2→ RDS for Oracle / EFS
冗長 オンプレミス→ Site-to-Site VPN→ VPC（DX 障害時の迂回）

移行フェーズ: MGN のレプリケーションエージェントを APP サーバーに導入し、ブロック単位で EC2 のステージング領域へ継続同期する。DMS は Oracle の REDO ログを読み取る CDC（Change Data Capture）で、フルロード完了後も差分を RDS へ適用し続ける。DataSync は NAS（NFS/SMB）をスケジュール同期で EFS / S3 に転送し、増分のみを送ることで回線負荷を抑える。

切替（カットオーバー）: 両環境のデータ同期が追いついた状態で、Route 53 の加重ルーティングにより AWS 側への流量を段階的に増やす（例: 10% → 50% → 100%）。問題発生時はレコードの重み変更だけでオンプレへ即時切り戻し可能。

設計のポイント

1. 7R に基づくワークロード別の移行方式選定 — 全てを一律リフト&シフトせず、APP サーバーは改修コストを避けて Rehost（MGN）、Oracle DB は運用負荷削減効果が大きい Replatform（RDS for Oracle） を選択。DB をマネージド化することでパッチ適用・バックアップ・フェイルオーバーの運用をオフロードし、移行を「単なる引っ越し」ではなく運用改善の機会にしている。

2. 継続レプリケーションによるダウンタイム最小化 — MGN のブロックレプリケーションと DMS の CDC はいずれも「動かしたまま同期し続ける」方式。停止を伴うのはカットオーバーの最終静止点のみで、業務停止時間を数時間から数分オーダーに圧縮できる。切替自体も Route 53 加重ルーティングで段階的に行い、切り戻し手段を常に確保する。

3. 回線の冗長化と帯域設計 — 初回フルロード（DB 全量 + ファイル全量）は転送量が大きく、帯域保証のある Direct Connect を主回線とした。VPN は暗号化オーバーヘッドと帯域変動があるためバックアップに限定。DataSync の帯域スロットリングで業務時間帯の回線圧迫を防ぐ。

4. データ特性による格納先の使い分け — 複数 EC2 が同時参照する共有ファイルはマネージド NFS の EFS、参照頻度の低いアーカイブは S3（+ Glacier ライフサイクル）へ振り分け、性能要件とコストを両立させる。

5. Oracle ライセンス戦略 — RDS for Oracle は SE2 の license-included を採用し、ライセンス管理を月額費用に内包。EE 機能（パーティショニング等）に依存している場合は BYOL か、将来的な Aurora PostgreSQL への Refactor（SCT による変換評価）を次段階の検討事項とする。

前提条件

24/7 稼働、移行期間 3 ヶ月想定（DMS / DataSync / MGN の費用は移行完了後に発生しなくなる）。DB 200GB・ファイル 1TB の初回転送を想定。Direct Connect はパートナー経由のホスト型接続の概算。

コスト最適化のポイント

移行完了後は DMS レプリケーションインスタンスと MGN ステージングリソースを必ず削除する（消し忘れが最頻の無駄コスト）。定常化後は EC2 / RDS に 1 年 Savings Plans / リザーブドインスタンス適用で 30-40% 削減。Dev 環境は夜間・休日停止のスケジューリングで約 60% 削減。S3 は Glacier ライフサイクルでアーカイブ費用を 1/5 に圧縮。

学習ポイント

  • 移行の 7R フレームワーク — Rehost / Replatform / Refactor / Repurchase / Relocate / Retain / Retire の選択肢からワークロード単位で方式を決める。全システム一括ではなく「APP は Rehost・DB は Replatform」のように分解して判断することが、移行コストと改善効果のバランスを取る普遍的な出発点になる。
  • CDC（Change Data Capture）による無停止データ移行 — トランザクションログ（Oracle なら REDO ログ）を読み取って差分を伝搬する仕組みは、DMS に限らず DB 移行・レプリケーション・イベント駆動連携（Debezium 等）に共通する基盤技術。「フルロード + 差分追従 + 静止点切替」の 3 段構えはあらゆるデータ移行に応用できる。
  • 段階的カットオーバーと切り戻し設計 — DNS 加重ルーティングによるカナリア切替は、移行に限らずブルー/グリーンデプロイやリージョン間フェイルオーバーでも同じ原理。切替の成功条件よりも「失敗したときに何分で戻せるか」を先に設計するのが実務の鉄則。
  • ハイブリッド接続の冗長設計 — 帯域保証が必要な定常転送は専用線（Direct Connect）、暗号化された即席の迂回路は VPN という役割分担は、オンプレ連携が残るすべてのエンタープライズ構成に共通するパターン。単一回線障害を単一障害点にしないことが移行期間中の事業継続を支える。
  • 移行は運用改善の機会 — 単純な引っ越し（Rehost のみ）では TCO 削減効果は限定的で、マネージドサービスへの載せ替え（Replatform）を組み合わせて初めてパッチ・バックアップ・冗長化の運用工数が削減される。「何を変えずに運び、何をマネージドに委ねるか」の線引きがクラウド移行設計の本質。
