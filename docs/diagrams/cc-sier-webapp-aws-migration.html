概要

cc-sier-organization Web アプリの Phase 4「AWS 移行検討」フェーズに対応する将来構成図。Phase 1（Vercel + Railway 版）が利用人数 50 人超 / 月額 API > $500 / セキュリティ要件アップ等で発動した場合に到達する想定アーキテクチャ。

Phase 1 の各 PaaS を AWS マネージドサービスへ等価置換: Vercel→App Runner、Railway Docker→ECS Fargate、Neon Postgres→Aurora PostgreSQL Serverless v2 (pgvector)、Vercel Blob→S3、Upstash Redis→ElastiCache for Redis Serverless、Anthropic API 直叩き→Amazon Bedrock (Claude on Bedrock)、Microsoft Entra ID は Cognito User Pool で OIDC Federation。Phase 1 の多層 Budget Cap・KILL_SWITCH・Audit Log・3 種バッチは AWS ネイティブ機能（Secrets Manager / SSM Parameter Store / CloudWatch Logs / EventBridge Scheduler）で再実装し、運用観測性とセキュリティ境界を強化する。

データフロー

Employees→ Route 53→ WAF→ CloudFront→ App Runner (Next.js)→ Internal ALB→ Fargate (Hono + Agent SDK)→ Bedrock

認証フロー: ユーザー → App Runner → Cognito User Pool（OIDC）→ Microsoft Entra ID（Federation）→ Cognito → JWT 発行 → App Runner

データ永続化フロー: Fargate → Aurora（run_events / runs / cost_pool / case_bank vector）/ S3（artifact: PNG / .drawio / IaC zip）/ ElastiCache（rate-limit KVS）

運用イベントフロー: EventBridge Scheduler → Fargate（cron: Catalog upsert / Case Bank Indexer / Blob TTL Cleanup）。SSM Parameter Store の KILL_SWITCH=on 切替 → EventBridge → Fargate に即停止指示。

監査・アラートフロー: Fargate audit logs → CloudWatch Logs（90 日保持）→ CloudWatch Alarms（コスト > $10/日 / エラー率 > 10%）→ SNS → AWS Chatbot → Slack Webhook。

設計のポイント

  • App Runner で Next.js を最小運用負荷でホスト — Vercel から最小の差分で移行できる SSR + Server Actions + SSE 対応のマネージド ECS。VPC Connector で Backend Tier (ALB) へ private 接続し、IGW 経由の余計な出口を作らない。
  • Bedrock 化で個人 API キー暴走リスクを完全排除 — Phase 1 の Anthropic API 直叩きを Bedrock InvokeModel に置換し、IAM ポリシー + Service Quota + CloudWatch Bedrock メトリクスで「per-request / per-user / pool」3 段階の Budget Cap を AWS ネイティブに再実装。Phase 1 の `maxBudgetUsd` ロジックは Application 側に残しつつ二重防御。
  • Aurora Serverless v2 で pgvector を維持しつつ自動スケール — Neon の課金体系から Aurora に移行することで Multi-AZ + 自動バックアップ + IAM 認証を獲得。pgvector 拡張は v15 以降ネイティブ対応で Phase 1 の Case Bank 索引互換性を維持。最小 0.5 ACU から課金、夜間スリープ可。
  • SSM Parameter Store + EventBridge で KILL_SWITCH を即時伝播 — Phase 1 の Railway 環境変数 `KILL_SWITCH=on` を SSM Parameter Store の /cc-sier/kill-switch に置換。値変更を EventBridge で検知して Fargate にイベント送信、再デプロイなしで即停止。Phase 1 の「朝起きて青ざめないための装置」設計思想を維持。
  • EventBridge Scheduler で 3 バッチを統一管理 — Phase 1 の Catalog 自動 upsert / Case Bank Indexer / Blob TTL Cleanup を EventBridge Scheduler の単一管理画面に集約。CloudWatch メトリクス連動で SLA 監視、失敗時 SNS → Slack 通知。Lambda 化も将来オプションとして残す。

前提条件

東京リージョン (ap-northeast-1) で全サービスを構成。Prod は 30 ユーザー × 平均 5 runs/h × 8h = 1,200 runs/日想定。Bedrock は Claude Sonnet 4.6 主体（Opus 4.7 は管理者承認時のみ）で計算。Aurora は I/O 最適化を有効化、ElastiCache は Multi-AZ。

コスト最適化

Aurora Serverless v2 の最小 ACU を夜間 0.5 にスケールダウン（年間 ~30% 削減）、CloudFront のキャッシュ最適化で WAF リクエスト数削減、Bedrock の Provisioned Throughput を Prod でも避け On-Demand 維持（小規模では不利）、S3 Intelligent-Tiering で artifact の自動階層化、未使用時間帯は App Runner を on-pause に。

学習ポイント

  • PaaS to AWS 移行の本質はコスト構造の置換 — Vercel/Railway/Neon は使った分だけ + Pro プラン課金で最小コスト、AWS は IGW・NAT・ALB・VPC Endpoint 等の「インフラ常駐コスト」が乗る。利用人数 50 人超で API コスト > $500/月になったときに「常駐コストの相対比率が下がる」損益分岐点を計算してから移行判断する。本構成では Phase 1 $272 → Phase 4 $818（3 倍）。
  • 個人 API キーから IAM への移行はガバナンスの段階的成熟 — Phase 1 は ANTHROPIC_API_KEY 個人キー + アプリ層 Budget Cap という「実装で守る」設計。Phase 4 は Bedrock IAM ポリシー + Service Quota + CloudWatch Bedrock メトリクスで「インフラで守る」設計に進化。3 段階 Budget Cap（per-req / per-user / pool）の概念は両方で維持しつつ実装層が変わる。
  • マネージドサービスの粒度選択が運用負荷を決める — Next.js を Lambda + API Gateway にせず App Runner を選んだのは「Server Actions / SSE / 中規模トラフィック」のスイートスポット。逆に Backend は ECS Fargate でコンテナ単位の柔軟性を維持し、cold start を避ける。1 つのアプリでも Tier ごとに最適なサーバーレス度合いが異なる。
  • VPC Endpoint で外部接続コストとリスクを最小化 — Bedrock / S3 / Secrets Manager / CloudWatch Logs を VPC Endpoint 化することで NAT Gateway 経由のデータ転送料金を削減 + プライベート通信化。本構成では月 4 × $7 = $28 のエンドポイント代でデータ転送 $50+ を削減。セキュリティ境界が IGW から VPC 内に閉じる副次効果も。
  • Phase 移行は「同じ設計思想を別の実装で再現」する練習 — KILL_SWITCH（env var → SSM）/ Budget Cap（app code → IAM Quota）/ Audit Log（DB → CloudWatch Logs）/ Batch（自前 cron → EventBridge）といったコンセプトは普遍的で、PaaS と AWS で実装が変わるだけ。「動く設計を別レイヤで再構築できるか」が SIer の腕の見せどころで、本図はその対比教材としても利用できる。
