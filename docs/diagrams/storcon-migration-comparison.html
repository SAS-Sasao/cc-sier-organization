概要

コンビニエンスストアのストアコンピューターシステムをオンプレミスからAWSクラウドへ移行する全体構成を1図で可視化。左上の On-Premise（EOS/EOB、Oracle DB、SQL Server DWH、Store Computer）から、DMS/SCTを経由して、AWS Cloud（ECS Fargateマイクロサービス、Aurora、DynamoDB、Redshift等）へ移行するデータフローと、移行後のクラウドネイティブアーキテクチャを表現。

設計のポイント

  • DMS/SCTによる段階的DB移行 — Oracle → Aurora（トランザクション）と SQL Server → Redshift（分析）を DMS で並行移行。CDC（Change Data Capture）でリアルタイム同期し、カットオーバー時のダウンタイムを最小化
  • マイクロサービス分割 — モノリスの EOS/EOB を 4つのドメインサービス（発注/在庫/売上/従業員）に分割。各サービスが独立してスケール・デプロイ可能
  • SQS FIFO による発注二重処理防止 — 発注処理は exactly-once 配信が必要なため SQS FIFO を採用。Standard キューでは at-least-once のため二重発注リスクあり
  • ElastiCache によるマスタキャッシュ — 商品マスタは読み取り頻度が極めて高いため Redis キャッシュを配置。DynamoDB への直接アクセスを軽減し、レイテンシ改善とコスト削減を両立

学習ポイント

  • DMS/SCTの使い分けと移行戦略 — SCT（Schema Conversion Tool）はスキーマ変換、DMS（Database Migration Service）はデータレプリケーションを担当する。異種DB間移行（Oracle → Aurora）ではSCTが必須。CDC による継続的レプリケーションでカットオーバー時のダウンタイムを数分に短縮できる
  • マイクロサービスにおけるデータベース分離パターン — モノリスの単一DBを、サービスごとに最適なDBへ分離する「Database per Service」パターン。トランザクションはAurora、高頻度参照はDynamoDB+ElastiCache、分析はRedshiftと、ワークロード特性に応じたポリグロット永続化が大規模システムでは標準
  • SQS FIFOとStandardの選択基準 — 発注処理のような「二重実行が許されない」処理にはFIFO（exactly-once配信）が必須。ログ転送のような「多少の重複が許容される」処理にはStandard（高スループット）を選択。コスト差は約4倍だが、業務要件に基づいて判断する
  • Direct Connectとハイブリッド構成 — オンプレ→AWS移行期間中はDirect Connect（専用線）でハイブリッド構成を維持する。インターネットVPNより低レイテンシ・高帯域で、大量データの移行トラフィックにも耐える。移行完了後も本部DC接続に継続利用するケースが多い
  • 移行コストの段階的最適化 — 移行直後はDev構成（$430-710/月）で運用開始し、トラフィック増加に応じてProd構成へスケール。Savings Plans やReserved Instancesは6ヶ月以上の安定運用後に適用するのがベストプラクティス
