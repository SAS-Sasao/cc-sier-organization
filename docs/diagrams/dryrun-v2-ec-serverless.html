概要

小売 EC サイトのサーバーレスバックエンド参照構成。フロントエンドの SPA は S3 + CloudFront で静的配信し、商品閲覧・カート・注文 API は API Gateway + Lambda + DynamoDB の同期パスで処理する。認証は Cognito ユーザープールで JWT を発行・検証する。

注文確定のような時間のかかる後続処理は SQS で受け止めて非同期化し、worker Lambda がデキューして在庫引当・注文ステータス更新（DynamoDB）と SNS による確定通知を行う。これによりユーザーへの API 応答は即時返しつつ、ピーク時のスパイクを SQS が吸収する。CloudWatch でメトリクス・ログを集約し、Lambda 実行は最小権限の IAM Role で制御する。本ファイルは company-diagram-v2（draw.io XML 直接生成方式）のドライ実行成果物。

データフロー

同期 買い物客→ Route 53→ CloudFront→ S3 静的サイト / 商品画像
同期 CloudFront→ API Gateway→ Lambda (商品/注文API)→ DynamoDB
非同期 Lambda (API)→ SQS (注文キュー)→ Lambda (worker)→ SNS 通知→ 買い物客

認証フロー: クライアントが Cognito で取得した JWT を Authorization ヘッダで送信 → API Gateway の Cognito オーソライザが検証 → 通過したリクエストのみ Lambda へ到達。

非同期処理フロー: 注文確定 API は注文を SQS にエンキューして即時 202 を返す。worker Lambda がイベントソースマッピングでデキューし、DynamoDB の注文ステータスを更新し、SNS トピックへ確定イベントを発行。SNS が購読者（メール / モバイルプッシュ）へファンアウトする。失敗メッセージは DLQ（maxReceiveCount=5）へ退避。

設計のポイント

  • 同期パスと非同期パスを明確に分離 — 商品照会のような即応性が要る処理は API Gateway→Lambda→DynamoDB の同期パスに、注文確定後の在庫引当・通知のような遅延許容処理は SQS→worker Lambda の非同期パスに分ける。ユーザー体感のレイテンシを守りつつ、後段の重い処理をキューで平準化する。
  • SQS + DLQ で注文ロストを防ぐ — 注文確定イベントは SQS に永続化してから worker が処理するため、worker 障害時もメッセージは消えない。maxReceiveCount=5 を超えた毒メッセージは DLQ に隔離し、本処理を止めずに後追い調査できる。注文＝失ってはいけないデータの典型パターン。
  • S3 は OAC でプライベート化し CloudFront 経由のみ公開 — 静的サイト・商品画像バケットはパブリックアクセスを全ブロックし、CloudFront Origin Access Control 経由でのみ到達可能にする。バケット直叩きを塞ぎ、WAF/署名 URL などの制御点を CloudFront に集約する。
  • 認可を API Gateway のオーソライザに寄せる — JWT 検証を各 Lambda に実装せず Cognito オーソライザに一元化することで、Lambda 関数は業務ロジックに専念できる。認証ロジックの重複と実装ミスを減らす。
  • 従量課金サービスで構成しアイドルコストを最小化 — Lambda・DynamoDB On-Demand・SQS・SNS はいずれもリクエスト課金で、トラフィックが無ければほぼ無料。EC の波のあるトラフィックに対し、固定費の薄いサーバーレス構成が相性良い。

前提条件

Dev は数十ユーザーの検証規模（月 ~50万リクエスト）、Prod は中規模 EC（月 ~1,000万リクエスト / ~10万注文）を想定。全サービス従量課金で、夜間・閑散期はアイドルコストがほぼ発生しない。CloudFront 転送量は 100GB/月（Prod）で試算。IAM・SQS DLQ は実質無料。

コスト最適化のポイント

DynamoDB はアクセスパターンが安定したら On-Demand → Provisioned + Auto Scaling へ切替で 2〜4 割削減。CloudFront のキャッシュ TTL を伸ばしてオリジン転送とリクエスト数を削減。Lambda は ARM (Graviton) ランタイムで ~20% 単価削減。S3 は Intelligent-Tiering で画像の自動階層化。SQS バッチサイズを大きくして worker Lambda の起動回数を削減。

学習ポイント

  • キューは「失ってはいけない処理」を守る緩衝材 — 同期 API の中で重い処理まで完結させると、後段障害がユーザー応答に直結し、リトライでデータが二重化するリスクもある。SQS を挟むと「受け付け」と「実行」が分離され、worker 障害時もメッセージは残り、DLQ で毒メッセージを隔離できる。注文・決済・在庫など失ってはいけないイベントに普遍的に効くパターン。
  • 従量課金サービスの組み合わせはアイドルコストを限りなくゼロに近づける — Lambda / DynamoDB On-Demand / SQS / SNS はリクエストが無ければ課金されない。トラフィックに波がある EC・予約・キャンペーン系では、常時起動のサーバー（EC2/RDS）より損益分岐が低くなりやすい。逆に常時高負荷ならプロビジョン型の方が安くなる損益分岐の見極めが設計判断の核心。
  • 認可・暗号化・最小権限は「点」ではなく「層」で設計する — Cognito オーソライザ（認可の集約点）、S3 OAC（直アクセス遮断）、IAM Role の最小権限（横移動の封じ込め）、KMS 暗号化（保管時保護）を組み合わせて初めて多層防御になる。どれか一つでは破られたとき後がない。セキュリティは最初から各層に織り込む。
  • 非同期化は「結果整合性」を受け入れる設計判断 — SQS を挟むと注文受付と確定通知の間に遅延が生まれる。即時に「確定」を見せたいのか、「受付しました」で十分なのかという UX 要件が、同期/非同期の境界を決める。技術選定の前にユーザー体験の許容遅延を定義することが先。
  • マネージドサービス間の結合はイベント駆動で疎結合に保つ — Lambda が SNS/SQS を介して連携することで、購読者（メール・プッシュ・分析）を後から追加してもプロデューサー側を変更せずに済む。ファンアウトで拡張余地を残す設計は、機能追加の速度を落とさないための投資。
