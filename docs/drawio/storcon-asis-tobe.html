概要

コンビニエンスストアのストアコンピューター（ストコン）をオンプレミスからAWSクラウドへ移行する際の、現行構成（As-Is）と移行後構成（To-Be）を左右対比で可視化した図。中央に3フェーズの移行ステップ（Rehost → Replatform → Refactor）と主要コンポーネントの対応関係を示す。

設計のポイント

  • 段階的移行（3フェーズ） ― Phase 1: Rehost（EC2へのLift&Shift）でリスク最小化 → Phase 2: Replatform（コンテナ化・マネージドDB化）→ Phase 3: Refactor（マイクロサービス＋イベント駆動化）
  • オフライン耐性の維持 ― 移行後もローカルDBを維持し、店舗ネットワーク障害時の業務継続を保証。差分同期で復旧時にクラウドと整合
  • データ層の用途別最適化 ― 単一RDBMSからAurora（ACID）、DynamoDB（高頻度アクセス）、ElastiCache（キャッシュ）、Redshift（分析）に分割し、各ワークロードに最適化
  • 温度監視のIoT化 ― 既存の店舗内温度監視をIoT Coreに移行し、センサーデータのクラウド収集・異常検知を実現

学習ポイント

  • 7R移行戦略の段階的適用 ― 24/365稼働のシステムでは一括移行は高リスク。Rehost→Replatform→Refactorの段階適用が王道パターン。各フェーズでロールバック可能な状態を維持する
  • モノリスからマイクロサービスへの分解基準 ― ストコンの業務機能（発注・在庫・売上・従業員）はそれぞれ独立したビジネスドメインを持つ。ドメイン境界に沿った分割がマイクロサービス設計の基本
  • ハイブリッド期間の設計 ― 移行期間中はオンプレとAWSが共存する。Direct Connectによる専用線接続と、ストラングラーフィグパターンによるAPIルーティング層が共存を可能にする
  • 非機能要件の移行先マッピング ― 24/365可用性 → Multi-AZ + Auto Scaling、PCI DSS → WAF + KMS + GuardDuty、スケーラビリティ → SQS背圧制御 + DynamoDB On-Demand。As-Isの課題がTo-Beのどの要素で解消されるかを対応付けることが重要
  • エッジコンピューティングの継続 ― クラウド移行後もローカルDBを残す設計は、コンビニ特有のオフライン耐性要件に起因する。すべてをクラウドに寄せるのではなく、業務継続要件に応じたエッジ処理の残し方が実案件では問われる
