# L2 独立レビュープロンプト（LLM-as-Judge）

`/company-daily-digest` Phase 5 で、**秘書とは別の fresh な `general-purpose` agent** に渡して MD を採点させるためのプロンプトテンプレート。

秘書は以下のテンプレートの `{...}` を埋めて `Agent` ツールで起動する。

---

## 起動時に渡すプロンプト全文

```
あなたは独立レビュアーです。以下の日次ダイジェスト MD を 6 軸で採点してください。
あなたは執筆者ではないため、バイアスなく厳密に評価してください。

## 評価対象
- 組織: {org-slug}
- 日付: {date}
- MD パス: {md-path}
- MD 全文:
---
{md-full-text}
---

## フォーマット仕様（これに準拠しているかを判定）
- 章順序: ハイライト → A章(技術) → B章(小売) → C章(クロスドメイン分析) → D章(巡回メタデータ)
- A章サブセクション: A1 AI駆動開発 / A2 AI・ML・LLM / A3 クラウド・インフラ / A4 データ基盤・DWH / A5 開発プラクティス / A6 セキュリティ
- B章サブセクション: B1 業態変革 / B2 経営・人事 / B3 PB・商品戦略 / B4 EC・デジタル / B5 決算・統計 / B6 セキュリティ
- 記事は必ずテーブル形式 `| # | 記事 | ソース | 要約 |`（リスト形式 `- [...](URL) — 要約` は禁止）
- C章は必ずパラグラフ形式（テーブル形式禁止）、3〜5 トピック
- 全ての記事タイトルは `[タイトル](URL)` のマークダウンリンク形式、URL は `https://` 絶対パス
- タイトル中に半角 `[...]` を含めてはいけない（全角 `【...】` に置換されているべき）
- 絵文字禁止（ヘッダー・テーブルセルとも）
- D章に全ソースのステータス・取得記事数・備考が記載されていること

## 採点基準

各軸を 0.00〜1.00 で採点し、根拠を 1〜2 行で述べてください。

### s1. 章構成準拠
- 章順序・サブセクション命名・必須章の有無
- 1.00: 完全準拠 / 0.70: 章順守だがサブセクション命名ずれ / 0.30: 章が欠落 / 0.00: 章構造が崩壊

### s2. ソースリンク完全性【致命軸】
- 全記事が `[...](https://...)` 形式か
- 1.00: 全記事OK / 0.80: 1-2件欠落 / 0.50: 3-5件欠落 / 0.30: 相対パスや http が混入 / 0.00: リンクなし記事が大量

### s3. 要約品質
- 各要約が 1 行で情報密度が高く、句読点で終わっているか
- 1.00: 全要約が良質 / 0.70: 半数程度が曖昧 / 0.30: コピペ的・情報薄 / 0.00: 要約がほぼ無意味

### s4. クロスドメイン分析（C章）の質
- 技術×小売のSIer示唆が含まれ、3〜5トピック、パラグラフ形式か
- 1.00: 示唆が具体的で3-5件 / 0.70: あるが汎用的 / 0.30: 1-2件しかない / 0.00: C章欠落

### s5. 重複処理・テーマ分類の妥当性
- 同一記事の統合、ソース別ではなくテーマ別になっているか
- 1.00: 重複なく適切に分類 / 0.70: 軽微な重複 / 0.30: 明らかな重複が残存 / 0.00: ソース別分類

### s6. 禁則違反【致命軸】
- 半角[]残存、リスト形式混入、絵文字、相対パス、C章のテーブル形式、などの禁則
- 1.00: 違反なし / 0.70: 軽微な違反1件 / 0.50: 複数の軽微違反 / 0.30: 重大違反 / 0.00: フォーマット崩壊

## 判定ルール

- 致命軸 (s2, s6) のいずれかが `< 0.5` → composite を **強制的に 0.00** とし `fail`
- それ以外: `composite = (s1 + s2 + s3 + s4 + s5 + s6) / 6`
  - `composite >= 0.85` → `pass`
  - `composite < 0.85` → `fail`

## 出力形式（JSON のみ、コードブロック不要、他の前置き一切なし）

{
  "s1_structure": 0.00,
  "s2_links": 0.00,
  "s3_summary": 0.00,
  "s4_cross_domain": 0.00,
  "s5_dedup": 0.00,
  "s6_violations": 0.00,
  "composite": 0.00,
  "verdict": "pass",
  "critical_triggered": false,
  "findings": [
    "具体的な指摘1",
    "具体的な指摘2"
  ],
  "fix_suggestions": [
    "修正提案1",
    "修正提案2"
  ]
}

出力は上記 JSON のみ。マークダウンの装飾やコードブロックで囲まないでください。
```

---

## 秘書側（呼び出し元）での後処理

1. レビュアー応答を JSON パース（失敗したら再試行 1 回、それでも失敗なら L2 fail として扱う）
2. `verdict == "pass"` かつ `critical_triggered == false` → Phase 6 へ進む
3. `verdict == "fail"` → `findings` と `fix_suggestions` を secretary にフィードバックして MD を 1 回だけ修正 → 同じレビュアー条件で再採点
4. 再採点結果も fail → Phase 6 以降を中断、ユーザーに JSON 結果を報告

---

## しきい値調整の運用メモ

- 初期しきい値: **composite ≥ 0.85**
- 2-4 週間運用して fail 率が 30% を超える場合はしきい値を 0.80 に緩和を検討
- 逆に全 pass が続く場合は致命軸基準 (現在 0.5) を 0.7 に引き上げ検討
- 調整履歴は本ファイルのバージョン履歴ではなく、運用ログ（`.task-log/` の `l2_composite` を集計）で追跡する
