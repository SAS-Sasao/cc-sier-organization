# 壁打ちメモ: 提案書・見積作成プロセスの標準化

- **日付**: 2026-03-19
- **オーナー**: 笹尾豊樹
- **ステータス**: 壁打ち完了・着手前

---

## 背景・課題

受託案件に対する提案書や見積作成の粒度が属人化している。

## 前提条件

| 項目 | 内容 |
|------|------|
| **最優先課題** | 提案書の品質ばらつき + 見積の積算根拠不透明 |
| **案件規模** | 5000万〜数億（大規模） |
| **案件種別** | 新規開発 × 新規顧客 |
| **既存資産** | 提案書10〜20件（SharePointに散在、参照先不明あり） |
| **合意プロセス** | 社内スキル標準化チームのPM承認が必要 |
| **追加要望** | 過去提案書のDB化 |

## ポイント

- 大規模 × 新規顧客 × 新規開発 → 標準化の効果が最も高い領域
- 金額が大きい → 見積根拠の透明性が経営判断に直結
- 新規顧客 → 提案書が第一印象。品質ばらつきは失注に直結
- PM承認プロセスがあるため、承認ゲートを2回（方針承認 + 最終承認）見込む

## アプローチ案（4フェーズ / 約3.5〜4ヶ月）

### フェーズ0: 提案書DB化 + 現状分析（Week 1-3）

| # | 作業 | 成果物 |
|---|------|--------|
| 0-1 | SharePointから提案書・見積を収集・一覧化 | 提案書台帳 |
| 0-2 | 各提案書の構成・粒度を分析 | 構成比較マトリクス |
| 0-3 | 「良い提案書」のリファレンス2〜3件を選定 | リファレンス提案書リスト |
| 0-4 | 属人化ポイントの特定 | 課題一覧 |

DB化は最初は提案書台帳（md/Excel）で十分。本格的なDB化はテンプレート定着後。

### フェーズ1: 方針策定 + ゲート1承認（Week 4-5）

| # | 作業 | 成果物 |
|---|------|--------|
| 1-1 | 標準化方針書の作成（課題→目標→スコープ→アプローチ） | 標準化方針書 |
| 1-2 | スキル標準化チームPMへの方針提案・承認取得 | **ゲート1通過** |

### フェーズ2: テンプレート・基準策定（Week 6-9）

| # | 作業 | 成果物 |
|---|------|--------|
| 2-1 | 提案書テンプレート作成 | 提案書テンプレート |

| 2-4 | PJ計画への引き継ぎチェックリスト | 引き継ぎチェックリスト |

### フェーズ3: パイロット + ゲート2承認（Week 10-13）
