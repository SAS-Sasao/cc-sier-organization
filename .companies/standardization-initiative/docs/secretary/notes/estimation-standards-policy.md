# 見積標準化 方針書

作成日: 2026-03-19
ステータス: ドラフト
作成者: 秘書室（壁打ちに基づく整理）

---

## 1. 背景と課題

### 現状の問題

- 見積作成に明確な根拠がなく、属人的な判断に依存している
- 見積が実績を下回る（上振れしない）傾向があり、利益率を圧迫している可能性が高い
- 顧客への説明責任を果たしにくい状態

### 前提条件

- 見積対象規模: 3,000万円〜数億円
- 見積作成者: 5名
- 過去実績データ: 約20件（うち工程別データあり10件、総額・フェーズ別のみ10件）
- 見積作成者間で「根拠がない」ことへの共通認識あり

---

## 2. 目的

見積に根拠を持たせ、組織全体で標準化する。

具体的には:
1. 過去実績に基づく見積基準（ベースライン）を策定する
2. 基準に従った見積テンプレートを導入する
3. フィードバックループにより継続的に精度を向上させる

---

## 3. 見積基準の構造

見積基準は **3層構造** で設計する。

### 3.1 第1層: 工程別工数比率（ベースライン）

過去10件の工程別データから中央値を算出し、標準比率として定義する。
（平均ではなく中央値を採用。外れ値の影響を避けるため）

| 工程 | 基準比率 | 算出根拠 |
|------|---------|---------|
| 要件定義 | ??% | 10件の中央値 |
| 基本設計 | ??% | 10件の中央値 |
| 詳細設計 | ??% | 10件の中央値 |
| 開発・実装 | ??% | 10件の中央値 |
| 単体テスト（UT） | ??% | 10件の中央値 |
| 内部結合テスト（IT） | ??% | 10件の中央値 |
| 外部結合テスト（ET） | ??% | 10件の中央値 |
| 総合テスト（ST） | ??% | 10件の中央値 |
| 受入テスト（UAT） | ??% | 10件の中央値 |
| PM・管理 | ??% | 10件の中央値 |
| **合計** | **100%** | |

> **注**: 受入テストについて、顧客実施で見積対象外とするケースも多い。自社の扱いを確定する必要がある。

> **注**: 規模帯（3,000万台 / 5,000万台 / 1億超）で比率が異なる可能性がある。データ数が集まれば規模帯別の基準値も検討する。

### 3.2 第2層: 補正係数（案件特性による調整）

ベースラインに掛ける係数。「なぜこの金額か」を説明する根拠の核となる。

#### 技術要素

| 条件 | 係数 |
|------|------|
| 実績あり技術のみ | ×1.0 |
| 一部未経験技術あり | ×1.2〜1.3 |
| 大半が未経験技術 | ×1.5 |

#### 要件の確度

| 条件 | 係数 |
|------|------|
| RFPあり・要件明確 | ×1.0 |
| 概要レベル・要件曖昧 | ×1.2 |
| 構想段階・要件未定 | ×1.3〜1.5 |

#### 顧客特性

| 条件 | 係数 |
|------|------|
| 既存顧客・関係構築済 | ×1.0 |
| 新規顧客 | ×1.1 |
| 意思決定が複雑（多部門関与） | ×1.2 |

#### 非機能要件

| 条件 | 係数 |
|------|------|
| 標準的 | ×1.0 |
| 高可用性・セキュリティ要件あり | ×1.2〜1.3 |

### 3.3 第3層: リスクバッファ（予備費）

| プロジェクトリスク評価 | バッファ率 |
|---------------------|----------|
| 低リスク（実績豊富・要件確定） | +10% |
| 中リスク（一部不確実性あり） | +20% |
| 高リスク（未経験領域・大規模） | +30% |

**算出方法:**
```
基本見積額 = 第1層（工程別工数） × 第2層（補正係数）
バッファ額 = 基本見積額 × リスクバッファ率
提示額     = 基本見積額 + バッファ額
```

## 4. 単価テーブル

自社原価ベースで6ランクを定義する。

| ランク | 役割例 | 単価（万円/人月） |
|--------|-------|-----------------|
| PM | プロジェクトマネージャー | 要確定 |
| PL | プロジェクトリーダー | 要確定 |
| SE上級 | アーキテクト・上級SE | 要確定 |
| SE | 設計・SE | 要確定 |
| PG | 開発・実装 | 要確定 |
| テスター | テスト実行 | 要確定 |

> **注**: 単価は自社の原価情報から確定する。本方針書では構造のみ定義。

## 8. 推進体制

| 役割 | 担当 | 責務 |
|------|------|------|
| 推進・仕組み設計 | とたろさん | 方針策定、データ分析、テンプレート設計、導入推進 |
| 運用オーナー | **要選定** | 基準書の維持更新、見積レビュー、フィードバック集約 |
| 利用者 | 見積作成者5名 | テンプレートに従った見積作成 |

### 運用オーナーの選定基準

| パターン | 向いているケース |
|---------|---------------|
| 見積経験が最も豊富な人 | 実績に基づく判断ができる。説得力がある |
| PM寄りの立場の人 | 案件横断で実績データを集めやすい |
| 品質管理・経営企画的な立場 | 中立的にレビューできる |
