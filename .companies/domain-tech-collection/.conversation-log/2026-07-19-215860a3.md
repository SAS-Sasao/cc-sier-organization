## 3. 運営モード

マスタが存在する場合の通常運用フローです。
アクティブ組織のパスは常に `.companies/{org-slug}/` を基点とします。

### 3.0 ユーザー識別

起動時に `git config user.name` を実行してユーザー名を取得し、`{operator}` 変数として保持する。
取得できない場合は `anonymous` をデフォルトとする。

```bash
git config user.name || echo "anonymous"
```

`{operator}` はコミットメッセージ、タスクログ、PR本文に記録される。

### 3.1 状態読み込み

起動時に以下のマスタを読み込みます:

1. `.companies/{org-slug}/masters/organization.md` — コスト設定、組織ポリシー
2. `.companies/{org-slug}/masters/departments.md` — 部署一覧とトリガーワード
3. `.companies/{org-slug}/masters/workflows.md` — 定義済みワークフロー

### 3.2 依頼内容の分析とルーティング

**Step 1: ワークフロー照合**
- `masters/workflows.md` のトリガーと依頼を照合
- 一致するワークフローがあれば、そのワークフローの実行方式に従う

**Step 2: 部署照合**（ワークフロー不一致の場合）
- `masters/departments.md` のトリガーワードと照合
- 一致する部署が見つかったら、その部署の対応 Subagent を確認

**Step 3: 実行モード判定**

| 条件 | 実行モード | アクション |
|------|-----------|-----------|
| ワークフロー一致 + subagent | Subagent委譲 | 該当 Subagent を名前指定で起動 |
| ワークフロー一致 + agent-teams | Agent Teams | チーム編成して並列実行 |
| 部署一致 + 小規模作業 | 秘書が直接対応 | secretary Subagent で処理 |
| 部署一致 + 専門作業 | Subagent委譲 | 部署の対応 Subagent を起動 |
| 部署一致 + 大規模並列 | Agent Teams | Agent Teams適性を確認して編成 |
| どれにも一致しない | 秘書対応 | secretary Subagent で汎用対応 |

### 3.3 Subagent 呼び出し

```
「{agent-name}エージェントを使って、{依頼内容}を実行してください。
 成果物は .companies/{org-slug}/docs/{dept}/{path} に保存してください。」
```

対応する Subagent ファイルは `.claude/agents/{agent-name}.md` にあります。

### 3.4 Agent Teams 編成

Agent Teams を使用する場合（`references/agent-templates.md` を参照）:

1. ワークフローのチーム構成を確認
2. `masters/roles.md` から各ロールのテイメイト指示テンプレートを取得
3. チームリード（通常は secretary）を指定し、**ブランチ名を生成**（詳細は 3.6 参照）
4. 各テイメイトに「ブランチ `{branch-name}` 上で作業すること」を明示して並列実行
5. 結果を統合して `.companies/{org-slug}/` 配下に保存
6. チームリードがまとめてコミット → PR 作成（3.6 を参照）

**コスト管理**: `masters/organization.md` の `COST_AWARENESS` 設定に従う:
- `conservative`: Agent Teams は明示指示時のみ。デフォルトは Subagent
- `balanced`（推奨）: workflows.md の実行方式に従う
- `aggressive`: 並列可能な場面では積極的に Agent Teams 使用

### 3.6 Gitワークフロー

ファイル生成を伴う作業の前後で実行します。詳細は `references/git-workflow.md` を参照。

**ファイル生成を伴わない作業（壁打ち、ダッシュボード表示、口頭相談等）では実行しません。**

**作業前:**
1. 現在のブランチを確認（main でない場合は警告）
2. 未コミットの変更がないか確認（あれば stash するか警告）
3. ブランチ作成: `{org-slug}/{type}/{YYYY-MM-DD}-{summary}`
4. ブランチに切り替え

**作業後:**
1. **LLM-as-Judge 評価**: 成果物が `docs/` 配下にある場合、secretary.md の「タスク完了後の品質評価」手順に従い、completeness / accuracy / clarity の3軸評価を実行し `.task-log/{task-id}.md` に `## judge` セクションを追記する（詳細は secretary.md 参照）。成果物がない場合はスキップ。
2. `git add .companies/{org-slug}/`（組織ディレクトリのみ）
3. `git commit -m "{type}: {概要} [{org-slug}] by {operator}"`
4. `git push origin {branch-name}`
5. PR 作成（gh CLI）: タイトルと本文に組織名・変更概要を含める
6. PR の URL をユーザーに報告
7. `git checkout main` で元のブランチに戻る

### 3.7 タスクログと Issue 作成

ファイル生成を伴うタスクの実行過程を記録し、完了時に GitHub Issue として可視化する。
詳細なテンプレートは `references/task-log-template.md` を参照。

**スキップ条件**: Gitワークフロー（3.6）と同じ。ファイル生成を伴わない作業（壁打ち、ダッシュボード等）ではログ作成・Issue作成ともにスキップ。

**タスクログの記録フロー**:

1. **タスク受付時**: `.companies/{org-slug}/.task-log/{task-id}.md` を作成（task-id: `YYYYMMDD-HHMMSS-{概要slug}`）
2. **秘書の判断時**: 実行モード・アサインロール・判断理由をログに記録
3. **Subagent委譲 / Agent Teams編成時**: 委譲先・指示内容をログに記録
4. **エージェント作業完了時**: 各Subagentの作業サマリー・成果物パスをログに追記
5. **タスク全体完了時**: ステータスを completed に更新

**Issue 作成フロー**（タスク完了時に実行）:

1. `.task-log/{task-id}.md` を読み込む
2. `references/task-log-template.md` の Issue テンプレートに従い本文を組み立てる
3. ラベルを決定（`references/task-log-template.md` のラベル決定ルール参照）
4. `gh issue create` で Issue 作成（gh CLI が使えない場合はスキップ）
5. `.task-log/{task-id}.md` に issue_number を追記

**.task-log/ の配置**: `.companies/{org-slug}/.task-log/` に配置。Git管理対象としてPRに含める。docs/ 配下には置かない（成果物の可読性を保つ）。

## 🤖 Claude

記事2本と ai-virtual-office の要件定義（§5.4 の `.claude` 構成計画）を突き合わせました。壁打ちとして、私の見立てと具体案を出します。

## まず大きな見立て：この2記事は「別の層」の話をしている

- **Osmani 記事** = ループの**実行系**の設計論。Automations / Worktrees / Skills / Connectors / Sub-agents の5部品 + 状態ファイル（背骨）で「人がプロンプトを打つ代わりに、仕事を発見→実行→検証→次を決めるシステムを作れ」
- **LayerX 記事** = ループの**学習系**の設計論。SKILL.md を学習パラメータと見なし、検証信号なしに書いたスキルはほぼ無効（49本中39本が改善なし、平均+1.2%）。「**ドキュメントを書く前に評価を作れ**」が核心

つまり「hooks/SKILL を追加するべき」という笹尾さんの直観は正しいのですが、順序が重要です。**先に検証信号（機械的に合否が出る仕組み）を hooks で作り、スキルはそれを参照する形で最小限に書く**。逆順（スキルを先に大量に書く）は LayerX の言う「無効スキルの山」になります。

## 気づき：ai-virtual-office 自体がループエンジニアリングの欠けたピース

Osmani の最重要警告は「**無人で回るループは、無人で間違え続けるループでもある**」——人間の観測・トリアージ層を消すな、です。ai-virtual-office のオフィスビュー + 待ちパネルは、まさにその「ループの観測層」そのもの。要件 §1.1 に「**ループエンジニアリングにおける human-in-the-loop 観測面**」という位置づけを一文足すと、製品の差別化ストーリーが一段強くなります（「可愛いだけで終わる」リスクへの回答にもなる）。

---

## 具体案：`.claude` 構成への追加

### 1. hooks は「観測用」と「検証用」の2系統に分ける（最優先）

§5.4 で計画済みの hooks は Relay への **観測用**（dogfooding）だけです。ループエンジニアリング的に足りないのは **検証用** hooks——CLAUDE.md に書く予定の規約を、文章ではなく機械チェックに落とすことです：

| 規約（CLAUDE.md に書く予定） | 機械化案 |
|---|---|
| `game/` は React 非依存 | PostToolUse hook: `game/` 配下編集時に `grep -r "from 'react'" game/` → 検出で block + 理由返却 |
| protocol 変更時は relay/web 両テスト | Stop hook: `packages/protocol` に diff があれば relay/web 両方の vitest 実行、fail なら完了させない |
| 型安全 | PostToolUse (Edit/Write on `*.ts`): 該当パッケージの `tsc --noEmit` を即時実行しエラーをフィードバック |

これは cc-sier の L0 機械レビュー（review-drawio.js）と同じ思想の応用で、「ルールをプロンプト空間から決定論的検証に移す」= ループの損失関数を作る作業です。安価で即効性が高いので最初にやるべき。

### 2. スキルは「評価シナリオ先行」で最小3本

- **office-verify**（計画済み）— これを Osmani の言う `/goal` の**停止条件**に昇格させる。「起動 → ダミーイベント投入 → 描画確認」を LLM の目視ではなく `scripts/verify.sh` として同梱し、exit code で合否が出る形に。これが M0/M1 の受入基準の機械化版になります
- **protocol-change**（新規提案）— Zod スキーマ変更 → 型再生成 → relay/web 両テスト → バージョン整合、の定型手順。protocol はこのアプリで最も壊れると痛い境界なので、スキル化の費用対効果が最大
- **office-release**（計画済み）— M1 以降でよい

LayerX の規律を運用ルールにする：各スキルに「スキルなしで失敗した具体例3つ」を書いてから本文を書く。cc-sier で経験済みの「SKILL.md 全面リライトで参照取りこぼし」（PR #251→#254 の教訓）は、まさに「編集量制限（テキスト学習率）」の話なので、**スキルは全面書き換え禁止・差分編集のみ**を新リポの CLAUDE.md に明記するとよいです。

### 3. Subagent に maker-checker 分離を追加

計画中の4種（game-engine-dev / pipeline-dev / ui-dev / org-adapter-dev）は全員 **maker**（モジュール別実装者）です。Osmani の指摘どおり、検証者が同じ視点だとグループシンクになるので：

- **office-qa.md**（verifier、強いモデル）— office-verify スキル実行 + 実装者の成果物への敵対的レビュー専任。cc-sier の「fresh general-purpose に L2 させる」思想と同じで、実装 Subagent 自身にレビューさせない
- 探索は既存の Explore agent（read-only・高速）で足りるので新設不要

### 4. Automations（夜間ループ）は M1 達成後に

朝の triage 自動化（CI 失敗 + open issues → worktree で修正 → PR）は Osmani アーキテクチャの完成形ですが、検証信号が整う前に回すと「無人で間違えるループ」になるだけ。cc-sier の nightly workflow で得た教訓（max-turns 60、`CLAUDE_CODE_DISABLE_BACKGROUND_TASKS`、pipefail 等）はそのまま移植できるので、始めるときのコストは低いです。

### 5. 状態ファイル（ループの背骨）

cc-sier の `.task-log/` 規約を ai-virtual-office にも軽量移植するのが自然です。Case Bank/報酬スコアまでは初期は過剰で、まず「試したこと・通ったこと・残っていること」が次セッションで読める task-log があれば十分。しかも task-log はこのアプリ自身のリプレイ素材になるので、**開発ログを書くこと自体がテストデータ生成**という一石二鳥の構造になります。

1. **検証 hooks 3本**（game/React ガード・protocol 両テスト・tsc 即時）← 今すぐ、数時間で効く

3. **office-qa subagent** 追加（maker-checker）
4. protocol-change スキル（評価シナリオ先行）
5. M1 後に夜間 automations

## 現状の整理

今の設計で「合否」が定義されているのは機械層だけです：

| 層 | 合否の定義 |
|---|---|
| 検証 hooks（5本） | あり — exit 0 / 2 で決定論的 |
| office-verify | あり — scripts/verify.sh の exit code |
| **office-qa（LLM レビュー）** | **なし** — 「敵対的レビュー」「妥当性判断」と責務は書いたが、何点で合格か・何を見たら即 fail か・出力形式・リトライ回数が未定義 |

これはループエンジニアリング的にまさに致命的で、LayerX 記事の核心（「質の高い検証信号の設計が本質的課題」）に自分で違反しています。合格基準のない checker は**ノイズの多い損失関数**そのもので、① 同じ成果物でも実行ごとに判定が揺れる、② maker への差し戻しが「なんとなく直して」になり自己修正ループが収束しない、③ 「レビュー通った」の意味が毎回違うので自動化に組み込めない、という問題が出ます。

皮肉なことに cc-sier 本体には答えが既にあります。**L2 独立レビューの設計（6軸採点 + composite ≥ 0.85 + 致命軸 + JSON 出力 + リトライ1回）**がまさにこれで、設計書に移植し忘れていました。

## 提案：office-qa 合格基準（cc-sier L2 パターンの移植）

```json
{
  "s1_mechanical": 0.00,   // 機械検証通過（hooks/office-verify/テスト green）★致命軸
  "s2_design":     0.00,   // 設計準拠（protocol 正本・game React非依存・NFR-8 テスタビリティ）
  "s3_test_integrity": 0.00, // テスト妥当性（assertion 弱体化なし・受入基準トレースあり）★致命軸
  "s4_quality":    0.00,   // 実装品質（可読性・エラー処理・スコープ遵守）
  "s5_traceability": 0.00, // 設計書/要件との対応が説明可能か
  "s6_security":   0.00,   // 機微情報の取り扱い（NFR-4: プロンプト本文・ファイル内容の流出なし）★致命軸
  "composite": 0.00,
  "verdict": "pass|fail",
  "critical_triggered": false,
  "findings": [],
  "fix_suggestions": []    // maker への差し戻し指示（office-qa は自分で直せないため必須）
}
```

- **判定ルール**: cc-sier と同一 — 致命軸 < 0.5 で composite 強制 0.00・fail。それ以外は等重み平均 ≥ 0.85 で pass
- **office-qa の作法**: 採点前に必ず機械検証（office-verify・テスト）を**実行**し、s1 は感想ではなく実行結果で埋める（LLM の主観を機械信号で錨止め）
- **リトライ**: fail → findings/fix_suggestions を maker に差し戻し → 1回だけ再採点 → それでも fail なら人間へエスカレーション（PR は auto-merge しない）
- **E2E spec レビュー特則**: assertion 削除・skip を含む diff は、理由明示 + 対応する受入基準の変更が示されない限り s3 = 0（即 fail）

## 反映が必要な箇所（3つ）

1. **設計書** `ai-virtual-office-loop-engineering-design.md` §4 に「office-qa 合格基準」を追記（cc-sier 側 → ai-vir 側へ同期）
2. **プロンプト txt** — office-qa.md のシステムプロンプトに採点 JSON と判定ルールを含める指示を追加
3. 将来の CI 自動化（設計書 §5.5 のループ B）で verdict JSON をゲート条件に使う旨を一文

反映してよければ、この3点をまとめて対応します（設計書は cc-sier 側改訂 → ai-vir 同期 → txt 更新の順）。
