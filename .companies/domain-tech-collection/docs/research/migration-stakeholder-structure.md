## 2. 影響度×関心度マトリクス

### 判定基準

- **影響度（Impact）**: プロジェクトの成否・スコープ・予算・スケジュールに与える影響の大きさ
- **関心度（Interest）**: プロジェクトの進捗・成果物への能動的な関与・関心の高さ

### マトリクス表

```

                     │   Keep Satisfied     │   Manage Closely     │
                     │   （満足維持）        │  （密接管理）         │
                     │                      │                      │
 影                  │  C-01: CIO/IT役員    │  C-02: IT部門責任者  │
 響                  │  C-15: 監査・法務    │  C-03: IT企画担当    │

                     │  C-13: 物流パートナー│  C-06: 商品部責任者  │
                     │                      │  C-07: 店舗運営責任者│
                     │                      │  C-09: SV            │

                     │                      │  V-01: プログラムMgr │
```

### 象限別サマリー

| 象限 | 管理方針 | 代表ステークホルダー | 主なリスク |
|------|---------|-------------------|----------|
| **Manage Closely（密接管理）** | 週次以上の双方向コミュニケーション。課題・判断事項を即日エスカレーション | IT部門責任者、各機能PM、SV、セキュリティ担当 | 情報連携遅延によるブロッカー発生 |
| **Keep Satisfied（満足維持）** | 経営報告は月次。進捗サマリーをシンプルに。問題が起きたら即座に報告 | CIO/IT役員、決済代行会社、物流パートナー、監査部門 | 突然のスコープ変更要求・プロジェクト中断判断 |
| **Keep Informed（情報提供）** | 定例または節目報告。一方向の情報共有が中心 | IT運用担当、店長、2次請け、AWS TAM | 運用移管の失敗、現場展開時の混乱 |
| **Monitor（監視）** | 必要時に連絡。変化があれば関心度を再評価 | 店舗スタッフ、3次請け | 現場混乱の兆候見落とし |

## 3. コミュニケーション計画テンプレート

### 3-A. Manage Closely（密接管理）象限

**対象**: IT部門責任者（C-02）、IT企画担当（C-03）、IT開発担当（C-04）、商品部責任者（C-06）、店舗運営部責任者（C-07）、SV（C-09）、情報セキュリティ担当（C-14）、プログラムマネージャー（V-01）、各機能PM（V-02〜V-05）、AWS SA（V-08）

| 項目 | 内容 |
|------|------|
| **頻度** | 週次定例（原則毎週火曜10:00〜11:00）+ 課題発生時随時 |
| **手段** | 対面 or ビデオ会議（Microsoft Teams / Zoom）+ 議事録メール |
| **報告内容** | ①進捗サマリー（WBS達成率）②課題リスト（RAID Log）③判断依頼事項④次週の予定 |
| **資料形式** | 週次ステータスレポート（A4 2枚以内）+ RAID Logスプレッドシート |
| **エスカレーション** | 重要度High課題 → 48時間以内にメール通知 → PMが直接電話確認 |
| **議事録管理** | 24時間以内に共有フォルダへ格納・担当者メール配信 |

### 3-B. Keep Satisfied（満足維持）象限

**対象**: CIO/IT担当役員（C-01）、決済代行会社（C-12）、物流パートナー（C-13）、監査・コンプライアンス（C-15）

| 項目 | 内容 |
|------|------|
| **頻度** | 月次定例（月末最終週）+ マイルストーン完了時 + 重大課題発生時（即時） |
| **手段** | 対面会議（役員は対面必須）+ メール。Slack等は使わない |
| **報告内容** | ①全体進捗サマリー（数値・グラフ中心）②予算執行状況③重大リスクと対策④次フェーズの概要 |

| **注意事項** | 課題は「問題と対策セット」で報告。問題のみの報告は不可 |

### 3-C. Keep Informed（情報提供）象限

**対象**: IT運用担当（C-05）、物流部担当（C-08）、店長（C-10）、2次請け（V-06）、AWS TAM（V-09）、既存ベンダー（V-10）、セキュリティ専門会社（V-11）、回線ベンダー（V-12）

| 項目 | 内容 |
|------|------|
| **頻度** | 隔週または月次 + 関与フェーズ移行時 |
| **手段** | メール配信 or プロジェクト管理ツール（Confluence/Backlog）への掲載 |
| **報告内容** | フェーズに関連する進捗・依頼事項・スケジュール変更 |
| **資料形式** | 標準フォーマットのステータスメール。1画面で読める分量 |
| **エスカレーション** | 依頼事項の返答期限超過 → PM からリマインド → 1週間後に上位者にエスカレーション |
| **注意事項** | 関与フェーズが近づいたら「Keep Informed」から「Manage Closely」へ格上げを検討 |

### 3-D. Monitor（監視）象限

**対象**: 店舗スタッフ（C-11）、3次請け（V-07）

| 項目 | 内容 |
|------|------|
| **頻度** | 必要時のみ（店舗展開前後のアンケート等） |
| **手段** | マニュアル配布、研修実施（C-11）/ 作業指示書（V-07） |
| **報告内容** | 操作手順・変更点の通知 |
| **エスカレーション** | 現場からの問い合わせが急増した場合は「Keep Informed」へ格上げし情報収集強化 |

## 5. Go/No-Go判定キーパーソン

### パイロット移行 Go/No-Go（最重要）

大規模展開前の最大のゲート。以下のキーパーソン全員の承認が必須。

| # | キーパーソン | 確認観点 | 典型的な懸念事項 |
|---|------------|---------|---------------|
| 1 | **C-02 IT部門責任者** | 全体品質・スケジュール妥当性 | バグ件数・未解決課題数が許容範囲内か |
| 2 | **C-01 CIO/役員** | ビジネスリスク | 店舗オペレーションへの影響・ブランドリスク |
| 3 | **C-07 店舗運営責任者** | 現場運用可否 | 店舗スタッフが正常に操作できるか |
| 4 | **C-06 商品部責任者** | 業務継続性 | 発注・廃棄処理が正常に動作するか |
| 5 | **C-14 情報セキュリティ担当** | セキュリティ適合 | 脆弱性診断クリア・PCI DSS準拠確認 |

### Go/No-Go判定基準（テンプレート）

```
【Go/No-Go 判定チェックリスト】（パイロット移行前）

□ 機能テスト完了率: 100%（未完了 = 即 No-Go）
□ 重大バグ（Sev1/Sev2）残件数: 0件（Sev1残件 = 即 No-Go）
□ 性能テスト: 全シナリオで基準値クリア
□ セキュリティ診断: クリティカル・ハイ脆弱性 0件
□ 運用引き継ぎ資料: レビュー完了・承認済み
□ 店舗研修マニュアル: 完成・SV確認済み
□ ロールバック手順: 作成・テスト完了
□ 監視・アラート設定: 動作確認済み
□ 外部連携（決済・物流）: インターフェーステスト完了
□ 並行稼働計画: 確定・関係者合意済み

【判定】
  全項目 ✓ → GO
  1項目でも × → NO-GO（例外なし）
  ※ NO-GO の場合は課題解消後に再判定ゲートを設ける
```

### ウェーブ展開中の Go/No-Go

各ウェーブ完了後にも簡易判定を実施（大規模案件では週次）。

| 判定者 | 頻度 | 判断材料 |
|--------|------|---------|
| C-02 IT部門責任者 | 各ウェーブ完了時 | 移行成功率・インシデント件数 |
| C-07 店舗運営責任者 | 各ウェーブ完了時 | SV/店長からのフィードバック |
| V-01 プログラムマネージャー | 週次 | 展開ペース・リソース状況 |

## 6. マルチベンダー体制の責任境界

### 体制図（テキスト表現）

```
クライアント（情報システム部）
    ↕ 定例報告・承認
SIer PMO（V-01: プログラムマネージャー）
    ↕
  ┌────────────────────────────┐
  │ SIer 機能チーム PM         │
  │  V-02: インフラPM          │
  │  V-03: アプリケーションPM  │
  │  V-04: データ移行PM        │
  │  V-05: テストPM            │
  └────────────────────────────┘
         ↕ 作業指示・報告
  ┌──────────────┬──────────────┐
  │2次請け(V-06) │3次請け(V-07) │
  │（開発パートナー）           │
  └──────────────┴──────────────┘

外部パートナー（SIer PMOが調整窓口）
  V-08: AWS SA ─── アーキレビュー
  V-09: AWS TAM ── 障害・コスト
  V-10: 既存ベンダー ─── 現行仕様
  V-11: セキュリティ会社 ─── 診断
  V-12: 回線ベンダー ─── 回線SLA
```

### 責任分界点（RACI マトリクス抜粋）

R=実行責任, A=説明責任, C=協議, I=情報共有

| 作業・成果物 | SIer PMO | インフラPM | APPM | 既存ベンダー | クライアント | AWS SA |
|-------------|---------|----------|------|------------|------------|--------|
| 要件定義書 | A | I | C | C | R/A | I |
| インフラ設計書 | A | R | I | I | C | C |
| AP設計書 | A | I | R | C | C | I |
| データ移行計画 | A | C | C | R | C | I |
| テスト計画 | A | C | C | I | C | I |
| 移行実行（本番） | A | R | R | C | C | I |
| 運用手順書 | A | R | R | I | R（承認） | I |

### ベンダー間インターフェース定義

| 境界点 | 管理責任 | 課題発生時の対応フロー |
|--------|---------|---------------------|
| インフラ ↔ AP間 | SIer PMO（V-01） | インフラPM・APPM合同で原因切り分け → PMOが調停 |
| AP ↔ 既存システム | SIer APPM（V-03） | 既存ベンダー（V-10）へ仕様照会 → PMO経由でクライアント承認 |
| AP ↔ 決済代行（C-12） | SIer APPM（V-03） | クライアント（C-02）同席の3者会議で解決 |
| 店舗回線 ↔ AWS | SIer インフラPM（V-02） | 回線ベンダー（V-12）とAWS TAM（V-09）を同時招集 |
| テスト ↔ 各開発 | テストPM（V-05） | バグチケットで担当PM特定・SLAで修正期限設定 |

## 7. よくある落とし穴と対策

### 落とし穴 1: 現場（SV・店長）のキーパーソン不在

**事例パターン**: IT部門とは合意できているのに、実際に店舗展開を始めたら「そんな話は聞いていない」とSVや店長から反発が起きる。研修を拒否したり、移行作業の受け入れを断られたりする。

**根本原因**: 要件定義フェーズで「IT部門からの承認 = 現場の合意」と誤認。SV・店長は「管理下」でなく「実質的なステークホルダー」であるにもかかわらず Manage Closely に含めていない。

**対策**:
- Ph.1 の段階でパイロット店舗のSVを「ユーザー代表」として要件ヒアリングに参加させる
- 店舗運営責任者（C-07）を通じて、移行計画・作業手順を事前共有するルートを確立する
- SV向けの「移行ブリーフィング会」をパイロット前に必ず開催する

---

### 落とし穴 2: 既存ベンダーの情報共有拒否・遅延

**事例パターン**: 現行システムの仕様書が不完全で、既存ベンダーに問い合わせると「そのドキュメントはない」「確認に時間がかかる」と言われ、設計フェーズが数週間ブロックされる。既存ベンダーが「仕事を奪われる」と感じて非協力的になるケース。

**根本原因**: 既存ベンダーに対してステークホルダーマネジメントを行わず、単なる「情報提供元」として扱っている。移行後の保守契約が既存ベンダーから取り上げられる場合、インセンティブが働かない。

**対策**:
- 既存ベンダーとの「引き継ぎ協力合意書」をプロジェクト開始前に締結する（クライアント主導）
- 現行仕様ドキュメントの収集を Ph.1 の最初のタスクとして計画に組み込む
- 必要に応じて、移行後の一部保守業務を既存ベンダーに残す交渉をクライアントに打診する

---

### 落とし穴 3: 外部連携先（決済代行・物流）のスケジュール対応遅延

**事例パターン**: 決済代行会社・物流パートナーとのインターフェーステストのスケジュール調整が難航し、本番直前まで結合テストが完了しない。決済・物流側は「自社開発リソースが別案件で使えない」と言い続ける。

**根本原因**: 外部連携先を「Always Keep Satisfied のみ」として扱い、テスト計画に組み込んでいない。決済代行・物流側のプロジェクト計画に、このプロジェクトのテスト工程が組み込まれていない。

**対策**:
- Ph.1 の段階で決済代行・物流パートナーの技術担当を特定し、テストスケジュールを合意する
- 四半期ごとにスケジュール確認MTGを設定し、外部連携先の工数を押さえる
- テスト環境（サンドボックス）を早期に準備し、先行してインターフェーステストを開始する

---

### 落とし穴 4: 経営層への「悪いニュース」報告遅延

**事例パターン**: スケジュール遅延・品質問題が発生しているのに、プロジェクト内部で「何とかなる」と楽観的見通しを維持し、経営層への報告を先送りにする。カットオーバー1ヶ月前に初めて問題が露呈し、経営層から「なぜ今頃言う」と信頼を失う。

**根本原因**: PM・PMOが経営層を「Keep Satisfied = 問題を知らせない」と誤解している。Keep Satisfied の本来の意味は「安心させる情報提供」であり「問題の隠蔽」ではない。

**対策**:
- 月次レポートに「課題と対策セット報告」ルールを徹底する（課題だけの報告は不可）
- 「信号機ルール」（Green/Yellow/Red）をプロジェクト開始時に経営層と合意する
- スケジュール2週間遅延 = 即 Yellow 報告、4週間 = 即 Red 報告・対策会議の開催をルール化する

---

### 落とし穴 5: マルチベンダー体制での「責任のたらいまわし」

**事例パターン**: インフラとAP間の障害が発生した際に、インフラPM・APPMの双方が「うちの責任ではない」と言い合い、クライアントの前で責任の所在が曖昧になる。クライアントがどのベンダーに何を言えばいいか分からなくなる。

**根本原因**: ベンダー間の責任境界点が文書化されていない。または文書化はされているが関係者に徹底されていない。クライアントへの単一窓口（SPOC）が機能していない。

**対策**:
- RACI マトリクスをプロジェクト開始時に全ベンダーで合意・署名する
- クライアントへの SPOC は SIer PMO（V-01）のみとする。他ベンダーはクライアントと直接合意しない
- 障害発生時の「15分ルール」：原因不明でも15分以内に PMO からクライアントへ第一報を入れ、責任所在ではなく「現在調査中」の事実を報告する

---

### 落とし穴 6: ウェーブ展開中の「疲弊による品質低下」

**事例パターン**: 100店舗規模のウェーブ展開が進むにつれて、移行作業チームが疲弊し、後半ウェーブで移行手順の省略・確認不足が発生する。また、初期ウェーブで発見された問題の対応を後回しにしたまま次ウェーブへ進み、問題が蓄積する。

**根本原因**: ウェーブ計画にバッファとレビューポイントが組み込まれていない。移行チームのローテーションや人員追加の計画がない。

**対策**:
- 各ウェーブ完了後に必ずレトロスペクティブ（振り返り）を実施し、手順改善を次ウェーブに反映する
- ウェーブ間に1〜2日のバッファを設け、未解決問題の対応期間とする
- 移行実行チームのローテーション計画を展開計画と同時に策定する

## 8. 推奨事項まとめ

### 優先度高（プロジェクト開始前に必ず実施）

| # | 推奨事項 | 理由 |
|---|---------|------|
| 1 | ステークホルダー登録簿の作成と四半期レビュー | フェーズが進むと関与者が増減する。都度更新しないと管理漏れが発生 |
| 2 | 既存ベンダーとの引き継ぎ協力合意書の締結 | 現行仕様ドキュメント収集は最大のリードタイムリスク。早期対処必須 |
| 3 | SV・店長のパイロット段階からの巻き込み | 現場の反発は展開フェーズで致命的になる。投資対効果が最も高い予防策 |
| 4 | SPOC（単一窓口）の明確化と全ステークホルダーへの通知 | マルチベンダー特有の「たらいまわし」を構造的に排除する |
| 5 | 外部連携先のテストスケジュール早期合意 | 決済・物流ベンダーの工数は半年前から押さえないと確保できないことが多い |

### 優先度中（Ph.1 完了までに実施）

| # | 推奨事項 | 理由 |
|---|---------|------|
| 6 | Go/No-Go 判定基準の事前合意・文書化 | 判定時に基準が曖昧だと判断が長引く。特にCIOとの事前合意が重要 |
| 7 | 「信号機ルール」の経営層との合意 | 悪いニュース報告の心理的障壁を仕組みで下げる |
| 8 | RACI マトリクスの全ベンダー合意・署名 | 責任境界の曖昧さはプロジェクト後半で顕在化する。前半での合意が必須 |

### 優先度中（展開フェーズ開始前に実施）

| # | 推奨事項 | 理由 |
|---|---------|------|
| 9 | ウェーブ展開計画へのバッファとレビューポイントの組み込み | 展開フェーズの疲弊・品質低下は計画段階での対策が最も効果的 |
| 10 | SV向けブリーフィング会の定例化 | 展開情報の現場浸透には一方向配布ではなく双方向のセッションが必要 |
