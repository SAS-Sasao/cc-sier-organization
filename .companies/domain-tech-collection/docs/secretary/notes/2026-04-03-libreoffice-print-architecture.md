# LibreOffice帳票印刷のレイテンシ改善 — 壁打ちメモ

- **日付**: 2026-04-03
- **種別**: アーキテクチャ壁打ち
- **参加者**: 笹尾豊樹, 秘書（CC-SIer）

---

## 現状のアーキテクチャ

```
[販売管理システム] → CSV出力 → [データシート(LibreOffice)]
                                       ↓
                              [テンプレブック]
                              ├── 帳票テンプレートシート
                              └── データシート（CSV貼付）
                                       ↓
                              [タスクスケジューラ] → バッチ処理 → 印刷
```

- LibreOfficeのテンプレブック（帳票テンプレートシート + データシート）を使用
- 販売管理システムからCSV形式でデータシートにデータを添付
- バッチ処理でLibreOfficeの印刷操作を実行
- タスクスケジューラでバッチ処理を管理

## 課題

- タスクスケジューラのポーリング間隔に依存するため、アプリケーションから帳票印刷を実行した際に **最大1分以上** のレイテンシが発生する
- 本質的に **ポーリング型であること** が遅延の根本原因

## 制約

- LibreOfficeをテンプレートとして使用する方針は変更不可

---

## 改善案

### 案1: ファイルシステム監視（推奨・最小改修）

```
[販売管理システム] → CSV を所定フォルダに出力
                          ↓ (FileSystemWatcher が即時検知)
                    [常駐サービス] → LibreOffice バッチ印刷
```

- Windows なら `FileSystemWatcher`（.NET）や PowerShell の `Register-WmiEvent`
- CSVが置かれた瞬間にトリガーされるので **待ち時間はほぼゼロ**
- タスクスケジューラを廃止し、Windows Serviceとして常駐させる
- **改修範囲**: 販売管理システム側は変更不要（CSV出力先が同じなら）

### 案2: HTTPエンドポイント化（アプリ連携を強化したい場合）

```
[販売管理システム] → HTTP POST /print {csv_path, template}
                          ↓ (即時)
                    [印刷サービス (Flask/Express等)]
                          ↓
                    LibreOffice --headless で印刷
```

- アプリ側のボタン押下 → API呼び出し → 即時印刷
- 印刷ステータスをレスポンスで返せる（成功/失敗/キュー位置）
- **改修範囲**: 販売管理システム側にHTTPクライアント呼び出しの追加が必要

### 案3: メッセージキュー（複数拠点・大量印刷がある場合）

```
[販売管理システム] → Redis/RabbitMQ に印刷ジョブ投入
                          ↓ (即時 pop)
                    [Worker プロセス] → LibreOffice 印刷
```

- 大量の同時印刷要求をキューで順序制御できる
- Workerを複数立てればスケールアウト可能
- **改修範囲**: キュー基盤の導入 + 販売管理システム側の改修

---

## 共通施策: LibreOffice コールドスタート回避

どの案を選んでも、LibreOfficeの起動自体に数秒かかる問題が残る。

**UNO Listenerモードで常駐させる**:

```bash
soffice --headless --accept="socket,host=localhost,port=2002;urp;" --norestore
```

- LibreOfficeをデーモンとして起動しっぱなしにする
- 印刷時はUNO API経由でドキュメントを開いて印刷 → 都度プロセス起動が不要
- 体感で **3〜5秒の短縮** が見込める

---

## 比較まとめ

| | 案1: ファイル監視 | 案2: HTTP API | 案3: メッセージキュー |
|---|---|---|---|
| **応答速度** | ~数秒 | ~数秒 | ~数秒 |
| **販売管理側の改修** | 不要 | HTTP呼出追加 | キュー投入追加 |
| **インフラ追加** | なし | なし | Redis等 |
| **印刷ステータス通知** | 難しい | 容易 | 容易 |
| **同時大量印刷の制御** | 工夫が必要 | 工夫が必要 | 得意 |
| **おすすめ度** | 最小改修で即効 | バランス良い | 大規模向け |

## 所感・注意点

- **販売管理システム側を触れるなら案2（HTTP API）、触れないなら案1（ファイル監視）** が現実的
- いずれの場合もUNO Listenerの常駐化は併用すべき
- **排他制御**: LibreOfficeはシングルインスタンスなので、同時に複数印刷が来た場合のキューイングは必須。案1・2でも簡易キュー（ロックファイルやインメモリキュー）は入れた方がいい
- **障害時のリカバリ**: 常駐サービスが落ちたときの検知・自動再起動の仕組み（Windows Service + 回復オプション設定）
- **LibreOfficeのメモリリーク**: 長時間常駐させると不安定になる事例がある。定期的なプロセス再起動（日次等）を入れておくと安心
