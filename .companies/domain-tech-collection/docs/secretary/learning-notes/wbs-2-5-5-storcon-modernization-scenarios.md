## 1. ストコン機能のマイクロサービス分解案

### 1-1. ドメイン境界の定義

ストコンは歴史的にモノリシック（一体型）なアーキテクチャで構築されてきた。その背景には、1990年代のWindows系OSへの移行時に、すべての業務機能を1つのアプリケーションに集約する設計思想が採用されたことがある。これをドメイン駆動設計（DDD）の観点から「境界付けられたコンテキスト（Bounded Context）」として分解すると、以下の6ドメインが識別できる。

#### 発注ドメイン（Order Domain）

発注に関わる業務を担う。コンビニ経営の要であり、ストコンの中核機能。

| 機能 | 概要 | 業務上の重要度 |
|------|------|--------------|
| 自動発注（オート発注） | 販売実績・在庫・リードタイムを元にシステムが算出 | 最高 |
| 推奨発注（スマート発注） | 推奨数量を提示し店員が確認・修正 | 最高 |
| 手動発注 | 品番・数量を直接入力。新商品・イベント品に使用 | 高 |
| 発注締め管理 | 1日2〜3回の発注締め時刻の管理・送信 | 最高 |
| 発注履歴管理 | 過去の発注量・実売量の参照 | 中 |
| 発注制限の受信・適用 | 本部からの発注上限・下限設定の受信と適用 | 高 |

**集約ルート**: 発注伝票（OrderSlip）
**主要エンティティ**: 発注明細（OrderLine）、発注制限（OrderConstraint）、推奨数量（RecommendedQuantity）

#### 在庫ドメイン（Inventory Domain）

理論在庫の維持と実在庫との突合を担う。

| 機能 | 概要 | 業務上の重要度 |
|------|------|--------------|
| 理論在庫計算 | 入荷数 - 販売数 - 廃棄数の継続計算 | 最高 |
| 棚卸管理 | HTデータ取込・実在庫登録・差異計算 | 高 |
| 在庫照会 | SKU別・温度帯別の在庫確認 | 高 |
| 廃棄管理 | 廃棄スキャン登録・廃棄ロス計上 | 最高 |
| 返品管理 | メーカー・DC返品の処理 | 中 |
| 検品処理 | 発注データとの突合・欠品/過剰処理 | 最高 |

**集約ルート**: 在庫ポジション（StockPosition）
**主要エンティティ**: 入荷記録（ReceiptRecord）、廃棄記録（WasteRecord）、棚卸差異（InventoryVariance）

#### 売上ドメイン（Sales Domain）

日次精算と売上分析を担う。POSからデータを受け取り、本部へ送信する橋渡し役。

| 機能 | 概要 | 業務上の重要度 |
|------|------|--------------|
| 日次精算（EOB） | 売上・廃棄・現金の日次締め処理 | 最高 |
| 単品管理・売上分析 | SKU別ABC分析・時間帯別・曜日別分析 | 高 |
| 精算書作成 | 日計表・精算書の印刷・本部送信 | 最高 |
| 現金実査突合 | 現金実査とレジ記録の差異（オーバー・ショート）確認 | 高 |
| 電子マネー別集計 | Suica・nanaco等の決済種別ごとの集計 | 高 |

**集約ルート**: 日次精算（DailyClosure）
**主要エンティティ**: 売上サマリー（SalesSummary）、決済種別集計（PaymentTypeAggregation）

#### 本部通信ドメイン（HQ Communication Domain）

店舗と本部の双方向データ通信を担う。EOSとEOBが主要プロトコル。

| 機能 | 概要 | 業務上の重要度 |
|------|------|--------------|
| 発注データ送信（EOS） | 発注締め時刻に本部へ発注データを送信 | 最高 |
| EOB送信 | 精算・廃棄・勤怠等の日次データ一括送信 | 最高 |
| 商品マスタ受信 | 新商品・価格変更・廃番情報の受信と適用 | 最高 |
| 棚割マスタ受信 | プラノグラムの受信と店舗展開 | 高 |
| セキュリティパッチ配信 | ストコン・POSのソフトウェア更新 | 高 |
| キャンペーン情報受信 | POSへの割引設定の配信 | 高 |

**集約ルート**: 通信セッション（CommunicationSession）
**主要エンティティ**: 送信キュー（TransmissionQueue）、受信マスタ（ReceivedMaster）

#### サービスドメイン（Service Domain）

公共料金収納代行・チケット発券・マルチメディア端末との連携を担う。

| 機能 | 概要 | 業務上の重要度 |
|------|------|--------------|
| 収納代行処理 | 払込票バーコードスキャン・現金受取・領収書発行 | 高 |
| チケット発券連携 | Loppi/Famiポートとの発券数・売上連携 | 中 |
| 収納代行精算 | 当日の収納代行金額の本部報告 | 高 |

**集約ルート**: サービス取引（ServiceTransaction）
**特記**: 収納代行は金融取引に準じる。未収金管理が重要。

#### 従業員ドメイン（Employee Domain）

シフト管理・勤怠管理を担う。他ドメインとの依存関係が最も少ない。

| 機能 | 概要 | 業務上の重要度 |
|------|------|--------------|
| シフト作成 | 週次・月次のシフト表作成・人件費概算 | 高 |
| 打刻管理 | 出退勤記録（タイムレコーダー連携） | 高 |
| 労働時間集計 | 深夜割増・時間外の自動計算 | 高 |
| 給与データ出力 | 給与計算システムへのCSVエクスポート | 中 |

**集約ルート**: シフト（Shift）
**特記**: 労働基準法・最低賃金法への準拠が必須。改正頻度が高い。

### 1-2. 各ドメインの依存関係

ドメイン間の依存関係を整理する。矢印は「依存する方向」を示す。

```
[本部通信ドメイン]
    ↑ 発注データを送信
    |  売上・廃棄・検品データを送信
    |  商品マスタ・棚割・キャンペーンを受信
    |
[発注ドメイン] ←── [在庫ドメイン]
    |                  |
    |  在庫水準を参照  |  理論在庫を更新
    |                  ↑
                  [売上ドメイン] ←── POSレジ（外部）
                       |
                  日次精算データを送信
                       |
               [本部通信ドメイン]

[サービスドメイン] ── 売上計上 ──→ [売上ドメイン]
[従業員ドメイン]   ── 勤怠データ ─→ [本部通信ドメイン]
```

#### 依存関係の特徴と注意点

| 依存関係 | 内容 | 分解時の注意 |
|---------|------|------------|
| 売上ドメイン → 在庫ドメイン | POSの販売データで理論在庫をリアルタイム更新 | 同期処理かイベント経由か設計判断が必要 |
| 発注ドメイン → 在庫ドメイン | 発注点判定のために現在庫を参照 | 在庫サービスのAPIを呼び出す形にする |
| 本部通信ドメイン → 全ドメイン | 通信ドメインがすべてのデータの出口・入口 | メッセージブローカー経由に疎結合化 |
| 在庫ドメイン → 商品マスタ | SKU単位の在庫管理に商品マスタが必須 | 商品マスタは読み取り専用の共有データとして扱う |

### 1-3. 分解の優先順位と推奨アプローチ

ストコンのモノリスをいきなり全面分解するのはリスクが高い。**Strangler Figパターン**（セブン-イレブンも採用）を用い、機能を段階的にマイクロサービスとして切り出すことを推奨する。

#### Phase 1: 優先分解ドメイン（リスク低・効果高）

| ドメイン | 推奨理由 | 切り出しの難易度 |
|---------|---------|--------------|
| 従業員ドメイン | 他ドメインへの依存が最も少ない。SaaS（勤怠管理システム）への置換も選択肢 | 低 |
| サービスドメイン | 収納代行・チケット機能は独立性が高い | 低〜中 |
| 売上ドメイン（分析部分） | 単品管理・ABC分析は読み取り専用。別サービスに分離しやすい | 中 |

#### Phase 2: 中期分解ドメイン（効果大・慎重設計が必要）

| ドメイン | 推奨理由 | 切り出しの難易度 |
|---------|---------|--------------|
| 発注ドメイン | AI発注最適化との統合で効果大。ただし発注締め時刻の厳守が必須 | 中〜高 |
| 在庫ドメイン | 廃棄削減・棚割最適化のデータ基盤になる。理論在庫の整合性維持が難しい | 高 |

#### Phase 3: 慎重に検討するドメイン（影響大・リスク高）

| ドメイン | 理由 | 代替アプローチ |
|---------|------|--------------|
| 本部通信ドメイン | 発注締めの遅延は店舗の欠品に直結。EOBの失敗は会計に影響 | APIゲートウェイを前段に置き、既存通信ロジックは当面維持 |
| 売上ドメイン（精算部分） | 日次精算はフランチャイズ会計の根幹。バグは加盟店トラブルに発展 | 最後に移行。徹底した並行稼働テストを実施 |

#### PM向け落とし穴

- 発注ドメインの分解は「在庫ドメインとの整合性」を先に設計しないと、在庫水準の不一致で誤発注が発生する
- 商品マスタは複数ドメインが参照するため、「共有データ」として慎重に扱う必要がある（分散モノリスに陥るリスク）
- チェーンごとに独自の発注フォーマット・通信プロトコルがあるため、本部通信ドメインの標準化には調整コストがかかる

## 2. イベント駆動化の候補業務

### 2-1. 現行バッチ処理の棚卸し

ストコンの業務処理は歴史的に「定時バッチ」が中心である。これは通信コスト・処理能力の制約から生まれた設計であり、クラウド化に伴いイベント駆動へ移行できるものが多い。

| バッチ処理 | 実行タイミング | 遅延の業務影響 |
|-----------|-------------|--------------|
| POSデータ本部送信 | 1時間ごと or 日次 | リアルタイム分析ができない |
| 廃棄アラート生成 | 日次（EOB処理内） | 廃棄対応が翌日になる場合がある |
| 在庫水準チェック（発注点判定） | 発注締め時刻に一括 | 締め時刻の間に発生した欠品を拾えない |
| 棚割マスタ受信・反映 | 夜間バッチ | 棚替え指示の反映に最長24時間かかる |
| 商品マスタ更新 | 夜間バッチ | 新商品の即日取扱いが困難 |
| 温度アラート通知 | 閾値超過時（一部システム） | 食品衛生上のリスク |

---

### 2-2. イベント駆動移行の候補と優先度

#### 優先度A（最高）: 廃棄アラート

**現状**: 消費期限管理は日次バッチで処理される。販売期限切れ商品のアラートが翌日にならないと本部に集約されないケースがある。

**イベント駆動後の姿**:
```
消費期限残り X時間の商品が存在
  → イベント発火（WasteAlertTriggered）
  → 店舗端末に即時通知（値引き・廃棄作業指示）
  → 本部ダッシュボードへリアルタイム集計
  → 需要予測モデルへのフィードバック
```

**業務効果**: 廃棄作業の見逃し防止。値引きタイミングの最適化による販売機会増。廃棄ロス率の改善。

**業務上の注意**: 値引き実施の判断権限が店長にあるチェーンでは、通知後の承認フローが必要。

---

#### 優先度A（最高）: 在庫発注点トリガー

**現状**: 発注量の計算は発注締め時刻に一括実行される。締め時刻の間に予期せぬ欠品（急な特需等）が発生しても、次の締め時刻まで自動発注されない。

**イベント駆動後の姿**:
```
販売取引が発生（POS → ストコン）
  → 理論在庫をリアルタイム更新
  → 在庫 ≤ 発注点 の条件を判定
  → 発注トリガーイベント発火（StockBelowReorderPoint）
  → 推奨発注数量を算出・店員へ通知
  → 発注締め時刻に自動送信（または店員が確認後送信）
```

**業務効果**: 欠品（チャンスロス）の削減。発注業務の分散化（締め時刻直前の作業集中を緩和）。

**業務上の注意**: 発注締め時刻は配送センターとの契約に紐づく。イベント駆動で発注点を検知しても、送信は所定の締め時刻に行う必要がある。「検知」と「送信」を分離して設計すること。

---

#### 優先度B（高）: POSデータのリアルタイム本部送信

**現状**: 多くのチェーンでは1時間ごとまたは日次バッチで本部へ送信している。

**イベント駆動後の姿**:
```
POS取引が発生（1件ごと）
  → ストコン内で売上を即時集計
  → 本部へリアルタイムストリーミング送信
  → 本部DWHへの即時反映
  → 全店売上ダッシュボードのリアルタイム更新
```

**業務効果**: 本部によるリアルタイム全店モニタリングが可能。異常（不正取引・POS障害）の早期検知。AI需要予測の精度向上（最新データを即時フィード）。

**業務上の注意**: 全国5万店規模のストリーミングは通信コストが大幅に増加する。重要度の高いデータ（売上サマリー等）と詳細データを分けて送信頻度を最適化する設計が必要。

---

#### 優先度B（高）: 棚割変更の即時反映

**現状**: 本部からのプラノグラム（棚割指示）は夜間バッチで店舗に配信される。

**イベント駆動後の姿**:
```
本部で棚割マスタが更新される
  → 変更イベント発火（PlanogramUpdated）
  → 対象店舗のストコンへ即時プッシュ配信
  → 店員の端末に棚替え指示が表示
  → 実施完了を店舗からフィードバック
```

**業務効果**: キャンペーン・新商品対応の即日実施が可能。棚替え実施状況の本部からのリアルタイム把握。

---

#### 優先度C（中）: 温度逸脱アラートの即時通報

**現状**: 一部チェーンでは実装済みだが、本部への集約が遅延しているケースがある。

**イベント駆動後の姿**:
```
温度センサーが閾値を超過
  → 即時アラートイベント発火（TemperatureThresholdExceeded）
  → 店員への端末通知
  → 本部への自動通報（食品衛生法の記録義務対応）
  → エスカレーション（一定時間応答なし → SV・設備業者へ自動連絡）
```

**法的注意**: 食品衛生法に基づく温度記録の保管義務がある。イベントログをタイムスタンプ付きで不変ストレージに記録する設計が必須。

## 3. コンテナ化の適性評価

### 3-1. サブシステム別評価マトリックス

以下の観点でコンテナ化の適性を評価する。
- **ステートレス性**: 状態を持たない処理はコンテナ化に適する
- **スケーラビリティ要件**: 負荷変動が大きい処理はコンテナのオートスケールと親和性が高い
- **依存関係**: OS・ハードウェア固有の依存があるとコンテナ化が困難
- **変更頻度**: 頻繁に改修が入る処理はコンテナ化でデプロイを高速化できる

| サブシステム | ステートレス性 | スケーラビリティ | ハード依存 | 変更頻度 | コンテナ化適性 |
|------------|-------------|--------------|----------|---------|-------------|
| 本部向けバッチ処理（EOB送信） | 高 | 中 | なし | 低 | ◎ 高 |
| AI発注推奨エンジン | 高 | 高 | なし | 高 | ◎ 高 |
| 売上分析・レポート生成 | 高 | 高 | なし | 中 | ◎ 高 |
| 商品マスタ配信API | 中 | 高 | なし | 中 | ○ 高 |
| 在庫照会API | 中 | 高 | なし | 中 | ○ 高 |
| 発注管理API | 中 | 中 | なし | 中 | ○ 中〜高 |
| 棚卸データ取込処理 | 高 | 低 | なし | 低 | ○ 中 |
| 従業員シフト管理 | 中 | 低 | なし | 中 | ○ 中（SaaS移行も選択肢）|
| POS-ストコン間のローカル通信処理 | 低 | 低 | 店内LANに依存 | 低 | △ 要検討 |
| POSレジアプリ本体 | 低 | 低 | 専用ハード | 低 | × 困難 |
| 電子マネー決済処理 | 低 | 低 | 認定端末必須 | 低 | × 困難（PCI DSS制約）|
| ストコン本体（レガシー） | 低 | 低 | 専用OS・ドライバ | 低 | × 困難（段階移行が必要）|
| 温度管理センサー連携 | 低 | 低 | センサーHWに依存 | 低 | △ エッジ処理必要 |

### 3-2. 推奨移行ロードマップ

#### Step 1: 本部系サービスのコンテナ化（6〜12ヶ月）

本部側で動作する処理から着手する。店舗システムへの影響がなく、リスクが最も低い。

| 対象 | アプローチ | 業務継続への影響 |
|------|----------|--------------|
| AI発注推奨エンジン | 既存ロジックをAPI化・コンテナ化。店舗からのリクエストを受け付ける | ほぼなし（推奨値の提供のみ） |
| 売上分析・レポート生成 | 読み取り専用処理のため安全に移行可能 | なし |
| 商品マスタ管理システム | 段階的にAPIを公開し、既存の夜間バッチ配信と並行運用 | 低 |

#### Step 2: 店舗系APIの先行コンテナ化（12〜18ヶ月）

ストコン本体をいじらず、その前段にAPIゲートウェイを設置して新機能をコンテナとして提供する（Strangler Figパターン）。

| 対象 | アプローチ | 業務継続への影響 |
|------|----------|--------------|
| 在庫照会API | ストコンのDBに読み取り専用で接続するAPIをコンテナで提供 | なし |
| 発注管理API | 新規発注画面をコンテナ化。既存ストコンの発注送信機能は当面維持 | 低（並行稼働） |
| 棚卸データ取込処理 | HTからのデータ取込APIをコンテナ化 | 低 |

#### Step 3: ストコン本体の段階的置き換え（18ヶ月〜）

ストコン本体の機能を1つずつAPIに置き換え、最終的に本体を廃止していく長期フェーズ。

| 対象 | アプローチ |
|------|----------|
| 精算処理（EOB） | 並行稼働テストを十分に実施後に切り替え |
| レガシーバッチ（PL/SQL等） | AWS DMS等でDBを移行後、バッチをコンテナ化 |
| ストコン本体 | Thin Client化（セブン-イレブン方式）または完全クラウド移行 |

#### PM向け落とし穴

- Step 1〜2は技術的なリスクが低いが、**既存ベンダー（NEC・富士通等）との契約関係**を先に整理する必要がある。保守契約が独自システムの改変を禁じている場合がある
- 電子マネー決済処理は**PCI DSSの認定端末・認定ソフトウェア**が必要であり、コンテナ化は容易でない。スコープ外として扱うことを強く推奨する
- 24時間365日稼働の店舗システムは**ブルーグリーンデプロイメント**が事実上必須。テストが不十分なデプロイは全国店舗に即影響する

## 4. ストコンデータの全体像

### 4-1. データカテゴリと発生源

ストコンが生成・収集するデータは以下の6カテゴリに分類される。

#### POSデータ（売上トランザクションデータ）

| 項目 | 内容 |
|------|------|
| 発生源 | POSレジ（1店舗あたり2〜8台） |
| 主要データ項目 | 取引ID、日時、SKU（JANコード）、数量、金額、決済種別、割引情報 |
| 発生頻度 | 1店舗あたり平均800〜1,200件/日（コンビニの1日平均客数より） |
| リアルタイム性 | 取引ごとに発生。現状は1時間バッチで集約するケースが多い |

#### 発注データ

| 項目 | 内容 |
|------|------|
| 発生源 | ストコン発注処理（1日2〜3回の締め時刻に送信） |
| 主要データ項目 | 発注ID、SKU、発注数量、推奨数量、実発注数量、配送区分（朝/昼/夕便） |
| 発生頻度 | 1店舗あたり1日あたり150〜300SKU分の発注データ |
| 特記 | 推奨数量と実発注数量の差分が「店員修正量」として分析に使える |

#### 在庫データ

| 項目 | 内容 |
|------|------|
| 発生源 | ストコン理論在庫計算、HTによる棚卸 |
| 主要データ項目 | SKU、理論在庫数、実在庫数（棚卸時）、在庫差異、温度帯 |
| 発生頻度 | 理論在庫は取引ごとに更新。棚卸は月1〜2回（全棚卸）または日次（循環棚卸） |

#### 廃棄データ

| 項目 | 内容 |
|------|------|
| 発生源 | 廃棄スキャン処理（人手） |
| 主要データ項目 | SKU、廃棄数量、廃棄理由（期限切れ/破損/誤廃棄等）、廃棄時刻 |
| 発生頻度 | 1店舗あたり1日30〜80件程度（デイリー品の廃棄が主体） |
| 特記 | 廃棄時刻と消費期限の差（何時間前に廃棄したか）が重要分析軸 |

#### 顧客・決済データ

| 項目 | 内容 |
|------|------|
| 発生源 | POS決済処理、ポイントカード読取 |
| 主要データ項目 | 匿名または会員ID、購買商品、購買金額、決済種別、ポイント利用・付与 |
| 特記 | 個人情報保護法・匿名加工情報の取扱いが必要。設計初期からのデータガバナンス検討が必須 |

#### 機器・設備データ

| 項目 | 内容 |
|------|------|
| 発生源 | 温度センサー、POS稼働ログ、ストコン稼働ログ |
| 主要データ項目 | センサーID、温度値、タイムスタンプ、エラーコード |
| 特記 | 食品衛生法上の温度記録保管義務あり。改ざん不能なストレージへの記録が必要 |

### 4-2. データ量の概算

#### 1店舗あたりの日次データ量

| データカテゴリ | 件数/日 | 推定データサイズ |
|-------------|--------|--------------|
| POSトランザクション | 800〜1,200件 | 約1〜2 MB/日 |
| 発注データ | 150〜300件 | 約50〜100 KB/日 |
| 在庫更新ログ | 3,000〜5,000件（取引ごと更新） | 約2〜5 MB/日 |
| 廃棄データ | 30〜80件 | 約10〜20 KB/日 |
| 温度センサーデータ | 約1,440件（1分ごと × センサー数）| 約1〜2 MB/日 |
| 機器稼働ログ | 数千件 | 約500 KB〜1 MB/日 |
| **1店舗合計** | | **約5〜10 MB/日** |

**根拠**: コンビニの1日平均客数は約900〜1,000人（JFA統計）。買い回り・単品購入を考慮すると取引件数は客数の0.8〜1.2倍程度。

#### 全国規模（5万店舗想定）

| 指標 | 概算 |
|------|------|
| 全店舗 日次データ量 | 約250〜500 GB/日 |
| 全店舗 年間データ量 | 約90〜180 TB/年 |
| 1年間のPOS取引件数 | 約150〜220億件（全国） |
| 保管期間（税務上7年） | 最大 約700〜1,300 TB の累積データ |

**PM向け留意点**: このデータ量はDWHのストレージコスト計算の基礎となる。頻繁にアクセスする直近データ（1〜2年）と長期保管データ（3〜7年）でストレージ階層を分けてコストを最適化することが現実的。

## 5. データ分析ユースケース

### 5-1. 需要予測・自動発注最適化

**ビジネス背景**: コンビニの廃棄ロスとチャンスロスは表裏の関係にある。発注量が多ければ廃棄が増え、少なければ欠品になる。AIによる需要予測はこのトレードオフを最小化する最重要ユースケース。セブン-イレブンはすでに本稼働しており、廃棄削減率15〜20%改善の実績がある。

**分析に使うデータ**:

| データ | 役割 |
|-------|------|
| 過去のPOS売上（SKU別・時間帯別・店舗別） | 基本的な販売パターンの学習 |
| 曜日・祝日・時間帯 | 周期パターンの考慮 |
| 気象データ（気温・天気・降水量） | 季節商品・デイリー品の需要変動 |
| 地域イベント情報（花火大会・スポーツ等） | スポット需要の予測 |
| 棚割変更履歴 | フェイス変更による売上影響の除去 |
| 近隣競合店の動向 | 間接的な需要変化の把握 |

**業務フローへの組み込み**:
```
[前日21時頃] 翌日の気象・イベント情報を取得
  → 需要予測モデルが翌日のSKU別需要を算出
  → 推奨発注数量を更新
[翌朝 発注締め前] 店員が推奨数量を確認・修正
  → 発注データを送信
[実績収集] 実売数・廃棄数をモデルにフィードバック
  → 継続的にモデルを再学習
```

**業務上の注意**:
- 推奨数量はあくまでも「推奨」。最終判断は店員・店長であり、現場の感覚（地域の行事・常連客の動向等）を反映できる仕組みが必要
- モデルが「外れた」場合の原因分析ができるよう、特徴量とその寄与度を記録しておくこと
- 新商品・季節限定商品は過去データがないため、類似商品の実績を使ったコールドスタート対策が必要

---

### 5-2. 廃棄ロス削減

**ビジネス背景**: 廃棄ロスはコンビニ加盟店オーナーの収益を直接圧迫する。全国コンビニの廃棄ロス総額は年間数千億円規模と推定される。廃棄削減はESG（食品廃棄削減）の観点でも注目されている。

**主要分析シナリオ**:

#### シナリオA: 消費期限ベースの廃棄リスクスコアリング

```
[入力] 各SKUの現在庫数・消費期限・残り時間・過去の時間帯別販売実績
[処理] 「消費期限までに売り切れる確率」をスコア化
[出力] リスクスコア上位商品に対する行動提案
       - 値引き推奨（残り3時間・販売確率60%以下等）
       - 本部への廃棄予測データ送信
       - 翌日の発注量への自動反映
```

#### シナリオB: 廃棄パターン分析

```
[分析対象] SKU別・曜日別・時間帯別の廃棄率
[目的] 「構造的に廃棄が多いSKU」の特定と棚割見直し提案
[アクション] 死に筋商品の取り扱い削減・フェイス削減の提案
```

**業務上の注意**:
- 値引き実施の権限はチェーンによって異なる（本部管理チェーンでは店舗単独で値引きできない場合がある）
- 廃棄データの入力は人手によるスキャン。入力ミスや登録漏れが一定数ある前提でデータ品質管理が必要
- デイリー品（弁当・おにぎり）は複数の消費期限区分が同時に陳列される。SKUだけでなく「入荷ロット」単位での管理が廃棄精度を高める

---

### 5-3. 棚割最適化

**ビジネス背景**: 限られた棚スペースに何をどう配置するかは、コンビニの売上を大きく左右する。単品管理の思想に基づき、SKU単位でフェイス数と配置を最適化することで売上・粗利を改善できる。

**主要分析シナリオ**:

#### シナリオA: フェイス弾力性分析

```
[分析] フェイス数を変化させた際の売上変化量の推定（弾力性）
[活用] フェイス1枚増やすと売上がどれだけ増えるかを予測
       → 最も弾力性の高い商品のフェイスを優先的に増やす
```

#### シナリオB: 店舗クラスタ別棚割推奨

```
[入力] 立地（駅前・住宅地・オフィス・ロードサイド等）・客層データ・売上構成
[処理] 類似店舗をクラスタリング
[出力] クラスタごとに最適化された棚割案の生成
```

#### シナリオC: 棚替え前後の効果測定

```
[データ] 棚替え実施日・変更内容・変更前後のSKU別売上
[分析] 棚替えによる売上増減の定量評価
[活用] 効果のあった棚割パターンのベストプラクティス化
```

**業務上の注意**:
- 棚割は本部が策定するものであり、店舗が独自に変更できる範囲は限られる。分析結果は本部MDチームへのインプットとして位置づける
- 交叉比率（粗利率 × 在庫回転率）を最大化する商品を優先する視点が重要
- 定番商品の棚を変えると常連客のクレームにつながることがある。導入時に現場の合意形成が必要

---

### 5-4. リアルタイム売上ダッシュボード

**ビジネス背景**: 本部のSV（スーパーバイザー）は複数店舗を巡回するが、全店の状況をリアルタイムで把握する手段が限られている。リアルタイムダッシュボードにより、異常店舗の早期発見・迅速な対処が可能になる。

**主要機能**:

| 機能 | 表示内容 | 更新頻度 |
|------|---------|---------|
| 全店売上マップ | 地図上に全店の売上状況を色分け表示 | 5〜15分ごと |
| 時間帯別売上進捗 | 前日・前週比での売上ペース確認 | 30分ごと |
| 異常店舗アラート | 売上急落・POS停止の疑い等を通知 | リアルタイム |
| 新商品売行き速報 | 新商品導入初日からの売上追跡 | 1時間ごと |
| 廃棄速報 | 廃棄量が多い店舗・SKUのランキング | 日次 |

**業務上の注意**:
- POSデータのリアルタイム送信が実現していない店舗では、ダッシュボードのデータは「遅延あり」になる。システム移行中は混在状態を明示する設計が必要
- SVが担当する店舗数は1人あたり20〜50店舗程度。ダッシュボードは「気づき」を提供するものであり、過剰な通知は逆効果（アラート疲れ）

---

### 5-5. 異常検知

**ビジネス背景**: POSデータの不正取引・在庫差異の大きい店舗・機器障害の早期検知は、損失防止・店舗運営品質向上に直結する。

**主要異常検知シナリオ**:

#### POS不正取引の検知

```
[対象] 特定店員の取引パターン（値引き率・取消頻度・高額取消等）
[手法] ルールベース（閾値超過）+ 機械学習（異常スコアリング）
[アクション] SVへのアラート・取引記録の保全
```

**業務上の注意**: 従業員の不正疑いに関わるため、データの取扱い・通報フローは就業規則・労働法との整合性が必要。誤検知時の対処フローも事前に設計すること。

#### 在庫差異（ロス）の異常検知

```
[対象] 理論在庫と実在庫の差異が大きいSKU・店舗
[手法] 店舗規模・立地ごとのベースライン差異を設定し、統計的外れ値を検知
[アクション] 棚卸の追加実施提案・万引き多発商品の対策
```

#### POS・ストコン障害の検知

```
[対象] POSからのデータが途絶えた店舗・エラーログの急増
[手法] ハートビート監視・エラー頻度の異常検知
[アクション] 保守チームへの自動通報・ストコン再起動のリモート試行
```

---

### 5-6. その他のユースケース

#### 顧客購買分析（バスケット分析）

| 分析内容 | 活用方法 |
|---------|---------|
| よく一緒に買われるSKUの組み合わせ | 関連商品の棚隣接配置・クロスセル提案 |
| 時間帯・曜日別の購買パターン | 時間帯限定キャンペーンの設計 |
| 定番購買者の離脱検知 | ポイント還元・クーポン配信でリテンション |

**注意**: 顧客IDが紐づかない現金取引は分析対象外。電子マネー・ポイントカード利用者のみが分析可能。個人情報保護法・プライバシーポリシーへの対応が前提。

#### 新商品効果測定

```
[データ] 新商品導入日・導入店舗・初週/初月の販売実績
[分析] 類似商品・類似店舗との比較による「新商品インクリメント」の算出
[活用] 全店展開判断・フェイス数調整・発注量見直し
```

#### 店舗間ベンチマーク分析

```
[手法] 立地・規模・競合環境が類似した店舗を自動クラスタリング
[分析] クラスタ内の売上・粗利・廃棄率の分布
[活用] 低パフォーマンス店舗の改善余地特定・SVの改善提案材料
```

#### 物流最適化

```
[データ] 発注量・配送ルート・DC在庫・配送車の積載量
[目的] 配送頻度・ルートの最適化による物流コスト削減
[連携先] DC（配送センター）側の物流システムとの連携が必要
```

**注意**: 物流最適化はストコン単独では実現できず、DCシステム・配送管理システムとのデータ連携が前提。プロジェクトスコープを超える可能性があるため、関係ベンダーとの調整が必要。

## 6. データパイプライン要件

### 6-1. バッチ vs リアルタイムの分類

| 分析ユースケース | 処理方式 | 理由 |
|---------------|---------|------|
| 需要予測・発注推奨 | バッチ（発注締め前に実行） | 発注は定時送信であり、リアルタイム更新は不要 |
| 廃棄リスクスコアリング | ニアリアルタイム（30分〜1時間） | 廃棄対応の判断は数時間単位で行うため |
| 棚割最適化 | バッチ（日次〜週次） | 棚割変更は月1〜2回程度のため |
| リアルタイム売上ダッシュボード | リアルタイム〜準リアルタイム（5〜15分） | SVの即時対応に活用 |
| POS不正取引検知 | リアルタイム | 不正は即時検知・保全が重要 |
| 在庫差異検知 | バッチ（棚卸後） | 棚卸実施タイミングに依存 |
| バスケット分析 | バッチ（日次〜週次） | 傾向分析であり即時性不要 |
| 新商品効果測定 | バッチ（日次） | 日次で進捗を確認すれば十分 |
| 温度逸脱アラート | リアルタイム | 食品衛生上の即時対応が必要 |

---

### 6-2. 要件マトリックス

| ユースケース | データ鮮度要件 | レイテンシ上限 | データ量（日次） | 主要データソース |
|------------|-------------|-------------|--------------|--------------|
| 需要予測・発注推奨 | 前日まで（発注締め前） | 締め時刻まで（時間単位） | 全店舗 × 数百SKU | POS売上、在庫、気象、イベント |
| 廃棄リスクスコアリング | 30分〜1時間 | 1時間以内 | 廃棄候補商品のみ（軽量） | 在庫、POSリアルタイム、消費期限マスタ |
| 棚割最適化 | 日次で十分 | 翌日中 | 全店舗の売上・在庫 | POS売上、棚割マスタ、廃棄データ |
| リアルタイム売上ダッシュボード | 5〜15分以内 | 15分以内 | 全店舗 POS集計 | POS売上サマリー |
| POS不正取引検知 | 即時（秒単位） | 数秒以内 | 全トランザクション | POS取引明細 |
| 在庫差異検知 | 棚卸後 | 1〜2時間 | 棚卸差異データ | 理論在庫、棚卸実績 |
| バスケット分析 | 日次 | 翌日 | 全POSトランザクション（購買セット） | POS取引明細（同一取引ID） |
| 温度逸脱アラート | 即時（秒単位） | 1分以内 | 温度センサーデータ | 温度センサー |

#### データパイプラインのアーキテクチャ方針

```
[データ発生源]
  POS取引 / 在庫更新 / 温度センサー
       ↓
[収集レイヤー]
  リアルタイム: ストリーミング取込
  バッチ: EOB処理・日次ファイル転送
       ↓
[格納レイヤー]
  生データ格納（S3等）
  ストリーミングバッファ（Kinesis等）
       ↓
[処理レイヤー]
  リアルタイム処理: ストリーム処理（秒〜分単位）
  バッチ処理: ETL/ELT（時間〜日単位）
       ↓
[分析・AI レイヤー]
  需要予測モデル / 異常検知モデル / BI/ダッシュボード
       ↓
[アクション レイヤー]
  店舗端末への通知 / 本部SVダッシュボード / 発注推奨
```

**業務要件として押さえるべき点**:
- リアルタイムと批次の処理を**Lambda Architecture**または**Kappa Architecture**で整理する。どちらを採用するかはユースケースの複雑さとチームの技術力で決める
- 全国5万店のリアルタイムデータ収集はピーク時（昼12〜13時・夜18〜20時）に集中する。平常時の5〜10倍のスパイク吸収設計が必要
- POSデータは個人購買履歴を含む可能性があるため、取込み段階で**匿名化・仮名化**の処理を組み込む必要がある

## 7. まとめ：PM視点でのチェックポイント

### 7-1. プロジェクト開始前に確認すべき事項

| 確認項目 | 理由 |
|---------|------|
| 対象チェーンの現行ストコンベンダーとの契約内容 | 改変禁止条項がある場合、移行設計が根本的に変わる |
| 発注締め・EOBの時刻と遅延許容範囲 | これを外せない制約として設計の出発点にする |
| PCI DSS準拠範囲の確認 | 決済系を移行スコープに含めるかどうかで難易度が激変する |
| 温度管理データの保管法的要件 | 食品衛生法に基づく保管義務の確認と設計への反映 |
| 個人情報・購買データの取扱いポリシー | 分析ユースケースの範囲がここで決まる |
| 加盟店（フランチャイジー）との関係性 | 新システムの強制導入には加盟店オーナーの合意が必要な場合がある |

### 7-2. スコープ管理の注意点

| リスク | 対処方針 |
|-------|---------|
| 「データ分析」がいつの間にか「AI発注の完全自動化」に拡大する | フェーズ0はダッシュボード・可視化のみと明示し、AIによる意思決定自動化は次フェーズとする |
| 物流最適化がスコープに入り、DC側のシステムまで改修が必要になる | ストコン側の「発注データ出力」までをスコープとし、DC側は別プロジェクトとする |
| コンテナ化・マイクロサービス化と業務移行が同時進行してプロジェクトが混乱する | 技術的なモダナイゼーションと業務システムの切り替えを別フェーズに分離する |
| 全チェーン共通基盤を目指して標準化が難航する | まず1チェーン・1機能での成功事例を作り、横展開の際に標準化を進める |

### 7-3. 業務継続性（BCP）の観点

コンビニは**24時間365日**の無停止営業が前提であり、システムの停止は即座に売上損失・顧客信頼の毀損に繋がる。

| リスクシナリオ | 業務への影響 | 対処方針 |
|-------------|-----------|---------|
| 発注システムの停止 | 欠品→チャンスロス。翌日以降も影響が続く | 手動発注フローを必ず維持する。ストコン上での緊急発注手段を残す |
| EOB処理の失敗 | 日次精算が未送信→会計・ロイヤリティ計算ができない | EOBのリトライ・再送信ロジックと、失敗時の本部への通報フローを設計する |
| 商品マスタ配信の遅延 | 新商品のスキャンが通らない・価格エラー | 前回マスタでの暫定運用手順を店舗に事前周知する |
| POSとストコンの通信断 | 売上データが集計できない | 店内ネットワーク障害時のPOS単独動作モードを確保する |
| クラウド接続断（インターネットVPN障害） | 本部との通信全停止 | 店舗ローカルでの業務継続（スタンドアロン動作）の要件を明確化する |

### 7-4. ステークホルダー管理の特有事項

| ステークホルダー | 関心事 | PM対応のポイント |
|---------------|-------|---------------|
| チェーン本部IT部門 | セキュリティ・稼働率・ベンダー管理 | NDA締結後の詳細情報開示。定期的な進捗報告 |
| 既存ベンダー（NEC・富士通等） | 既存保守契約の維持・新システムへの参画 | 対立ではなく連携。既存ナレッジの引継ぎ協力を得る |
| 加盟店オーナー | 店舗運営への影響最小化・利便性向上 | 移行前の十分な研修・移行中のホットライン設置 |
| 本部SVチーム | 巡回業務のデジタル化への期待と不安 | 早期にダッシュボードの試作版を見せてフィードバックを得る |
| 配送センター担当 | 発注データの受信フォーマット変更への対応 | フォーマット変更は最小限にとどめ、変換レイヤーで吸収する |
