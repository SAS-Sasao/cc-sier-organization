# 技術者出身PMがハマる失敗パターン集

**作成日**: 2026-03-27
**作成者**: 秘書室（壁打ち深掘り）
**対象者**: 笹尾豊樹
**目的**: 技術者→PMの視座切替で起きやすい失敗を事前に学び、参画後の地雷を避ける
**関連資料**: [PMスキルマップ](pm-skill-map.md) の伸びしろ4領域と連動

---

## 本資料の使い方

各パターンは以下の構造で記載:
1. **症状** — 外から見える問題行動
2. **根本原因** — 技術者マインドとのギャップ
3. **ケーススタディ** — SIer移行案件での具体シナリオ（ストコン案件想定）
4. **処方箋** — 具体的な対策・習慣
5. **セルフチェック** — 自分がハマっていないか確認する問い

---

## パターン1: 「正論パンチ」— 正しさで押し切る

### 対応領域: ステークホルダー調整（最優先）

### 症状
- 会議で技術的に正しい結論を論理的に説明し、相手が黙る
- 黙った＝合意だと思っているが、実は不満が蓄積している
- 後日「そんなの聞いていない」「納得していない」と覆る

### 根本原因
技術の世界では「正しいものが勝つ」。コードは動くか動かないか。
しかしPMの世界では「正しくても受け入れられなければ意味がない」。
相手の立場・メンツ・政治的事情を考慮せずに正論を出すと、合意形成が崩壊する。

### ケーススタディ

> **状況**: ストコン移行で、本部IT部門が「既存のバッチ処理をそのままAWSに載せたい（Rehost）」と主張。笹尾PMは技術的にRefactorが正しいと判断。
>
> **失敗パターン**: 「Rehostだとコスト効率が悪く、将来の拡張性もありません。Refactorすべきです」と会議で正面から否定。本部IT部門長のメンツを潰す結果に。
>
> **その後**: 本部IT部門がプロジェクト全体に非協力的になり、要件ヒアリングが停滞。3週間のスケジュール遅延。

### 処方箋

| やりがち | こうする |
|---------|---------|
| 「Rehostは間違いです」 | 「Rehostのメリットを整理した上で、もう一つの選択肢も検討してよいですか」 |
| 会議でいきなり結論を出す | 事前に1on1で相手の懸念を聞き、落としどころを作ってから会議に臨む |
| 資料に正解だけ書く | 複数案を並べ、トレードオフ表を作り、相手に「選んでもらう」形にする |

### セルフチェック
- [ ] 会議で相手が黙ったとき「合意した」と解釈していないか？
- [ ] 自分の提案に反対意見が出ないとき「完璧だから」と思っていないか？
- [ ] 技術的に劣る案を相手が推したとき、まず「なぜそう思うか」を聞いているか？

## パターン2: 「一人で解決マン」— 技術力で抱え込む

### 対応領域: リスク・課題管理

### 症状
- 課題が発生すると「自分で調べて解決する方が速い」と思い、手を動かし始める
- 課題管理表に載らない「隠れ課題」が増える
- PMなのにコードを書いている/設計書を直接書いている時間が多い
- チームメンバーが暇にしている一方、PMだけが残業

### 根本原因
技術者は「問題を自分の手で解決する」ことに快感を覚える。
PMの仕事は「問題を適切な人に振り、全体の流れを管理する」こと。
自分で解決すると1つの課題は速く片付くが、他の10個の課題が放置される。

### ケーススタディ

> **状況**: ストコンのデータ移行テストでDMS（Database Migration Service）のレプリケーションエラーが発生。笹尾PMはAWSの知識があるため、原因調査を自分で開始。
>
> **失敗パターン**: 3日間DMS調査に没頭。その間にPMO週次報告を飛ばし、別チームからのIF仕様レビュー依頼を放置。PMOから「報告がない。プロジェクト状況が見えない」と警告。
>
> **その後**: 技術的には解決したが、PMOからの信頼を失い、報告頻度を日次に増やされる（管理コスト増大）。

### 処方箋

**15分ルール**: 技術課題に遭遇したら、15分だけ状況を把握する。それ以上は担当者に振る。

| 時間 | PMがやること |
|------|------------|
| 0-15分 | 課題の概要を把握し、影響範囲を判断する |
| 15分〜 | 担当者をアサインし、期限を設定する |
| 以降 | 進捗を確認する（自分では手を動かさない） |

**エスカレーション判断マトリクス**:

| | 影響：小 | 影響：大 |
|---|---------|---------|
| **緊急度：低** | 担当者に任せる（週次で確認） | 課題管理表に登録、担当者+期限を設定 |
| **緊急度：高** | 担当者に即振り、日次で確認 | **即エスカレーション**（PMOに報告+対策会議設定） |

### セルフチェック
- [ ] 今週、技術的な作業に何時間使ったか？（目安: PM業務の20%以下が健全）
- [ ] 課題管理表に載っていない「自分だけが知っている課題」はないか？
- [ ] 「自分がやった方が速い」と思った回数を数えているか？

## パターン3: 「スコープ・イエスマン」— 技術的にできるから断れない

### 対応領域: スコープ管理（政治面）

### 症状
- 顧客から「これも追加で」と言われると、技術的に可能なので「できます」と答える
- 見積もり工数が膨らんでいるが「頑張ればいける」と思っている
- 気づいたらスコープが当初の1.5倍になっている
- チームから「いつの間にこんな機能が増えた？」と言われる

### 根本原因
技術者は「できるかどうか」で判断する。PMは「やるべきかどうか」で判断する必要がある。
技術的に可能 ≠ プロジェクトとして実行すべき、という区別ができない。
また「断る＝能力不足」と感じるマインドがある。

### ケーススタディ

> **状況**: ストコン移行の要件定義フェーズで、顧客業務部門から「ついでに発注画面のUI刷新もやってほしい」とリクエスト。技術的にはReactで作り直せば可能。
>
> **失敗パターン**: 「できますよ！」と即答。変更管理プロセスを通さずにWBSに追加。2ヶ月後、UIテスト工数が想定の3倍かかることが判明。スケジュール遅延。
>
> **その後**: PMOから「なぜ変更管理を通さなかった？」と追及。顧客にもスケジュール見直しを説明する羽目に。

### 処方箋

**断り方5パターン**:

| # | パターン | 言い方の例 |
|---|---------|-----------|
| 1 | **スコープ確認型** | 「当初スコープとの差分を整理させてください。次の会議で影響を報告します」 |
| 2 | **トレードオフ提示型** | 「追加可能ですが、決済機能の移行が2週間後ろ倒しになります。どちらを優先しますか？」 |
| 3 | **フェーズ分割型** | 「Phase 1では現行踏襲とし、Phase 2でUI刷新を検討しませんか？」 |
| 4 | **代替案提示型** | 「UI刷新の代わりに、設定変更で対応できる範囲があります。まずそこから始めませんか？」 |
| 5 | **プロセス誘導型** | 「変更管理プロセスに乗せて、CCBで優先度を判断していただけますか？」 |

**共通ポイント**: いずれも「NO」と言わない。代わりに「判断材料を揃えてから決めましょう」に持ち込む。

### セルフチェック
- [ ] 直近1週間で「できます」と即答したリクエストはないか？
- [ ] そのリクエストは変更管理プロセスを通っているか？
- [ ] 追加スコープのコスト・スケジュール影響を定量的に示したか？

## パターン4: 「技術語翻訳ミス」— 相手に合わせた説明ができない

### 対応領域: ステークホルダー調整

### 症状
- PMO報告で「DMS replication latency」「Aurora failover」等の技術用語を連発
- 顧客が質問しても「つまりDBの同期が…」とさらに技術で説明
- 経営層から「で、スケジュールは間に合うの？」と聞かれて初めて気づく
- 報告後に「何を言っているかわからなかった」と陰で言われる

### 根本原因
技術者は正確性を重視するため、省略や比喩を使うことに抵抗がある。
しかしステークホルダーが知りたいのは「技術の詳細」ではなく「ビジネスへの影響」。

### ケーススタディ

> **状況**: ストコンのデータ移行テストで、DynamoDBの書き込みキャパシティ超過が発生。PMO週次報告で状況を説明。
>
> **失敗パターン**: 「DynamoDBのWCUが設定値を超過し、スロットリングが発生しました。On-Demand Capacityへの切替を検討しています」と報告。
>
> **その後**: PMO長は技術者ではないため「で、どうなるの？」。笹尾PMがさらに技術説明を重ねて会議時間超過。結局「移行テストが予定通り完了するかどうか」だけが知りたかった。

### 処方箋

**3層翻訳フレームワーク**: 同じ事象を3つのレイヤーで説明できるように準備する。

| レイヤー | 対象者 | 説明の仕方 | 例 |
|---------|--------|-----------|-----|
| **L1: ビジネス** | PMO、経営層 | 「スケジュール・コスト・品質への影響」 | 「移行テストが2日遅延する見込み。カットオーバーへの影響はなし」 |
| **L2: マネジメント** | 他チームPM、SV | 「何が起きて、何をすれば解決するか」 | 「データ量が想定を超えたため、DB設定を変更して再テスト。2日で完了予定」 |
| **L3: 技術** | 開発チーム | 「正確な技術詳細」 | 「DynamoDBのWCUスロットリング。On-Demand切替+バッチサイズ縮小で対応」 |

**PMO報告の鉄則**: L1から話し始め、質問されたらL2に降りる。L3は聞かれない限り出さない。

### セルフチェック
- [ ] PMO報告を「技術用語ゼロ」で説明できるか？
- [ ] 報告資料の1ページ目に「ビジネスインパクト」が書いてあるか？
- [ ] 相手の職種・役割に合わせて説明レベルを変えているか？

## パターン5: 「完璧主義の罠」— 80点で出せない

### 対応領域: 計画・進捗管理 / チーム運営

### 症状
- 設計書のレビューで細部の指摘が多すぎて進まない
- 「もう少し精度を上げてから出したい」と報告が遅れる
- メンバーの成果物に対して「自分ならこう書く」と全面書き換えする
- 品質は高いが、スケジュールが常にギリギリ

### 根本原因
技術者は「正確で美しいコード」を追求してきた。その品質基準をPM業務やチーム成果物にも適用してしまう。
PMが追求すべきは「プロジェクト全体の最適化」であり、個々の成果物の完璧さではない。

### ケーススタディ

> **状況**: ストコンの基本設計フェーズ。ベンダーから提出された設計書をレビュー。
>
> **失敗パターン**: 設計書の図の体裁、命名規則の揺れ、日本語の表現まで指摘して50件のコメント。ベンダーPMから「重要な指摘はどれですか？」と聞かれる。修正に1週間かかり、設計フェーズが遅延。
>
> **その後**: ベンダーが「レビューが厳しすぎて出しづらい」と委縮。ドラフト段階での早期共有が減り、手戻りリスクが逆に増加。

### 処方箋

**レビュー3分類ルール**:

| 分類 | 基準 | 対応 | 例 |
|------|------|------|-----|
| **Must** | これがないと動かない/誤りがある | 即修正を依頼 | IF仕様の不整合、セキュリティ要件の欠落 |
| **Should** | 品質向上に寄与するが、なくても動く | 次回修正でOK | エラーハンドリングの網羅性、パフォーマンス考慮 |
| **Nice** | 好みの問題/体裁 | 指摘しない | 図の配置、文章表現、命名規則の揺れ |

**80点リリースの習慣**: 「Mustが0件になったらリリースOK」と決める。ShouldとNiceは後続イテレーションで改善。

### セルフチェック
- [ ] レビューコメントの7割以上が「Must」になっているか？（Niceが多いなら完璧主義のサイン）
- [ ] 「もう少し」と思って出し渋っている報告書・資料はないか？
- [ ] メンバーの成果物を書き換えた回数を数えているか？

## パターン6: 「Bad News隠し」— 悪い報告を先送りする

### 対応領域: リスク・課題管理 / ステークホルダー調整

### 症状
- 「来週までに自分で解決すれば報告不要」と考える
- 解決できず、2週間後にPMOから「なぜ早く言わなかった？」と詰められる
- 小さな遅延の報告を避けた結果、積み重なって大遅延になる
- 「問題ありません」が口癖になっている

### 根本原因
技術者は「問題を報告する＝解決できない自分の無能さを露呈する」と感じがち。
PMの世界では「早期に報告する＝リスクコントロールできている証拠」として評価される。

### ケーススタディ

> **状況**: ストコンのAPI開発で、外部ベンダーが提供するPOS連携IFの仕様書に不明点が3件。問い合わせ中だが回答が遅い。
>
> **失敗パターン**: 「自分で仕様を推測して進める。回答が来たら修正すればいい」と判断。PMO週次報告では「順調」と報告。2週間後、推測が外れて手戻り発生。
>
> **その後**: 「2週間前にリスクとして上げてくれていれば、PMO経由でベンダーに圧力をかけられた」とPMO長。信頼低下。

### 処方箋

**Bad News First の型**:

```
1. 結論（何が起きているか）
   「POS連携IFの仕様確認が3件未回答で、開発着手がブロックされています」

2. 影響（放置するとどうなるか）
   「来週までに回答がないと、結合テスト開始が1週間遅延します」

3. 対策案（自分はどう考えているか）
   「案A: PMO経由でベンダーにエスカレーション
    案B: 推測で進めて後日修正（手戻りリスクあり）」

4. 依頼（相手に何をしてほしいか）
   「PMO長からベンダーPMに催促していただけますか？」
```

**報告タイミングの基準**:

| 状態 | アクション |
|------|----------|
| 課題発生から3営業日以内に解決見込みあり | 自チームで対応、週次で報告 |
| 3営業日で解決見込みなし | 即PMOに報告 |
| スケジュール影響がある | 即PMOに報告（規模を問わず） |
| 他チームへの影響がある | 即関係者全員に報告 |

### セルフチェック
- [ ] 今週の報告で「問題ありません」と言った回数は？（0回が理想ではない）
- [ ] 「来週までに解決すれば大丈夫」と思っている課題はないか？
- [ ] 直近1ヶ月で「もっと早く言ってほしかった」と言われたことはないか？

## パターン7: 「会議ファシリ不在」— 議論に参加してしまう

### 対応領域: チーム運営 / ステークホルダー調整

### 症状
- 技術会議で議論が白熱すると、PMなのに意見を言い始める
- 気づいたら自分が一番発言している
- 議論の結論が出ず、時間切れで次回持ち越し
- 参加者から「結局何が決まったの？」と言われる

### 根本原因
技術者として「正解を知っている」ため、議論に参加したくなる。
しかしPMの役割は「議論を整理し、結論に導く」こと。プレイヤーとファシリテーターの二足のわらじは履けない。

### ケーススタディ

> **状況**: ストコンのアーキテクチャレビュー会議。ECSかLambdaかの選定議論。
>
> **失敗パターン**: 笹尾PMが「Lambdaのコールドスタートを考えるとECSが良いと思います」と発言。ベンダーAが反論、ベンダーBも参戦。笹尾PMも議論に参加。1時間経過して結論なし。
>
> **その後**: 会議のゴール・判断基準・タイムボックスが設定されていなかったため、全員が言いたいことを言って終了。

### 処方箋

**会議の型（PMが必ずやること）**:

| タイミング | PMの行動 |
|-----------|---------|
| **会議前** | ゴール（「ECS or Lambdaの選定を決定する」）と判断基準（コスト/レイテンシ/運用性）をアジェンダに明記 |
| **冒頭3分** | ゴールと判断基準を宣言。タイムボックスを設定（「30分で結論を出します」） |
| **議論中** | ホワイトボード役に徹する。意見を整理し、判断基準に沿って論点を可視化する |
| **意見を求められたら** | 「PMとしての意見は最後に言います。まず皆さんの見解を聞かせてください」 |
| **残り5分** | 「判断基準に照らすと、Xという結論でよいですか？」と合意確認 |
| **会議後** | 決定事項・担当・期限を5分以内にチャットで共有 |

### セルフチェック
- [ ] 直近の会議で自分の発言時間は全体の何%か？（20%以下が目安）
- [ ] 会議のゴールと判断基準を事前に明文化しているか？
- [ ] 会議後に決定事項を即共有しているか？

## パターン横断: ストコン案件タイムラインでの発生しやすい時期

| 時期 | 発生しやすいパターン | 理由 |
|------|-------------------|------|
| **参画直後（6月）** | パターン4（技術語翻訳ミス） | 自己紹介代わりに技術力をアピールしがち |
| **要件定義（7-8月）** | パターン3（スコープ・イエスマン） | 顧客との関係構築期で断りづらい |
| **基本設計（9-10月）** | パターン5（完璧主義の罠）、パターン7（会議ファシリ不在） | 設計レビューが本格化 |
| **開発・テスト（11-2月）** | パターン2（一人で解決マン）、パターン6（Bad News隠し） | 技術課題が多発し手を出したくなる |
| **移行・カットオーバー（3-5月）** | パターン1（正論パンチ）、パターン6（Bad News隠し） | プレッシャーの中で余裕がなくなる |

| 7 | 会議ファシリ不在 | プレイヤーとファシリは兼務できない | チーム運営 | 会議の型（ゴール→判断基準→タイムボックス） |
