## 1. コンビニ物流の全体構造と特徴

### 1-1. コンビニ物流の定義と他業態との違い

コンビニの物流は「**多頻度小口配送（たひんどしょうくちはいそう）**」と呼ばれ、一般的な食品小売（スーパー等）と大きく異なる。

| 比較項目 | コンビニ | スーパーマーケット | 一般EC |
|---------|---------|----------------|------|
| 1日の配送回数 | 1〜3便（温度帯による） | 1〜2便 | 1便（翌日以降） |
| 発注リードタイム | 同日〜翌日 | 1〜3日 | 1〜7日 |
| 1回の配送量 | 小口（数ケース〜数十ケース） | 中〜大口（パレット単位） | 個口（数点） |
| 発注単位 | 最小単位（バラ/ケース） | ケース〜パレット | 個 |
| 返品の仕組み | 原則なし（廃棄処理） | ケース単位で返品可 | 商品による |
| 物流の主体 | 本部管理の共配センター | 各チェーンによって異なる | 物流会社（3PL等） |

**多頻度小口配送が必須な理由**:

1. **小売面積の制約**: 標準的なコンビニ店舗の売場面積は30〜50坪。バックルームも狭く、大量在庫を抱えられない
2. **商品回転速度**: 弁当・おにぎりは日3便で回転する。在庫=廃棄リスクのため最低限しか持てない
3. **鮮度と品質管理**: チルド品は温度管理の観点から長時間保管が禁止
4. **チャンスロス防止**: 品切れはそのまま販売機会の喪失。頻繁な補充で陳列を維持する

### 1-2. コンビニ物流全体の構造図

```
[上流: 生産・製造]
  メーカー・ベンダー（食品・飲料・日用品・雑誌等）
         |
         | 納品（メーカー→共配センター）
         |
[中間: 物流拠点]
  共配センター（Distribution Center）
  ├── チルドDC（5〜8℃）       弁当・惣菜・牛乳・デザート
  ├── 米飯DC（20℃前後）       おにぎり・サンドウィッチ
  ├── 常温DC                  加工食品・スナック・飲料・日用品
  ├── フローズンDC（−25℃）    アイスクリーム・冷凍食品
  └── 雑誌・書籍DC            雑誌・書籍・タバコ
         |
         | 配送（DC→店舗）温度帯管理トラック
         |
[下流: 店舗]
  コンビニ店舗（全国約55,000店）
  └── ストアコンピュータ（ストコン）が発注・検品・在庫管理を担当

[情報の流れ（逆方向）]
  店舗（POS売上 / ストコン発注） → 本部発注管理システム
  本部AI需要予測 ← 販売実績・在庫データ
  本部 → 共配センターへ発注指示 → ベンダーへ発注
```

### 1-3. コンビニ物流の3つの特性

#### 特性1: 在庫レス設計（クロスドッキング）

共配センターは原則として**在庫を持たない**設計。メーカーから纏めて受け入れた商品を仕分けし、即日店舗へ出荷する「クロスドッキング型」の物流拠点である。

```
[クロスドッキングの流れ]
メーカーA →┐
メーカーB →┤ センターに入荷 → 仕分け → 配送車に積み込み → 各店舗へ
メーカーC →┘
（センターでの滞留時間: 数時間〜1日以内）
```

在庫を持たない設計には以下の利点と制約がある。

| 利点 | 制約 |
|------|------|
| センター保管コストゼロ | 発注精度が低いと即品切れになる |
| 廃棄リスクをセンターが持たない | 欠品補充に時間がかかる |
| 鮮度を最大化できる | 緊急追加発注に対応しにくい |
| 回転率を高いまま維持できる | センター稼働が止まると全店に影響する |

#### 特性2: 共同配送による効率化

複数メーカーの商品を一つの配送車にまとめて積載し、1便で複数店舗をまわる「**共配（共同配送）**」がコンビニ物流の根幹。

- 1970〜1980年代の初期コンビニでは、各メーカーが個別に店舗へ配達していた（直送方式）
- 1店舗への1日の納品トラック台数が70台以上になる問題が発生
- これを解決するため1980〜1990年代にかけて共配体制が整備された
- 現在は1店舗への1日配送車台数が**5〜8台程度**（温度帯別）まで削減

#### 特性3: 本部主導の物流管理

コンビニの物流は加盟店（店舗オーナー）ではなく、**本部が全体を設計・管理**する。

- 加盟店は「どこから何を買うか」を自由に選べない（指定ベンダー制度）
- 本部指定の共配センターのみが店舗へ配送できる
- 配送時刻・便数・配送コストはすべて本部が決定
- 加盟店は配送費用として「物流費（配送料）」を本部へ支払う

この構造により、本部はサプライチェーン全体の可視性を持ち、発注・在庫・廃棄を一元管理できる。一方で加盟店は仕入れの自由を持たず、本部依存のビジネス構造に組み込まれている。

## 2. 配送センター（DC）の種類と機能

### 2-1. 温度帯別DCの分類

コンビニの共配センターは**温度帯**によって物理的に分離されている。これは食品衛生法・HACCP基準への対応と、商品品質維持の両面から必須の設計である。

| DC種別 | 温度設定 | 取扱商品 | 配送頻度 | 代表的な商品 |
|--------|---------|---------|---------|-----------|
| **チルドDC** | 5〜8℃ | デイリー品全般 | 日3便（朝・昼・夕） | 弁当・おにぎり・惣菜・牛乳・ヨーグルト・デザート |
| **米飯DC** | 15〜20℃（常温でも管理） | 米飯類 | 日2〜3便 | おにぎり（単品）・サンドウィッチ・パン類 |
| **常温DC** | 常温（15〜25℃管理） | 加工食品・日用品 | 週2〜3便 | カップ麺・缶詰・菓子・飲料・日用品・雑貨 |
| **フローズンDC** | −25℃前後 | 冷凍食品 | 週1〜2便 | アイスクリーム・冷凍食品・冷凍お菓子 |
| **雑誌・書籍DC** | 常温 | 印刷物・タバコ | 週1〜2便（雑誌は発売日に合わせ） | 雑誌・コミック・タバコ・文房具 |

**チルドDCが日3便必要な理由**:

```
朝便（05:00〜09:00頃到着）: 朝食帯のおにぎり・サンド・ヨーグルトを補充
昼便（10:00〜12:00頃到着）: 昼食帯の弁当・惣菜を補充
夕便（15:00〜18:00頃到着）: 夕食・夜食帯の弁当・惣菜を補充

理由:
- 消費期限が1〜2日のため、前日製造→当日販売→期限切れで廃棄のサイクルが高速
- 1日3回補充しないと午後以降に棚が空く（チャンスロス）
- 過剰に仕入れると廃棄ロスが発生する
→ 3便でギリギリの在庫水準を維持するのが最適解
```

### 2-2. DCの立地・規模・担当エリア

| 項目 | 標準的な数値 | 補足 |
|------|-----------|------|
| 1DCの担当店舗数 | 50〜300店舗 | 都市部は密集するため担当店舗数が多い |
| DCの数（全国） | チェーンによって50〜150カ所程度 | チルドのみ・常温のみ等の専用DCも存在 |
| DCの延床面積 | 1,000〜5,000坪程度 | チルドは冷凍設備が必要で建設コスト高 |
| センターあたりの配送車数 | 10〜50台 | 温度帯維持のため専用車両が必要 |
| メーカーからの入荷時間 | 配送便の2〜4時間前 | 仕分け・ピッキングに要する時間 |

**DC立地の原則**:
- 担当エリアの中心に近い場所（配送距離の均等化）
- 幹線道路へのアクセスが良い場所（早朝・深夜の大型トラック通行が可能）
- 冷凍・冷蔵設備の電力供給が安定している場所
- 物流会社が運営しやすい敷地面積の確保

### 2-3. DCにおける主要業務プロセス

```
[入荷フェーズ]
メーカー・ベンダーが指定時間に納品
  ↓
商品受付・検品（数量・品温・賞味期限確認）
  ↓
JAN/ITFコードをスキャンしてDCの在庫管理システム（WMS）に登録

[仕分けフェーズ]
店舗別・便別に商品を仕分け（ソーター設備を使用）
  ↓
店舗ごとの出荷単位（通函=かご車/折り畳みコンテナ）に梱包
  ↓
配送車に積み込み（積み順は配送コースに合わせて最適化）

[出荷フェーズ]
配送車が出発（便時刻に合わせて逆算管理）
  ↓
各店舗に納品（ストコンの検品処理と連携）

[WMSが管理するデータ]
- 入荷予定・実績（本部発注システムと連携）
- 店舗別出荷明細（ストコンの検品との照合に使用）
- 配送ルート・配送時刻の実績管理
- 温度管理ログ（入荷〜出荷の品温履歴）
```

### 2-4. 通函（かご車・折り畳みコンテナ）の仕組み

コンビニでは商品を「**通函（つうかん）**」と呼ばれる専用容器に入れて配送する。

| 容器種別 | 用途 | 特徴 |
|---------|------|------|
| **かご車（台車型）** | チルド品・常温品の大口 | 折り畳み可能。店舗で回収してDCへ返却 |
| **折り畳みコンテナ（折コン）** | 常温品・小口品 | 積み重ね可能。チェーン専用色の場合多い |
| **チルドコンテナ** | チルド品の少量 | 保冷機能付き。フタ付きで温度管理 |

通函は**店舗→DC→メーカー→DC→店舗**という循環利用が基本。店舗での通函管理もストコンの業務の一部（入荷検品時に通函数を記録する場合がある）。

## 3. 発注データの流れ: ストコンからベンダーまで

### 3-1. 発注データの流れ全体図

```
[店舗（ストコン）]
  店長・スタッフが発注操作（または自動発注確認）
       |
       | EOS通信（発注データ送信）
       ↓
[本部: 発注管理システム（EDI基盤）]
  全店舗からの発注データを集約・処理
  ├── 発注データの検証（MOQ・マスタ整合性チェック）
  ├── 本部一括発注分との統合
  └── 共配センター別に仕分けて発注指示を生成
       |
       | 発注指示（共配センターへ）
       ↓
[共配センター（物流管理システム / WMS）]
  入荷予定データを受信
  ├── メーカー・ベンダーへ発注指示を転送
  └── 出荷計画を生成（店舗別・便別の仕分け計画）
       |
       | 発注指示（ベンダーへ）
       ↓
[メーカー・ベンダー（生産・製造）]
  製造・ピッキング・梱包・配送準備
       |
       | 納品（センターへ）
       ↓
[共配センター（再登場）]
  入荷・検品 → 仕分け → 積み込み
       |
       | 配送（店舗へ）
       ↓
[店舗（ストコン: 入荷検品処理）]
  納品された商品をストコンで検品登録
  ├── 出荷明細との照合（センターが送信した出荷データと突合）
  ├── 理論在庫への加算
  └── 差異があれば本部へレポート
```

### 3-2. 発注締め時刻（はっちゅうしめ）の仕組み

各便には「発注締め時刻」があり、これを過ぎると次便扱いになる。コンビニの発注締め管理は**1日に複数回**発生する。

| 配送便 | 到着目標時刻 | 発注締め時刻（目安） | 締め〜到着の間隔 |
|-------|-----------|----------------|--------------|
| 朝1便（チルド） | 05:00〜07:00 | 前日 22:00〜24:00 | 約5〜7時間 |
| 昼便（チルド） | 10:00〜12:00 | 当日 04:00〜06:00 | 約5〜6時間 |
| 夕便（チルド） | 15:00〜17:00 | 当日 10:00〜12:00 | 約4〜5時間 |
| 常温便 | 午前中〜午後 | 前日または数日前 | 約12〜24時間以上 |

**発注締め管理においてストコンが担う機能**:
1. 各便の締め時刻をストコン内部に保持（マスタデータ）
2. 締め時刻が近づくとアラート表示（未確認発注がある場合）
3. 締め時刻を過ぎた場合は次便への転送または受付拒否の処理
4. 締め後に発生した緊急需要は手動発注で翌便に対応

### 3-3. 発注データの内容（EDIメッセージ構造）

ストコンから本部へ送信される発注データは、**EDI（Electronic Data Interchange）**の標準フォーマットに準拠している。

```
発注EDIデータの主要フィールド:
  ┌─────────────────────────────────────────┐
  │ ヘッダー部                              │
  │   送信元店舗コード  : 8〜10桁の店舗識別番号 │
  │   送信日時         : YYYYMMDDHHmmss      │
  │   発注便識別子     : 便コード（朝1/昼/夕等）│
  │   総行数           : 発注明細件数         │
  ├─────────────────────────────────────────┤
  │ 明細部（商品ごとに繰り返し）             │
  │   JANコード        : 13桁（商品識別）      │
  │   発注数量         : 整数（バラ数またはケース数）│
  │   発注単位区分     : B=バラ / C=ケース      │
  │   発注方式区分     : A=自動 / M=手動 / H=本部│
  │   推奨発注数（参考）: AI算出値            │
  │   修正理由コード   : 修正した場合のコード  │
  └─────────────────────────────────────────┘
```

このデータが本部に届くと、本部EDIシステムが以下の検証を行う。

| 検証項目 | 内容 | エラー時の対応 |
|---------|------|-------------|
| 商品マスタ照合 | JANコードが商品マスタに存在するか | 発注却下・店舗へアラート |
| MOQ検証 | 発注数量が最小発注単位以上か | 自動補正またはエラー返却 |
| 取扱店舗検証 | その商品をその店舗が取り扱っているか | 発注却下 |
| 二重発注チェック | 同便に同商品を重複発注していないか | マージまたはエラー |
| 締め時刻検証 | 該当便の発注受付時間内か | 次便に振り替え |

## 4. EOS の仕組みと技術的詳細

### 4-1. EOS の定義と歴史

**EOS（Electronic Ordering System）**は、店舗と本部（または配送センター）間で発注データを電子的に交換する仕組みの総称。コンビニの発注業務をペーパーレス化・自動化した中核技術である。

| 時代 | EOS の方式 | 特徴 |
|-----|----------|------|
| 1970〜1980年代 | 電話・FAX発注 | 手書き・口頭による発注 |
| 1980〜1990年代 | 電話回線EOS | 店舗から電話回線でホストコンピュータへ接続 |
| 1990〜2000年代 | ISDN/広帯域EOS | 高速化・大容量化。データの複雑化に対応 |
| 2000〜2010年代 | IP-VPN EOS | インターネット技術ベース。コスト削減 |
| 2010年代〜現在 | クラウドEOS | 本部システムのクラウド移行。リアルタイム連携 |

### 4-2. ストコンにおけるEOS通信の仕組み

```
[ストコン内部の処理フロー]

1. 発注確定（店員の操作または自動確定）
       ↓
2. 発注データの組み立て（EDIフォーマット変換）
       ↓
3. 通信制御モジュールが送信キューに格納
       ↓
4. 定刻または即時に本部サーバーへ送信
   （KDDI IP-VPN回線経由 / ローソンの場合）
       ↓
5. 本部から受信確認（ACK）を待機
       ↓
6. ACK受信 → 送信完了をストコン画面に表示・ログ記録
   タイムアウト → リトライ処理（3回程度が一般的）
       ↓
7. 送信済み発注データを「発注残」として理論在庫に計上
```

**通信の信頼性設計**:
- EOS通信はコンビニにとって**ミッションクリティカル**な通信。通信障害は即品切れにつながる
- 主回線（光ファイバーまたはIP-VPN）と**モバイル回線バックアップ（LTEまたは5G）**を冗長構成
- 本部サーバーも通常は二重化・DR構成
- ストコンはオフライン時でも発注データを蓄積し、回線復旧時に自動再送する「バッファリング」機能を持つ

### 4-3. EOB（Electronic Order Book）との違い

EOS と混同されることが多い **EOB（Electronic Order Book）** との区別:

| 用語 | 意味 | 使用場面 |
|------|------|---------|
| **EOS** | Electronic Ordering System。店舗から本部への発注システム全体 | 発注通信全般を指す際に使う |
| **EOB** | Electronic Order Book。店舗の発注端末・ハンディターミナル | 発注入力デバイスを指す際に使う |

現在はストコン（PC端末）に EOS 機能が統合されており、ハンディターミナル（EOB）は補助的な入力デバイスとして位置付けられる。入荷検品や棚卸では EOB（ハンディ）が活用される。

### 4-4. EDI 標準とコンビニの発注通信

コンビニの発注通信は**流通業界標準の EDI（JEDICOS / JCA手順 / ①②手順）**をベースに、各チェーンが独自に拡張した仕様を採用している。

```
主要な EDI 標準:
  JCA手順（Japan Chain store Association）
    └─ 流通業界が定めた通信手順の標準。1980年代に策定
    └─ 今でも一部レガシーシステムで使用

  JEDICOS（Japan EDI for Commerce and distribution）
    └─ JCA手順を発展させた現代的EDI仕様
    └─ 商品マスタ・受発注・請求等の電子帳票を定義

  コンビニ独自拡張
    └─ 多頻度発注・温度帯管理・便管理に対応するため独自項目を追加
    └─ チェーンごとの仕様差異が大きく、ベンダーはチェーン別対応が必要
```

## 5. 商品コード体系: JAN コードと ITF コード

### 5-1. JAN コードの概要

**JAN コード（Japan Article Number）**は、日本の商品に付与される13桁のバーコードシンボル。国際的にはEAN-13（欧州）やGTIN-13として統一されている。

```
JANコード（13桁）の構成例:
  4 9012345 678901 2
  │ │       │      └─ チェックデジット（1桁）
  │ │       └────────── アイテムコード（5桁、メーカーが付番）
  │ └─────────────────── メーカーコード（7桁、GS1が付与）
  └───────────────────── 国コード（日本=45または49）
```

**コンビニにおけるJANコードの役割**:

| 使用場面 | 内容 |
|---------|------|
| 発注 | ストコンでの発注操作時に商品特定に使用（バーコードスキャンまたは商品マスタ検索） |
| 入荷検品 | 納品商品のJANをハンディ（EOB）でスキャンして入荷データと照合 |
| 廃棄登録 | 廃棄商品のJANをスキャンして廃棄数を記録 |
| 棚卸 | 棚卸時に在庫商品のJANをスキャンして実在庫数を記録 |
| POSレジ | レジ読み取りで商品特定・価格決定・販売実績記録 |
| マスタ管理 | 商品マスタ（名称・価格・分類・仕入先等）のキーとして使用 |

### 5-2. ITF コードの概要

**ITF コード（Interleaved Two of Five）**は、ケース（ダース・カートン）単位の商品に付与されるバーコード。JANコードが小売単位（1個）であるのに対し、ITFコードは**物流単位（ケース）**を識別する。

```
ITFコード（14桁）の構成例:
  0 4 9012345 678901 2
  │ │ │              └─ チェックデジット
  │ │ └─────────────── JANコード（13桁）
  │ └───────────────── インジケーター（入数・梱包段階を示す）
  └─────────────────── ゼロパディング（フィルキャラクタ）
```

| 比較項目 | JAN コード | ITF コード |
|---------|-----------|----------|
| 桁数 | 13桁 | 14桁 |
| 単位 | 個（小売単位） | ケース（物流単位） |
| 表示場所 | 商品パッケージ（消費者向け） | ケース外箱（物流向け） |
| 主な読み取り場所 | POSレジ・ストコン（発注・棚卸） | 物流センター・検品（入荷時） |
| 関係 | JANコードにプレフィックスを付けてITF化 | JANコードから派生 |

**コンビニにおけるITFコードの使用場面**:

入荷検品時にケース単位でITFコードをスキャンすることで、ケース入数に基づいて自動的にバラ数（JAN単位）に換算し、理論在庫を更新できる。

```
例: おにぎり（1ケース=10個入）
  ITFスキャン: 1回のスキャンで10個入荷として理論在庫に加算
  JANスキャン: 1個ずつ10回スキャンが必要（精度は高いが手間がかかる）
```

多くのコンビニでは入荷検品時にITFスキャンを優先し、差異があった場合のみJANスキャンで個数確認を行う。

### 5-3. GS1 コードへの移行動向

近年、流通業界全体で **GS1（Global Standards One）**が定める統合的なコード体系への移行が進んでいる。

| コード | 正式名称 | 位置付け |
|--------|---------|---------|
| GTIN-13 | Global Trade Item Number (13桁) | JANコードの国際標準名称 |
| GTIN-14 | Global Trade Item Number (14桁) | ITFコードの国際標準名称 |
| GS1-128 | GS1 Application Identifier | 賞味期限・ロット番号等の追加情報を含むバーコード |
| GS1 QR コード | 2次元コード版GS1標準 | 食品ロス対策・トレーサビリティ強化で普及中 |

**コンビニでの GS1 QR コード活用**:
- 2025年頃から弁当・惣菜に賞味期限・製造ロットを含む GS1 QR コードを印字する動きが加速
- ストコン・ハンディ端末も GS1 QR 対応への更新が必要になりつつある
- 食品トレーサビリティ法（食品衛生法改正）の要件強化に対応するための整備

## 6. 共同配送と温度帯別配送の仕組み

### 6-1. 共同配送の成立条件と仕組み

**共同配送（共配）**とは、複数のメーカー・ベンダーの商品を一括して共配センターに集め、そこから店舗へ配送する仕組みである。これが成立するためには以下の条件が必要。

```
共同配送が成立するための3条件:
1. 標準化された通函（容器）
   → メーカー・チェーン問わず同一規格の折コン・かご車を使用
   
2. 本部による情報一元管理
   → 店舗発注データを本部が集約し、センターへ一括発注指示

3. 物流子会社または委託物流会社の存在
   → センターを管理・運営する専門会社が介在することで中立性を確保
```

### 6-2. 温度帯別配送車の種類と管理

コンビニ配送では、商品の温度帯に合わせた**専用車両**が使用される。

| 車両種別 | 荷室温度 | 積載商品 | 1台あたりの担当店舗数（目安） |
|---------|---------|---------|--------------------------|
| チルド車（冷蔵） | 5〜8℃ | 弁当・惣菜・乳製品 | 10〜20店舗 |
| 常温車 | 15〜25℃（外気温管理） | 加工食品・日用品 | 20〜40店舗 |
| フローズン車（冷凍） | −25℃以下 | アイス・冷凍食品 | 30〜60店舗（1日1〜2便で多め） |
| 複合温度帯車 | 複数の温度ゾーンを区切り | チルド+常温を1台で | 特定エリア向け |

**温度管理の法的要件**:
- 食品衛生法に基づく温度基準は「食品の特性に応じた適切な温度で保存」
- 冷蔵食品: 10℃以下（実務上は5〜8℃で厳格管理）
- 冷凍食品: −15℃以下（実務上は−25℃以下で管理）
- 車両・センターには温度ロガー（データロガー）が設置され、常時記録

### 6-3. 配送ルート最適化

1センターが担当する50〜300店舗を、効率よく巡回する配送ルートの最適化は、物流コスト削減の最重要テーマ。

**配送ルート最適化の変数**:
1. 時間窓制約: 各店舗の納品可能時間帯（例: 朝1便は05:00〜07:00到着必須）
2. 積載重量・容積: 1台の積載可能量から積み込み順序を逆算
3. 道路渋滞: 都市部では時間帯別の渋滞予測が必須
4. 優先順位: 日販が高い大型店・アイテム数が多い店舗を優先
5. 環境規制: ディーゼル車の規制地域（東京等）への対応

**AIによる動的ルート最適化（2020年代〜）**:
- 配送当日の交通情報・気象データを取り込んでルートをリアルタイム変更
- 複数便の積み込み順序・配送順序を同時最適化（組み合わせ最適化問題）
- ローソンでは三菱商事系物流会社（後述）がシステム投資を進めている

## 8. サプライチェーンのデジタル化動向（2024〜2026年）

### 8-1. AI活用による需要予測・発注最適化の高度化

2024〜2026年のコンビニ物流における最大のトレンドは、**AI発注の精度向上と適用範囲の拡大**である。

| フェーズ | 時期 | 内容 |
|---------|------|------|
| 第1世代AI発注 | 2015〜2020年 | 販売実績・気象データを使った基本的な需要予測。大手3チェーンが段階導入 |
| 第2世代AI発注 | 2020〜2024年 | 人流データ・イベントカレンダー・エリア特性を統合した高精度予測。全店展開完了 |
| 第3世代AI発注（進行中） | 2024年〜 | 生成AI活用による説明可能な発注推奨・店長との対話型調整・商品ライフサイクル考慮 |

**第3世代の特徴（ローソンの事例）**:
- AI.CO（発注AI）が廃棄率・欠品率・売上のバランスを考慮した多目的最適化を実施
- 店長が「なぜこの発注推奨なのか」を理解できる説明機能（Explainable AI）
- 特売・値引きを考慮した動的な発注量調整

### 8-2. RFID（Radio Frequency Identification）の本格活用

**RFID**は商品個別にICタグを貼付し、無線通信で一括読み取りを行う技術。コンビニ物流では以下の場面での活用が拡大中。

```
[コンビニでのRFID適用フロー]

製造工場: 商品製造時にRFIDタグを貼付
  ↓
共配センター: ゲート型リーダーでケース単位の入出荷を一括記録
  ↓
配送車両: 積み込み・荷降ろし時の自動照合（紛失防止・誤積防止）
  ↓
店舗: 検品不要化（RFIDゲートを通過するだけで自動検品）
       棚卸の自動化（棚ごとにRFIDリーダーを設置）
       欠品の自動検知（棚前のRFIDリーダーが在庫ゼロを検知）
```

**コンビニ各チェーンのRFID動向**:

| チェーン | RFID動向 |
|---------|---------|
| セブン-イレブン | 2024年から一部商品（酒類・加工食品）にRFID試験導入 |
| ファミリーマート | Azure IoT × RFIDによる在庫管理実証実験（2023〜） |
| ローソン | Azure IoT連携のRFID在庫管理実証。2025〜2026年に本格展開予定 |

**ローソンのRFIDとストコンの連携**:
```
RFIDタグ付き商品が入荷
  ↓
店舗のRFIDゲートが一括読み取り
  ↓
ストコンに入荷データが自動転送（検品操作が不要化）
  ↓
理論在庫が自動更新
  ↓
Azure IoT → クラウドにリアルタイム在庫データが蓄積
  ↓
発注AI（AI.CO）が在庫データを参照して発注推奨を精緻化
```

RFIDが普及するとストコンの「検品業務」が大幅に削減・自動化される。これはストコンの機能設計・UI設計にも影響する大きなトレンド。

### 8-3. デジタルトレーサビリティの強化

食品安全・食品ロス対策・消費者の安心安全ニーズから、**商品のトレーサビリティ（追跡可能性）**の強化が法規制・業界自主規制の両面で進行中。

**トレーサビリティの仕組み**:
```
製造工場（製造日・ロット番号・原材料情報）
  ↓  ← GS1コード（バーコード/QRコード）にロット情報を付帯
共配センター（入荷記録・仕分け記録）
  ↓
配送車両（出荷記録・温度管理ログ）
  ↓
店舗（入荷検品記録・在庫記録・廃棄記録）
  ↓
消費者（購入記録 → アプリで原材料・産地確認可能に）
```

**ストコンへの影響**:
- GS1 QR コード対応のスキャナー搭載が求められるようになりつつある
- 入荷検品時にロット番号・製造日を記録するフィールドの追加
- 廃棄記録にロット番号を紐付けて、原因追跡を可能にする設計

### 8-4. ドローン・自動配送ロボットによる最終配送

コンビニの物流「ラストワンマイル（DCから店舗への配送）」の自動化は、将来の技術として研究・実証段階。

| 技術 | 実証状況 | 課題 |
|------|---------|------|
| ドローン配送 | 過疎地・島嶼部での実証実験（2022〜） | 積載重量・航続距離・規制の制約 |
| 自動配送ロボット（低速） | 大学・住宅地での実証（2024〜） | 悪天候・狭隘道路・コスト |
| 自動運転トラック | 高速道路でのレベル4実証（2024年〜） | 市街地対応・センサーコスト |

現時点ではコンビニの通常物流には適用されておらず、2030年代以降の実用化が期待される段階。

## 9. ストコン移行案件との関連（PM視点）

2026年6月のNECストコン移行案件PM参画にあたり、物流・サプライチェーンのドメイン知識が直接役立つポイントを整理する。

### ポイント1: EOS通信の移行はミッションクリティカル

ストコンをクラウド移行する際、**EOS通信（発注データ送信）は最もリスクの高いコンポーネント**のひとつ。

```
リスクの所在:
  EOS通信の障害 → 発注データが本部に届かない
    ↓
  共配センターへの発注指示が出ない
    ↓
  翌日以降の配送便に商品が積まれない
    ↓
  全国数百店舗が同時に品切れ状態（チャンスロスと顧客不満）

PMとしてチェックすべき項目:
  [ ] EOS通信のリトライ・バッファリング仕様の継承確認
  [ ] 移行後の通信プロトコル・EDIフォーマット互換性確認
  [ ] 本部側EOS受信システムとの連携試験計画
  [ ] 通信障害時の手動代替手段（電話・FAX発注）の用意
  [ ] 移行直後の監視強化（EOS送受信ログのリアルタイム監視）
```

### ポイント2: 発注締め時刻のSLA設計

クラウド移行後もEOS通信は**発注締め時刻という厳格なSLA**のもとで動作しなければならない。

| SLA要件 | 現行 | 移行後に維持すべき要件 |
|---------|------|-------------------|
| 発注締め前の通信完了率 | ほぼ100%（クリティカル） | 99.9%以上 |
| EOS送受信のレイテンシ | 数秒〜数十秒 | 同等以下 |
| 通信エラー時のリトライ | 自動リトライあり | 継承が必須 |
| 障害検知から復旧まで | — | 30分以内（業界慣行） |

ストコンのクラウド化によって本部サーバーへの通信経路が変わるため、**新しい通信経路でのレイテンシ・信頼性の事前検証**が移行計画の重要タスクとなる。

### ポイント3: 入荷検品データとWMS連携の設計

ストコンの入荷検品データは、共配センターの WMS（倉庫管理システム）が送信する出荷明細と**突合（照合）**される。この連携設計は移行後も機能しなければならない。

```
連携フロー（現行）:
  センターWMS → 出荷明細を本部に送信
       ↓
  本部が各店舗のストコンに出荷明細をダウンロード
       ↓
  店舗での入荷検品時: ストコンが出荷明細と実入荷数を照合
       ↓
  差異があれば本部・センターにレポート

移行後の確認ポイント:
  [ ] 出荷明細のダウンロード通信が継続するか
  [ ] 照合ロジック（差異判定の閾値設定）が移行後も同一か
  [ ] 差異レポートの送信先・フォーマットが変わらないか
```

### ポイント4: 物流データの分析基盤との連携

クラウド移行の大きな動機のひとつは、ストコンが収集するデータをリアルタイムで分析基盤（AWS Kinesis・S3・Redshift等）に流すこと。物流・在庫データは分析の中核となる。

| データ種別 | 物流・サプライチェーンでの活用 |
|----------|--------------------------|
| 発注データ（EOS） | 発注精度の分析・過発注・過小発注パターンの特定 |
| 入荷実績データ | DC〜店舗間の配送遅延・欠品率の可視化 |
| 廃棄データ | 廃棄ロスの傾向分析・AI発注精度のフィードバック |
| 検品差異データ | センター仕分けエラー・配送ミスの発生率追跡 |

PMとしては、**分析基盤との連携がアーキテクチャ設計のどの段階で組み込まれているか**を早期に確認し、スコープ漏れを防ぐことが重要。

### ポイント5: RFIDへの対応を移行スコープに含めるか否かの判断

ローソンではRFIDによる自動検品の実証が進んでいる。ストコン移行と同時にRFID対応を盛り込むかどうかは、移行プロジェクトのスコープ判断として重要。

```
判断基準:
  「ストコン移行」と「RFID対応」は別プロジェクトとして分離する方が安全

理由:
  RFID対応 → ストコンのUI・データモデル・I/F仕様が変わる
  同時実施 → 変更点が複合して品質管理・テスト範囲が爆発する

PMとしての推奨アクション:
  [ ] RFID実証プロジェクトのタイムラインを早期に確認
  [ ] RFID対応APIをストコン移行後の「拡張ポイント」として設計に組み込む
  [ ] RFID本番展開のタイミングで、ストコンの追加改修が必要になることをステークホルダーに合意
```

## 10. 用語集

| 用語 | 読み | 定義 |
|------|------|------|
| **EOS** | イーオーエス | Electronic Ordering System。店舗から本部への電子発注通信システム |
| **EOB** | イーオービー | Electronic Order Book。発注用ハンディターミナル（現在はストコンに統合） |
| **DC** | ディーシー | Distribution Center。配送センター。コンビニでは温度帯別に設置 |
| **共配センター** | きょうはいセンター | 複数メーカーの商品を一括配送する共同配送センター |
| **クロスドッキング** | — | センターに在庫を持たず、入荷した商品を即日仕分けて出荷する物流手法 |
| **多頻度小口配送** | たひんどしょうくちはいそう | コンビニ物流の特徴。1日複数回・少量ずつ配送 |
| **発注締め** | はっちゅうしめ | 各配送便の発注受付締切時刻 |
| **JAN コード** | ジャンコード | Japan Article Number。13桁の商品バーコード（個品単位） |
| **ITF コード** | アイティーエフコード | Interleaved Two of Five。14桁のケース単位バーコード |
| **GS1** | ジーエスワン | Global Standards One。流通・物流バーコード標準化団体 |
| **EDI** | イーディーアイ | Electronic Data Interchange。企業間でデータを電子交換する仕組み |
| **WMS** | ダブリューエムエス | Warehouse Management System。倉庫管理システム |
| **通函** | つうかん | 配送用の専用コンテナ・かご車。循環利用が前提 |
| **RFID** | アールエフアイディー | Radio Frequency Identification。ICタグによる無線識別技術 |
| **リードタイム** | — | 発注から入荷までの所要時間（コンビニは通常1日） |
| **単品管理** | たんぴんかんり | 商品をSKU単位で発注・管理し廃棄とチャンスロスを最適化するコンビニ独自概念 |
| **チャンスロス** | — | 品切れによる販売機会の喪失 |
| **発注残** | はっちゅうざん | 発注済みだが未入荷の在庫（理論在庫の計算に含める） |
| **配荷** | はいか | 本部が各店舗へ商品を割り当てて配送すること（本部一括発注で使用） |
| **三菱食品** | みつびししょくひん | 三菱商事系の食品卸売会社。ローソン物流の主要パートナー |
| **AI.CO** | エーアイコ | ローソンの発注AI（フューチャーアーキテクト構築）。2024年7月全店展開完了 |
| **トレーサビリティ** | — | 食品の製造〜消費までの追跡可能性 |
