## 1. 店舗システム全体像

### 1-1. コンポーネント構成図

```
═══════════════════════════ 店舗層 ═══════════════════════════
│                                                             │
```

### 1-2. 主要コンポーネントの役割

| コンポーネント | ベンダー | 主な役割 | 店舗数/規模 |
|--------------|---------|---------|-----------|
| **POS端末** | NECプラットフォームズ（共同開発） | 決済・売上収集・多言語対応 | 全約14,663店舗 |
| **セルフレジ** | NEC | 省人化・自動釣銭・24h対応 | **4,500店舗以上** |
| **ストアコンピュータ** | 不明（NEC/富士通推定） | 店舗基幹業務（発注・在庫・精算） | 全店舗 |
| **天井カメラ** | 不明（Azure連携） | 顧客行動追跡・棚前滞留分析 | 8〜12台/店 |
| **Loppi** | NECグループ系 | マルチメディア（発券・各種申込） | 全店舗 |
| **AI.CO** | フューチャーアーキテクト構築 | 発注AI（品揃え・値引き・廃棄） | 全店（2024年7月完了）|

### 1-3. クラウド基盤の多層構造

ローソンは単一クラウドではなく **4種のクラウド/基盤が並存** する複雑なマルチクラウド環境を運用している。

| クラウド | 用途 | 主なパートナー |
|---------|------|-------------|
| **AWS** | 新規開発システム全般、Webサイト、アプリバックエンド | 日立ソリューションズ、アイレット |
| **NSSOL absonne** | 既存業務システム（リフト&シフト） | NSSOL |
| **Azure** | IoT在庫管理実証（RFID）、店舗DX（カメラAI） | Microsoft |
| **KDDI WAKONX** | AIサービス基盤、データ分析・会員統合 | KDDI |

> **PM として知っておくべきポイント**
> 店舗層は「NECが主体の専用ハードウェア群」、本部層は「マルチクラウドの複合環境」という二層構造になっている。**ストコン AWS 移行はこの二層の境界線を書き換える作業**であり、単なるサーバー移転ではない。移行後は店舗層のデータが直接 AWS に流れる設計となるため、既存の absonne・Azure・WAKONX との連携設計が必要になる。

## 3. ストアコンピュータの現行アーキテクチャ

### 3-1. 2019年刷新の概要

**ストアコンピュータ（ストコン）** はコンビニ店舗のバックルーム（事務所）に設置されたPC端末型の基幹システム。

| 刷新項目 | 内容 |
|---------|------|
| **刷新期間** | 2019年1月〜6月（全約14,663店舗完了） |
| **CPU性能** | 従来比 **約2倍**（Intel Skylake→Coffee Lake世代更新相当） |
| **消費電力** | 従来比 **約6割削減** |
| **画面起動時間** | 平均 **1〜3秒短縮** |
| **OS** | Windows Embedded / Windows Pro系（推定。未公開） |
| **ベンダー** | NEC または富士通製（推定。業界標準ベンダー） |

### 3-2. ストコンが担う機能

```
[ストアコンピュータの主要機能]

  発注・在庫管理          売上確認
  ├─ 発注入力             ├─ 日次売上集計
  ├─ 入荷検品             ├─ 時間帯別分析
  ├─ 在庫照会             └─ AI.CO との連携（発注推奨受信）
  └─ 廃棄・返品処理

  タイムキーピング        精算・伝票処理
  ├─ 出退勤管理           ├─ 日次締め処理
  ├─ シフト管理           ├─ 精算確認
  └─ 時給計算支援         └─ 各種伝票出力

  管理者機能（2019年追加）
  ├─ オーナーポータル（複数店舗一覧表示）
  └─ 複数店リモートログイン（別店舗へのリモートアクセス）
```

### 3-3. 24/365要件とオフライン耐性

コンビニは **24時間365日無停止** が絶対要件。この要件はストコンのアーキテクチャ設計に大きく影響している。

| 要件 | 現行の対応 | AWS移行後の課題 |
|------|---------|--------------|
| **24/365稼働** | オンプレ自営機器で完結。ネットワーク障害でも継続稼働 | クラウド依存度が上がると通信障害時のリスクが増す |
| **オフライン耐性** | 店舗内ローカル処理が基本。本部との通信が途絶えても売上・在庫管理を継続 | **オフライン時の動作仕様の再設計が必須** |
| **瞬時起動** | ハードウェア最適化済みの専用機器 | クラウド上での起動遅延をゼロにする設計が必要 |
| **ネットワーク冗長化** | IP-VPN ＋ モバイルバックアップ回線 | クラウドへの接続断リスク対策が別途必要 |

### 3-4. セブン-イレブンとの比較

| 項目 | ローソン（現状） | セブン-イレブン（2025年完了） |
|------|--------------|---------------------------|
| ストコンの扱い | **オンプレ存続**（2019年ハード刷新のみ） | **廃止**（クラウド移行完了） |
| 店舗端末 | 専用POS + 専用ストコン | 汎用端末（iPad/Android）化 |
| データ処理場所 | ストコンでローカル集約→本部送信 | クラウドで直接処理 |
| クラウド基盤 | マルチクラウド（AWS+absonne+Azure） | Google Cloud（一本化） |
| 移行方式 | 未着手（今回の案件が相当） | フルクラウド・マイクロサービス |

> **PM として知っておくべきポイント**
> セブン-イレブンは2025年にストコン廃止・汎用端末化を完了させた。ローソンはまだオンプレ存続中であり、**業界の遅れを取り戻す案件**として位置付けられる。ただしセブンはGoogle Cloud一本化でシンプルだったのに対し、ローソンは既存のマルチクラウド環境（AWS・absonne・Azure・WAKONX）との整合が必要で、複雑度が高い。

### 4-1. 店舗↔本部のデータ連携全体図

```
═══════════ 店舗 ═══════════       ══════════ 本部・クラウド ══════════
                                   
POS端末                            AI.CO（発注AI）
  │                                  │
  ├─[売上データ（リアルタイム）]──→  │ 発注推奨数算出
  │                                  ↓
  │                                AWS（SCM/CRM系）
ストアコンピュータ ←──[発注推奨]─── │
  │                                  │
  ├─[発注データ（バッチ）]─────────→ │ → サプライヤー（EOS）
  │                                  │
  ├─[POS売上（日次締め）]──────────→ │ → 統合マスタシステム
  │                                  │
  ├─[在庫データ（定期）]───────────→ │ → 在庫分析
  │                                  │
  └─[廃棄・返品データ（日次）]──────→ │ → 食品ロス管理
                                   
天井カメラ                          Azure（店舗DX）
  └─[顧客行動映像（準リアルタイム）]→ │
                                     │
温度センサー                        本部監視システム
  └─[温度ログ（リアルタイム）]──────→ │ → 温度管理アラート
                                   
Loppi                               本部サービス基盤
  └─[各種申込・決済データ]─────────→ │
═══════════════════════════════════════════════════════
```

### 4-2. バッチ vs リアルタイムの分類

| データ種別 | 送信方式 | 頻度 | 重要度 |
|-----------|---------|------|-------|
| **POS売上（リアルタイム）** | リアルタイム/準リアルタイム | 販売都度 | 高（AI.COへのフィードバック） |
| **発注データ** | **バッチ** | 1日1〜2回（締め時刻） | 高（在庫・廃棄に直結） |
| **POS売上（日次締め）** | **バッチ** | 日次 | 高（経営管理） |
| **在庫データ** | **バッチ** | 日次〜週次 | 中 |
| **廃棄・返品データ** | **バッチ** | 日次 | 中（食品ロス管理） |
| **温度管理ログ** | リアルタイム | 常時 | 高（法令・食品安全） |
| **顧客行動映像（カメラ）** | 準リアルタイム | 常時 | 中（DX施策） |
| **Loppi各種データ** | リアルタイム | 取引都度 | 中 |

### 4-3. AI.CO（発注AI）との連携フロー

```
[AI.CO発注フロー]

販売実績（POS）──→
天候・気温データ──→
Pontaカードデータ──→  AI.CO ──→ 発注推奨数 ──→ ストコン画面表示
同立地他店データ──→  (AWS上)     （商品別）       ↓
本部施策データ──→               発注推奨数 = 販売予測数 − 在庫予測数 + 安全在庫数
（約100因子）

2024年〜: 値引き推奨機能追加（強化学習アルゴリズム）
```

> **PM として知っておくべきポイント**
> 現状、AI.COはAWS上でフューチャーアーキテクトが構築・運用している。**ストコンがAWS移行した後も、AI.COとのデータI/F（発注推奨の受信・POS売上の送信）が継続して機能することが移行成功の必須条件**。フューチャーアーキテクトとのI/F仕様合意が、移行プロジェクト序盤の重要マイルストーンとなる。

## 5. AWS 移行で変わるポイント

### 5-1. 移行対象 vs 非移行（推定）

| コンポーネント | 移行区分 | 根拠・補足 |
|-------------|---------|-----------|
| **ストアコンピュータ（基幹機能）** | **移行対象**（今回の案件） | オンプレ→AWSクラウド化が今回の主テーマ |
| **POS端末（ハードウェア）** | 非移行（当面維持） | 専用ハードウェア（NECプラット製）を店舗に設置済み。短期での汎用端末化は困難 |
| **セルフレジ** | 非移行（当面維持） | 4,500店舗以上への設置済みNEC製ハードウェア |
| **天井カメラ** | 非移行（当面維持） | ハードウェアはそのまま。データ送信先はAzure |
| **Loppi** | 非移行 | 独自端末・独自サービス基盤 |
| **ネットワーク** | 一部刷新 | KDDI参画により通信インフラの刷新が進む可能性 |

### 5-2. ストコン移行で変わる構造

```
[現状（オンプレ）]
  店舗内：ストコン（Windows端末）
            ↓ バッチ・都度通信
          KDDI IP-VPN
            ↓
          本部DC（オンプレ）または既存クラウド

[移行後（AWS）]
  店舗内：シン・ストコン（軽量クライアント or Webブラウザベース？）
    or    汎用タブレット端末（セブンの汎用端末化パターン）
            ↓ 常時接続（オフライン対応機能付き）
          KDDI IP-VPN + au回線バックアップ
            ↓
          AWS（東京/大阪リージョン）
            ├─ ストコン機能（発注・在庫・精算等）
            ├─ AI.CO（既存・フューチャー管理）
            └─ 統合マスタシステム
```

### 5-3. 移行後に新たに必要になる設計領域

| 設計領域 | 内容 | 難易度 |
|---------|------|-------|
| **オフライン動作設計** | 通信断時のローカルキャッシュ・バッファリング・再接続処理 | 高 |
| **24/365 HA設計** | Multi-AZによる冗長化、RTO/RPO定義 | 高 |
| **14,600店舗ロールアウト設計** | ウェーブ展開計画（パイロット→段階展開）、ロールバック計画 | 高 |
| **マルチクラウド連携設計** | absonne・Azure・WAKONXとのAPI設計、データ整合性 | 高 |
| **POS-ストコン間通信設計** | 既存POS端末との通信プロトコル維持または変更 | 中〜高 |
| **端末管理（MDM）設計** | 店舗端末のOSアップデート・障害対応の中央管理 | 中 |

> **PM として知っておくべきポイント**
> 移行後のストコンが「どの形態のクライアント端末になるか」はまだ公開情報がない。セブンのように汎用タブレット化するのか、専用端末は維持してバックエンドのみクラウド化するのか。**参画直後にこの「端末形態の方針」を確認することが、WBS全体の工数見積もりを決める最重要事項**の一つ。

## 6. 移行リスクの所在

### 6-1. FMEA 観点によるリスク分析

WBS 4.1.3（リスク管理基礎）で学んだFMEA（Failure Mode and Effects Analysis）の観点でストコン移行の主要リスクを評価する。

**評価基準**: S（重大度）× O（発生頻度）× D（検出可能性）= RPN（リスク優先数）
- S/O/D: 1〜10スケール。Dは検出困難なほど高スコア
- **RPN > 125 を高リスクゾーン**として優先対処する

| # | 故障モード | 影響（Effect） | 根本原因 | S | O | D | **RPN** | 優先度 |
|---|-----------|--------------|---------|---|---|---|---------|--------|
| 1 | **カットオーバー中の本番障害** | **全店舗が発注・売上処理不能に**（24h×14,600店） | 移行手順の不備、リハーサル不足 | 10 | 5 | 4 | **200** | **最高** |
| 2 | **AI.COとのI/F断絶** | 発注AI機能停止。手動発注に切り替え（過剰発注・欠品リスク） | フューチャーとのI/F仕様合意漏れ | 8 | 6 | 5 | **240** | **最高** |
| 3 | **オフライン時の動作不全** | 通信障害時に店舗業務が停止（現行はオフライン継続可能） | オフライン設計の考慮漏れ | 9 | 5 | 3 | **135** | 高 |
| 4 | **POS-ストコン間プロトコル非互換** | 売上データがストコンに届かない。発注精度低下 | 既存POSのI/F仕様変更 | 8 | 4 | 6 | **192** | 高 |
| 5 | **マルチクラウド間のデータ不整合** | 在庫データがabsonne・AWSで乖離。二重管理の混乱 | マルチクラウド連携設計の複雑性 | 7 | 6 | 7 | **294** | **最高** |
| 6 | **温度管理データの欠損** | 食品安全法違反リスク。賞味期限管理の根拠データ喪失 | リアルタイムIoT連携の移行設計漏れ | 9 | 3 | 5 | **135** | 高 |
| 7 | **ロールアウト中の店舗展開失敗** | 一部店舗で旧システムと新システムが混在。並行運用コスト増大 | ウェーブ展開計画の不備 | 6 | 7 | 4 | **168** | 高 |
| 8 | **KDDI通信インフラ切替タイミングのずれ** | ストコン移行と通信基盤刷新のタイミングがずれて二重コスト発生 | KDDIの通信インフラ刷新との非同期 | 6 | 5 | 6 | **180** | 高 |

### 6-2. 最高リスク3件の詳細分析

#### リスク2：AI.COとのI/F断絶（RPN=240）

```
[現状の依存関係]
  ストコン ←── AI.CO発注推奨（フューチャー管理）
    └─ POSデータ ──→ AI.CO

[移行後のリスク]
  新AWS上ストコン ←?── AI.CO（設計変更要）
    └─ 移行中の二重連携が必要（旧ストコン + 新ストコンが並行稼働）
```

**対策案**:
- フューチャーアーキテクトとのI/F仕様書取得を、キックオフ後30日以内の必須アクションに設定
- 移行期間中の「二重連携期間」（旧ストコンと新ストコンが並行稼働する期間）の設計を早期に合意

#### リスク5：マルチクラウド間のデータ不整合（RPN=294）

```
[ローソンのマルチクラウド複雑性]

AWS ─────────→ 新ストコン機能
    ↕ 同期?
NSSOL absonne → 既存業務システム（ストコン関連含む可能性）
    ↕ 同期?
Azure ──────→ 在庫管理・カメラDX
    ↕ 同期?
WAKONX ─────→ AI基盤・会員データ
```

セブンがGoogle Cloud一本化でシンプルだったのとは対照的に、ローソンは**4つのクラウドにまたがる整合性管理**が必要。

**対策案**:
- アーキテクチャ設計フェーズで「データ主権マップ」（どのデータのMaster of Recordはどのシステムか）を作成
- absonne・Azure・WAKONXとの連携はAPI Gateway等を通じた疎結合設計で変更容易性を確保

#### リスク1：カットオーバー中の本番障害（RPN=200）

**コンビニ特有の制約**：24時間365日稼働のため、「深夜メンテナンス時間帯」が存在しない。

**対策案**:
- 並行稼働期間を長めに設定（旧ストコンのフォールバックを維持）
- 展開ウェーブを極小化（最初のパイロット店舗を1〜5店舗に限定）
- RTO（目標復旧時間）を厳密に定義（例：障害発生から30分以内に旧ストコンへ戻せること）

### 6-3. リスク分布サマリー

```
        影響度（Severity）
    高 │ ④POS非互換  ③オフライン ①CO失敗
       │             ⑥温度管理
       │      ⑤マルチCloud  ②AI.CO断絶
    中 │ ⑦ロールアウト ⑧KDDI タイミング
       │
    低 │
       └──────────────────────────
         低      発生確率       高
```

> **PM として知っておくべきポイント**
> **最優先で対策すべきリスクは「AI.COとのI/F断絶（RPN=240）」と「マルチクラウド間データ不整合（RPN=294）」の2件**。どちらも設計フェーズの初期に対策を確定しないと、後から直すコストが急増する。参画直後の30日間で、フューチャーアーキテクトとのI/F仕様合意、および既存マルチクラウド連携のアーキテクチャレビューを必須アクションとして計画に組み込む。

## 7. 用語集

| 用語・略語 | 正式名称 / 意味 |
|----------|---------------|
| **ストコン** | ストアコンピューターの略。コンビニ店舗のバックルームに設置されたPC型基幹システム。発注・在庫・売上・精算管理等を担う。**今回の移行対象** |
| **POS** | Point of Sale（販売時点情報管理）。レジで商品が売れた際にリアルタイムで販売データを収集するシステム・端末 |
| **EOS** | Electronic Ordering System（電子発注システム）。バーコードやIDで商品を特定し、本部や卸業者へ発注データを電子送信する仕組み |
| **EOB** | Electronic Order Book。EOS用の発注端末。ストコンまたはハンディ端末で代替されることが多い |
| **Loppi** | ローソン独自のマルチメディア端末。コンサートチケット・旅行・各種行政サービス等の申込・発券ができる |
| **WAKONX** | KDDIが推進するDXプラットフォーム。「つながる社会」の実現を目指し、ローソンでは会員データ統合・AI基盤として活用 |
| **AI.CO** | AI Customized Order。ローソン全店導入済みの発注AIシステム（フューチャーアーキテクト構築）。品揃え・発注・値引き・廃棄の4課題に対応 |
| **LDI** | ローソンデジタルイノベーション。ローソン×シグマクシスの共同出資IT戦略子会社。内製化の中核。NEC側の主要窓口になると推定 |
| **absonne** | NSSol（日本製鉄ソリューションズ）が提供するマネージドクラウドサービス。ローソンは既存業務システムのリフト移行先として活用 |
| **RFID** | Radio Frequency Identification。無線電波を用いた自動認識技術。ローソンではAzure IoTと連携したリアルタイム在庫管理に活用 |
| **MDM** | Mobile Device Management。モバイル・端末を中央管理するシステム。セブン-イレブンはOmnissa Workspace ONEを採用 |
| **RTO** | Recovery Time Objective（目標復旧時間）。障害発生からシステムが復旧するまでの目標時間 |
| **RPO** | Recovery Point Objective（目標復旧時点）。障害時にどの時点のデータまで復旧できるかの目標 |
| **RPN** | Risk Priority Number（リスク優先数）。FMEA における S × O × D の積。数値が高いほど優先度が高い |
| **FMEA** | Failure Mode and Effects Analysis（故障モード影響解析）。システムの「どのように失敗するか」を事前分析する手法 |
| **EVA** | NEC の映像分析基盤（Edge Video Analytics）。ヒートマップ・人物行動分析・立ち寄り率等の映像AI機能を提供 |
| **AVACOM** | ローソンが導入しているアバター接客システム。無人店舗運営を実現 |
| **Real×Tech Lawson** | ローソンが掲げる次世代コンビニビジョン。デジタル（Tech）とリアル店舗体験の融合。KDDIのWAKONXと連動 |
