## 1. 大規模移行プロジェクトのPM体制

### 1-1. なぜ多層PM体制が必要か

大規模システム移行（特にコンビニのような全国数万店舗規模）では、以下の理由から単一PMでは管理しきれない:

| 複雑性の要因 | 具体例（ストコン案件） |
|------------|---------------------|
| **スコープの広さ** | 発注、POS連携、決済、在庫、温度管理等 機能ドメインが多数 |
| **ステークホルダーの多さ** | 発注元企業、SIer複数社、AWSパートナー、店舗SV、本部IT部門 |
| **ロールアウトの段階性** | パイロット店→地域展開→全国展開のウェーブ型 |
| **技術的な深さ** | AWS移行、ネットワーク設計、セキュリティ（PCI DSS）、エッジ処理 |
| **リスクの高さ** | 24/365運用、決済停止不可、廃棄ロスへの直接影響 |

このため、**PMO（全体統括）+ 機能チームPM（機能ドメイン別）+ ベンダーPM（協力会社別）** の多層構造が採用される。

### 1-2. 多層PM体制の全体像


## 2. PMO / 機能チームPM / ベンダーPM の役割分担

### 2-1. PMO（Project Management Office）

PMOはプロジェクト全体のガバナンスを担う組織横断的な管理体。

| 責任範囲 | 具体的なタスク |
|---------|-------------|
| **全体計画管理** | マスタースケジュール策定、マイルストーン管理、クリティカルパス監視 |
| **進捗統合** | 各機能チーム・ベンダーからの進捗を統合し、全体ステータスを可視化 |
| **課題・リスク管理** | 横断課題の検知、エスカレーション判断、リスク対応方針の決定 |
| **品質管理** | テスト戦略策定、品質ゲート定義、レビュー体制の整備 |
| **変更管理** | スコープ変更の影響評価、承認プロセスの運営 |
| **コミュニケーション** | ステアリングコミティ運営、月次報告書作成、全体会議のファシリテーション |
| **標準化** | テンプレート整備、プロセス標準化、ナレッジ管理 |

**PMOの構成メンバー例**:
- PMO長（プロジェクトダイレクター）: 1名
- PMOアナリスト（進捗・コスト管理）: 1〜2名
- 品質管理マネージャー: 1名
- 変更管理担当: 1名
- コミュニケーション担当（議事録・報告書）: 1名

### 2-2. 機能チームPM（Function Team PM）

各機能ドメインを担当するPM。笹尾さんが参画する「特定機能チームのPM」はこのレイヤー。

| 責任範囲 | 具体的なタスク |
|---------|-------------|
| **担当領域の計画策定** | 機能スコープ定義、WBS詳細化、見積もり、スケジュール策定 |
| **チーム運営** | SE/開発者のアサイン調整、デイリー/ウィークリー運営、モチベーション管理 |
| **要件管理** | 機能要件の詳細化、業務部門との要件確認、要件変更管理 |
| **IF管理** | 他チームとのインターフェース仕様合意、依存関係の調整 |
| **テスト管理** | 単体/結合テスト計画、テスト実施管理、不具合管理 |
| **リスク管理** | 担当領域固有のリスク識別・対応計画策定 |
| **PMOへの報告** | 週次進捗報告、課題・リスクのエスカレーション |

**機能チームPMの日常**:
- 朝会: チーム内の進捗確認・ブロッカー共有（15分）
- 週次: PMOへの進捗報告書作成・提出
- 随時: 他チームPMとのIF調整会議
- 月次: チームの振り返り（レトロスペクティブ）

### 2-3. ベンダーPM（Vendor PM）

SIer案件特有の役割。多社協業構造において各ベンダーの窓口となるPM。

| 責任範囲 | 具体的なタスク |
|---------|-------------|
| **契約管理** | 請負/準委任の契約範囲の明確化、追加見積もり、検収条件 |
| **ベンダー内リソース管理** | 要員計画、スキルセット管理、ローテーション対応 |
| **成果物管理** | 納品物の品質担保、レビュー対応、指摘事項の是正 |
| **エスカレーション** | 自社内の上位者への報告・判断依頼 |
| **コスト管理** | 予実管理、追加費用発生時の交渉 |

### 2-4. 3つのPMの関係性マトリクス

| 項目 | PMO | 機能チームPM | ベンダーPM |
|------|-----|------------|-----------|
| **視点** | プロジェクト全体 | 機能ドメイン | 契約・自社リソース |
| **計画の粒度** | マスタースケジュール | 機能別WBS（L3-L4） | ベンダー内作業計画 |
| **管理対象** | 全チーム横断 | 担当機能チーム | 自社要員・成果物 |
| **報告先** | スポンサー/ステコミ | PMO | 機能チームPM + 自社上位者 |
| **意思決定** | Go/No-Go、スコープ | 機能内の優先順位 | 自社リソース配置 |
| **会議運営** | ステコミ、全体定例 | チーム朝会、IF会議 | 社内進捗会 |
| **主な帳票** | 全体進捗報告書 | 週次報告・課題管理表 | 工数実績・品質報告 |

## 3. 大規模移行プロジェクトの典型的なWBS構成

### 3-1. L1-L2 レベル（全体構造）

大規模システム移行プロジェクトの典型的なWBS（IPA「システム移行ガイドライン」ベース）:

```
1. プロジェクト管理
   1.1 計画策定（移行戦略、マスタースケジュール）
   1.2 進捗管理（定例運営、報告書）
   1.3 品質管理（品質ゲート、レビュー体制）
   1.4 リスク・課題管理
   1.5 変更管理
   1.6 コミュニケーション管理
   1.7 ベンダー管理

2. 要件定義
   2.1 移行対象資産の棚卸し
   2.2 現行システム調査（As-Is分析）
   2.3 移行要件定義（To-Be設計の前提条件）
   2.4 非機能要件定義
   2.5 移行方式決定（7Rの適用判断）

3. アーキテクチャ設計
   3.1 移行先インフラ設計（AWS構成）
   3.2 ネットワーク設計（Direct Connect、VPN）
   3.3 セキュリティ設計（IAM、暗号化、PCI DSS）
   3.4 運用設計（監視、バックアップ、DR）
   3.5 データ移行設計

4. 機能別設計・開発
   4.1 機能チームA: 発注・在庫管理
   4.2 機能チームB: POS連携・決済
   4.3 機能チームC: 売上分析・レポーティング
   4.4 機能チームD: 基盤・共通機能
   （各機能チーム内に 設計→開発→UT→CT のサブWBSが展開）

5. テスト
   5.1 結合テスト（IT: Integration Test）
   5.2 総合テスト（ST: System Test）
   5.3 ユーザー受入テスト（UAT）
   5.4 性能テスト
   5.5 セキュリティテスト
   5.6 移行リハーサル

6. 移行実行
   6.1 データ移行（本番データ移行計画・実行）
   6.2 パイロット移行（先行店舗群）
   6.3 ウェーブ展開（地域別・段階的）
   6.4 全店展開完了

7. カットオーバー
   7.1 Go/No-Go判定
   7.2 並行稼働（デュアルラン）
   7.3 旧システム停止
   7.4 安定化運用（ハイパーケア期間）

8. プロジェクト完了
   8.1 運用引き継ぎ
   8.2 ポストモーテム
   8.3 ナレッジ移管
```

### 3-2. 機能チームPMが特に注視すべきWBS要素

機能チームPMとして参画する場合、以下のWBS要素が直接の責任範囲となる:

| WBS要素 | 機能チームPMの関与度 | ポイント |
|---------|-------------------|---------|
| 2.1 移行対象資産の棚卸し | **主担当** | 担当機能の現行仕様・データ量・依存関係の洗い出し |
| 2.3 移行要件定義 | **主担当** | 機能要件の移行判断（As-Is踏襲 vs 刷新） |
| 4.x 機能別設計・開発 | **主担当** | 自チームのWBS詳細化・進捗管理が最重要 |
| 5.1-5.3 テスト | **共同** | IF結合テスト計画、UAT支援 |
| 6.2-6.3 ウェーブ展開 | **共同** | 機能観点でのロールアウト確認・店舗サポート |

## 4. ストコンAWS移行案件に当てはめたPM体制イメージ

### 4-1. 想定されるプロジェクト体制

ストコンAWS移行案件の規模・特性から想定される体制:

```
【発注元】コンビニチェーン本部
    └── IT部門（プロジェクトスポンサー）
         │
         ├── PMO（プライムベンダーSIer or 発注元直属）
         │    ├── PMO長
         │    ├── PMOアナリスト x2
         │    ├── 品質管理マネージャー
         │    └── 変更管理担当
         │
         ├── 機能チームA: 発注・在庫管理（←★ 笹尾さん参画想定）
         │    ├── 機能チームPM ★
         │    ├── アーキテクト
         │    ├── SE x3-5
         │    └── テスター x2
         │
         ├── 機能チームB: POS連携・決済
         │    ├── 機能チームPM
         │    └── ...
         │
         ├── 機能チームC: クラウド基盤・ネットワーク
         │    ├── 基盤PM
         │    └── インフラエンジニア x3-5
         │
         ├── 機能チームD: データ移行・分析基盤
         │    ├── 機能チームPM
         │    └── ...
         │
         └── ベンダー群
              ├── SIer A社（プライムベンダー）— ベンダーPM
              ├── SIer B社（サブベンダー）— ベンダーPM
              ├── AWSパートナー — テクニカルPM
              └── 運用保守会社 — ベンダーPM
```

### 4-2. 機能チームPMとしての想定業務

笹尾さんが「発注・在庫管理」機能チームPMとして参画する場合の想定:

| フェーズ | 主な業務 | 具体的なアクション |
|---------|---------|------------------|
| **参画直後（6月）** | キャッチアップ | 現行システム仕様の理解、チームメンバーとの1on1、PMOルール把握 |
| **要件定義** | As-Is整理 | 発注3方式（自動/推奨/手動）の現行仕様書レビュー、データフロー図作成 |
| **設計** | IF調整 | POSチーム・基盤チームとのIF仕様合意、API設計レビュー |
| **開発** | 進捗管理 | スプリント管理、ブロッカー除去、PMOへの週次報告 |
| **テスト** | テスト計画 | 結合テストシナリオ策定、UAT支援、不具合トリアージ |
| **ロールアウト** | 展開支援 | パイロット店舗での機能検証、フィードバック反映 |

### 4-3. ストコン案件固有の留意点

| 観点 | 留意点 |
|------|--------|
| **24/365運用** | 店舗は深夜も営業。メンテナンスウィンドウが極端に狭い。発注バッチの実行時刻に制約 |
| **店舗数規模** | 全国数千〜2万店。ウェーブ展開が前提。1ウェーブあたり200〜500店が目安 |
| **オフライン耐性** | 通信断時にも発注・売上登録が必要。ローカルDB+差分同期の設計が必須 |
| **業務影響** | 発注ミス＝廃棄ロス or 機会損失。テスト・ロールバック計画が特に重要 |
| **マルチベンダー** | NEC、富士通等の既存ベンダーとAWSパートナーの協業。IF管理が複雑 |
| **PCI DSS** | 決済系との連携がある場合、PCI DSS v4.0準拠が必須。監査対応も |

## 5. 参画後に即活用できる実践知識

### 5-1. 会議体の典型パターン

大規模移行プロジェクトで設定される主な会議体:

| 会議名 | 頻度 | 参加者 | 目的 | 成果物 |
|--------|------|--------|------|--------|
| **ステアリングコミティ** | 月次 | スポンサー、PMO長、各チームPM代表 | Go/No-Go判断、大方針決定 | 議事録、決定事項一覧 |
| **PMO定例** | 週次 | PMO全員、各チームPM | 全体進捗確認、横断課題共有 | 全体進捗報告書 |
| **機能チーム定例** | 週次 | 機能チームPM、SE、テスター | チーム内進捗、ブロッカー共有 | 週次報告、課題一覧 |
| **朝会（デイリー）** | 毎日 | 機能チーム全員 | 当日のタスク確認、障害共有 | （口頭） |
| **IF調整会議** | 週次 or 随時 | 関連チームPM、アーキテクト | チーム間IF仕様の合意・変更 | IF仕様書、合意記録 |
| **品質レビュー** | フェーズ毎 | 品質管理Mgr、各チームPM | 品質ゲート通過判定 | レビュー結果報告 |
| **リスクレビュー** | 隔週 | PMO、各チームPM | リスク登録簿の更新、対応状況確認 | リスク登録簿更新版 |
| **変更管理委員会（CCB）** | 随時 | PMO長、関連チームPM、スポンサー代理 | スコープ変更の承認/却下 | 変更要求書、判定結果 |

### 5-2. エスカレーションルール

```
Level 0: チーム内で解決（機能チームPMの判断）
   ↓ 解決不能 or 他チーム影響あり
Level 1: PMO定例で共有・調整依頼
   ↓ PMO判断で解決不能 or コスト/スケジュール影響大
Level 2: PMO長→スポンサーにエスカレーション
   ↓ 事業影響・契約変更を伴う
Level 3: ステアリングコミティで審議・決定
```

**機能チームPMのエスカレーション判断基準**:
- スケジュール遅延: 2週間以上の遅延見込み → Level 1
- 品質問題: 重大不具合（P1）が3件以上滞留 → Level 1
- リソース不足: 離脱等で計画の50%以上の影響 → Level 1
- IF仕様の不整合: 2チーム間で合意できない → Level 1
- コスト超過: 見積もりの10%超 → Level 2

```

  - スケジュール: {影響の有無と内容}
  - コスト: {影響の有無と金額}
  - 品質: {影響の有無と内容}
  - 他チームへの影響: {影響の有無と内容}
■ 推奨対応: 承認 / 却下 / 保留
■ CCB判定: {結果}
■ 判定日: YYYY/MM/DD
```

### 5-4. 機能チームPMの1日（参画初期のイメージ）

```
09:00-09:15  朝会（チーム全員）
             └─ 昨日やったこと / 今日やること / ブロッカー
09:15-10:00  課題・メール対応、タスクボード更新
10:00-11:00  IF調整会議（他チームPMと）
11:00-12:00  設計レビュー or 要件確認（業務部門と）
12:00-13:00  昼休み
13:00-14:00  PMO週次定例（週1回）
14:00-16:00  チームメンバーとの1on1、ブロッカー除去
16:00-17:00  週次報告書作成、課題管理表更新
17:00-17:30  翌日の準備、メール返信
```

### 5-5. 参画初日〜1週間で押さえるべきこと

1. **プロジェクト標準ドキュメント**: コーディング規約、テスト方針、ブランチ戦略、CI/CD
2. **マスタースケジュール**: 全体のフェーズ・マイルストーン・自チームの位置付け
3. **チームメンバー一覧**: 名前・役割・得意領域・勤務形態（リモート/出社）
4. **IF仕様書**: 自チームが関連する全IFの一覧と合意状況
5. **課題管理表・リスク登録簿**: 現在のオープン課題・リスクの把握
6. **過去の議事録**: 直近1ヶ月分のPMO定例・チーム定例の議事録
7. **開発環境**: 自分の開発・確認環境の構築（Slack/Teams、Jira/Redmine、Git等のアカウント）
8. **キーパーソンとの1on1**: PMO長、隣接チームPM、自チームのリードエンジニア
