## 1. ストアコンピューターの全体像

### 1-1. 定義と役割

**ストアコンピューター（ストコン）**とは、コンビニエンスストアの各店舗に設置される店舗情報管理システムの中核となる専用コンピューターである。店長・従業員が直接操作する業務端末としての役割を担い、店舗運営に必要なすべての情報処理を行う。

主な役割は以下の通りである。

| 役割カテゴリ | 具体的な機能 |
|------------|-------------|
| 発注管理 | 自動発注・推奨発注・手動発注の統合管理 |
| 在庫管理 | 商品在庫の把握・棚卸管理 |
| 売上管理 | 日次精算・売上分析・単品管理 |
| 本部通信 | 発注データ送信・マスタデータ受信 |
| サービス管理 | 公共料金収納・チケット発券の処理 |
| 従業員管理 | シフト管理・勤怠管理 |

### 1-2. 歴史的経緯

| 時期 | 出来事 |
|------|--------|
| 1974年 | セブンイレブン1号店開業（豊洲）。当初は手書き伝票による発注 |
| 1978年頃 | セブンイレブンがEOS（Electronic Ordering System）を導入し、電話回線経由の電子発注を開始 |
| 1982年頃 | POSレジの導入が本格化。販売データの電子化が進む |
| 1985年頃 | ストアコンピューターの原型となるターミナル型端末が普及 |
| 1990年代 | Windows系OSベースのストコンへの移行が進む。グラフィカルUIの採用 |
| 2000年代 | ブロードバンド化によりリアルタイム通信が可能に。本部との常時接続が実現 |
| 2010年代 | タブレット端末の補助的活用、クラウドサービスとの連携開始 |
| 2020年代 | クラウドネイティブへの移行検討が本格化。オンプレ専用機からのクラウド移行案件増加 |

### 1-3. コンビニ業界におけるストコンの位置付け

コンビニのシステム全体像は以下の3層で構成される。

```
[本部システム層]
  ├── 商品マスタ管理システム
  ├── 発注管理システム（EOS/EOB）
  ├── 在庫管理・物流システム
  ├── 会計・経営管理システム
  └── データウェアハウス（売上分析・需要予測）
          ↕（専用回線 / VPN / インターネットVPN）
[店舗システム層]  ← ストアコンピューターが中核
  ├── ストアコンピューター（ストコン）
  ├── POSレジ
  ├── マルチメディア端末（Loppi、Famiポート等）
  └── 電子マネー・決済端末
          ↕
[物流・サービス層]
  ├── 配送センターシステム
  ├── 公共料金収納代行システム
  └── チケット発券システム（イープラス等）
```

ストコンは**店舗と本部を結ぶハブ**であり、店舗内のすべてのシステムからデータを集約し、本部へ送信する役割を担う。逆に本部からの商品マスタ・価格変更・キャンペーン情報等を受信し、店舗内システムへ配信する役割も担う。

### 1-4. 主要チェーンのストコン事情

| チェーン | システム名称・特徴 | ハードウェア・開発元 |
|---------|------------------|-------------------|
| セブンイレブン | セブン情報システム（SIS）。業界で最も先進的なシステムとして知られる。独自のAI発注支援を導入済み | NECが主要ベンダー。富士通、日立も参画 |
| ファミリーマート | FamilyMart Store Management System。ファミポートとの連携が特徴 | 富士通が主要ベンダー |
| ローソン | ローソンストア100を含む複数フォーマットに対応。ポンタとの連携強化 | 日本IBM・日立が参画 |
| ミニストップ | イオングループのシステム基盤を活用。イートイン管理機能が独自 | 富士通系 |
| デイリーヤマザキ | ヤマザキグループの基幹システムと連携 | - |

**注意点**: 各チェーンのシステム詳細は非公開情報が多く、上記は公開情報・業界慣行に基づく一般的な情報である。実際の参画案件では機密保持契約（NDA）の範囲で確認すること。

## 2. 主要機能詳細

### 2-1. 商品発注機能

発注はコンビニ店舗運営の最重要業務のひとつである。ストコンは3種類の発注方式をサポートする。

#### 発注方式の種類

| 発注方式 | 概要 | 特徴 |
|---------|------|------|
| **自動発注（オート発注）** | 過去の販売実績・在庫数・発注リードタイムをもとにシステムが自動的に発注量を算出 | 人手不要。ただし異常値や特需に対応しにくい |
| **推奨発注（スマート発注）** | システムが推奨数量を提示し、店員が確認・修正して発注 | 最もバランスが良い。店員の判断を活かせる |
| **手動発注** | 店員が品番と数量を直接入力して発注 | 新商品・特注品・イベント商品等に利用 |

#### 発注に関わる主要パラメータ

- **発注点（リオーダーポイント）**: 在庫がこの水準を下回ると発注をトリガーする閾値
- **発注量（EOQ）**: 1回の発注で補充する数量。安全在庫を考慮して設定
- **リードタイム**: 発注から入荷までの時間。コンビニは1日1〜3回配送が基本
- **棚割数量（フェイス数）**: 棚に陳列する最大個数。在庫計算の基準値
- **最小発注単位（MOQ）**: 1回に発注できる最小数量。ケース単位・バラ単位が混在

#### 発注締め時刻（発注締め）

チェーンや配送センターによって異なるが、一般的には以下の区分がある。

| 配送区分 | 発注締め時刻の目安 | 配送時刻の目安 |
|---------|----------------|--------------|
| 朝便 | 前日20〜22時 | 当日4〜7時 |
| 昼便 | 当日6〜8時 | 当日10〜14時 |
| 夕便 | 当日10〜12時 | 当日15〜20時 |

### 2-2. 商品廃棄管理

#### 廃棄処理の業務フロー

```
1. 賞味期限・消費期限の確認（日次）
2. 販売期限切れ商品の特定（ストコンのアラート機能）
3. 廃棄スキャン（バーコードスキャンで廃棄登録）
4. 廃棄数量のストコンへの記録
5. 本部への廃棄データ送信（日次精算時）
6. 廃棄ロスとして会計計上
```

#### 廃棄関連の重要概念

- **ロス**: 廃棄・万引き・誤廃棄等による損失。コンビニ経営の重要指標
- **廃棄ロス率**: 売上に対する廃棄金額の割合。目標値はチェーンによって設定
- **チャンスロス**: 品切れにより販売機会を失った損失（廃棄の逆）
- **デイリー商品**: 日配品（弁当・おにぎり・惣菜等）。廃棄管理が最も重要なカテゴリ

### 2-3. 売上精算・日次精算

#### 精算処理の流れ

```
[日次精算の流れ]
 閉店または締め時刻
  ↓
 POSレジからの売上データ集計
  ↓
 ストコンで日計表を作成（売上・客数・客単価・商品別）
  ↓
 現金実査（レジ内現金のカウント）
  ↓
 売上金と現金実査の突合（オーバー・ショートの確認）
  ↓
 精算書の印刷・店長確認
  ↓
 本部への精算データ送信（EOB処理）
  ↓
 売上金の銀行入金処理（加盟店の場合）
```

#### 精算書に含まれる主要データ

| 項目 | 内容 |
|------|------|
| 総売上 | 当日の全取引売上合計 |
| 商品売上 | 商品カテゴリ別の売上内訳 |
| サービス売上 | 収納代行・チケット等のサービス売上 |
| 粗利 | 売上 - 原価 |
| 廃棄ロス | 当日の廃棄金額 |
| 客数・客単価 | 来店客数と1人当たり購入額 |
| 消費税 | 軽減税率対応の内訳も含む |
| 電子マネー別集計 | Suica、nanaco、楽天Edy等の決済種別ごとの集計 |

### 2-4. 在庫管理

#### 在庫管理の概念

コンビニの在庫管理は**理論在庫**と**実在庫**の2種類を管理する。

| 概念 | 説明 |
|------|------|
| **理論在庫（帳簿在庫）** | ストコンが計算で管理する在庫数。入荷数 - 販売数 - 廃棄数で計算 |
| **実在庫** | 棚卸で実際に確認した在庫数 |
| **在庫差異（ロス）** | 理論在庫 - 実在庫の差分。万引き・誤廃棄・検品漏れ等が原因 |

#### 在庫管理のポイント

- コンビニは**在庫回転率が極めて高い**（デイリー品は1日以内）
- 品番（JANコード）単位で在庫を管理
- 温度帯別管理（常温・冷蔵・冷凍・ホット）が必要
- デッドストック（売れ残り）の早期検出もストコンの役割

### 2-5. 棚卸

| 棚卸の種類 | 実施頻度 | 概要 |
|----------|---------|------|
| **定期棚卸（全棚卸）** | 月1〜2回 | 全商品の実数カウント。深夜〜早朝に実施が多い |
| **循環棚卸（ローリング棚卸）** | 日次・週次 | カテゴリを分割して日常的に実施 |
| **スポット棚卸** | 随時 | 差異が疑われる商品を個別にチェック |

棚卸時はハンディターミナル（HT）を使ってバーコードをスキャンし、ストコンへデータを転送して理論在庫と突合する。

### 2-6. 売上分析・単品管理

#### 単品管理（たんぴんかんり）

コンビニが確立した独自の経営手法。**SKU（品番）単位で販売動向を分析し、死に筋商品を排除して売れ筋商品に棚を集中させる**管理手法である。

ストコンでは以下の分析機能を提供する。

| 分析機能 | 内容 |
|---------|------|
| ABC分析 | 売上金額・数量で商品をA（上位）B（中位）C（下位）に分類 |
| 時間帯別販売分析 | 時間帯ごとの売れ行きを把握。品切れ防止に活用 |
| 曜日別・季節別分析 | 曜日・季節パターンを学習して発注精度を向上 |
| 交叉比率分析 | 粗利率 × 在庫回転率。本当に稼いでいる商品の特定 |
| 棚前販売比較 | 棚替え前後の販売実績比較 |

### 2-7. 検品・入荷処理

#### 検品フロー

```
1. 配送ドライバーが納品書（デジタル/紙）を持参
2. 商品と納品書の照合（数量・品番チェック）
3. ハンディターミナルで商品バーコードをスキャン
4. ストコンで受注データと突合（発注数 vs. 納品数）
5. 差異がある場合は「欠品処理」または「過剰入荷処理」
6. 検品完了データを本部へ送信
7. 理論在庫を更新
```

#### 検品時の主要チェック項目

- 品番・数量の一致確認
- 賞味期限・消費期限の確認
- 温度管理状態の確認（冷蔵・冷凍品）
- 破損・汚損の確認

### 2-8. フェイスアップ・棚割連携

| 用語 | 説明 |
|------|------|
| **フェイス（フェイシング）** | 棚の正面から見える商品の面数。1フェイス = 1商品が正面を向いた状態 |
| **フェイスアップ** | 売れた分を補充して棚を整え、すべての商品が正面を向いた状態に戻す作業 |
| **棚割（たなわり）** | どの棚のどの位置にどの商品を何フェイス並べるかを定義した計画 |
| **プラノグラム** | 棚割を視覚化した図面。本部からストコンを通じて店舗に配信される |

ストコンは本部が策定した棚割マスタを受信し、店舗での実施を管理する。

### 2-9. 公共料金収納代行・チケット発券

#### 収納代行サービス

コンビニは電気・ガス・水道・税金・保険料等の収納代行を行う。ストコンはこれらのサービスをマルチメディア端末（Loppi、Famiポート等）と連携して管理する。

| 処理 | 流れ |
|------|------|
| 支払い受付 | バーコード付き払込票をスキャン → 金額確認 → 現金受取 → 領収書発行 |
| 精算 | 当日の収納代行金額を本部へ報告 → 翌日以降に収納機関へ送金 |

#### チケット発券

- マルチメディア端末（Loppi、Famiポート）でチケット選択・購入
- ストコン側では売上計上・在庫（発券枚数）管理を行う
- イベント当日に大量発券が集中するため、処理能力の担保が重要

### 2-10. 従業員管理（シフト・勤怠）

| 機能 | 概要 |
|------|------|
| シフト作成 | 週次・月次のシフト表を作成。人件費の概算も表示 |
| 打刻管理 | 出退勤時刻の記録。ストコンまたは専用タイムレコーダーと連携 |
| 労働時間集計 | アルバイト別の労働時間・時間外・深夜割増の自動計算 |
| 給与データ出力 | 給与計算システムへのCSVエクスポート機能 |
| シフト最適化 | 曜日・時間帯別の来客数予測をもとに必要人員を提案（一部チェーンで導入）|

### 2-11. 本部とのデータ通信

#### 店舗→本部（上り通信）

| データ種別 | 送信タイミング | 主な用途 |
|-----------|--------------|---------|
| 発注データ | 発注締め時刻 | 配送センターの出荷指示 |
| POSデータ | リアルタイム / 日次 | 本部での売上分析・需要予測 |
| 精算データ | 日次締め | 会計・ロイヤリティ計算 |
| 廃棄データ | 日次 | 廃棄ロスの把握・棚割見直し |
| 検品データ | 入荷時随時 | 物流精度管理 |
| 棚卸データ | 棚卸実施時 | 在庫差異の把握 |
| 勤怠データ | 週次・月次 | 人件費管理 |

#### 本部→店舗（下り通信）

| データ種別 | 受信タイミング | 主な用途 |
|-----------|--------------|---------|
| 商品マスタ | 随時 | 新商品登録・価格変更・廃番処理 |
| 棚割マスタ（プラノグラム） | 棚替え前 | 店舗での棚割実施 |
| 発注制限情報 | 随時 | 欠品・過剰在庫の調整指示 |
| キャンペーン情報 | キャンペーン前 | POSへの割引設定配信 |
| セキュリティパッチ | 随時 | ストコン・POSのソフトウェア更新 |
| 価格変更 | 随時 | 定価改定・特売設定 |

## 3. システム構成とインテグレーション

### 3-1. 店舗内システム構成図

```
[店舗内ネットワーク（イントラネット）]
│
├── ストアコンピューター（ストコン）  ← 中核
│     ├── 業務アプリケーション（発注・精算・在庫等）
│     ├── ローカルDB（商品マスタ・在庫データ・販売データ）
│     └── 通信モジュール（本部接続）
│
├── POSレジ（2〜8台）
│     ├── レジアプリケーション
│     ├── バーコードスキャナー
│     ├── キャッシュドロワー
│     └── レシートプリンター
│
├── マルチメディア端末（Loppi / Famiポート等）
│     └── チケット・収納代行処理
│
├── 電子マネー・決済端末
│     ├── 交通系IC（Suica、ICOCA等）
│     ├── 流通系電子マネー（nanaco、WAON等）
│     ├── QRコード決済（PayPay、LINE Pay等）
│     └── クレジットカード端末
│
├── ハンディターミナル（HT）
│     └── 検品・棚卸・フェイスアップ確認
│
└── 温度管理システム
      └── ウォークイン・冷蔵・冷凍ケースの温度監視
```

### 3-2. POS連携

#### POSとストコンの役割分担

| 機能 | POS（レジ）| ストコン |
|------|-----------|---------|
| 商品スキャン・売上計上 | 担当 | - |
| 支払い処理 | 担当 | - |
| レシート発行 | 担当 | - |
| 商品マスタ保持 | ローカルキャッシュ | マスタ管理 |
| 売上データ蓄積 | 一時保持 | 集計・蓄積 |
| 本部へのデータ送信 | - | 担当 |
| 発注処理 | - | 担当 |
| 在庫計算 | 販売データ提供 | 演算・管理 |

#### POSとストコンの通信方式

- **リアルタイム通信**: 1取引ごとにPOS→ストコンへ売上データを送信
- **バッチ通信**: 時間帯区切り（1時間ごと等）でまとめて送信する方式もある
- **通信プロトコル**: TCP/IP（店内LAN）。独自フォーマットまたはXML/JSONが多い

### 3-3. 本部システムとの通信

#### 通信インフラ

| 時代 | 通信手段 |
|------|---------|
| 1980〜90年代 | 電話回線（アナログ）・ISDN |
| 2000年代 | ADSL・光回線 + 専用VPN（MPLS等） |
| 2010年代 | 光回線 + IPsec/SSL-VPN |
| 2020年代〜 | インターネットVPN・クラウド接続（AWS Direct Connect等）への移行検討 |

#### EOSとEOBの違い

| 用語 | 正式名称 | 概要 |
|------|---------|------|
| **EOS** | Electronic Ordering System | 電子発注システム。発注データを電子的に送信する仕組み全般 |
| **EOB** | End of Business | 日次締め処理。営業終了後に本部へ精算データ等を一括送信する処理 |

#### 通信スケジュール（例）

| 時刻 | 処理内容 |
|------|---------|
| 6:00 | 夜間バッチ処理完了・朝の商品マスタ更新受信 |
| 随時 | POSリアルタイム売上データの本部送信 |
| 8:00 | 昼便の発注締め（チェーン・カテゴリにより異なる） |
| 12:00 | 夕便の発注締め |
| 20:00 | 翌日朝便の発注締め |
| 22:00〜翌2:00 | EOB処理（日次精算・廃棄・勤怠データ一括送信） |

### 3-4. マルチメディア端末との連携

| 端末名 | チェーン | 主な機能 | ストコンとの連携 |
|-------|---------|---------|---------------|
| **Loppi** | ローソン | チケット購入・各種申込み・公共料金支払い | 収納代行の売上データをストコンへ連携 |
| **Famiポート** | ファミリーマート | チケット・旅行・行政サービス等 | 同上 |
| **マルチコピー機** | 全チェーン | 印刷・スキャン・コピー | 利用料売上をストコンへ連携 |
| **ATM** | 全チェーン（銀行系） | 現金引き出し・振込 | ストコンとの連携は限定的（利用料のみ） |

### 3-5. 電子マネー・決済端末との連携

コンビニはキャッシュレス化が最も進んでいる小売業態のひとつである。

#### 決済種別と精算

| 決済種別 | 代表例 | 精算方式 |
|---------|-------|---------|
| 交通系IC | Suica、ICOCA、PASMO | 各鉄道会社との精算センター経由 |
| 流通系電子マネー | nanaco（セブン）、WAON（ミニストップ） | 自社システムとの直接連携 |
| クレジットカード | VISA、MasterCard、JCB等 | クレジット決済センター経由 |
| QRコード決済 | PayPay、d払い、LINE Pay等 | 各サービス事業者との精算 |
| バーコード決済 | 同上 | 同上 |

ストコンは決済種別ごとの売上を集計し、EOB処理時に本部へ報告する。未精算の電子マネー売上は**未収金**として管理される。

### 3-6. 物流・配送システムとの関係

```
[発注から入荷までの流れ]
店舗（ストコン）
  発注データ送信
    ↓（EOS）
本部 発注管理システム
  注文確定・出荷指示
    ↓
配送センター（DC）
  ピッキング・積み込み
    ↓（配送車）
店舗
  検品・入荷
    ↓（ストコン）
理論在庫更新・検品データ送信
```

#### 配送センター（DC）の種類

| DC種別 | 温度帯 | 配送商品例 |
|-------|-------|---------|
| **常温DC** | 常温 | 加工食品・菓子・日用品 |
| **チルドDC** | 0〜5℃ | 弁当・おにぎり・惣菜・乳製品 |
| **冷凍DC** | -20℃以下 | アイス・冷凍食品 |
| **ベーカリーDC** | 常温〜チルド | パン・スイーツ |

### 3-7. 温度管理システム（ウォークイン・冷蔵・冷凍）

| 管理対象 | 設定温度帯 | 監視方法 |
|---------|----------|---------|
| ウォークイン（冷蔵庫） | 0〜5℃ | 温度センサー + 自動アラート |
| ウォークイン（冷凍庫） | -20℃以下 | 同上 |
| 冷蔵オープンケース | 2〜8℃ | 同上 |
| アイスケース | -18℃以下 | 同上 |
| ホットケース | 60〜65℃以上（食品衛生法） | 同上 |

温度管理システムはストコンと連携し、温度逸脱時にアラートを表示する。一部チェーンでは本部への自動通報機能も実装している。**食品衛生上の義務**として、温度記録の保管が求められる。

## 4. 業務プロセスとデータフロー

### 4-1. 1日の店舗業務サイクル

```
[早朝（2:00〜7:00）]
  ・EOB処理完了・本部からのマスタ更新受信
  ・朝便の入荷・検品
  ・デイリー商品（弁当・おにぎり）の陳列開始
  ・清掃・店内整備

[開店〜午前（7:00〜12:00）]
  ・通常営業（朝のピーク: 7〜9時）
  ・昼便発注締め（チェーン・カテゴリにより8時前後）
  ・POSデータのリアルタイム本部送信

[昼〜夕方（12:00〜18:00）]
  ・昼のピーク（12〜13時）営業
  ・昼便入荷・検品
  ・夕便発注締め（12時前後）
  ・デイリー商品の値引き・廃棄処理開始（チェーンにより異なる）

[夜（18:00〜24:00）]
  ・夕方・夜のピーク対応
  ・夕便入荷・検品
  ・翌日朝便の発注締め（20〜22時）
  ・廃棄処理（販売期限切れ商品）

[深夜（24:00〜2:00）]
  ・EOB処理（日次精算・廃棄・検品データ一括送信）
  ・棚割変更・棚卸（実施日）
  ・清掃・メンテナンス
```

### 4-2. 発注から廃棄までのサイクル

```
[発注サイクル]

① 発注
  ストコンが推奨数量を算出（販売実績・在庫・天気・イベント等を考慮）
  店長/従業員が確認・修正
  発注データを本部へ送信（EOS）

② 配送センター処理
  本部が受注確認・出荷指示
  DCでピッキング・トラックへ積み込み

③ 入荷・検品
  配送ドライバーが商品を搬入
  ハンディターミナルで検品スキャン
  ストコンで発注データと突合・理論在庫更新

④ 陳列
  バックヤードから売り場へ補充
  フェイスアップ（既存商品の整列）
  プラノグラムに従った配置

⑤ 販売
  POSでの販売記録
  リアルタイムで在庫データ更新

⑥ 廃棄
  販売期限切れ商品を特定
  廃棄スキャン実施
  廃棄データをストコンへ記録・本部へ送信
```

### 4-3. データフロー（店舗↔本部の双方向）

#### 上り（店舗→本部）データフロー

```
POSレジ
  │ リアルタイム売上データ
  ↓
ストアコンピューター
  │ 集約・加工
  ├── 発注データ ─────────────→ 発注管理システム（EOS）
  ├── 精算データ ─────────────→ 会計・ロイヤリティシステム
  ├── 廃棄データ ─────────────→ 在庫・棚割管理システム
  ├── 検品データ ─────────────→ 物流管理システム
  ├── POSデータ ──────────────→ データウェアハウス（売上分析）
  └── 勤怠データ ─────────────→ 人件費管理システム
```

#### 下り（本部→店舗）データフロー

```
本部各システム
  ├── 商品マスタ管理 ──→ ストアコンピューター → POSレジ
  ├── 価格・キャンペーン管理 → ストコン → POSレジ・マルチメディア端末
  ├── 棚割管理システム ─→ ストコン（プラノグラム）
  ├── 発注制限情報 ────→ ストコン（発注可能数量の上限設定）
  └── ソフトウェア更新 ─→ ストコン・POSへの自動適用
```

## 5. 業界用語集

### 5-1. システム・技術用語

| 用語 / 略語 | 正式名称・読み | 説明 |
|-----------|-------------|------|
| **ストコン** | ストアコンピューター | 店舗に設置される業務管理コンピューター本体 |
| **POS** | Point of Sale（ポイント・オブ・セール） | 販売時点情報管理。レジとセットで使われる用語 |
| **EOS** | Electronic Ordering System | 電子発注システム。発注データを電子的に本部へ送信する |
| **EOB** | End of Business | 日次締め処理。本部への精算データ一括送信 |
| **JAN** | Japan Article Number | 日本で使われる商品バーコード規格（13桁） |
| **SKU** | Stock Keeping Unit（エス・ケー・ユー） | 在庫管理の最小単位。品番・サイズ・色等で一意に識別 |
| **HT** | ハンディターミナル | 携帯型バーコードスキャナー。検品・棚卸に使用 |
| **DC** | Distribution Center（ディストリビューション・センター） | 配送センター。商品を集約してコンビニ店舗へ配送する拠点 |
| **MD** | マーチャンダイジング（Merchandising） | 商品政策。何を・いくらで・どのように売るかの計画と実行 |
| **DWH** | データウェアハウス（Data Warehouse） | 分析用に整備された大規模データ格納基盤 |

### 5-2. 商品・在庫管理用語

| 用語 | 説明 |
|------|------|
| **単品管理** | SKU単位で販売動向を分析・管理する手法。コンビニが確立した経営手法 |
| **棚割（たなわり）** | 棚のどの位置に何の商品を何フェイス並べるかを定義した計画 |
| **プラノグラム** | 棚割を図面化したもの。本部から配信される |
| **フェイス（フェイシング）** | 棚正面から見える商品の面数。1フェイス＝1列 |
| **フェイスアップ** | 売れた後に商品を手前に引き出して整列させる作業 |
| **理論在庫** | 帳簿上の在庫。入荷−販売−廃棄で算出 |
| **在庫差異（ロス）** | 理論在庫と実在庫の差。万引き・誤廃棄が主因 |
| **デイリー品** | 日配品。弁当・おにぎり・惣菜等、毎日配送される短期限商品 |
| **チルド品** | 冷蔵（0〜5℃）で管理する商品 |
| **発注点** | 在庫がこの水準を下回ると自動発注をトリガーする閾値 |
| **安全在庫** | 欠品防止のために保持する最低限の在庫バッファ |
| **チャンスロス** | 品切れにより販売機会を失った損失額 |
| **廃棄ロス** | 廃棄した商品の仕入原価相当額 |
| **死に筋商品** | 売れない商品。棚から排除する対象 |
| **売れ筋商品** | よく売れる商品。フェイス数を増やす対象 |
| **交叉比率** | 粗利率 × 在庫回転率。商品の総合的な収益性指標 |

| **ウォークイン** | 店員が中に入れる大型冷蔵・冷凍庫 |

## 付録: クラウド移行時の主要考慮事項

### A. 移行対象システムの整理

```
[クラウド移行の優先度分類（一般的な考え方）]

移行しやすい（高優先度）:
  ・分析系（DWH・BI）
  ・本部系の情報システム
  ・従業員管理・給与計算

移行が中程度:
  ・マスタ管理システム
  ・発注管理システム（EOS）
  ・ECサイト・デジタルマーケティング

移行が難しい（低優先度 / 慎重に検討）:
  ・POSシステム（リアルタイム性・無停止要件）
  ・電子マネー決済端末（PCI DSS対応）
  ・ストアコンピューター本体（レガシー・専用OS多い）
```

### B. コンビニ特有の非機能要件

| 要件 | 内容 | クラウド設計での考慮 |
|------|------|------------------|
| 可用性 | 99.99%以上（年間約53分以内の停止が許容上限） | マルチAZ・リージョン冗長 |
| レイテンシ | POS応答は300ms以内が目安 | エッジキャッシュ・CDN活用 |
| スループット | 全国5万店舗超が同時接続。ピーク時のスパイク対応 | オートスケーリング |
| データ保全 | 取引データは法定保管期間（消費税法上7年等）| S3 Glacier等の低コストアーカイブ |
| セキュリティ | PCI DSS Level 1（最高水準）への対応が必要 | WAF・暗号化・監査ログ |
| 温度データ保管 | 食品衛生法に基づく温度記録の保管 | タイムスタンプ付き不変ストレージ |
