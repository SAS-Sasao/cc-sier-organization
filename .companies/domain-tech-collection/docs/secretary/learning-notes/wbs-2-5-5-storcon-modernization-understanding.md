## 1. エグゼクティブサマリー

ストコンは歴史的にモノリシックなアーキテクチャで構築されており、AWS移行に際しては「リフト＆シフト（そのままクラウドへ）」か「リアーキテクチャ（マイクロサービス化）」かの戦略選択が最初の設計判断となる。ドメイン駆動設計で整理すると、ストコンの機能は発注・在庫・売上・本部通信・サービス・従業員の6ドメインに分解でき、それぞれ依存関係と移行リスクが異なる。

データ分析の観点では、需要予測・廃棄削減・棚割最適化・リアルタイム売上ダッシュボード・異常検知の5ユースケースが主要であり、1店舗あたり日次5〜10 MB、全国5万店規模では年間90〜180 TBに及ぶデータを活用することになる。

移行のリスク管理では「発注締め時刻の遵守」「EOB処理の信頼性」「PCI DSS準拠範囲の確認」「24時間365日の業務継続性」が不可侵の制約として設計の出発点となる。PM視点では、技術的モダナイゼーションと業務システム切り替えを別フェーズに分離し、スコープクリープ（AI自動化・物流最適化への拡大）を防ぐことが最重要課題である。

## 2. モダナイゼーション戦略：モノリスの段階的解体

### 2-1. ストコンのアーキテクチャ課題

ストコンは1990年代のWindows系OSへの移行時に、すべての業務機能を1つのアプリケーションに集約するモノリシック設計が採用された。この設計は以下の問題を引き起こしている。

- 一部の機能を変更しただけで全体への影響確認が必要
- 特定機能だけスケールアウトできない
- デプロイ単位が巨大で変更リードタイムが長い
- 最新のクラウドサービスとの連携が困難

### 2-2. リフト＆シフト vs リアーキテクチャ

| 戦略 | 概要 | メリット | デメリット |
|------|------|---------|-----------|
| **リフト＆シフト** | 既存ストコンをほぼそのままクラウドへ移行 | 移行リスク低・期間短 | クラウドのメリットを十分に享受できない |
| **リアーキテクチャ** | マイクロサービス化・コンテナ化を伴う全面刷新 | 長期的な保守性・拡張性向上 | 移行リスク高・期間長・コスト大 |
| **段階的移行** | まずリフト＆シフトで移行後、機能単位で段階的にリアーキテクチャ | リスク分散・効果を得ながら進化 | 計画の継続的なマネジメントが必要 |

**推奨アプローチ**: セブン-イレブンも採用している**Strangler Figパターン**を使い、まず本部系処理や分析系処理をコンテナ化し、ストコン本体は当面維持しながら段階的に機能を置き換えていく。

### 2-3. ドメイン境界の定義（DDD視点）

ストコンをドメイン駆動設計（DDD）の「境界付けられたコンテキスト（Bounded Context）」で整理すると、以下の6ドメインが識別できる。

| ドメイン | 集約ルート | 役割 | 移行優先度 |
|---------|----------|------|-----------|
| 発注ドメイン | 発注伝票（OrderSlip） | 自動発注・推奨発注・締め管理 | Phase 2（慎重設計要） |
| 在庫ドメイン | 在庫ポジション（StockPosition） | 理論在庫計算・廃棄管理・検品 | Phase 2（難易度高） |
| 売上ドメイン | 日次精算（DailyClosure） | 日次精算・単品管理・電子マネー集計 | Phase 3（精算は最後）|
| 本部通信ドメイン | 通信セッション（CommunicationSession） | EOS/EOB送信・マスタ受信 | Phase 3（影響最大） |
| サービスドメイン | サービス取引（ServiceTransaction） | 収納代行・チケット発券 | Phase 1（独立性高） |
| 従業員ドメイン | シフト（Shift） | シフト管理・勤怠・給与出力 | Phase 1（依存少） |

### 2-4. ドメイン間の依存関係

```
[本部通信ドメイン]
    ↑ 発注データ送信（EOS）
    ↑ 売上・廃棄・検品データ送信（EOB）
    ↓ 商品マスタ・棚割・キャンペーン受信

[発注ドメイン] ←── 在庫水準参照 ──── [在庫ドメイン]
                                              ↑ 理論在庫更新
                                       [売上ドメイン] ←── POSレジ（外部）

[サービスドメイン] ──売上計上──→ [売上ドメイン]
[従業員ドメイン]   ──勤怠データ─→ [本部通信ドメイン]
```

**設計上の注意点**:
- 売上→在庫の依存は同期か非同期（イベント経由）かの設計判断が必要
- 商品マスタは複数ドメインが参照する共有データとして扱う（分散モノリスに陥らないよう注意）
- 本部通信ドメインはすべてのデータの出口・入口であり、メッセージブローカー経由で疎結合化する

## 3. 業務シナリオ分析：6ドメインと主要業務フロー

### 3-1. 発注ドメイン（コンビニ経営の要）

| 機能 | 業務上の重要度 | 備考 |
|------|-------------|------|
| 自動発注（オート発注） | 最高 | 販売実績・在庫・リードタイムを元にシステムが算出 |
| 推奨発注（スマート発注） | 最高 | 推奨数量を提示し店員が確認・修正 |
| 発注締め管理 | 最高 | 1日2〜3回の発注締め時刻の管理・送信 |
| 発注制限の受信・適用 | 高 | 本部からの発注上限・下限設定 |
| 手動発注 | 高 | 新商品・イベント品に使用 |
| 発注履歴管理 | 中 | 過去の発注量・実売量の参照 |

**PM向け注意**: 発注締め時刻は配送センターとの契約に紐づく。移行後も送信タイミングの遵守が絶対制約。発注点の「検知」と発注データの「送信」を分離して設計すること。

### 3-2. 在庫ドメイン（理論在庫の維持）

| 機能 | 業務上の重要度 | 備考 |
|------|-------------|------|
| 理論在庫計算 | 最高 | 入荷数 - 販売数 - 廃棄数の継続計算 |
| 廃棄管理 | 最高 | 廃棄スキャン登録・廃棄ロス計上 |
| 検品処理 | 最高 | 発注データとの突合・欠品/過剰処理 |
| 棚卸管理 | 高 | HTデータ取込・実在庫登録・差異計算 |
| 在庫照会 | 高 | SKU別・温度帯別の在庫確認 |
| 返品管理 | 中 | メーカー・DC返品の処理 |

### 3-3. 売上ドメイン（精算の根幹）

| 機能 | 業務上の重要度 | 備考 |
|------|-------------|------|
| 日次精算（EOB） | 最高 | 売上・廃棄・現金の日次締め処理 |
| 精算書作成 | 最高 | 日計表・精算書の印刷・本部送信 |
| 現金実査突合 | 高 | 現金実査とレジ記録の差異確認 |
| 電子マネー別集計 | 高 | Suica・nanaco等の決済種別ごと集計 |
| 単品管理・売上分析 | 高 | SKU別ABC分析・時間帯別・曜日別分析 |

**PM向け注意**: 日次精算はフランチャイズ会計の根幹。バグは加盟店トラブルに発展するため、最後に移行し徹底した並行稼働テストを実施する。

### 3-4. 本部通信ドメイン（EOS/EOBの主要プロトコル）

| 機能 | 業務上の重要度 | 備考 |
|------|-------------|------|
| 発注データ送信（EOS） | 最高 | 発注締め時刻に本部へ発注データを送信 |
| EOB送信 | 最高 | 精算・廃棄・勤怠等の日次データ一括送信 |
| 商品マスタ受信 | 最高 | 新商品・価格変更・廃番情報の受信と適用 |
| セキュリティパッチ配信 | 高 | ストコン・POSのソフトウェア更新 |
| 棚割マスタ受信 | 高 | プラノグラムの受信と店舗展開 |
| キャンペーン情報受信 | 高 | POSへの割引設定の配信 |

## 4. イベント駆動化とコンテナ化の適性評価

### 4-1. 現行バッチ処理の課題と改善方向

ストコンの業務処理は歴史的に「定時バッチ」が中心であり、通信コスト・処理能力の制約から生まれた設計である。クラウド化に伴い、以下の処理がイベント駆動へ移行できる。

| バッチ処理 | 遅延の業務影響 | 優先度 |
|-----------|-------------|--------|
| 廃棄アラート生成（日次） | 廃棄対応が翌日になる場合がある | A（最高） |
| 在庫水準チェック（発注締め時刻一括） | 締め時刻の間に発生した欠品を拾えない | A（最高） |
| POSデータ本部送信（1時間ごとまたは日次） | リアルタイム分析ができない | B（高） |
| 棚割マスタ受信・反映（夜間） | 棚替え指示の反映に最長24時間かかる | B（高） |
| 商品マスタ更新（夜間） | 新商品の即日取扱いが困難 | B（高） |
| 温度アラート通知 | 食品衛生上のリスク | C（中） |

### 4-2. イベント駆動化の代表シナリオ

**廃棄アラート（優先度A）**:
```
消費期限残り X時間の商品が存在
  → イベント発火（WasteAlertTriggered）
  → 店舗端末に即時通知（値引き・廃棄作業指示）
  → 本部ダッシュボードへリアルタイム集計
  → 需要予測モデルへのフィードバック
```

**在庫発注点トリガー（優先度A）**:
```
POS販売取引が発生
  → 理論在庫をリアルタイム更新
  → 在庫 <= 発注点 の条件を判定
  → 発注トリガーイベント発火（StockBelowReorderPoint）
  → 推奨発注数量を算出・店員へ通知
  → 所定の発注締め時刻に送信（「検知」と「送信」を分離）
```

### 4-3. コンテナ化の適性マトリックス

| サブシステム | コンテナ化適性 | 理由 |
|------------|-------------|------|
| AI発注推奨エンジン | 高（◎） | ステートレス・スケーラビリティ高・変更頻度高 |
| 売上分析・レポート生成 | 高（◎） | 読み取り専用・ステートレス |
| 本部向けバッチ処理（EOB送信） | 高（◎） | ステートレス・ハード依存なし |
| 商品マスタ配信API | 高（○） | スケーラビリティ高・ハード依存なし |
| 在庫照会API | 高（○） | 読み取り専用・スケーラビリティ高 |
| 発注管理API | 中〜高（○） | 並行稼働による段階移行が可能 |
| 電子マネー決済処理 | 困難（×） | PCI DSS認定端末・認定ソフトウェア必須 |
| POSレジアプリ本体 | 困難（×） | 専用ハード依存 |
| ストコン本体（レガシー） | 困難（×） | 専用OS・ドライバ依存、段階移行が必要 |

## 5. データ分析基盤の設計

### 5-1. ストコンデータの全体像

**1店舗あたりの日次データ量**:

| データカテゴリ | 件数/日 | 推定データサイズ |
|-------------|--------|--------------|
| POSトランザクション | 800〜1,200件 | 約1〜2 MB/日 |
| 在庫更新ログ | 3,000〜5,000件 | 約2〜5 MB/日 |
| 温度センサーデータ | 約1,440件（1分ごと） | 約1〜2 MB/日 |
| 発注データ | 150〜300件 | 約50〜100 KB/日 |
| 廃棄データ | 30〜80件 | 約10〜20 KB/日 |
| 機器稼働ログ | 数千件 | 約500 KB〜1 MB/日 |
| **合計** | | **約5〜10 MB/日** |

**全国規模（5万店舗想定）**:

| 指標 | 概算 |
|------|------|
| 全店舗 日次データ量 | 約250〜500 GB/日 |
| 全店舗 年間データ量 | 約90〜180 TB/年 |
| 1年間のPOS取引件数 | 約150〜220億件（全国） |
| 保管期間（税務上7年）の累積 | 約700〜1,300 TB |

### 5-2. 主要データ分析ユースケース

#### ユースケース1: 需要予測・自動発注最適化

**ビジネス背景**: 廃棄ロスとチャンスロスは表裏の関係。セブン-イレブンはAI発注で廃棄削減率15〜20%改善の実績あり。

**主要入力データ**:
- 過去のPOS売上（SKU別・時間帯別・店舗別）
- 曜日・祝日・時間帯（周期パターン）
- 気象データ（気温・天気・降水量）
- 地域イベント情報（花火大会・スポーツ等）
- 棚割変更履歴（フェイス変更による影響除去）

**業務フロー**:
```
前日21時頃: 翌日の気象・イベント情報取得
  → 需要予測モデルがSKU別需要を算出
  → 推奨発注数量を更新
翌朝発注締め前: 店員が推奨数量を確認・修正
  → 発注データを送信
実績収集: 実売数・廃棄数をモデルにフィードバック
  → 継続的に再学習
```

#### ユースケース2: 廃棄ロス削減

**主要シナリオ**:
- 消費期限ベースの廃棄リスクスコアリング（残り時間×在庫数×時間帯別販売確率）
- 廃棄パターン分析（構造的に廃棄が多いSKUの特定と棚割見直し提案）

**注意**: 値引き実施の権限はチェーンによって異なる。本部管理チェーンでは店舗単独での値引きができない場合がある。

#### ユースケース3: 棚割最適化

- フェイス弾力性分析（フェイス1枚増やした際の売上変化量の推定）
- 店舗クラスタ別棚割推奨（立地・客層・売上構成でクラスタリング）
- 棚替え前後の効果測定（定量評価とベストプラクティス化）

**注意**: 棚割は本部MDチームが策定するもので、分析結果は本部へのインプットとして位置づける。交叉比率（粗利率×在庫回転率）を最大化する視点が重要。

#### ユースケース4: リアルタイム売上ダッシュボード

| 機能 | 更新頻度 |
|------|---------|
| 全店売上マップ（地図上に色分け表示） | 5〜15分ごと |
| 時間帯別売上進捗（前日・前週比） | 30分ごと |
| 異常店舗アラート（売上急落・POS停止） | リアルタイム |
| 新商品売行き速報 | 1時間ごと |
| 廃棄速報 | 日次 |

**注意**: SVが担当する店舗数は1人あたり20〜50店舗程度。過剰な通知はアラート疲れを招くため設計に留意。

#### ユースケース5: 異常検知

- POS不正取引検知（特定店員の取引パターンのルールベース+機械学習）
- 在庫差異（ロス）の異常検知（店舗規模・立地ごとのベースライン差異による統計的外れ値検知）
- POS・ストコン障害の検知（ハートビート監視・エラー頻度の異常検知）

### 5-3. データパイプラインのアーキテクチャ方針

**処理方式の分類**:

| ユースケース | 処理方式 | データ鮮度要件 |
|------------|---------|-------------|
| 需要予測・発注推奨 | バッチ（発注締め前） | 前日まで |
| 廃棄リスクスコアリング | ニアリアルタイム | 30分〜1時間 |
| 棚割最適化 | バッチ（日次〜週次） | 日次で十分 |
| リアルタイム売上ダッシュボード | 準リアルタイム | 5〜15分以内 |
| POS不正取引検知 | リアルタイム | 即時（秒単位） |
| 在庫差異検知 | バッチ（棚卸後） | 棚卸後1〜2時間 |
| 温度逸脱アラート | リアルタイム | 即時（1分以内） |

**パイプライン全体像**:
```
[データ発生源]
  POS取引 / 在庫更新 / 廃棄データ / 温度センサー
       ↓
[収集レイヤー]
  リアルタイム: ストリーミング取込（Kinesis等）
  バッチ: EOB処理・日次ファイル転送
       ↓
[格納レイヤー]
  生データ格納（S3等 / ホット・コールド階層化）
       ↓
[処理レイヤー]
  リアルタイム: ストリーム処理（秒〜分単位）
  バッチ: ETL/ELT（時間〜日単位）
       ↓
[分析・AI レイヤー]
  需要予測モデル / 異常検知モデル / BI・ダッシュボード
       ↓
[アクション レイヤー]
  店舗端末への通知 / 本部SVダッシュボード / 発注推奨
```

## 6. AWS サービスマッピング

ソースレポートが参照する技術リサーチ室レポートの知識も踏まえ、業務機能とAWSサービスの対応関係を整理する。

| 業務カテゴリ | 具体的な機能 | AWS サービス候補 | 備考 |
|------------|-----------|---------------|------|
| ストリーム収集 | POSリアルタイム送信・温度センサー | Amazon Kinesis Data Streams | 全国5万店スパイク吸収設計が必要 |
| バッチ収集・転送 | EOB送信・日次ファイル転送 | AWS Transfer Family / S3 | 発注締め時刻の確実な処理保証 |
| 生データ格納 | POSログ・温度記録等の長期保管 | Amazon S3（標準/IA/Glacier） | ホット/ウォーム/コールド階層でコスト最適化 |
| バッチ処理（ETL） | 需要予測の事前計算・棚割分析 | AWS Glue / Amazon EMR | 大規模データの日次バッチ処理 |
| ストリーム処理 | 不正取引検知・温度逸脱アラート | Amazon Kinesis Data Analytics / AWS Lambda | 秒単位のリアルタイム処理 |
| データウェアハウス | 売上・在庫・廃棄データの統合分析 | Amazon Redshift | 全国5万店の過去データを高速クエリ |
| AI/ML | 需要予測モデル・廃棄リスクスコアリング | Amazon SageMaker | モデル学習・推論・再学習の自動化 |
| ダッシュボード | リアルタイム売上ダッシュボード・異常アラート | Amazon QuickSight / Amazon Managed Grafana | SVがモバイルからも参照可能な設計 |
| コンテナ基盤 | AI発注エンジン・各種API | Amazon ECS / Amazon EKS | 本部系処理のコンテナ化から開始 |
| API管理 | 在庫照会API・発注管理API | Amazon API Gateway | Strangler Figパターンでの前段設置 |
| メッセージブローカー | ドメイン間の疎結合通信 | Amazon SQS / Amazon EventBridge | 発注トリガーイベント・棚割変更通知等 |
| 温度記録の不変保管 | 食品衛生法の記録義務対応 | Amazon S3 Object Lock（WORM） | タイムスタンプ付き改ざん不能ストレージ |
| DB（トランザクション） | 在庫ポジション・発注データ | Amazon Aurora / Amazon RDS | 高可用性・MultiAZ構成 |
| セキュリティ・匿名化 | POSデータの個人情報処理 | AWS Glue DataBrew / Amazon Macie | 取込み段階での匿名化・仮名化 |

## 7. 移行フェーズとロードマップ

### 7-1. Strangler Figパターンによる段階移行

**全体方針**: ストコン本体をいじらず、前段にAPIゲートウェイを設置して新機能をコンテナとして提供。徐々にストコン本体の機能を置き換える。

### 7-2. 3フェーズのロードマップ

**Phase 1（0〜12ヶ月）: 本部系・分析系のコンテナ化**

店舗システムへの影響がなく、リスクが最も低い処理から着手する。

| 対象 | アプローチ | 業務継続への影響 |
|------|----------|--------------|
| AI発注推奨エンジン | 既存ロジックをAPI化・コンテナ化 | ほぼなし（推奨値の提供のみ） |
| 売上分析・レポート生成 | 読み取り専用処理のため安全に移行 | なし |
| 商品マスタ管理システム | 段階的にAPIを公開し、既存夜間バッチと並行運用 | 低 |
| 従業員シフト管理 | SaaS移行またはコンテナ化 | 低（他ドメインへの依存が最少） |

**Phase 2（12〜18ヶ月）: 店舗系APIの先行コンテナ化**

ストコン本体の前段にAPIゲートウェイを設置して新機能を追加する。

| 対象 | アプローチ | 業務継続への影響 |
|------|----------|--------------|
| 在庫照会API | ストコンのDBに読み取り専用で接続するAPIをコンテナで提供 | なし |
| 発注管理API | 新規発注画面をコンテナ化。既存の発注送信機能は当面維持 | 低（並行稼働） |
| 棚卸データ取込処理 | HTからのデータ取込APIをコンテナ化 | 低 |
| サービスドメイン（収納代行等） | 独立性が高く切り出しやすい | 低〜中 |

**Phase 3（18ヶ月〜）: ストコン本体の段階的置き換え**

最もリスクの高い処理を並行稼働テストを経て段階的に切り替える。

| 対象 | アプローチ |
|------|----------|
| 精算処理（EOB） | 並行稼働テストを十分に実施後に切り替え |
| レガシーバッチ（PL/SQL等） | AWS DMS等でDB移行後、バッチをコンテナ化 |
| ストコン本体 | Thin Client化（セブン-イレブン方式）または完全クラウド移行 |

### 7-3. PM視点でのチェックポイント

**プロジェクト開始前に確認すべき事項**:

| 確認項目 | 理由 |
|---------|------|
| 対象チェーンの現行ベンダーとの契約内容 | 改変禁止条項がある場合、移行設計が根本的に変わる |
| 発注締め・EOBの時刻と遅延許容範囲 | 不可侵の制約として設計の出発点にする |
| PCI DSS準拠範囲の確認 | 決済系を移行スコープに含めるかで難易度が激変する |
| 温度管理データの保管法的要件 | 食品衛生法に基づく保管義務の設計への反映 |
| 個人情報・購買データの取扱いポリシー | 分析ユースケースの範囲がここで決まる |
| 加盟店との関係性 | 新システム強制導入には加盟店オーナーの合意が必要な場合がある |

**スコープクリープのリスクと対処方針**:

| リスク | 対処方針 |
|-------|---------|
| データ分析がいつの間にかAI発注の完全自動化に拡大 | フェーズ0はダッシュボード・可視化のみと明示 |
| 物流最適化がスコープに入りDC側改修が必要になる | ストコン側の「発注データ出力」までをスコープとし、DC側は別プロジェクト |
| コンテナ化と業務移行が同時進行してプロジェクトが混乱 | 技術的モダナイゼーションと業務システム切り替えを別フェーズに分離 |
| 全チェーン共通基盤を目指して標準化が難航 | まず1チェーン・1機能での成功事例を作り、横展開の際に標準化を進める |

## 8. ストコン移行案件との関連（PM視点）

### 8-1. プロジェクト設計への直接活用（5ポイント）

**ポイント1: ドメイン分解を移行計画の単位にする**
6ドメイン（発注・在庫・売上・本部通信・サービス・従業員）の分解は、移行フェーズの計画単位と一致させることができる。「どのドメインから着手するか」の判断軸として、依存関係の少ない従業員・サービスドメインを先行させ、発注・在庫・精算処理は後工程とする計画を立てる際の根拠として活用できる。

**ポイント2: 発注締め時刻とEOBを「不可侵の制約」として合意形成する**
発注締め時刻の遵守とEOBの信頼性は、フランチャイズ会計・配送センター連携に直結する絶対制約である。プロジェクト開始時にスコープ定義書や要件定義書に明記し、技術判断の変更があっても変えられない制約として関係者全員の合意を取ることが重要。

**ポイント3: PCI DSS準拠範囲を早期に確定し、スコープから明示的に除外する**
電子マネー決済処理・POSレジアプリ本体はPCI DSS認定端末・認定ソフトウェアが必要であり、コンテナ化は容易でない。移行プロジェクトの初期段階でPCI DSS関連機能をスコープ外として文書化し、後からスコープ拡大の要求が来た場合に明確に断れる根拠を準備しておく。

**ポイント4: データ量の概算をDWH・ストレージコスト見積もりに活用する**
1店舗あたり日次5〜10 MB、全国5万店で年間90〜180 TBという概算値は、AWS S3・Redshiftのストレージコスト見積もりの基礎数値として使える。頻繁にアクセスする直近1〜2年のデータと税務上保管義務がある3〜7年のデータでストレージ階層を分けることで、コストを大幅に削減できる。

**ポイント5: 業務継続性（BCP）シナリオをリスク管理表に組み込む**
コンビニは24時間365日の無停止営業が前提であり、以下のシナリオが最重要BCP要件である。

| リスクシナリオ | 業務への影響 | 対処方針 |
|-------------|-----------|---------|
| 発注システムの停止 | 欠品→チャンスロス。翌日以降も影響 | 手動発注フローを必ず維持する |
| EOB処理の失敗 | 日次精算が未送信→会計・ロイヤリティ計算不可 | リトライ・再送信ロジックと失敗時の通報フロー設計 |
| 商品マスタ配信の遅延 | 新商品のスキャンが通らない | 前回マスタでの暫定運用手順を店舗に事前周知 |
| クラウド接続断 | 本部との通信全停止 | 店舗ローカルでのスタンドアロン動作の要件明確化 |

### 8-2. ステークホルダー管理への活用

| ステークホルダー | 関心事 | PM対応のポイント |
|---------------|-------|---------------|
| チェーン本部IT部門 | セキュリティ・稼働率・ベンダー管理 | NDA締結後の詳細情報開示。定期的な進捗報告 |
| 既存ベンダー（NEC・富士通等） | 既存保守契約の維持・新システムへの参画 | 対立ではなく連携。既存ナレッジの引継ぎ協力を得る |
| 加盟店オーナー | 店舗運営への影響最小化 | 移行前の十分な研修・移行中のホットライン設置 |
| 本部SVチーム | 巡回業務のデジタル化への期待と不安 | 早期にダッシュボードの試作版を見せてフィードバックを得る |
| 配送センター担当 | 発注データ受信フォーマット変更への対応 | フォーマット変更は最小限にとどめ、変換レイヤーで吸収する |

## 10. キーワード整理

| 用語 | 説明 | 重要度 |
|------|------|--------|
| **Strangler Figパターン** | 既存システムの前段に新機能を追加し、徐々に古い機能を置き換えるリファクタリング手法。セブン-イレブンも採用。 | 最高 |
| **Bounded Context（境界付けられたコンテキスト）** | ドメイン駆動設計（DDD）の概念。モデルが一貫して適用される境界領域。ストコンでは発注・在庫・売上等の6ドメインに相当。 | 高 |
| **発注締め（EOS締め）** | 1日2〜3回設定された発注データの本部送信期限。配送センターとの契約に紐づく不可侵の制約。 | 最高 |
| **EOB（End of Business）** | 日次精算処理。売上・廃棄・勤怠等のデータを本部に一括送信する。フランチャイズ会計の根幹。 | 最高 |
| **EOS（Electronic Ordering System）** | 電子発注システム。発注データを本部/DCに電子送信する仕組み。 | 最高 |
| **理論在庫** | システム上で計算された在庫数（入荷数 - 販売数 - 廃棄数）。実在庫（棚卸時）との差異が在庫ロス。 | 高 |
| **廃棄ロス** | 消費期限切れ等で廃棄した商品の原価損失。加盟店オーナーの収益を直接圧迫する。 | 高 |
| **チャンスロス** | 欠品により販売機会を失うこと。廃棄ロスとトレードオフの関係にある。 | 高 |
| **単品管理（たんぴんかんり）** | SKU単位で発注量・売上・廃棄を管理し最適化する手法。コンビニ経営の基本思想。 | 高 |
| **フェイス（フェイシング）** | 棚の最前列に陳列する商品の数。フェイス数が多いほど視認性・販売数が上がる傾向がある。 | 高 |
| **交叉比率** | 粗利率×在庫回転率。棚割最適化で商品の優先度を決める際の指標。 | 中 |
| **プラノグラム** | 商品の棚への配置を図示した棚割指示書。本部MDチームが策定し店舗に配信する。 | 中 |
| **Lambda Architecture** | リアルタイム処理とバッチ処理を並列で行うデータ処理アーキテクチャ。 | 中 |
| **PCI DSS** | クレジットカード・電子マネー等の決済情報保護に関する国際セキュリティ基準。認定端末・認定ソフトウェアが必要。 | 高 |
| **ブルーグリーンデプロイメント** | 本番環境と同一の新環境を用意し、切り替えによりダウンタイムなしでデプロイする手法。24時間稼働の店舗システムには事実上必須。 | 高 |
| **棚卸差異（InventoryVariance）** | 理論在庫と実在庫の差。万引き・誤廃棄・誤入力等が原因。 | 中 |
| **コールドスタート問題** | 新商品・季節限定品等、過去データがない商品への需要予測ができない問題。類似商品の実績で補完する。 | 中 |
| **バスケット分析** | 同一取引で一緒に購入された商品の組み合わせを分析する手法。関連商品の棚配置等に活用。 | 中 |

## 11. 理解度セルフチェック

以下の問いに答えることで、本ノートの理解度を確認する。

**Q1: 発注ドメインのマイクロサービス分解を検討する際、なぜ発注ドメインと在庫ドメインの整合性を先に設計しなければならないのか？**

> ポイント: 発注点判定のために現在庫を参照する依存関係があるため、在庫サービスとの整合性が壊れると誤発注が発生する。また、POS売上で理論在庫をリアルタイム更新する設計（同期か非同期イベントか）も先に決める必要がある。

**Q2: イベント駆動で在庫発注点を検知した場合でも、発注データの送信タイミングを所定の締め時刻に維持しなければならない理由は何か？**

> ポイント: 発注締め時刻は配送センターとの契約に紐づいており、チェーン側だけでは変更できない制約である。「在庫発注点の検知（システム内処理）」と「発注データの送信（外部との契約）」を分離して設計することが必要。

**Q3: コンテナ化の適性評価で、電子マネー決済処理が「困難（×）」と評価される理由を説明せよ。**

> ポイント: 電子マネー決済処理はPCI DSSの認定端末・認定ソフトウェアが必要であり、任意のコンテナ環境では動作させられない。認定取得のコストと期間が膨大になるため、移行スコープ外として扱うことが強く推奨される。

**Q4: 全国5万店規模のリアルタイムデータ収集パイプライン設計で、特に注意すべきピーク対応の要件は何か？**

> ポイント: ピーク時（昼12〜13時・夜18〜20時）に平常時の5〜10倍のデータスパイクが発生する。このスパイクを吸収できるようKinesis等のストリーミングバッファのシャード数やスループット上限を設計する必要がある。また、重要度の高い売上サマリーと詳細トランザクションデータで送信頻度を分けることでコストを最適化する。

**Q5: ストコン移行プロジェクトで「データ分析基盤の構築」をスコープに含める場合、初期フェーズで最もスコープクリープしやすいリスクは何か、そして防ぐ方法は？**

> ポイント: 最大のリスクは「ダッシュボード・可視化」が「AI発注の完全自動化」にいつの間にか拡大すること。フェーズ0はダッシュボード・可視化のみと明示し、AIによる意思決定自動化は次フェーズとすることをスコープ定義書に明記して関係者全員の合意を取ることが有効な防止策。
