# ストアコンピューター クラウド移行案件 — 学習スケジュール

**作成日**: 2026-03-21
**作成者**: 秘書室（tech-researcher + retail-domain-researcher の調査結果を統合）
**対象者**: 笹尾豊樹
**案件概要**: コンビニエンスストアのストアコンピューターをオンプレミス→AWSクラウドに移行 + 機能刷新
**参画予定**: 2026年6月

---

## 全体方針

学習領域を **ドメイン知識**（コンビニ業務・ストコン）、**技術スタック**（AWS・移行技術）、**PM知識**（移行PM・大規模ロールアウト・機能チームPM）の3軸で整理し、3月下旬〜5月末の約10週間で段階的に習得する。

```
Week 1-3   ドメイン基礎 + AWS基礎 + PM基礎（3軸の土台を同時に作る）
Week 4-6   ドメイン深掘り + AWS設計パターン + PM中級（案件に直結する知識）
Week 7-9   移行技術 + 統合理解 + PM実践（カットオーバー・ロールアウト計画）
Week 10    総仕上げ + SAA受験 + PM参画準備
```

## PM視点の学習コンテンツ

### 1. 大規模システム移行プロジェクトのPM知識

#### WBS構成・マイルストーン設計

大規模移行プロジェクトのWBSは、通常のシステム開発WBSと異なる以下のフェーズ構造を持つ。

```
1. 移行準備フェーズ
   1.1 現行システム調査・棚卸し（機能・データ・インターフェース）
   1.2 移行方式設計（7Rのどの戦略を各機能に適用するか）
   1.3 移行ツール・環境構築
   1.4 移行リハーサル計画策定

2. 並行開発フェーズ
   2.1 新機能実装（機能チーム別）
   2.2 データ変換・クレンジング
   2.3 インターフェース変換基盤構築

3. 移行検証フェーズ
   3.1 移行リハーサル実施（データ整合性・性能・業務継続性確認）
   3.2 総合テスト（新旧機能等価確認）
   3.3 Go/No-Go 判定会議

4. カットオーバーフェーズ
   4.1 本番移行実施
   4.2 並行稼働（デュアルラン）
   4.3 旧システム廃止

5. 安定化フェーズ
   5.1 移行後モニタリング（KPI監視）
   5.2 問い合わせ対応（ヘルプデスク強化期間）
   5.3 後続展開（次ウェーブへの展開）
```

**重要マイルストーン（ゲートレビュー）の設計観点**:
- 各フェーズ終了時に「ゲートレビュー」を設定し、次フェーズ進行の可否を判定基準で判断する
- 判定基準は定量化すること（例: 「テスト合格率 95%以上」「移行リハーサル所要時間がメンテナンスウィンドウ内」）
- 移行リハーサル結果が判定基準を満たさない場合の追加ループ日程をWBSに組み込む

#### カットオーバー計画（Go/No-Go判定・ロールバック計画）

**Go/No-Go 判定の定量的指標例**:

| 判定項目 | Go 基準 | No-Go 条件 |
|---------|---------|-----------|
| データ移行完了率 | 100%（差異ゼロ） | 1件でも未解決の差異がある |
| 機能テスト合格率 | 必須機能 100% / 全体 95% 以上 | 必須機能に未解決バグ |
| 性能テスト | ピーク時レスポンスが基準値以内 | 基準値超過が1件以上 |
| 移行リハーサル所要時間 | メンテナンスウィンドウ（3h）以内 | 超過 |
| セキュリティ脆弱性 | Critical/High が 0 件 | 1件でも未解決 |

**ロールバック計画の構成要素**:
- **トリガー条件**: カットオーバー後に旧システムに戻す判断基準（例: 主要機能の障害が X 分以上継続）
- **権限者**: ロールバック判断を下す責任者（PMだけでなくクライアント承認が必要な場合が多い）
- **ロールバック手順**: データ逆移行手順・DNS切り替え戻し・旧システム再起動確認
- **判断タイムリミット**: カットオーバーから何時間以内にロールバック可能かを事前に定義
- **訓練**: リハーサルでロールバック手順も演習する

#### 並行稼働（デュアルラン）の管理

- **目的**: 新システムと旧システムを同時稼働させ、出力結果の等価性を確認する
- **期間設計**: 最低1週間、できれば月次バッチの完了サイクルをまたぐ期間を確保する
- **差異検出の仕組み**: 新旧の出力（在庫数・売上集計・発注データ等）を自動比較するスクリプトを用意する
- **差異解消プロセス**: 差異発生時の原因分析担当・修正手順・再確認サイクルを事前に定義する
- **終了判定基準**: 連続 X 営業日間、差異ゼロを確認後に旧システム廃止の申請を行う

#### リスク管理（移行リスクの特定・定量化・軽減策）

**移行プロジェクト特有のリスクカテゴリ**:

| カテゴリ | 代表的リスク | 軽減策 |
|---------|-----------|-------|
| データリスク | データ欠損・重複・変換ミス | 移行前後のレコード件数・チェックサム照合 |
| 業務停止リスク | カットオーバー中の取引停止超過 | リハーサルで所要時間を計測・短縮 |
| ロールバック不能 | 旧システムの起動失敗 | 本番カットオーバー前日まで旧環境を稼働状態で維持 |
| 依存システム障害 | 決済・物流等の外部連携が移行後に機能しない | 結合テストを十分に実施・移行当日の連携先サポート体制確保 |
| 人的リスク | キーパーソンの離脱・知識属人化 | ドキュメント整備・ペアワーク・手順書標準化 |
| スコープクリープ | 移行に便乗した機能追加要求 | 変更管理プロセス（CCB）を設置し、承認なき変更を防ぐ |

**定量化の手法**:
- リスク点数 = 発生確率（1〜5）× 影響度（1〜5） で数値化
- スコア 15 以上を「最優先対処リスク」として課題管理表に登録
- 月次でリスク再評価会議を実施し、スコアが変化したリスクを再採点する

### 2. コンビニ/小売業界のPM特有事項

#### 全国展開ロールアウト計画の設計

全国数千〜数万店舗への一斉展開は、段階展開（ウェーブ展開）が標準。

**ウェーブ展開の設計観点**:
1. **パイロット店舗（Wave 0）**: 5〜10店舗程度で完全な移行を実施し、課題を洗い出す
2. **エリア展開（Wave 1〜N）**: 地域・業態・FC/直営の区分などでグループ化し、各ウェーブに数十〜数百店舗を割り当てる
3. **ウェーブ間インターバル**: 前ウェーブの安定を確認してから次ウェーブを開始する（最低2週間推奨）
4. **完全展開（Wave Final）**: ロングテール（離島・特殊業態等）の対応

**ロールアウト計画の必須要素**:
- 店舗ごとの移行スケジュール管理台帳
- 移行担当者（フィールドSE）のアサイン計画
- 移行後の店舗サポート窓口（ヘルプデスク）の体制
- 移行失敗時の個店ロールバック手順

#### 24/365稼働システムのメンテナンスウィンドウ設計

コンビニのストコンは24時間365日稼働のため、メンテナンスウィンドウ（MW）の設計が特に重要。

**MW設計の原則**:
- **時間帯**: 深夜2:00〜5:00（最も客数が少ない3時間帯）が標準
- **曜日**: 月曜深夜（日〜月の切り替わり）または火曜深夜を避ける（週次バッチが集中するため）
- **例外日の排除**: 決算月末・大型連休・元日・入学シーズンはMWを設けない
- **縮退運転**: MW中の最低限の業務継続を設計する（POS売上はローカル保持 + 後で送信等）
- **時間超過時の対応手順**: 予定時間内に完了しない場合の中断判断基準と手順を事前定義

#### 店舗オペレーションへの影響最小化

- **店長・SV（スーパーバイザー）への事前周知**: 移行スケジュール・注意事項を文書で通知。SVを経由した伝達経路を活用する
- **マニュアル整備**: 店舗スタッフが参照できる「移行日の手順書」を平易な日本語で作成（IT用語を使わない）
- **フィールドSE派遣**: 初回移行店舗には現地立会いを徹底し、問題を即時対処できる体制を敷く
- **移行後サポート強化期間**: 移行後2週間はヘルプデスクの受電能力を通常の3倍程度に増強

#### チェーン本部との調整プロセス

- **承認ゲートの確認**: 本部IT部門の承認に加え、店舗運営部門（SV管轄）・情報システム部・経営層の各承認が必要な場合が多い。誰が最終承認者かを案件開始時に確定させる
- **週次定例の設計**: 本部IT担当者との週次進捗報告に加え、月次でのステアリングコミッティ（経営報告）を設計する
- **エスカレーションパス**: 課題が生じた際の即時エスカレーション先（本部の緊急連絡先）を事前に取得・整理する

### 3. 機能チームPMとしてのスキル

#### 機能チーム間の依存関係管理

大規模移行では複数の機能チームが並走するため、チーム間の依存管理が最重要課題のひとつ。

**依存管理の実践手法**:
- **インターフェース定義書（IF仕様書）の早期合意**: 各チームの境界となるAPIやデータ仕様を設計フェーズ初期に合意し、変更管理プロセス下に置く
- **依存マトリクスの作成**: 「チームA機能がチームBの何に依存しているか」を一覧化し、遅延の連鎖を可視化する
- **スタブ・モックの活用**: 依存先が未完成でも自チームの開発を進められるよう、スタブサーバーを立てることを依頼する
- **定期的なチーム横断同期**: 週次でチームPM全員が集まるIP（Integration Planning）会議を設ける

#### アジャイル/スクラムの大規模移行案件への適用

SIer案件では、ウォーターフォールの計画とアジャイルの実行を組み合わせた「ハイブリッドアプローチ」が現実的。

| レベル | 管理手法 | 対象 |
|-------|---------|------|
| プログラムレベル | ウォーターフォール的WBS | マイルストーン・カットオーバー日程 |
| チームレベル | スクラム（2週スプリント） | 機能実装・テスト |
| 統合レベル | PI（Program Increment）計画 | 四半期単位の目標設定と依存解消 |

**SAFe（Scaled Agile Framework）の活用**:
- 5チーム以上が並走する場合、SAFe の ART（Agile Release Train）概念が参考になる
- PI Planning（全チームが集まる四半期計画会議）を移行プロジェクトのフェーズ計画に組み込む

#### ベンダーマネジメント（SIer案件特有の多社協業）

**典型的な登場人物**:
- **プライムSI（元請け）**: 全体PM・アーキテクト
- **機能別SIer（下請け）**: 各機能チームのリード（自社が該当することが多い）
- **ハードウェアベンダー**: 店舗機器（POS端末・ストコン本体等）のメーカー
- **通信キャリア**: 店舗-センター間の回線
- **クライアント（チェーン本部）**: 最終発注者・要件オーナー

**ベンダーマネジメントの実践**:
- **契約種別の確認**: 請負（成果物責任）か準委任（役務提供）かで、仕様変更時の費用負担・スコープ管理の方法が変わる
- **SLA/KPIの合意**: ベンダーごとに「いつまでに何を納品するか」「品質基準は何か」を契約レベルで明確化する
- **定例会とリポート**: ベンダーごとに週次/隔週の進捗報告の場を設ける。フォーマットを統一して元請けPMがダッシュボードで管理できるようにする
- **問題ベンダーへの対応**: 遅延が続く場合のエスカレーション手順（ベンダー経営層へのアラート→元請けPM→クライアントIT部門）を事前に合意しておく

#### ステークホルダー管理

**ステークホルダーマップの例**:

| ステークホルダー | 関心事 | 影響力 | 関与レベル |
|---------------|--------|-------|---------|
| チェーン本部IT部門 | システム品質・コスト・スケジュール | 高 | 週次定例 |
| 店舗運営部門（SV） | 店舗オペレーション影響の最小化 | 高 | 月次報告 + MW前後個別調整 |
| 個店（店長） | 移行日の作業負荷・サポート体制 | 中 | マニュアル + ヘルプデスク |
| 元請けPM | 全体工程・品質・コスト | 高 | 週次IP会議 |
| 各機能チームPM | 自チームのスコープ・スケジュール | 中 | 週次IP会議 |
| ハードウェアベンダー | 機器納入・現地工事スケジュール | 中 | 隔週定例 |
