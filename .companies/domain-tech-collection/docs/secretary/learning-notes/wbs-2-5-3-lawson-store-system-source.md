## 1. 店舗システム全体像

### ローソン店舗の主要システム構成

```
[店舗層]
  ├── POS端末（NECプラットフォームズ共同開発）

## 4. 店舗-本部間ネットワーク構成

### 現状（推定）

公開情報からネットワーク構成の詳細は確認できなかったが、以下の状況証拠から推定:

| 要素 | 推定内容 | 根拠 |
|------|---------|------|
| 通信キャリア | **KDDI**が有力 | 2024年からKDDIが経営参画（議決権50%）、通信インフラ提供を明言 |
| 接続方式 | **IP-VPN**または**閉域網** | KDDIはIP-VPNサービスを法人向けに提供、小売店舗の標準構成 |
| バックアップ回線 | **モバイル回線（au 4G/5G）** | KDDI傘下のため、au回線をバックアップに使用する合理性 |
| 帯域 | **光回線ベース** | POS・ストコンのリアルタイムデータ送信、カメラ映像データに必要 |

### KDDI経営参画後のネットワーク変化（推定）

2024年以降、KDDIの経営参画により以下の変化が予想される:

1. **KDDI WAKONX基盤への統合**: AI推論・データ分析のためのネットワーク最適化
2. **5G活用**: 店舗内IoTデバイス（カメラ、RFIDリーダー等）の高速通信
3. **エッジコンピューティング**: KDDI基地局側でのリアルタイム処理
4. **統合ネットワーク管理**: 約14,600店舗 + 約2,200 au Styleの統合管理

### コンビニのネットワーク一般構成

```
[店舗]
  POS端末 ──┐
  ストコン ──┼── 店舗内LAN ── ルーター ── IP-VPN/閉域網 ── [本部DC/クラウド]
  カメラ  ──┤                              │
  Loppi  ──┘                      モバイル回線（バックアップ）

## 6. AI.CO（発注AI）の技術詳細

### システム概要

| 項目 | 内容 |
|------|------|
| 正式名称 | AI.CO（AI Customized Order） |
| 前身 | セミオート発注システム（2015年全店展開、フューチャーアーキテクト構築） |
| 全店導入 | 2024年7月完了 |
| 開発パートナー | フューチャーアーキテクト |

### 分析因子（約100個）

- 過去の販売実績
- 天候・気温
- Pontaカード会員データ（1億1,305万人、2023年5月時点）
- 同立地の他店舗販売状況
- 本部の販売施策
- 曜日・時間帯パターン
- その他約90の因子

### 発注推奨数の計算式

```
発注推奨数 = 販売予測数 - 在庫予測数 + 安全在庫数
```

### AI.COの新機能（2024年〜）

- **値引き推奨**: 商品の販売許容時間の期限を基に、利益を出しながら売り切るための最適な値引き額・タイミングをAIが推奨
- **強化学習アルゴリズム**: 新たに強化学習を活用し、発注精度を向上
- **効果**: 2025年度の既存店粗利は70億円伸長見込み

## 9. セブン-イレブンのフルクラウド化との比較

### アーキテクチャ比較

| 項目 | ローソン | セブン-イレブン |
|------|---------|---------------|
| **クラウド基盤** | AWS（メイン）+ Azure + absonne | **Google Cloud**（フル） |
| **ストコン** | オンプレ存続（2019年刷新） | **廃止**（クラウド移行） |
| **アーキテクチャ** | マルチクラウド・段階移行 | **フルクラウド・マイクロサービス** |
| **店舗端末** | 専用POS + 専用ストコン | **汎用端末**（iPad/Android約30万台） |
| **認証** | 従来型（ID/パスワード） | **顔認証**（NEC、40万人対応） |
| **端末管理** | 個別管理 | **Omnissa Workspace ONE** |
| **運用管理** | 個別 | **ServiceNow ITSM** |
| **POS端末** | NECプラットフォームズ | 東芝テック |
| **SI主幹** | フューチャーアーキテクト | **NEC**（PMO + 発注システム） |
| **導入時期** | 段階的（2015年〜） | **2025年春〜年内全店** |

### 進捗差の分析

セブン-イレブンは2025年にフルクラウド化を完了する見込みだが、ローソンはストコンがオンプレミスに残存している。この差の背景として:

1. **経営体制の違い**: セブン-イレブンは自前IT部門（セブン&アイ・ネットメディア等）が強力。ローソンはKDDI参画が2024年と最近
2. **クラウド戦略の違い**: セブンはGoogle Cloudに一本化。ローソンはマルチクラウド（AWS+absonne+Azure）で複雑
3. **SI体制の違い**: セブンはNECがPMOとして中核に入りマルチベンダー統制。ローソンはフューチャーが基幹系、NECはPOS/AIと分散
4. **店舗数の違い**: セブン約21,000店 vs ローソン約14,600店。セブンの方が規模は大きいがフルクラウド化を先行

### アーキテクチャ差の要点

```
[セブン-イレブン] フルクラウド型
  店舗: 汎用端末(iPad/Android) → Google Cloud (マイクロサービス)
  認証: NEC顔認証
  端末管理: Omnissa Workspace ONE
  運用: ServiceNow ITSM

[ローソン] ハイブリッド型（現状）
  店舗: 専用POS + 専用ストコン → AWS/absonne（本部系のみ）
  認証: 従来型
  端末管理: 個別
  運用: 個別
```

## PMとしての示唆

### 1. ストコンAWS移行の技術的論点

ローソンのストコンはオンプレミスに残存しており、セブン-イレブンに比べてクラウド化が遅れている。移行に際して以下が論点になる:
- **既存ストコン機能のクラウド代替アーキテクチャ**: マイクロサービス化するか、モノリシックなリフト&シフトか
- **店舗端末の取り扱い**: セブンのように汎用端末化するか、既存POS端末は維持か
- **既存のマルチクラウド環境との整合**: AWS上に構築するが、absonne・Azure・WAKONXとのデータ連携が必要

### 2. セブン案件の知見転用

NEC社内にはセブン-イレブンのフルクラウド化知見が蓄積されているはず。以下が転用候補:
- **マイクロサービスアーキテクチャ設計**: Google Cloud→AWSへの技術変換が必要だが、設計思想は共通
- **マルチベンダー統制のPMOノウハウ**: セブン案件で約3,000人規模のマルチベンダー管理を実践
- **顔認証技術**: セブンで40万人対応の実績あり、ローソンにも展開可能
- **端末管理**: Omnissa Workspace ONEの導入ノウハウ

### 3. フューチャーアーキテクトとの関係構築

ストコンはAI.CO（発注AI）や統合マスタシステムとデータ連携している。これらはフューチャーアーキテクトが構築・運用しているため:
- **データI/F仕様の取得**: フューチャーアーキテクトからマスタデータ・発注データのI/F仕様を入手する必要
- **移行期間中の並行運用**: 既存ストコン→新クラウドシステムへの段階移行中、フューチャー管理の既存システムとの二重連携が発生

### 4. KDDI通信インフラの活用

KDDIが経営参画したことで、店舗ネットワークの刷新もストコン移行と同時に進む可能性がある:
- **KDDI IP-VPN/閉域網**: 店舗-クラウド間の通信基盤
- **5Gエッジコンピューティング**: 店舗側での軽量処理
- **WAKONX統合**: AI推論・データ分析基盤との連携

### 5. リスク要因

- **マルチクラウド環境の複雑性**: AWS+absonne+Azure+WAKONXの4環境間でのデータ整合性
- **約14,600店舗への展開リスク**: 全店展開のロールアウト計画とロールバック計画
- **24時間365日稼働**: コンビニの無停止要件に対するクラウド移行戦略
- **既存POS端末との互換性**: NECプラットフォームズ製POSとの通信プロトコル変更の影響
