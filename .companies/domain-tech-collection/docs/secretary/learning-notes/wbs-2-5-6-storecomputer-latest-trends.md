## 1. ストコンのクラウド移行事例

### 要点

- **セブンイレブン・ジャパン（SEJ）**: 独自開発のストコン「ROSS（Retail Operation Support System）」を刷新。2023〜2024年にかけてAWS活用を本格化し、発注最適化・需要予測基盤をクラウドに移行。2025年には全国約2万店規模での展開を推進中。AI需要予測エンジンの精度向上が主要KPI。
- **ファミリーマート**: マイクロソフトとの戦略的提携（2022年発表、2024年以降本格稼働）により、Azure AIを活用した発注・廃棄削減システムを展開。ストコンの一部機能をクラウドAPIとして外出し、店舗端末のシン化（Thin Client化）を段階的に推進。
- **ローソン**: 三菱商事×KDDIの親会社体制を背景に、KDDIのネットワーク・エッジインフラを活用した「スマートストア構想」を推進。店舗IoT（カメラ・センサー）データの収集・分析にAWS/GCPを活用。
- **ミニストップ**: イオングループのIT基盤共通化の流れで、グループ共通クラウド（主にAWS）への統合を推進。独自開発から汎用パッケージ＋クラウドへの転換。
- 各社共通で「**オンプレ専用機からの脱却**」が加速。ただし即時性・可用性が求められるPOS・決済部分はエッジ処理を維持し、分析・AI部分をクラウドに移行するハイブリッド構成が主流。

### 参考キーワード・企業名・技術名

- AWS、Azure（Microsoft）、GCP（Google）
- セブン&アイ・ホールディングス、ファミリーマート（伊藤忠商事）、ローソン（三菱商事・KDDI）
- ROSS（セブンイレブン独自システム）
- Thin Client化、クラウドネイティブ移行、段階的移行（Strangler Fig パターン）
- NEC、富士通（従来のストコンベンダー）、NECネクサソリューションズ

### PM視点で押さえるべき観点

1. **移行リスクの分離**: 決済・EOB等のミッションクリティカル機能と、発注支援・分析機能を分離して段階移行するアーキテクチャ判断が重要。「全部まとめて移行」は失敗リスクが高い。
2. **ベンダーロックインの回避**: 従来は「各社独自専用機」だったストコンがクラウド化されることで、特定クラウドベンダーへの依存リスクが発生する。マルチクラウド・コンテナ化戦略の検討が必要。
3. **本部・店舗間の役割再定義**: クラウド移行により「どの処理をどこで実行するか」のアーキテクチャ設計が問われる。従来は店舗ローカル処理だったものをどこまでクラウドに移せるかの判断基準の整理が必須。

## 2. 店舗システムのクラウド化トレンド

### 要点

- **Unified Commerce 基盤**: オンプレ×クラウドの2層構造から、クラウドを中心とした「Unified Commerce」基盤への移行が世界的トレンド。日本でも2024〜2025年にかけて大手小売の採用事例が増加。
- **MACH アーキテクチャの採用**: Microservices、API-first、Cloud-native、Headless の4原則からなるMACHアーキテクチャが小売IT基盤の設計指針として浮上。ストコン機能の単機能API化（発注API、棚割API等）の実現パス。
- **Kubernetes/コンテナ化**: 店舗システムの可搬性向上とデプロイ自動化のため、コンテナ化が進む。AWS EKS、Azure AKS上でのストコン後継システム構築事例が国内でも出始めている。
- **リアルタイムデータ基盤**: Apache Kafka、AWS Kinesis 等を使ったストリーミング処理で、店舗売上・在庫データのリアルタイム分析基盤を構築する案件が増加（2024〜2025年）。
- **SaaS型POS/ストコン**: Square、Lightspeed、SmartHR的なSaaS化の波がPOS/ストコン領域にも波及。中小小売向けから始まり、コンビニFCの個店管理ツールとして補完的に活用される動きが見られる。

### 参考キーワード・企業名・技術名

- MACH Alliance、Unified Commerce
- Kubernetes（K8s）、Docker、Helm
- Apache Kafka、AWS Kinesis、Azure Event Hubs
- AWS EKS、Azure AKS、GKE
- Commercetools、Salesforce Commerce Cloud（海外先行事例）
- NTTデータ、日立、富士通（国内SIer主要プレイヤー）

### PM視点で押さえるべき観点

1. **技術負債の見極め**: 20〜30年前のCOBOL/Windows系ストコンをモダン化する際、「スクラッチ開発 vs. SaaS置換 vs. リファクタリング」の3択の評価軸を明確にすること。コスト・期間・リスクのトレードオフを定量化して提案する。
2. **データ品質問題**: クラウド移行で各種データが統合される際、長年の運用で発生した商品マスタの重複・表記ゆれ等のデータ品質問題が表面化する。移行前のデータクレンジング工数が大幅に増加するリスクを予算に織り込むこと。
3. **回線障害時のフォールバック設計**: クラウド化により「回線断時の店舗運営継続」が重要な非機能要件となる。オフラインファースト設計（Service Worker、ローカルキャッシュ）の検討が必要。

## 3. エッジコンピューティング×店舗

### 要点

- **エッジ×クラウドのハイブリッド構成が標準化**: 決済処理（レイテンシ要件: 数百ms以内）はエッジ（店舗内ローカル）で完結させ、分析・AI推論はクラウドで実行するアーキテクチャが業界標準に。
- **AWS Outposts / Azure Stack Edge の店舗採用**: クラウドAPIをオンプレで動かすマネージドエッジサービスの採用事例が国内コンビニでも検討段階から実証実験フェーズへ移行（2025年）。ローソン・セブンで実証報告あり。
- **カメラ×エッジAIによる在庫可視化**: 棚カメラ＋エッジAIによるリアルタイム欠品検知システムが普及フェーズへ。NVIDIA Jetson、Intel OpenVINO等のエッジAIチップを使った画像認識を店舗内で完結させる構成。
- **5G/Wi-Fi 6による店舗内通信刷新**: 従来の有線LAN中心の店舗内ネットワークを5G専用線またはWi-Fi 6で刷新し、IoTデバイス（冷蔵ケース温度センサー、自動搬送ロボット等）との連携を強化する動きが加速。
- **デジタルサイネージ×エッジ配信**: 価格変更・プロモーション情報を電子棚札（ESL: Electronic Shelf Label）でリアルタイム更新。ストコンからの価格マスタ更新をトリガーにESLが自動更新される仕組みが標準化。

### 参考キーワード・企業名・技術名

- AWS Outposts、AWS IoT Greengrass、Azure Stack Edge
- NVIDIA Jetson、Intel OpenVINO
- ESL（Electronic Shelf Label）、電子棚札
- Wi-Fi 6（802.11ax）、5G専用線（ローカル5G）
- ソフトバンク（5G店舗ソリューション）、KDDI、NTTドコモ
- パナソニック コネクト（スマートストアソリューション）

### PM視点で押さえるべき観点

1. **エッジデバイスの運用管理コスト**: 店舗数が数百〜数千規模になると、エッジデバイスのソフトウェア更新・障害対応が運用コストの大部分を占める。MDM（Mobile Device Management）やAWS IoT Fleet Hubを活用した中央管理アーキテクチャの設計が不可欠。
2. **エッジとクラウドの整合性担保**: エッジ側のデータがクラウドに同期されるまでのラグ（数秒〜数分）の間に生じる不整合への対処（Eventual Consistency設計）をシステム設計の初期段階から議論すること。
3. **セキュリティの攻撃面拡大**: エッジデバイスの増加はサイバー攻撃の攻撃面（Attack Surface）を広げる。各デバイスへのゼロトラスト適用（デバイス認証、通信暗号化）の要件定義が重要。

## 4. コンビニ業界の最新課題と対応策

### 要点

- **慢性的な人手不足とDXによる省人化**: 2024〜2025年の業界調査では、コンビニ加盟店の約6〜7割がアルバイト採用難を深刻な経営課題として挙げている。セルフレジ・セミセルフレジの全店展開が加速し、1店舗あたりの最低人員数を削減するシステム整備が最優先課題。
- **セルフレジ・無人店舗の実装状況**: セブンイレブンはセルフレジを2025年末までに全店（約2.1万店）に展開する計画を推進。ファミリーマートは「ゼロアシスト店舗（省人化店舗）」の実証実験を継続中。完全無人化は法規制（酒類・タバコ販売の年齢確認義務）の壁で限定的。
- **AI需要予測・自動発注の精度向上**: 食品廃棄率削減と機会損失防止を両立するAI発注が普及。セブンイレブンは2023年に独自AI「発注AI」の精度を大幅改善し、廃棄削減率15〜20%を達成したと発表。気象データ・イベントカレンダー・地域特性を組み合わせた多変量予測が標準化。
- **フードロス削減とサステナビリティ**: 2025年4月施行の「食品ロス削減推進法」改正により、廃棄管理システムへの要件が強化。消費期限マネジメント・値引き自動化（ダイナミックプライシング）のシステム対応が急務。
- **キャッシュレス決済比率の急拡大**: コンビニのキャッシュレス決済比率は2024年時点で60〜70%（業界推定）。PayPay、楽天Pay、交通系ICカードの乱立から、決済端末の統合化・マルチ決済対応が必須要件に。

### 参考キーワード・企業名・技術名

- セルフレジ、セミセルフレジ、ゼロアシスト店舗
- AI需要予測（気象API連携、多変量時系列モデル）
- ダイナミックプライシング（消費期限連動値引き）
- フードロス削減、食品ロス削減推進法
- PayPay、楽天Pay、Suica/PASMO（交通系IC）、iD、QUICPay
- Panasonic、東芝テック（セルフレジ機器ベンダー）
- Preferred Networks（AI発注システム開発協力例）

### PM視点で押さえるべき観点

1. **省人化と顧客体験のバランス**: セルフレジ化は人件費削減だけでなく顧客の待ち時間短縮にも寄与するが、高齢者・外国人旅行者への配慮が必要。UI/UXの多言語対応・バリアフリー設計を要件に含めること。
2. **AI発注の「ブラックボックス問題」**: AI発注精度が高くても、店長・オーナーが発注根拠を理解できない場合は現場での信頼が得られない。XAI（説明可能AI）や発注根拠の可視化UIの設計が重要。
3. **決済基盤の標準化遅れ**: 日本の決済規格は乱立しており、新決済手段への対応コストが継続発生する。決済基盤のアーキテクチャとして「決済プラグイン型」（新決済手段をプラグインとして追加可能な設計）を採用することでコスト削減が可能。

## 5. PCI DSS v4.0 対応ポイント

### 要点

- **PCI DSS v4.0 完全移行期限（2025年3月31日）**: v3.2.1のサポートが2024年3月31日に終了し、2025年3月31日をもってv4.0への完全移行が義務化。コンビニ・小売業はカード決済処理（加盟店レベル1〜4）として対応が必須。
- **主要変更点①「カスタマイズドアプローチ」の導入**: v4.0では従来の規定要件準拠に加え、リスクベースで独自コントロールを定義できる「カスタマイズドアプローチ」が選択可能に。大手小売は自社のセキュリティ成熟度に合わせた柔軟な対応が可能になった反面、証跡整備・文書化の工数が増大する。
- **主要変更点②「ソフトウェアセキュリティ要件（Req.6）の強化」**: Webアプリケーションに対するWAF必須化（2025年3月31日適用）、スクリプトの完全性チェック（SRI: Subresource Integrity）の義務化がECサイト・セルフレジUIに影響。コンビニの決済画面・モバイルアプリも対象。
- **主要変更点③「認証強化（Req.8）」**: 多要素認証（MFA）の適用範囲がCDE（カードデータ環境）へのすべての非コンソールアクセスに拡大。ストコン管理画面・本部システムへのリモートアクセスにMFA必須。
- **「ターゲット日付（Targeted Risk Analyses）」の導入**: v4.0では一部要件に「組織がリスク分析に基づいて頻度を定義する」アプローチが導入。従来の「年1回」等の固定スケジュールから脱却し、リスクに応じた動的なセキュリティ活動頻度の設定が求められる。

### 参考キーワード・企業名・技術名

- PCI DSS v4.0、PCI SSC（Payment Card Industry Security Standards Council）
- CDE（Cardholder Data Environment、カードデータ環境）
- WAF（Web Application Firewall）、SRI（Subresource Integrity）
- MFA（Multi-Factor Authentication、多要素認証）
- QSA（Qualified Security Assessor、認定セキュリティ評価機関）
- SAQ（Self-Assessment Questionnaire、自己問診票）
- Visa、Mastercard、JCB、American Express（ブランドごとの追加要件に注意）
- NTTデータ先端技術、JPCERT/CC、PCIDSS.jp（国内情報源）

### PM視点で押さえるべき観点

1. **スコープ（CDE）の定義が最重要**: PCI DSS対応コストを最小化するには、カードデータが通過・保存・処理される範囲（CDE）を最小化するアーキテクチャ設計が鍵。P2PE（Point-to-Point Encryption）やトークナイゼーションでCDEをできるだけ小さくすること。
2. **v4.0の「将来日付要件（Future-dated Requirements）」への注意**: v4.0には2025年3月31日までに対応が必要な要件と、2026年3月31日まで猶予がある「将来日付要件」が混在している。プロジェクトのスコープ設定時に両者を区別して工程を組むこと（2025年3月31日以降も対応作業が残る）。
3. **コンビニ特有のスコープ広範化リスク**: コンビニはATM設置、マルチメディア端末（クレジット払い対応）、収納代行（公共料金・税金）等、一般小売より決済に触れるタッチポイントが多い。スコープ調査の初期段階でこれらすべてを洗い出すことが不可欠。

## 総括：PM向け重要示唆

| 課題領域 | 短期対応（〜2026年） | 中長期視点（2027年〜） |
|---------|------------------|-------------------|

| エッジ活用 | カメラ×AI在庫可視化 | ロボティクス×エッジ完全統合 |

### SIerとして狙える案件タイプ（2024〜2026年）

1. **ストコン刷新PMO支援**: クラウド移行計画策定・RFP作成・ベンダー選定支援
2. **PCI DSS v4.0 対応コンサル**: スコープ定義・ギャップ分析・QSA対応補佐
3. **AI発注基盤構築**: MLOps基盤（SageMaker/Azure ML）、需要予測モデル開発・保守
4. **エッジIoT基盤構築**: 店舗センサー統合・AWS IoT Greengrass基盤、ESL連携
5. **決済基盤モダナイズ**: キャッシュレス統合、マルチ決済プラットフォーム化
