## 1. 主要チェーンのクラウド移行最新状況

### セブン-イレブン

**移行戦略**: ROSS（Retail Operation Support System）を機能ドメインごとに切り出しクラウド化（Strangler Fig パターン）

| 機能ドメイン | 移行状況 | 活用サービス |
|-----------|---------|-----------|
| AI需要予測・発注最適化 | 本稼働（2023年〜全国展開） | Amazon SageMaker、AWS Lambda |
| 売上データ分析基盤 | 本稼働 | Amazon Redshift、Amazon S3 |
| 商品マスタ管理の一部 | 検証〜段階展開中 | Amazon RDS、AWS DMS |
| 店舗センサーデータ収集 | 実証実験中 | AWS IoT Greengrass、Amazon Kinesis |
| 決済・POSコア処理 | オンプレ継続 | — |

**AI発注実績**: 廃棄削減率15〜20%改善。気象API・地域イベント・ポイント利用履歴の多変量モデル。再学習サイクルを月次→週次に短縮（SageMaker Pipelines）。

**2025〜2026年の次フェーズ**: 店舗内ストコンのThin Client化。現行Windows専用機を軽量クライアントに置換し、業務ロジックをクラウド側で処理。全国2.1万店への一斉展開は2027年以降が現実的。

**ベンダー体制**: NEC（主要）、富士通（一部参画）、AWS Japan（クラウド基盤パートナー）

| 施策 | 概要 | 進捗 |

**Azure選択の背景**: 伊藤忠グループ全体のMicrosoft関係強化。SAP on Azureとの親和性・ERP統合を視野。

**ベンダー体制**: 富士通Japan（店舗IT運用保守継続）、Microsoft（クラウド基盤）

## 2. ストコン×POS連携アーキテクチャ

### データフロー

```
[POS端末（2〜8台）]
   │ ① 売上トランザクション（リアルタイム or バッチ）
   ▼
[ストアコンピューター]
   │ ② 内部処理: 理論在庫更新、売上累計、廃棄候補検知、AI発注フィード
   │ ← 商品マスタ・価格マスタをPOSへ配信（下り）
   ▼
[本部システム / クラウド]
   │ ③ リアルタイム売上、発注データ、EOBデータ（上り）
   │ → 商品マスタ・棚割マスタ・キャンペーン情報（下り）
   ▼
[DWH / AI基盤]
```

**POS→ストコン通信モード**:
- **リアルタイム（同期）**: 1取引完了ごと（数秒以内）→ 即時在庫更新・盗難検知
- **バッチ（非同期）**: 1時間ごと → 集計分析・レポート

**データ量概算（1店舗/日）**: 売上トランザクション1〜5MB、商品マスタ更新数百KB〜1MB、EOB数十MB。全国5.5万店で本部側はEOBだけで日次数TB規模。

**商品マスタ同期の難所**: 新商品発売日・価格変更日の「マスタ切り替えタイミング」管理。POS複数台への一斉配信の原子性担保が設計要件。

### リアルタイム性要件

| 機能 | 許容レイテンシ | 処理場所 |
|------|-------------|---------|
| 決済処理（現金） | 〜500ms | POSローカル |
| 電子マネー決済 | 〜1秒 | POS + 決済センター |
| 商品マスタ参照（スキャン時） | 〜200ms | POSキャッシュ |
| 在庫更新（販売後） | 〜数秒 | ストコンローカル |
| 売上の本部送信 | 〜数分許容 | ストコン→本部 |
| AI発注推奨更新 | 発注締め前（数時間） | クラウド |

**「ファスト/スロー」2層アーキテクチャ**: ファスト層（決済・在庫・廃棄）はエッジ完結、スロー層（AI・集計・マスタ同期）をクラウドに移行。Apache Kafka（Amazon MSK）によるイベントストリーミングの採用例が出始めている。

### 障害時フォールバック

| 障害シナリオ | フォールバック設計 |
|------------|----------------|
| 回線断 | ストコンがローカル処理継続。回線復旧後に蓄積データ一括送信 |
| ストコン停止 | POSはマスタキャッシュで独立動作継続。翌日の代替機交換が標準 |
| 商品マスタ配信障害 | ストコンのローカルマスタで継続。差分の遅延配信で追いつく |
| 決済センター通信断 | 現金決済への誘導。一部チェーンはオフライン承認蓄積方式 |
| クラウド障害 | ルールベース自動発注（従来方式）に自動フォールバック |

**設計原則**: 必須機能（販売・現金決済・廃棄管理・温度監視）はローカル完結。オンライン依存機能（収納代行・チケット発券・電子マネー）は通信依存で停止許容。

## 3. 次世代ストコンの方向性

### エッジコンピューティング

**エッジで処理**: 決済、マスタキャッシュ、廃棄管理、温度監視、エッジAI推論
**クラウドで処理**: 需要予測、全社分析、マスタ管理、MDM、モデルトレーニング

**エッジAI実装パターン**: 棚カメラ + NVIDIA Jetson Orinでリアルタイム欠品検知 → ストコン画面にアラート。モデル更新はクラウドからOTA配信。

**エッジデバイス管理（MDM）**: 5万店×5台=25万台規模。AWS IoT Fleet Hub / Azure Device Update for IoT Hub。SIer視点では「最も見落とされがちな要件」。

### AI統合

| AI活用領域 | 成熟度（2025年） |
|-----------|---------------|
| 需要予測・AI発注 | 実用段階（3社とも本稼働） |
| 欠品・在庫可視化 | 一部店舗で実用段階 |
| 廃棄削減・ダイナミックプライシング | 試験〜実用段階 |
| シフト最適化 | 試験段階 |
| LLMによる発注支援・Q&A | 実証実験段階 |

**生成AI統合**: ファミマがAzure AI Foundry（GPT-4o）で従業員支援ツール実証。ローソンがKDDI連携でGemini API検討中。

### クラウドネイティブ化

**MACHアーキテクチャへの移行パス**:
- フェーズ1（〜2026年）: 分析・AI系のクラウドリフト
- フェーズ2（2026〜2028年）: 店舗端末のシン化・API化
- フェーズ3（2028〜2030年）: フルクラウドネイティブ（専用HW廃止）

エッジ側K8sとして**K3s**（軽量Kubernetes）の活用が注目。RAM 512MB〜1GBで動作。

## 4. SIer視点での示唆

### 見落とされがちなアーキテクチャ要件

| 要件 | 対応タイミング |
|------|-------------|
| EOBの時間分散設計（全店一斉のクラウド輻輳防止） | アーキテクチャ設計 |
| マスタ同期の原子性（POS複数台への一斉配信） | 詳細設計 |
| オフライン継続動作の明示的要件定義 | 要件定義 |
| エッジデバイスの5〜7年TCO算出 | 提案 |
| AI需要予測のMLOpsサイクル設計 | システム設計 |

### ベンダー競合マップ

| ベンダー | ポジション | SIerとしての関係 |
|---------|-----------|---------------|
| NEC | セブン主要ベンダー | 競合 or 協業 |
| 富士通 | ファミマ主要ベンダー | 競合 |
| 日本IBM | ローソン既存ベンダー | 競合・要注意 |
| 日立 | ローソン参画。OT統合に強み | 協業余地あり |
| NTTデータ | 横断的参画。PMO・セキュリティ | 協業余地大 |

### 差別化できる領域

- **MLOps基盤構築と運用**: AI需要予測モデルの継続的改善サイクル
- **エッジデバイス管理基盤**: MDM・OTA更新・遠隔監視の統合管理
- **データガバナンス・マスタ管理**: マルチシステム商品マスタのゴールデンレコード化
- **クラウド移行PMO**: 段階移行計画の策定・ベンダー調整
- **セキュリティ（ゼロトラスト化）**: エッジデバイス増加に伴うPCI DSS v4.0対応
