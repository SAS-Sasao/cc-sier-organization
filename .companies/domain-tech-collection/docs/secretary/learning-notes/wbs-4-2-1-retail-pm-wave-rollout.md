## 1. サマリー（3行以内）

コンビニの大規模ロールアウトは「一斉展開ではなくウェーブ展開が絶対条件」であり、10,000〜20,000店舗を数十回のウェーブに分割して段階的に切り替えるPM設計が必要となる。MW（ミドルウェア）はPOS連携・決済・発注EOBなど複数のサブシステムが絡み合うためベンダー責任分界の合意が最重要論点であり、移行中の縮退運転・ロールバック計画を事前に確立しておかないと本部/加盟店からの猛烈なクレームに直面する。24/365運用の制約から「ゼロダウンタイム切替」は技術的には実現可能でも、コンビニ業界特有の繁忙期凍結・FC加盟店の合意形成・下請法遵守が現実的な展開ペースを規定する最大のボトルネックとなる。

## 2. ウェーブ展開（Wave Rollout）

### 2-1. 定義と背景

**ウェーブ展開（Wave Rollout）**とは、全対象拠点を複数のグループ（ウェーブ）に分け、グループごとに順次システムを切り替えていく展開方式のことである。一斉切替（Big Bang）の対義語として用いられる。

#### なぜコンビニ/小売でウェーブ展開が必然なのか

コンビニチェーンがウェーブ展開を選ぶのは「好ましいから」ではなく「それ以外の選択肢が現実的でないから」である。以下の要因が積み重なっている。

| 要因 | 詳細 |
|------|------|
| **規模の絶対的な大きさ** | セブンイレブン約21,700店、ローソン約14,600店、ファミリーマート約16,700店（2024年時点）。一斉切替では人員・機材・回線の物理的確保が不可能 |
| **24時間営業の継続義務** | 一般的なシステム移行のように「週末にまとめて停止して切り替える」ことが原則不可能。営業時間帯外（深夜〜早朝）しか作業窓がない |
| **FC加盟店への配慮** | 法的には独立した事業者である加盟店オーナーへの合意形成・トレーニングは一定の時間が必要。一斉展開ではSVのサポート能力が追いつかない |
| **ロールバックリスクの分散** | 致命的な不具合発生時に全店が影響を受ける「全滅リスク」を避けるため、被害を1ウェーブ分に封じ込める |
| **物流・現場作業の平準化** | ハードウェア交換・設置作業が伴う場合、現場作業員の稼働を平準化しなければ展開不可能 |
| **本部サポートの限界** | ヘルプデスク・SVのサポート能力には上限がある。一斉に全店で問い合わせが発生すると機能不全に陥る |

コンビニのシステム移行事例では、概ね **1回のウェーブあたり100〜500店舗程度**を対象とし、全店展開完了まで **1〜3年**を要するケースが多い。日本全国のコンビニは約55,000店（2024年時点）のため、仮に1ウェーブ300店・月2回ペースで展開しても完了まで約7.6年かかる計算になる。実際は1ウェーブあたりの対象店舗数を増やしたり、サポート体制を強化して展開速度を上げる施策を組み合わせる。

#### 参考: 日本の主要コンビニチェーン店舗数推移

各社アニュアルレポートおよびIR資料によると:
- セブン-イレブン・ジャパン: 約21,700店（2024年度末）— [セブン&アイ・ホールディングス 統合レポート](https://www.7andi.com/ir/library/ar.html)
- ローソン: 約14,600店（2024年3月末）— [ローソン IR資料室](https://www.lawson.co.jp/company/ir/library/)
- ファミリーマート: 約16,700店（2024年度末）— [ファミリーマート IR情報](https://www.family.co.jp/company/ir.html)

上位3チェーンの合計でも約53,000店に達し、大規模ロールアウトの絶対規模が一般的な産業用SIと比べ桁違いであることがわかる。

### 2-2. 一般的なウェーブ分割ロジック

ウェーブをどのような基準でグループ分けするかは、プロジェクトの性質・リスク許容度・組織構造によって異なる。コンビニ/小売では以下のパターンが一般的に採用される。

#### パターン1: 地域軸によるウェーブ分割

最もシンプルで管理しやすいアプローチ。地理的に近い店舗を束ねることでSVの現場サポートが効率化される。

```
[地域軸ウェーブ展開の例]

Wave 1: Pilot（5〜10店舗）
  └── 特定エリアの直営店 or 協力加盟店
  └── 目的: 本番環境での動作確認・運用手順の確立

Wave 2: 地域A（関東・東京都心部）300店
  └── SVエリア単位で束ねる
  └── ヘルプデスク要員: +30名体制

Wave 3: 地域B（関東・郊外+東北）400店
  └── 地域Aの安定確認後に着手
  └── 教訓をロールアウトマニュアルに反映

Wave 4: 地域C（中部・東海）350店
  ：
Wave N: 最終（北海道・沖縄・離島等）150店
  └── 通信回線が特殊な離島等は最後のウェーブに配置
```

**地域軸の利点**: SVが担当エリアに集中できる、方言対応や地域慣習に配慮した店舗オペレーション指導が可能、交通・宿泊手配が効率的

**地域軸の落とし穴**: 本部データセンターとの通信レイテンシが地域によって異なる問題が後半ウェーブで出現することがある。また「先行地域は最新版、後発地域は旧版が動いている」という並行期間が長くなるほど、両バージョンの維持コストが増大する。

#### パターン2: FC vs 直営による分割

オーナーシップと意思決定構造が異なる直営店と加盟店を別ウェーブで扱う方式。

| ウェーブ属性 | 理由 |
|------------|------|
| **直営店先行** | 本部がオペレーションを完全にコントロールできる。問題発生時の意思決定が速い。「加盟店への説明材料」となる成功事例を先に作れる |
| **加盟店後続** | 直営での安定実績を元に加盟店オーナーへ説明。法的トラブル回避（加盟店契約上の義務を明確化してから展開） |

日本のコンビニ加盟店比率は概ね 直営 10〜15%、加盟店 85〜90% 程度（チェーンによる）。ローソンは特に直営比率が低く、加盟店オーナーへの丁寧な合意形成が不可欠。

#### パターン3: 売上規模・店舗種別による分割

```
大型店（都市型・24h・高売上）: Wave 1〜
  └── システム負荷が高く、問題が起きやすい。早期に実態を確認
ロードサイド店・郊外店: 中間ウェーブ
  └── 最も標準的なオペレーション。マニュアルのブラッシュアップ期
特殊形態店（高速SA・病院・空港内）: 最終ウェーブ
  └── 営業規制・常駐スタッフ制約等が複雑。個別対応が必要
```

#### パターン4: SVエリア（スーパーバイザー担当区域）軸

**現場作業の観点から最も実用的**とされるパターン。SVは担当エリアの加盟店オーナーと既に信頼関係があり、新システムの説明・トラブル対応での連絡が取りやすい。SVエリア単位でウェーブを組むことで、SVが担当エリア全店を連続してサポートできる。

SVは通常 7〜10店舗を担当するため、1エリア（25〜50SV担当分）= 175〜500店をまとめて1ウェーブにするイメージとなる。

#### パターン5: ハイブリッド（複数軸の組み合わせ）

実際のプロジェクトでは複数軸を組み合わせる。典型的な構成例:

```
Wave 0 (Pilot): 直営店から10店 → 検証・マニュアル確立
Wave 1: 関東A区（SVエリア10区） 直営+主要加盟店 250店
Wave 2: 関東B区+東北（SVエリア12区） 350店
  ：（以降、地域×SVエリア軸で展開）
```

### 2-3. PM論点

#### 論点1: ペース配分（展開速度の設計）

ウェーブの展開ペース（1ウェーブの店舗数・間隔）は以下のトレードオフから決定する。

| 速度を上げる要因 | 速度を下げる要因 |
|----------------|----------------|
| 並行運用コスト削減（旧バージョン維持費）| ヘルプデスクの対応能力 |
| 早期に全店展開完了して新機能を全体適用 | SVのサポート工数 |
| 旧ベンダーへの保守料金の早期解約 | 加盟店オーナーへのトレーニング時間 |
| 開発チームの集中期間を短縮 | ロールバック対応チームの体制維持 |

実務上は、**Pilot Wave の結果を受けて本展開のペースを確定**する。Pilot で予想外の問題が多発した場合は展開ペースを落とす（ステークホルダーへの再説明が必要）。

#### 論点2: パイロット店選定の基準

パイロット店の選定は「成功しやすい店を選んで早く終わらせたい」と「実際の代表的な条件を確認したい」の間のトレードオフ。

**パイロット店選定の原則**:
1. **直営店を最優先**: コントロールが効く。問題発生時の意思決定が速い
2. **「普通の店舗」を選ぶ**: 特殊形態店（高速SA店、病院内店等）は後回し。標準的な立地・売上・POSレジ台数の店舗が検証に適する
3. **協力的な加盟店オーナーを1〜2店含める**: 直営だけだと加盟店特有の問題（オーナーへの通知方法・夜間対応等）が見えない
4. **地理的なアクセスが良い店**: 問題発生時にSEを即座に現地に送れる首都圏を選ぶ
5. **通信環境が特殊でない店**: 光回線標準のシンプルな構成の店を選ぶ（特殊回線の問題は後回し）

#### 論点3: ロールバック計画

ウェーブ展開で最も軽視されがちで、最も重要な計画。**展開前に「何があったらロールバックするか」の判定基準（Go/No-Go criteria）を合意しておく**ことが必須。

ロールバック判定基準の例:
- 展開後 24時間以内に重大インシデント（売上計上不可・発注データ消失等）が3件以上発生
- 決済処理の失敗率が 0.1%（千件に1件）を超える状態が 30分以上継続
- 本部との通信断が 15分以上継続し、縮退運転に移行した店舗が展開対象店の 10%超
- 加盟店オーナーから「営業継続困難」申告が1件でも発生

ロールバックには技術的復旧（旧バージョンへの切戻し）と業務的復旧（手書き発注・電話精算への移行）の2段階がある。コンビニの場合、技術的ロールバックが困難な場合でも「業務的縮退」で最低限の営業継続ができる状態を維持することがより重要。

#### 論点4: 広告・POP連動と店舗オペレーション変更の同期

ストコン移行では「単なるITシステムの入れ替え」で終わらないことが多い。UI/UXの変更が発注業務フローに影響し、それが発注書・納品書のレイアウト変更、ひいてはベンダー配送票の変更にまで連鎖する。

PMとして管理すべき「非ITの連動作業」:
- 新システム画面の店舗内マニュアル更新（紙・ラミネート版）
- スタッフへの操作研修（オーナー研修・アルバイト向け簡易説明）
- 本部からの告知メール・SVからの事前説明のスケジューリング
- 発注フォーマット変更がある場合のベンダーへの事前通知（最低30日前）
- 決済端末の設定変更（クレジット・電子マネー）の同期

これらを「ITチームのみが管理するタスク」として扱うと、展開当日に店舗側が「聞いていない」「画面が違う」という混乱が起きる。**店舗運営部門・商品部門・物流部門との横断プロジェクト管理**が不可欠。

#### 論点5: 展開凍結期間の設計

コンビニには業界慣行として「システム変更禁止期間（フリーズ期間）」がある。年末年始（12月20日〜1月10日頃）とお盆（8月10日〜16日頃）は最大需要期であり、不具合発生時のリスクが極めて高い。これらの期間はウェーブ展開スケジュールから外すことが原則。

実質的に「展開できない期間」:
- 12月1日〜1月15日（年末年始）: 売上・需要のピーク
- 8月1日〜8月20日（お盆）: 夏季繁忙
- ゴールデンウィーク前後（4月25日〜5月10日）: 準繁忙
- 4月初旬（新入学・新年度）: 商品入れ替えが多い
- 月末・月初（25日〜5日）: 月次締め・給与支払後の需要増

1年のうち「展開可能な期間」は実質 7〜8ヶ月程度しかない。これをスケジュール計画の前提として最初から組み込まないと、全体工期が大幅に圧迫される。

## 3. MW（ミドルウェア）設計

### 3-1. コンビニ店舗のMWスタック全体像

コンビニ店舗のシステムは単一のアプリケーションではなく、複数のサブシステムがミドルウェアによって疎結合に連携する構造を持つ。これがストコン移行における「技術的複雑性の源」であり、PMにとっての最大の管理対象である。

```
[コンビニ店舗 MWスタック全体図]

━━━━━━━━━━━━━ 本部系（バックエンド） ━━━━━━━━━━━━━

本部基幹系                  本部通信サーバー
 ├── 商品マスタ管理           ├── JCA通信（EOB受信）
 ├── 需要予測・発注管理        ├── ISDN→VPN/Internet 移行
 ├── チャージ計算（OA）       ├── 流通BMS対応
 └── 日次集計・財務連携       └── EDI（メーカー/DC間）

              ↑↓ WAN（専用VPN or インターネットVPN）

━━━━━━━━━━━━━ 店舗系 ━━━━━━━━━━━━━━━━━━━━

[ストアコンピューター（ストコン）]
  │
  ├── POS連携MW（POSブリッジ/POSミドル）
  │    └── POS端末（各レジ台）からの売上データをリアルタイム収集
  │    └── インターフェース: SUNPOZ, JMA, StoreWorks等（POS機種依存）
  │
  ├── 決済連携MW（ペイメントゲートウェイ）
  │    └── クレジット（CAFIS経由）/ 電子マネー（Edy/Suica/nanaco/WAON）
  │    └── QRコード決済（PayPay/d払い/楽天Pay等）
  │    └── インターフェース: PCI DSS準拠の専用プロトコル
  │
  ├── 発注EOBミドルウェア（End of Business）
  │    └── 閉店後の発注データ集計・本部送信
  │    └── JCA手順（旧来）→ 流通BMS/Web-EDI（移行中）
  │    └── タイムアウト管理・再送制御が重要
  │
  ├── EAI/ESB（Enterprise Application Integration）
  │    └── 複数サブシステム間のデータ変換・ルーティング
  │    └── 例: TIBCO EMS, IBM MQ, ActiveMQ, Amazon SQS（クラウド型）
  │
  ├── 在庫管理連携（検品・棚卸）
  │    └── ハンディスキャナーとの通信
  │    └── 検品データ自動照合ロジック
  │
  └── スタッフ管理連携
       └── シフト管理・タイムカード連携
       └── 労働時間集計→本部給与システムへ
```

#### MWの「見えにくさ」問題

コンビニ店舗のMWが移行プロジェクトで問題化しやすい理由は、**MW自体が長年に渡って「暗黙の前提」として積み重なってきた**ためである。典型的な状況:

- 20年前に導入したJCA手順を使った発注MW が今もそのまま稼働
- MWのバージョンが古く、ソースコードの存在すら不明な状態
- ベンダーが廃業・合併しており、保守担当者が社内に1名しかいない
- 仕様書が存在せず、動作実績のみが「仕様」として機能している

これを適切に把握しないまま上位層（ストコンアプリ）だけを移行しようとすると、移行後に「MW経由のデータが届かなくなった」「決済MWとの接続が切れた」という問題が発生する。

### 3-2. 24/365運用を前提とするMW要件

#### 要件1: 冷結（コールドスタンバイ）・温結（ウォームスタンバイ）・縮退運転

| モード | 定義 | 適用場面 |
|--------|------|---------|
| **通常運転** | 全MWが稼働、本部連携も正常 | 平常時 |
| **縮退運転（ローカルモード）** | 本部との通信が途絶しても、店舗単体での発注・POS・決済が継続 | WAN障害時（1日数時間程度） |
| **温結スタンバイ** | バックアップサーバーがアクティブ状態で待機。フェイルオーバーが数十秒〜数分で完了 | 主系サーバー障害時 |
| **冷結スタンバイ** | バックアップサーバーは停止状態。手動起動が必要（通常15〜30分） | コスト削減が優先の場合 |

コンビニ店舗では、通常「温結＋縮退運転」の組み合わせを採用する。フルスタンバイ（ホットスタンバイ）は コスト対効果の観点から採用されないケースが多い。

縮退運転のポイント:
- 縮退時でも **POS売上（現金・クレジット・電子マネー）は継続**できること
- 縮退時の発注データは**ローカルにバッファリング**し、通信回復後に一括送信
- 縮退モードへの切替・復帰は自動検知と手動確認の二段階とする（誤検知による不要な縮退を防ぐ）

#### 要件2: 順序保証とメッセージキューの設計

コンビニのMWでは「メッセージの順序が保証されること」が多くの業務に影響する。

| 業務 | 順序保証が必要な理由 |
|------|------------------|
| 発注データ | 「発注→取消→再発注」が逆順で処理されると在庫過多・欠品に直結 |
| 決済取消 | 「決済→取消」が逆順になると二重支払い・残高不整合 |
| 在庫変動 | 「入庫→売上→廃棄」の順序がずれると在庫マイナスが発生 |
| 精算データ | 「本日分→翌日分」のバッチが逆順で処理されると月次集計が狂う |

クラウド移行後のMWとして Amazon SQS FIFO キューや Kafka を利用する場合でも、**順序保証の要件を事前に棚卸ししておく**ことが設計ミスを防ぐ。特に Kafka の場合はパーティション設計が順序保証の鍵となる。

#### 要件3: リトライ設計と二重起動防止

ネットワーク不安定時のMW動作設計として以下の2点が必須:

**リトライ設計**:
- 最大リトライ回数（例: 3回）と間隔（指数バックオフ: 30秒、60秒、120秒）を明示的に定義
- タイムアウト値を業務の重要度に応じて設定（決済 < 3秒、発注データ < 30秒）
- デッドレターキュー（DLQ）によるエラーデータの隔離と手動再処理フロー

**二重起動防止（冪等性の確保）**:
- バッチ処理（発注EOBなど）が重複実行されないよう、ロック機構またはユニークな処理IDによる冪等性チェックが必須
- 特に閉店処理バッチ（日次サマリー送信）が障害後の再起動で二重実行されると、発注量が2倍になるという致命的バグが発生する

#### 参考: 流通EDI標準

GS1 Japan（旧・流通システム開発センター）が推進する流通業界EDI標準「流通BMS」について:
- [GS1 Japan — 流通BMS 概要](https://www.gs1jp.org/standard/edi/ryutsu-bms.html): コンビニ・スーパー・メーカー間のEDIメッセージ標準
- [GS1 Japan — EDI（電子データ交換）](https://www.gs1jp.org/standard/edi/): 流通業界全体のEDI推進情報

旧来のJCA手順（テキスト固定長フォーマット + ISDN/VPN通信）からWeb-EDI（流通BMS/XML形式 + Internet VPN）への移行が進んでいる。ストコン移行プロジェクトでは、この**EDI移行**をシステム移行と同時に行うのか、別フェーズとするのかの判断が重要なスコープ論点となる。

### 3-3. PM論点

#### 論点1: MWベンダー複数混在での責任分界点

コンビニ店舗のMWは歴史的経緯から複数ベンダーの製品が混在していることが多い。典型的な構成例:

```
[責任分界点の典型的構成]

ベンダーA（ストコンベンダー）
  担当: ストコンアプリ + EAI/ESB製品
  責任範囲: ストコン↔EAI間のインターフェース

ベンダーB（POSベンダー）
  担当: POSシステム + POS-MW間プロトコル
  責任範囲: POS端末↔POSブリッジMW間

ベンダーC（決済代行会社）
  担当: ペイメントゲートウェイ
  責任範囲: 店舗決済MW↔クレジット/電マネネットワーク間

ベンダーD（通信インフラ）
  担当: WAN回線（VPN）
  責任範囲: 店舗〜本部間の通信品質

ベンダーE（新規クラウドSIer）
  担当: AWS基盤 + 移行後のクラウドMW
  責任範囲: クラウドMW↔既存MW間アダプター
```

**PMが必ず押さえる責任分界点の合意内容**:
1. インターフェース仕様書（IFD: Interface Definition）の共同策定・相互サイン
2. 障害発生時の「誰がファーストコンタクト」かのエスカレーションフロー
3. 接続試験（結合テスト）の費用・工数の分担
4. 証跡ログの取得範囲と相互開示範囲（セキュリティ上開示できないログ範囲も事前合意）

#### 論点2: ライセンス方式と移行コスト

オンプレMWからクラウドネイティブMWへの移行では、ライセンスコストが大幅に変動する。

| MW種別 | オンプレライセンス | クラウド移行後 | 移行時の注意点 |
|--------|-----------------|-------------|-------------|
| メッセージブローカー（IBM MQ等）| CPUコア数×年間費 | Amazon MQ / SQS（使用量課金）| オンプレは高額年間ライセンス。クラウドは低コストだが初期移行でMQ設定の再構築が必要 |
| EAI/ESB（TIBCO等）| サーバー台数×年間費 | AWS Step Functions / EventBridge | 既存フロー定義の移植が必要。TIBCO→EventBridgeは定義言語が全く異なる |
| データベース接続MW | 接続プール数課金 | RDS Proxy（AWSマネージド）| ほぼ自動化可能だが、オンプレDB特有の設定（Oracle hint等）は引継が必要 |

#### 論点3: 運用引継ぎとサポート窓口SLA

MWは「動いていて当然」とされがちで、運用引継ぎが後回しになるリスクがある。PMとして確認すべき項目:

- MWの監視アラート設計（閾値・通知先）は誰が設定するか
- 運用手順書（OS再起動後のMW起動シーケンス等）は誰が作成するか
- MWベンダーの保守SLA（障害対応開始時間・解決目標時間）の確認
- 保守期限切れMWがある場合の対応方針（延長サポート購入 or 移行加速）
- 夜間・休日対応の体制（MWベンダーのオンコール体制の確認）

### 3-4. レガシーMW→クラウドネイティブMW移行時の互換性課題

#### 課題1: JCA手順の廃止問題

JCA手順（旧・日本チェーンストア協会が策定したEDIプロトコル）は1970〜80年代に策定された通信プロトコルで、現在も多くのコンビニ・スーパーの発注データ通信基盤として現役稼働している。

[日本チェーンストア協会 — 協会の案内](https://www.jcsa.gr.jp/) によると、JCA手順の後継として流通BMSへの移行を業界全体で推進しているが、移行は容易ではない。

JCA手順の特徴と廃止問題:
- **固定長テキストフォーマット**: XMLやJSONと全く異なる古い形式
- **ISDN/VPN前提の設計**: インターネットVPN上での動作に改修が必要
- **相手方（メーカー/DC）側の対応が必要**: 自社だけ移行しても相手が旧JCA手順のままでは接続できない
- **移行コスト**: 大手メーカーは対応済みが多いが、中小サプライヤーは未対応が多い

**PM判断**: JCA→流通BMS移行をストコン移行と同時に行うか、別フェーズとするかは初期スコープ合意で明確化必須。スコープを切り分ける場合は、移行期間中の「両プロトコル並行運用」の期間とコストを計画に含める。

#### 課題2: 決済MWのPCI DSS準拠継続

決済MWは **[PCI DSS（Payment Card Industry Data Security Standard）](https://www.pcisecuritystandards.org/)** の準拠が義務付けられている。クラウド移行で決済MWが新バージョンになる場合:

- PCI DSS準拠のスコープが変わるため、QSA（Qualified Security Assessor）による再審査が必要
- AWS環境でPCI DSS準拠を達成するためのAWSのガイドライン: [AWS PCI DSS Compliance](https://aws.amazon.com/jp/compliance/pci-dss-level-1-faqs/)
- 審査には通常 3〜6ヶ月を要し、本番稼働前に完了させる必要がある

#### 課題3: レガシーMWの「知識の空白」

長年稼働してきたMWは、開発者・運用者の退職・異動により「なぜこうなっているのかを知っている人が誰もいない」状態になっていることが多い。移行プロジェクト開始時に「現行MWの仕様理解フェーズ」を独立したフェーズとして設けることが重要。

具体的アクション:
1. 現行MWベンダーのキーパーソンへのヒアリング（退職予定者がいれば優先）
2. ログ解析による実際のメッセージフロー把握（仕様書より実ログが正確）
3. 「動いているが誰も意味を知らない処理」のリスト化
4. ネットワークトレース（Wireshark等）による通信の実態把握

## 4. 24/365運用

### 4-1. コンビニの運用制約

コンビニの運用が「24/365」である、という事実は一般的なSI案件との差を生む。具体的な制約を以下に整理する。

#### 制約1: 計画メンテナンス窓がほぼ存在しない

| 業態 | メンテナンス窓 |
|------|-------------|
| 一般企業基幹システム | 毎週土日夜間 + 年2回長期メンテ |
| 銀行・証券 | 休日深夜 3〜6時 + 四半期定期 |
| EC サイト | 深夜 2〜5時（最低限） |
| **コンビニ店舗** | **原則ゼロ**（閉店しない、売上発生を止められない） |

コンビニにおける「作業可能時間帯」は店舗での人的作業（ハードウェア交換など）が可能な「来客の少ない時間帯」（深夜 2〜5時）に限定される。ただしシステム自体は停止できない。

**実務上の対応**:
- ソフトウェアのリリースは「ローリングリリース（1台ずつ）」か「Blue-Green（別環境へ切替）」で進める
- ハードウェア交換は店舗が最も空いている深夜帯に現地作業員を配置
- 深夜作業の場合、労働基準法の深夜割増賃金（22時〜翌5時は25%増）が発生する

#### 制約2: 「売上」は止められない

コンビニ店舗の運営で「絶対に止めてはいけない処理」は以下の優先順位で定義される:

```
優先度1（絶対死守）: POSレジでの販売・決済処理
優先度2（死守）: 電子マネー・クレジットカード決済の処理
優先度3（重要）: 発注データの送信（閉店後のEOBバッチ）
優先度4（重要）: 売上データの本部送信（日次集計）
優先度5（通常）: 在庫管理・検品データの更新
優先度6（後回し可）: スタッフ管理・シフト連携
```

「優先度1〜2が停止した場合」はコンビニの業務が実質停止するため、最大限の冗長化設計が必要。縮退運転でも「現金での販売」が継続できる状態を最低ラインとする。

### 4-2. 運用窓の確保方法

#### 手法1: Blue-Green デプロイメント

2つの本番環境（Blue: 現行稼働、Green: 次バージョン）を用意し、切替えを瞬時に行う手法。

```
[Blue-Green の切替フロー]

移行前: 全トラフィック → Blue環境（旧バージョン）

切替時:
  1. Green環境でアプリケーション起動・ウォームアップ完了を確認
  2. ロードバランサーのルーティングをGreen向けに切替（数秒〜数十秒）
  3. BlueをStandby状態で保持（問題発生時のロールバック先）

問題発生時:
  4. ロードバランサーをBlueに切戻し（数秒で復元）
```

**コンビニ適用時の注意**: 店舗側ストコンのBlue-Greenはハードウェア制約上困難。クラウド（本部側）でBlue-Greenを採用し、店舗はAPIを叩く形（バージョン切替はAPIのインターフェースで吸収）とする設計が現実的。

#### 手法2: Feature Flag（機能フラグ）によるロールアウト制御

新機能を「フラグ」によって段階的に有効化する手法。ウェーブ展開と組み合わせることで精細なコントロールが可能。

```
例: 新発注画面の段階展開

Wave 1 店舗: feature_flag.new_order_ui = true  → 新画面表示
その他の店舗: feature_flag.new_order_ui = false → 旧画面のまま

問題発生時:
  feature_flag.new_order_ui = false に一括設定 → 全店で即座に旧画面に戻る
```

Feature Flagの管理には AWS AppConfig（[AWS AppConfig Overview](https://docs.aws.amazon.com/appconfig/latest/userguide/what-is-appconfig.html)）等のマネージドサービスが利用可能。コンビニ規模では「どの店舗グループにどのフラグを適用するか」のグルーピング管理が重要。

#### 手法3: DB オンラインマイグレーション

DBスキーマ変更を「全サービス停止なし」で行うための段階的手法。

```
[Expand/Contract パターン]

Step 1 (Expand): 旧スキーマ + 新カラムを両方持つ状態でデプロイ
  → 旧アプリ・新アプリの両方が動作可能

Step 2 (Data Migration): バックグラウンドで旧データを新カラムに移行
  → 業務を止めずに実施（数時間〜数日かかる場合も）

Step 3 (Contract): 旧カラム・旧テーブルを削除
  → 全店で新バージョンが安定後に実施
```

コンビニのDBは数万店の日次トランザクションを集約するため、DBマイグレーション中の性能影響（Lock待ち、CPU上昇）を必ず事前試験で確認する。[AWS Database Migration Service (DMS)](https://aws.amazon.com/jp/dms/) を活用したオンラインマイグレーションが有効。

### 4-3. インシデント対応体制

#### インシデント重篤度の分類

コンビニシステム特有の重篤度分類（一般的なSIと異なる点は「売上への直接影響」を最高重篤度とする点）:

| 重篤度 | 定義 | 対応時間目標 | 例 |
|--------|------|------------|---|
| **P1（緊急）** | 売上・決済が停止。複数店舗に影響 | 初動 15分以内、暫定対処 1時間以内 | 決済ゲートウェイダウン、発注システム全停止 |
| **P2（重大）** | 単一店舗での売上停止、または複数店舗でのデータ不整合 | 初動 30分以内、暫定対処 2時間以内 | 1店舗でのPOS-ストコン通信断 |
| **P3（中）** | 業務継続は可能だが、オペレーション効率が低下 | 業務時間内での解消 | 発注推奨機能の表示エラー |
| **P4（低）** | 軽微な表示ミス・機能制限。業務影響なし | 次回定期リリースで対処 | 画面レイアウトの崩れ |

#### コンビニ特有の対応体制構造

```
[インシデント対応フロー]

店舗スタッフ
  ↓ 気づいた問題を店長へ報告
店長
  ↓ 担当SVへ電話/チャット
スーパーバイザー（SV）
  ↓ ストコンの遠隔監視画面確認 → P2以上ならエスカレーション
店舗ヘルプデスク（24h）
  ↓ ログ確認・リモート操作による切り分け
SIer 一次サポート
  ↓ 解決できなければ
SIer 二次サポート / MWベンダー
```

**コンビニ特有の難しさ**: 深夜 2時に全国で同時に問題が発生した場合、数百〜数千のSVが一斉にヘルプデスクに電話する「電話殺到」状態が発生する。これに対応するための「ステータスページ（インシデント情報一括公開）」の整備が重要。SVが「今どうなっているか」をリアルタイムで確認できる仕組みがないと、電話が殺到して対応チームが機能不全に陥る。

#### FC加盟店への対応

直営店と異なり、加盟店オーナーはシステムのトラブルを「自分のビジネスへの損失」として受け取る。インシデント対応では:

- 加盟店オーナーへの状況説明: **SVが担当する**（本部やSIerが直接加盟店に連絡しない）
- 復旧見込み時間のコミュニケーション: 不確実でも「現在調査中、30分後に更新します」という定期更新が重要
- 売上損失の補填基準: あらかじめ本部のポリシーとして確立しておく（システム障害による売上損失に対して本部がどこまで補填するか）

### 4-4. 月次・日次・時次の運用カレンダー

コンビニシステムの運用は「時間との戦い」である。業務プロセスがどの時間帯に走るかを把握していないと、メンテナンス作業やデプロイタイミングを誤る。

#### 時次の処理（1時間以内の定期処理）

| 処理 | タイミング | 内容 |
|------|---------|------|
| POSデータ集計 | 毎時 | POSレジからの売上データをストコンに集約 |
| 在庫アラート確認 | 毎時 | 品切れ・低在庫商品の自動発注トリガー |
| 決済バッチ | 毎1〜2時間 | クレジット・電子マネーの承認データ照合 |
| リアルタイム発注 | 随時（手動も可） | 品切れ時の即時補充発注 |

#### 日次の処理（毎日1回）

| 処理 | タイミング | 内容 |
|------|---------|------|
| **EOBバッチ（閉店後発注）** | 通常 21〜23時頃（店舗・商品によって異なる） | 1日の発注データを集計して本部/DC/ベンダーへ送信。最重要バッチ |
| **日次精算（POS閉店処理）** | 深夜 1〜3時頃 | POSデータを確定。売上合計・チャージ計算の基準データ |
| **本部データ送信** | 深夜 2〜5時 | 売上サマリー・在庫データを本部基幹システムへ送信 |
| **マスタ更新受信** | 深夜 3〜6時 | 本部からの商品マスタ更新データを受信・適用 |
| **ログバックアップ** | 深夜 3〜6時 | 業務ログ・トランザクションログをオフサイトに保存 |

**EOBバッチの重要性**: コンビニ店舗でもっとも重要なバッチ処理。このバッチが失敗すると翌日の商品が入荷せず、欠品・廃棄過多・売上損失に直結する。EOBバッチの成功を監視する専用のアラートと、失敗時の緊急対応フロー（手動発注への切替え、DC側への口頭連絡等）を整備しておくことが必須。

#### 月次の処理

| 処理 | タイミング | 内容 |
|------|---------|------|
| **月次締め処理** | 月末深夜〜月初早朝 | 売上・仕入・在庫の月次集計。会計データ確定 |
| **チャージ計算** | 月末翌月初（5日頃まで）| OA（オープンアカウント）の月次精算計算 |
| **棚卸** | 月1回（店舗ごとに設定）| 実在庫と帳簿在庫の照合。差異がある場合は調査 |
| **スタッフ給与データ確定** | 月末〜月初 | 勤怠データ集計・給与計算システムへの連携 |
| **システムパッチ適用** | 月初〜月中の平日 | セキュリティパッチ・定期アップデートの適用 |

**月次締め処理の注意**: 月末月初（25日〜5日）はこの処理が集中するため、システムメンテナンス・デプロイは原則禁止。特に月初3〜5日は月次集計の最終確認期間であり、この時期にインシデントが発生すると財務処理に影響する。

## 5. コンビニ/小売業界特有のPM落とし穴

### 落とし穴1: 本部×加盟店（FC）構造での意思決定遅延

コンビニのFCモデルでは「加盟店オーナーは法的に独立した事業者」である。本部のシステム変更に加盟店が同意しない場合、強制的な展開はFC契約上の問題を引き起こす可能性がある。

**典型的な遅延パターン**:
- 新システムの操作性変更について加盟店オーナーが反発
- 「研修が足りない」「聞いていない」という抗議が一部店舗から起き、SV経由で展開に待ったがかかる
- 本部法務が「加盟店への合意形成」を要求し、スケジュールが数ヶ月後退

**対策**:
- 展開前の加盟店代表者（FC組合等）への説明・合意形成プロセスをスケジュールに明示的に組み込む（2〜3ヶ月の余裕が必要）
- 直営店での成功事例と「加盟店にとってのベネフィット」を具体的な数値で説明する資料を事前に準備
- 苦情ルート（SV → 本部 → プロジェクト）を整備し、加盟店からのフィードバックをプロジェクトに反映する仕組みを作る

### 落とし穴2: ベンダー多重構造での責任の「すき間」

コンビニのストコン移行はほぼ必ず「SIer × MWベンダー × ハードウェアベンダー × 通信ベンダー × 決済代行会社」という多重請負構造になる。

**すき間の典型例**:
- 「ストコンアプリとMWの接続不具合」について、SIerとMWベンダーが互いに「相手側の問題」と主張
- 「ハードウェアの設置作業ミス」について、現場作業会社が「仕様と違う」と言い、ハードウェアベンダーが「仕様通り」と主張
- 通信回線の遅延について、通信ベンダーが「SLA内」、SIerが「業務要件を満たしていない」と主張

**対策**:
1. **責任分界点（DIV: Demarcation In View）マトリクス**を全ベンダーと合意してプロジェクト開始時にサインオフ
2. 接続試験（IT: Integration Test）は全ベンダーが参加する「合同テスト」を設ける
3. インシデント発生時の「最初の調査担当」を仕様書で事前に決めておく（誰もが「自分の担当外」と言えないようにする）
4. 主ベンダー（SIer）に「全体調整責任」を明示的に付与し、サブコン間の問題はSIerが調停する仕組みを作る

### 落とし穴3: 法規制への対応

コンビニシステム移行では、以下の法規制への対応が「PM管理領域」に入る:

| 法規制 | 影響 |
|--------|------|
| **食品衛生法** | 温度管理データの記録・保存義務。ストコンの温度管理機能が移行後も正常動作することの証明が必要 |
| **割賦販売法・資金決済法** | 決済データの保護・ログ保存義務。クラウド移行後も同等の保護が担保されることをシステム監査で証明 |
| **景品表示法** | POSシステムでの値引き・ポイント計算が正確に行われることの保証 |
| **労働基準法（深夜作業規制）** | 店舗での深夜作業は25%割増賃金。夜間作業計画策定時にコストに計上 |
| **個人情報保護法** | 電子マネー・クレジットの顧客情報保護。クラウド移行後のデータ所在地確認 |

### 落とし穴4: 下請法・公正取引委員会

大規模IT発注では公正取引委員会の指針が適用される。[公正取引委員会 — 下請法の概要](https://www.jftc.go.jp/shitauke/index.html) によると:

- 発注書面を作成していない状態での作業開始は「下請法違反のリスク」
- 大幅なスコープ削減を後から追加費用なしで要求することは「不当な経済上の利益の提供要請」に該当する可能性
- PMとして「変更要件のリクエスト→見積もり→合意→発注書→実施」のサイクルを文書で管理する必要がある

コンビニの大規模プロジェクトでは「本部の担当者が口頭で追加要件を依頼する」という文化が根強く残る場合がある。これを口頭で引き受け続けると、下請法上の問題と同時に「際限ないスコープクリープ」が発生する。

### 落とし穴5: 業界慣行（フリーズ期間・展開禁止）

前述の通り、以下のフリーズ期間はスケジュール作成の前提として組み込む（プロジェクト開始時に確認・合意が必要）:

```
展開禁止期間 (業界慣行)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
12月1日〜1月15日   年末年始（全てのシステム変更禁止）
4月25日〜5月10日   ゴールデンウィーク（準禁止）
8月1日〜8月20日    夏季繁忙（準禁止）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

```

追加で「チェーン固有の禁止期間」が存在する場合もある（新商品大量投入時期・棚替え大規模実施時期等）。PMはプロジェクト開始直後にこれらの制約を確認し、マスタースケジュールに「グレーアウト期間」として明示する。

### 6-3. 想定リスク TOP5 と対策

**リスク1: EOBバッチの移行期間中の二重送信・欠落**

| 項目 | 内容 |
|------|------|
| 発生シナリオ | 切替当日、旧ストコンと新ストコンが同時にEOBを送信する、またはどちらも送信しない |
| 影響 | 翌日入荷量が2倍になる（在庫過多・廃棄増）または入荷が発生しない（品切れ） |
| 確率 | 高（EOBの二重送信は過去事例でも多数報告） |
| 対策 | 切替日当日の「発注データ送信ロック機構」を実装。新ストコン起動直後は自動発注を無効化し、確認後に手動で有効化するフローを確立 |
| 関連: | 「[WBS 5.1 コンビニサプライチェーン](wbs-5-1-convenience-supply-chain.md)」の発注データフローも参照 |

**リスク2: FC加盟店オーナーの反発による展開遅延**

| 項目 | 内容 |
|------|------|
| 発生シナリオ | 新システムの操作性変更・トレーニング不足に加盟店オーナーが反発。FC組合が集団で本部に「展開の一時停止」を要求 |
| 影響 | 展開スケジュールが数ヶ月〜半年後退 |
| 確率 | 中（直営先行で成功事例を作り、説明を丁寧に行えば軽減可能） |
| 対策 | 直営Wave完了時点でFC組合向け説明会を開催。展開前に「30日間トレーニング保証」を提供。加盟店向け操作マニュアルはSVが作成に関与（現場言葉で書く） |

**リスク3: 決済MWの接続不具合による売上停止**

| 項目 | 内容 |
|------|------|
| 発生シナリオ | 新ストコン→ペイメントゲートウェイのプロトコル変更部分で接続不具合が発生。クレジット・電子マネー決済が停止 |
| 影響 | 売上の大幅減少（コンビニの現金以外決済比率は60〜70%） |
| 確率 | 中（接続試験で十分に検証していれば低下） |
| 対策 | Wave 0（Pilot）で全決済パターンを徹底的にテスト。PCI DSS再審査を本番前に完了。決済停止時の「現金のみ受付」への縮退手順を店舗スタッフに周知 |
| 参考: | [PCI Security Standards Council — PCI DSS v4.0](https://www.pcisecuritystandards.org/document_library/) |

**リスク4: 年末年始前の展開強行による繁忙期障害**

| 項目 | 内容 |
|------|------|
| 発生シナリオ | スケジュール遅延のプレッシャーにより「年末ギリギリまで展開」を試みて、12月中旬に問題が発生。年末年始の最繁忙期に障害対応を迫られる |
| 影響 | 12月の売上損失（コンビニは1年で最大の需要期）+ 対応チームの疲弊 |
| 確率 | 高（スケジュール圧迫時にPMが追いつめられるケースが多い） |
| 対策 | **12月1日を絶対的な展開停止ラインとして合意書に明記**。11月末時点で展開中のウェーブがある場合は強制停止して年明けまで待機するポリシーを確立 |

**リスク5: 旧ベンダー（現行ストコンベンダー）の非協力**

| 項目 | 内容 |
|------|------|
| 発生シナリオ | 現行ストコンを担当する旧ベンダーが、移行後の契約がなくなることを理由に仕様開示・移行支援に消極的になる |
| 影響 | 現行システムの「暗黙仕様」が不明なまま移行作業が進み、移行後に予期しない不具合が多発 |
| 確率 | 高（ほぼ全ての大規模移行プロジェクトで発生） |
| 対策 | 移行期間中の「現行ベンダーへの保守継続契約」を延長（経済的インセンティブを維持）。仕様調査フェーズを独立したフェーズとして設け、現行ベンダーに有償で仕様書作成・ヒアリング対応を依頼。契約上の情報開示義務を最初の契約に明記 |

    → 現状・リスク・計画を経営層・本部IT部門に報告
    → 追加リソース・予算の必要性を早期に提示
