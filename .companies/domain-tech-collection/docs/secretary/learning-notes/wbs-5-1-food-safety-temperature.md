## 1. 食品安全規制の体系と法的根拠

### 1-1. コンビニを規律する主要法令

コンビニの食品安全管理に関係する法令・基準は複数存在し、それぞれが異なる視点から規制する。

| 法令・基準 | 所管省庁 | コンビニへの主な影響 |
|-----------|---------|-------------------|
| **食品衛生法（昭和22年法律第233号）** | 厚生労働省 / 消費者庁 | HACCP 義務化の根拠法。施設・設備・衛生管理の最低基準 |
| **食品表示法（平成25年法律第70号）** | 消費者庁 | 消費期限・賞味期限・アレルゲン表示の義務化 |
| **食品安全基本法（平成15年法律第48号）** | 食品安全委員会 | フードチェーン全体のリスク評価・管理の基本方針 |
| **JAS 法（農林物資の規格化等に関する法律）** | 農林水産省 | 食品品質規格・表示の標準化 |
| **廃棄物の処理及び清掃に関する法律** | 環境省 | 食品廃棄物（フードロス）の適正処分義務 |
| **小売業に係る食品衛生管理の手引き** | 厚生労働省（通知） | コンビニ・スーパー等の HACCP 実施に向けた業種別ガイドライン |

### 1-2. 食品衛生法の改正（2018年・2021年）

2018年の食品衛生法大改正は、日本の食品安全管理に構造的な変化をもたらした。

```
【2018年改正の主要ポイント】

1. HACCP に沿った衛生管理の制度化（第51条の2）
   - 食品等事業者（コンビニを含む）へのHACCP義務化が法的根拠に
   - 施行: 2020年6月1日（1年間の経過措置あり）
   - 完全施行: 2021年6月1日

2. 特定食品等の輸出入規制強化
3. 乳製品・水産食品の衛生基準刷新
4. FSSC22000等の食品安全規格との整合化

【コンビニへの実務的影響】
コンビニ本部・加盟店 → 「小規模な営業者等」として
「HACCPの考え方を取り入れた衛生管理」（旧A類型）に準拠が必要
（HACCPのフルプランとは区別されるが、温度管理記録は必須）
```

### 1-3. 消費期限と賞味期限の法的定義

食品表示法・食品表示基準（内閣府令）により、以下が義務付けられる。

| 区分 | 法的定義 | コンビニでの主な対象商品 | 表示義務 |
|------|---------|----------------------|---------|
| **消費期限** | 定められた方法で保存した場合、食べても安全な期限（おおむね5日以内を目安）| 弁当・おにぎり・惣菜・サンドウィッチ・生菓子 | 義務（年月日で表示） |
| **賞味期限** | 定められた方法で保存した場合、品質が十分に保たれる期限 | 乳製品・ドリンク・菓子・加工食品 | 義務（3ヶ月超は年月のみ可） |

**重要**: 消費期限の食品（弁当等）は、期限切れで販売すると**食品衛生法違反**。コンビニ店舗における消費期限管理の徹底はコンプライアンス要件である。

## 2. HACCP 義務化とコンビニへの影響

### 2-1. HACCP とは

HACCP（Hazard Analysis and Critical Control Points：ハサップ）は、食品の製造・加工・調理・販売の各プロセスにおいて、食品安全上の危害要因（生物的・化学的・物理的）を科学的に分析し、重要管理点（CCP）を設定して連続的に監視・記録する食品安全管理手法。

```
【HACCP の7原則12手順（国際規格 Codex 委員会）】

手順1:  HACCP チームの編成
手順2:  製品の記述（原材料・添加物・包装等の確認）
手順3:  意図する用途の確認（誰が食べるか）
手順4:  工程フロー図の作成
手順5:  工程フロー図の現場確認

原則1（手順6）: 危害要因分析（HA）
原則2（手順7）: 重要管理点（CCP）の決定
原則3（手順8）: 管理基準（CL: Critical Limit）の設定
原則4（手順9）: モニタリング方法の設定    ← 温度ログ収集が該当
原則5（手順10）: 改善措置の設定           ← 温度逸脱アラートが該当
原則6（手順11）: 検証方法の設定           ← 定期記録レビューが該当
原則7（手順12）: 記録・文書管理           ← ストコン温度ログ保管が該当
```

### 2-2. コンビニに適用される HACCP の類型

2021年6月の完全施行において、食品事業者はその規模・業態に応じて2種類の対応が求められる。

| 類型 | 対象 | 求められる対応 | コンビニへの適用 |
|------|------|-------------|----------------|
| **HACCP に基づく衛生管理** | 大規模事業者、と畜場、食鳥処理場等 | 7原則12手順のフルHACCPプラン | コンビニ本部（食品製造・加工部門）に適用される場合あり |
| **HACCP の考え方を取り入れた衛生管理** | 小規模事業者（従業員50人未満の食品製造業、コンビニ店舗等） | 業種別手引書に基づく衛生管理の実施・記録 | **コンビニ各店舗に適用**（主体） |

**コンビニ店舗における実務的要求事項**（「小売業に係る食品衛生管理の手引き」より）:

```
【実施・記録が求められる事項】

1. 施設・設備の衛生管理
   - 冷蔵・冷凍設備の温度管理と記録（最重要）
   - 厨房・調理エリアの清掃・消毒記録

2. 温度管理の記録（原則として毎日実施）
   - 冷蔵庫・冷凍庫の設定温度と実測温度の記録
   - フライヤー・ホット什器の加熱温度記録
   - 入荷商品の品温確認記録

3. 従業員の衛生管理
   - 健康チェック記録（手洗い実施、体調不良者の排除）
   - 衛生教育の実施記録

4. 原材料の衛生管理
   - 仕入先の確認記録
   - 消費期限・賞味期限の確認記録

【記録の保存期間】
食品衛生法に明確な期限規定はないが、厚生労働省ガイドラインは
「製品の賞味期限を考慮した適切な期間」と規定。
業界標準として、コンビニ各チェーンは概ね1〜2年間の保存を
本部規定で義務付けているケースが多い。
```

### 2-3. 2021年6月完全施行がコンビニにもたらした変化

| 変化の領域 | 施行前（2021年5月以前） | 施行後（2021年6月以降） |
|-----------|----------------------|----------------------|
| 温度記録の方法 | 手書き台帳が主流 | 自動記録（センサー連携）への移行が加速 |
| 記録の管理 | 店舗内保管（紙） | 電子化・本部へのリアルタイム送信が標準化 |
| アラート対応 | 店員の目視発見が主体 | ストコン連携による自動アラートへ移行 |
| 本部の管理 | 抜き打ち実地確認 | リモート監視・ダッシュボード管理が可能に |
| 監査対応 | 紙台帳の提出 | 電子記録のエクスポート・提出 |

**実務上の転換点**: HACCP 義務化を契機に、従来「任意対応」だったセンサー自動記録・クラウド連携が「事実上の標準」として普及。特に大手チェーンでは全国店舗への展開が急速に進んだ。

## 3. 温度管理の法的要件と管理基準

### 3-1. 温度帯別の法的管理基準

コンビニの温度管理は、食品衛生法の基準と各チェーン本部が定める自主基準の二重構造になっている。法的最低基準よりも自主基準の方が厳格な場合が多い。

| 温度帯 | 法的根拠（食品衛生法施行規則等） | 法定基準温度 | 主要チェーンの自主管理温度 | 対象什器・設備 |
|-------|--------------------------------|------------|--------------------------|--------------|
| **常温（ドライ）** | 特段の基準なし（品質劣化防止） | 規定なし | 15〜25℃（夏期管理） | 通常棚・バックルーム |
| **チルド** | 食品衛生法施行規則 別表第2（食肉・魚介類等は10℃以下） | 10℃以下 | 5〜8℃ | オープン冷蔵ケース・扉付冷蔵庫 |
| **フローズン（冷凍）** | 食品衛生法施行規則（−15℃以下） | −15℃以下 | −18〜−25℃ | 冷凍ケース・冷凍庫 |
| **ホット（温蔵）** | 食品衛生法第58条準拠（65℃以上で保存） | 65℃以上 | 68〜75℃（什器種別により異なる） | ホットケース・肉まんウォーマー |
| **フライ・中食調理** | 食品衛生法・各都道府県条例 | 中心温度75℃1分以上（特定加熱食品は） | 中心温度80℃以上を確認 | フライヤー・電子レンジ |

### 3-2. 温度管理に関する厚生労働省通知の要点

「食品等の冷凍・冷蔵保存」に関する厚生労働省のガイドラインでは、以下の管理基準が示されている。

```
【冷蔵設備の管理ポイント】
- 設定温度: 食品の種類に応じた適切な温度設定
- 温度計: 最高・最低温度が記録できる自記温度計の使用を推奨
- 確認頻度: 原則として1日1回以上（開店時・営業中・閉店時）
- 過積載禁止: 冷気循環を妨げる詰め込み過ぎを防止
- 点検・清掃: 定期的な温度計の校正・ドアパッキンの点検

【記録の要件】
- 記録日時・測定者・測定温度・異常の有無・対応内容
- 異常発生時は対応内容を必ず記録
- 記録は一定期間保存（各チェーン本部規定に準拠）
```

### 3-3. 温度ゾーン別管理のリスクマトリックス

| 温度帯 | 主な危害要因 | 発生しうる食中毒菌 | 管理の緊急度 |
|-------|-----------|------------------|------------|
| チルド（5〜8℃） | 保存温度の上昇（ドア開け放し・停電） | リステリア菌・腸炎ビブリオ（温度上昇時） | 高（日3便の食品を扱うため） |
| フローズン（−18℃以下） | 解凍・再凍結 | サルモネラ菌（解凍後急激繁殖） | 中〜高（解凍検知が重要） |
| ホット（68℃以上） | 温度低下（ヒーター故障・ドア開放） | ウェルシュ菌・黄色ブドウ球菌 | 高（4時間以内の廃棄が原則） |
| 常温 | 高温多湿による品質劣化 | カビ・酵母（主に品質問題） | 低（直接の食中毒リスクは低い） |

## 4. ストコンが担う HACCP 関連機能

### 4-1. 機能マップ: HACCP の7原則とストコン機能の対応

```
HACCP原則                       ストコン（および連携システム）の機能
─────────────────────────────────────────────────────────────────
原則3: 管理基準(CL)設定  →    各温度帯の設定温度をマスタデータとして保持
原則4: モニタリング方法   →    IoTセンサーからの温度ログ自動収集・記録
原則5: 改善措置          →    温度逸脱アラート・廃棄指示生成
原則6: 検証方法          →    定期的な温度記録レビュー機能・レポート出力
原則7: 記録・文書管理     →    温度ログの保管・監査向けエクスポート機能
```

### 4-2. 温度管理機能の詳細

| 機能カテゴリ | 詳細機能 | ストコン内（ローカル処理） | 本部連携（クラウド/WAN） |
|------------|---------|------------------------|----------------------|
| **温度ログ収集** | IoTセンサーからの定期取得（10分〜30分間隔） | ストコン内DB（ローカルバッファ） | 本部クラウドへリアルタイム送信 |
| **閾値管理** | 各什器の上限・下限温度設定 | ストコン内部マスタ | 本部から一括更新・変更 |
| **アラート生成** | 温度逸脱（上限超過・下限超過）時の警告 | ストコン画面表示・音声アラート | 本部への自動通報 |
| **廃棄判定支援** | ホット什器の温度低下→廃棄タイミング通知 | 廃棄指示画面表示 | 廃棄実績の本部送信 |
| **記録管理** | 温度ログの保存・帳票出力 | ローカルストレージ（直近） | 本部長期保管（監査対応） |
| **機器異常検知** | センサー断線・通信エラーの検知 | 障害アラート | 保守部門へエスカレーション |

### 4-3. 温度ログのデータ構造（標準的な例）

```
【温度ログレコードの主要フィールド】

┌────────────────────────────────────────────────────────┐
│ ヘッダー情報                                           │
│   店舗コード       : 8桁（全国一意）                   │
│   什器ID          : 店舗内の什器を識別する番号          │
│   什器種別コード   : 01=チルドケース / 02=冷凍庫 / ...  │
│   記録日時        : YYYYMMDDHHmmss（秒単位）           │
├────────────────────────────────────────────────────────┤
│ 測定値                                                 │
│   測定温度（℃）   : 小数点1桁（例: 6.8）              │
│   設定温度（℃）   : 该当什器の目標温度                  │
│   逸脱フラグ      : 0=正常 / 1=上限超過 / 2=下限超過   │
│   センサーステータス: 00=正常 / 01=断線 / 02=精度異常   │
├────────────────────────────────────────────────────────┤
│ 対応情報（逸脱時のみ）                                  │
│   アラート発報日時 : 逸脱を検知した時刻                 │
│   対応者コード     : 対応したスタッフのID               │
│   対応内容コード   : 01=設定確認 / 02=商品移動 / ...   │
│   対応完了日時     : 復旧確認時刻                      │
└────────────────────────────────────────────────────────┘

【1店舗あたりの温度ログ量の目安】
  什器数: 10〜30台（店舗規模による）
  記録間隔: 10分または30分
  1日のレコード数: 什器数 × (24h × 60分 / 間隔) 件
    例: 20台 × 144件/日 = 2,880件/日/店舗
  全国5万店: 約1.44億件/日（30分間隔の場合）
  データサイズ: 1レコード約200バイト → 約28 GB/日（全国）
```

## 5. IoT センサーによる自動温度記録の仕組み

### 5-1. センサーシステムの全体構成

```
【店舗内温度監視システムの構成（2020年代標準）】

[冷蔵・冷凍ケース]
     ↓ 温度センサー（熱電対 or サーミスタ）
[センサーユニット（ゲートウェイ）]
     ← 有線（RS-485/Modbus）または近距離無線（Zigbee/BLE）
     ↓
[店舗ゲートウェイ / IoTコントローラー]
     ← 設備管理システム（BAS: Building Automation System）
     ↓ LAN（有線Ethernet）
[ストアコンピューター（ストコン）]
     ← 温度データ受信・ローカル記録・閾値チェック・アラート
     ↓ WAN（Internet VPN / 専用線）
[本部 温度管理クラウドシステム / MES]
     ← 全店舗リアルタイム監視・ダッシュボード・アラート一元管理
     ↓ エスカレーション
[エリアマネージャー / SV スマートフォン通知]
[設備保守会社 / 修理依頼システム]
```

### 5-2. センサーの種類と特性

| センサー種別 | 測定原理 | 測定範囲 | 精度 | コンビニでの用途 |
|-----------|---------|---------|------|--------------|
| **熱電対（サーモカップル）** | 2種金属の接触電位差 | −200〜+1,300℃ | ±0.5〜2℃ | 高温調理設備（フライヤー等） |
| **測温抵抗体（PT100）** | 金属の抵抗変化 | −200〜+850℃ | ±0.1〜0.5℃ | 精密な冷凍・冷蔵管理 |
| **サーミスタ** | 半導体の抵抗変化 | −50〜+150℃ | ±0.1〜1℃ | 冷蔵ケース・冷凍庫（低コスト・小型） |
| **赤外線センサー** | 放射熱量測定 | −50〜+500℃ | ±1〜3℃ | 商品表面温度の非接触測定（入荷検品） |
| **BLE（Bluetooth Low Energy）センサー** | サーミスタ+無線送信 | −30〜+85℃ | ±0.5℃ | 移動什器・通函の温度追跡 |

### 5-3. 通信プロトコルと標準化動向

コンビニの温度管理センサー通信は、過去は各ベンダー独自プロトコルが主流だったが、標準化・クラウド連携の観点からオープンプロトコルへの移行が進んでいる。

| プロトコル | 特性 | コンビニへの採用状況 |
|-----------|------|-------------------|
| **RS-485 / Modbus** | 有線・低遅延・長距離対応・工業標準 | 既存設備で主流。レガシーシステムの大半が採用 |
| **Zigbee** | 低消費電力無線・メッシュネットワーク | 後付けセンサー追加時に採用されるケースあり |
| **BLE（Bluetooth Low Energy）** | スマートフォン連携・低コスト | 携帯型温度計・ハンディ端末との連携で普及 |
| **Wi-Fi（802.11）** | 高帯域・既存LAN活用可能 | 店舗Wi-Fi環境が整備された店舗で採用 |
| **ECHONET Lite** | スマートホーム・国内標準（IEC化済み） | 設備機器の統合管理。家電/HVAC連携で採用増加 |
| **MQTT（クラウド連携）** | 軽量メッセージング・IoT標準 | AWS IoT Core / Azure IoT Hub との連携で採用拡大中 |

### 5-4. 自動記録システムが解決した課題

**HACCP 義務化以前の手動記録の問題点**:

```
問題1: 記録の「もれ」と「まとめ書き」
  → 多忙な深夜帯・早朝に実測せず、前の時間帯の値をコピーして記録
  → 監査で発覚するケースがあり、チェーン全体の信頼性リスクに

問題2: 異常の発見遅れ
  → 30分ごとの目視確認では、停電・機器故障後に数時間気づかない
  → 商品の品質劣化・廃棄ロスが発生

問題3: 記録の保管・検索
  → 紙台帳は紛失・劣化リスク
  → 特定日の記録を遡って検索するのに数時間かかる

【自動記録システムによる解決】
  → 10分ごとの自動取得で「記録漏れ」を物理的に防止
  → 閾値を超えた瞬間にアラート → 最大15分以内に検知・対応可能
  → 電子データのため全店・全期間の横断検索が数秒で完了
  → クラウド保存で物理的紛失リスクを排除
```

## 6. 温度逸脱時のアラート・エスカレーションフロー

### 6-1. アラートの分類と対応優先度

| アラートレベル | 条件 | 初動対応時間の目安 | 対応者 |
|-------------|------|----------------|--------|
| **レベル1（注意）** | 設定温度から±2℃以内の逸脱・5分未満 | 30分以内確認 | 在店スタッフ |
| **レベル2（警告）** | 設定温度から±2℃超の逸脱・または15分以上継続 | 15分以内確認 | 店長 / 在店リーダー |
| **レベル3（緊急）** | 設定温度から±5℃超・または30分以上継続 | 5分以内対応開始 | 店長 + SV（スーパーバイザー）通報 |
| **設備故障アラート** | センサー断線・機器停止を検知 | 即時対応 | 設備保守会社への発報 + 店長 |

### 6-2. エスカレーションフロー（詳細）

```
【温度逸脱発生時のフロー】

STEP 1: センサーが閾値超過を検知（リアルタイム）
     ↓
STEP 2: ストコン画面にアラート表示 + 音声通知（店内）
     ↓
STEP 3: 本部クラウドへ自動通報（同時並行）
     ↓                           ↓
STEP 4a: 在店スタッフが          STEP 4b: 本部監視システムが
         該当什器を確認                   全店舗アラートを受信
     ↓
STEP 5: 原因確認
   ├── 一時的な温度上昇（ドア開放等）→ ドアを閉めて様子見
   ├── 機器故障の疑い              → 設備保守会社へ連絡
   └── 停電・断電                 → 緊急商品移動手順の実施
     ↓
STEP 6: 対応内容をストコンに入力（対応者コード・対応内容・完了時刻）
     ↓
STEP 7: 温度が基準内に戻ったことを確認
     ↓
STEP 8: 対応完了を記録 → 本部へ完了通知
     ↓
STEP 9: 【長時間逸脱の場合】商品の廃棄判断
   - チルド品: 10℃超で2時間以上 → 本部指示に基づき廃棄
   - フローズン品: 部分解凍が確認された場合 → 廃棄
   - ホット品: 65℃未満が4時間以上 → 廃棄
   - 廃棄実績をストコンに記録（廃棄数量・廃棄理由コード）

【SVへのエスカレーション条件（レベル3）】
  - 30分以上の温度逸脱継続
  - 機器故障の可能性
  - 廃棄商品が発生した場合
  - 食品安全インシデントの可能性
```

### 6-3. ホット什器の特殊管理（肉まん・惣菜ウォーマー）

ホット什器の温度管理は、食品衛生法の加熱保存基準（65℃以上）に直結するため、特別な管理が必要。

| 什器種別 | 管理基準温度 | 保存上限時間 | ストコンの機能 |
|---------|------------|------------|-------------|
| 肉まん・中華惣菜ウォーマー | 68〜75℃ | 4〜6時間（チェーン規定による） | 投入時刻記録・廃棄タイマー表示 |
| ホットスナックケース | 65℃以上 | 3〜4時間 | 廃棄タイムアラート |
| コーヒーマシン（豆） | 特段の基準なし | 1日1回の豆交換 | 豆補充アラート（オプション） |
| フライヤー油 | 調理時に180℃以上 | 毎日交換が推奨 | フライヤー使用回数カウント |

## 7. 入荷検品時の食品安全確認プロセス

### 7-1. 検品時の温度確認プロセス

コンビニへの食品納品においては、**入荷品温確認**がHACCP管理上の重要な管理点（CCP）となる。

```
【入荷検品フロー（食品安全重点手順）】

STEP 1: 納品車到着確認
   - ドライバーと品目・便数を照合
   - 通函（かご車・折コン）の外観確認（破損・汚染の有無）

STEP 2: 品温確認（最重要ステップ）
   - 赤外線温度計またはプローブ型温度計を使用
   - サンプリング（全数または指定サンプル数）で品温を実測
   - 確認基準:
     ┌─────────────────────────────────────────┐
     │ チルド品（弁当・惣菜・乳製品）: 10℃以下   │
     │ フローズン品（冷凍食品）    : −15℃以下    │
     │ ホット商品（既製品）        : 65℃以上     │
     └─────────────────────────────────────────┘

STEP 3: ストコンへの検品入力
   - ハンディスキャナーで商品バーコードをスキャン
   - 納品数量を入力（出荷明細との照合）
   - 品温確認結果の入力（正常/異常）
   - 異常品温の場合は返品・受取拒否の処理

STEP 4: 棚への陳列・冷蔵庫への収納
   - チルド品は速やかに冷蔵ケースへ（30分以内が目安）
   - 先入れ先出し（FIFO: First In First Out）の徹底
   - 賞味期限・消費期限の確認と並べ替え

STEP 5: 検品完了記録
   - 検品完了日時・担当者・入荷数量・差異の記録
   - 本部へ検品完了の送信（ストコン自動送信）
```

### 7-2. 温度異常品の受取拒否基準と処理フロー

| 品温状態 | 判定 | 処理方法 | ストコンでの処理 |
|---------|------|---------|--------------|
| 規定温度内 | 受取OK | 通常検品・陳列 | 通常入荷登録 |
| 規定温度から+2℃以内 | 要確認 | SVまたは本部に確認 | 保留登録・確認入力 |
| 規定温度から+2℃超 | 受取拒否推奨 | ドライバーに返品・本部報告 | 返品処理・異常報告 |
| 明らかな異常（解凍・腐敗） | 受取拒否 | 即時返品・本部報告 | 緊急報告送信 |

## 8. 消費期限・賞味期限管理（3分の1ルールと販売期限）

### 8-1. 3分の1ルールとは

日本の食品流通業界が自主的に採用してきた商慣行で、製造日から賞味期限までの期間を3等分し、最初の3分の1を納品期限、次の3分の1を販売期限とする慣行。

```
【3分の1ルールの例】
製造日: 1月1日
賞味期限: 4月1日（製造から90日間）

     1月1日        2月1日         3月1日        4月1日
       ├─────────────┼──────────────┼─────────────┤
       │  第1フェーズ  │  第2フェーズ   │  第3フェーズ  │
       │（製造〜30日）  │（30〜60日）    │（60〜90日）   │
       │    ↑          │     ↑          │     ↑         │
       │ 納品期限       │  販売期限      │  消費者使用期間│
       │（製造から1/3まで）│（製造から2/3まで）│（残り1/3）   │

コンビニへの納品は 2月1日まで に限定（第1フェーズ内）
コンビニでの販売は 3月1日まで に限定（第2フェーズ内）
```

### 8-2. 3分の1ルールの見直し動向

フードロス削減の観点から、3分の1ルールの緩和が業界全体で進んでいる。

| 時期 | 動向 |
|-----|------|
| 2013年頃〜 | 農林水産省・消費者庁がフードロス削減の観点から業界への見直し要請 |
| 2018年〜 | セブン・ファミマ・ローソンが順次、3分の1ルールの緩和を本部指示で開始 |
| 2022年〜 | 経済産業省・農水省・環境省が連携してフードロス削減ガイドライン改訂 |
| 2023〜現在 | 大手コンビニ各社が「2分の1ルール」または「期限日前日まで販売可能」への移行を推進 |

**ストコンへの影響**: 販売期限マスタの変更、商品ごとの期限設定の更新が必要。移行時には本部からのマスタデータ一括更新が必須。

### 8-3. 消費期限管理のシステム的な実装

コンビニの消費期限管理は「時刻管理（時間チェック）」と「日付管理（日付チェック）」の組み合わせ。

| 管理対象商品 | 管理の単位 | ストコンの機能 |
|-----------|---------|-------------|
| 弁当・おにぎり（チルド） | 時間単位（例: 製造から24時間以内） | 廃棄時刻アラート（商品ごと） |
| 惣菜・サラダ | 日時単位 | 廃棄予定リスト表示・廃棄入力 |
| 乳製品・豆腐 | 年月日単位 | 消費期限超過検知（POS連携） |
| デザート・スイーツ | 年月日単位 | 販売期限管理（棚確認ガイド） |
| パン類 | 年月日単位 | 廃棄予定リスト表示 |

```
【ストコンの消費期限・廃棄管理機能】

1. 廃棄予定リスト生成
   - 廃棄時刻が近い商品の一覧を自動表示（例: 3時間前からリスト化）
   - 廃棄順序を棚の陳列位置順に並べ替えて効率化

2. 廃棄入力処理
   - ハンディスキャナーで商品をスキャン → 廃棄数量を入力
   - 廃棄理由コード（消費期限/品質劣化/温度異常等）の記録
   - 廃棄金額の自動計算（本部への廃棄ロス報告）

3. POS連動による期限超過防止
   - 消費期限超過商品のバーコードをレジでスキャンすると販売禁止アラート
   - 一部チェーンではリアルタイム販売可否判定をPOSが行う

4. 廃棄実績の本部送信
   - 廃棄商品のJANコード・数量・廃棄理由・廃棄日時を本部へ送信
   - 本部は廃棄率を分析してAI発注精度の改善に活用
```

### 8-4. 販売期限の値引き販売（マークダウン）施策

フードロス削減の観点から、販売期限前の値引き販売（マークダウン）が大手チェーンで本格導入されている。

| チェーン | 施策内容 | ストコンへの影響 |
|--------|---------|-------------|
| **セブンイレブン** | 消費期限3〜6時間前から自動値引き（アプリ経由） | 値引きJANコードの管理・POS連携 |
| **ファミリーマート** | アプリ連携型の「ファミマフードドライブ」（値引きクーポン） | クーポン管理システムとストコンの連携 |
| **ローソン** | 「ローソンスマホレジ」での最終値引き対応 | スマホレジ連携・在庫管理との同期 |

## 9. 廃棄管理とフードロス削減施策

### 9-1. 廃棄管理の全体構造

```
【コンビニ廃棄管理の構造】

発生源別廃棄の分類:
  A. 消費期限切れ廃棄（時間管理廃棄）
     - 弁当・おにぎり・惣菜の期限廃棄
     - 最大廃棄カテゴリ（デイリー品全体の5〜15%が廃棄）

  B. 温度異常廃棄（食品安全廃棄）
     - 冷蔵・冷凍機器の故障による廃棄
     - 配送中の温度逸脱が発覚した場合の廃棄

  C. 品質劣化廃棄（外観・衛生廃棄）
     - 消費期限内だが外観異常（カビ・変色等）
     - 通常は少量だが、災害・停電後に増加

  D. 返品・回収廃棄
     - メーカー自主回収（食品リコール対応）
     - 本部指示による店頭回収（緊急自主回収）
```

### 9-2. 廃棄コストの構造と加盟店への影響

コンビニの廃棄ロスは加盟店（フランチャイジー）が原則として負担する重大な経営課題。

| 項目 | 内容 |
|------|------|
| 廃棄コストの負担者 | 原則として加盟店が全額負担（FC契約の基本構造） |
| 廃棄率の目安 | デイリー品（弁当等）で売上の3〜7%程度が廃棄 |
| 廃棄ロス額（1店舗・月） | 数十万円〜100万円超（品揃え・立地により大きく異なる） |
| 本部への廃棄ロス報告 | ストコン経由で日次送信（廃棄分の仕入原価を記録） |
| 廃棄削減へのインセンティブ | 一部チェーンで廃棄率改善への加盟店インセンティブ制度あり |

### 9-3. AI 需要予測によるフードロス削減

ストコンが発注するデータは、本部のAI需要予測システムの学習データにもなっており、廃棄削減とチャンスロス防止の両立を目指している。

```
【AI需要予測が活用するデータソース】

1. 販売実績データ（ストコン・POS経由）
   - 商品別・時間帯別・曜日別の販売数
   - 天候・気温との相関

2. 廃棄実績データ（ストコン経由）
   - 廃棄商品・廃棄数量・廃棄理由
   - 廃棄率の推移トレンド

3. 外部データ（本部が取得）
   - 気象データ（気温・降水確率）
   - 周辺イベント情報（スポーツ・コンサート等）
   - 地域の人口動態・近隣施設の変化

【AI活用による廃棄削減の実績（業界水準）】
  - AI発注導入前後で廃棄率が10〜20%改善（チェーンによって異なる）
  - 特に天候依存度の高い商品（冷たい飲料・アイス等）で効果大
```

### 9-4. フードロス削減施策（2024〜2026年の最新動向）

| 施策 | 内容 | ストコンへの影響 |
|-----|------|-------------|
| **動的値引き（ダイナミックプライシング）** | 廃棄時刻に近づくほど値引きを拡大 | 値引き価格マスタの動的更新・POSとの連携強化 |
| **フードバンク連携** | 販売期限前の食品を福祉施設等に寄付 | 寄付数量管理・廃棄処理との区別記録 |
| **ロスゼロ発注（機械学習精度向上）** | 発注数を需要予測と一致させる精度向上 | AI推奨発注の採用率向上に向けたUI改善 |
| **消費期限延長（技術革新）** | 包材・保存技術改良による期限延長 | 商品マスタの期限情報更新 |

## 10. ストコン移行案件との関連（PM 視点）

クラウド移行プロジェクトにおいて、食品安全・温度管理機能は技術的な難易度が相対的に高く、法的要件（HACCP・食品衛生法）との整合性確認が必須となる。以下の5点を PM として特に重視すべきである。

### ポイント1: 温度ログの連続性とデータ移行要件

クラウド移行の「カットオーバー瞬間」において、温度ログの欠落が HACCP 上の記録要件違反につながるリスクがある。

```
【リスク】
  オンプレ → クラウド移行の切り替え時に、
  温度センサーからのデータ収集が一時停止する可能性

【対策（移行設計での考慮点）】
  1. 移行期間中のダブルライト（オンプレ + クラウドに同時記録）
  2. ストコンローカルバッファの容量確認（停電・通信断でも最低24時間分を保持）
  3. カットオーバー後の温度ログ連続性を監査ログで証明できる設計
  4. 移行前の既存温度ログの完全移行（法定保存期間 + αを確保）

【スコープ確認事項】
  - 過去何年分の温度ログを新システムに移行するか（本部・監査要件の確認）
  - 移行対象は全国一斉か段階的か（段階的の場合は混在期間の管理方法）
```

### ポイント2: リアルタイム監視のクラウド化とSLA要件

温度アラートは食品安全に直結するため、クラウドシステムの可用性要件が通常のSLAよりも厳格となる。

```
【要求される可用性水準】
  温度監視システム: 99.9%以上（年間8.76時間以内のダウンタイム）
  アラート通報機能: 99.99%以上（年間52.56分以内のダウンタイム）
  ※ただし、ストコンがローカルでもアラートを出す二重化設計が前提

【クラウド化での設計要点】
  - AWS IoT Core + CloudWatch を活用したリアルタイム温度監視
  - Lambda によるアラート即時発報（5秒以内）
  - マルチAZ構成による高可用性
  - DynamoDB または RDS へのログの永続化

【障害時のフォールバック設計（重要）】
  - WAN 断線時: ストコンローカルで温度記録を継続
  - クラウド障害時: 店舗ストコンがアラートを出す（クラウド依存しない設計）
  - 復旧時: ローカルバッファの温度ログをクラウドへ自動同期
```

### ポイント3: 温度ログの保管期間と監査対応要件

HACCP 記録の電子保存に関して、監査対応のためのデータ保持設計が必要。

```
【保管期間の要件整理】
  法定要件: 食品衛生法に明確規定なし
  業界標準: 1〜2年間（チェーン本部の内部規定による）
  監査対応: 行政調査（食中毒発生時等）で過去2年分の提出を求められるケースあり
  推奨設計: 3年間のホット保管 + アーカイブ（低コストストレージ）で5年間保持

【AWS での実装例】
  - S3 Standard: 直近1年分（高頻度アクセス）
  - S3 Infrequent Access: 1〜3年分
  - S3 Glacier Instant Retrieval: 3〜5年分（監査要求時に時間）
  - S3 Glacier Deep Archive: 5年超（長期保存・低コスト）
  - S3 Lifecycle Policy で自動的に階層移動

【監査対応機能の設計要点】
  - 特定店舗・特定期間の温度ログを数秒で検索・出力できるUI
  - 改ざん防止: CloudTrail + S3 Object Lock（WORM モード）
  - 出力形式: CSV / PDF（行政提出に対応できる形式）
```

### ポイント4: 食品リコール・緊急回収時のストコン対応

食品事故・リコールは緊急性が高く、ストコンが最前線の対応ツールとなる。クラウド移行後も同等以上の機能継続が必要。

```
【食品リコール発生時のフロー（ストコン関与部分）】

本部 → 緊急リコール通知 → ストコン（全店同時受信）
      ↓
対象商品のJANコード・製造日・ロット番号を特定
      ↓
ストコンの在庫照合機能で該当商品の在庫数を即時確認
      ↓
POS連動で「販売禁止」フラグをレジに反映（スキャン時にエラー）
      ↓
店頭の対象商品を回収・隔離（スタッフがハンディで回収数量入力）
      ↓
回収数量・回収完了日時を本部へ送信

【移行後の設計要件】
  - リコール通知の受信機能を維持（本部→ストコンのプッシュ通知）
  - JANコード単位の在庫照合機能（リアルタイム）
  - POS との即時連携（販売禁止フラグ反映の遅延を最小化）
```

### ポイント5: 各都道府県の食品衛生条例への対応

HACCP の要件は国の法令だけでなく、各都道府県が独自に定める食品衛生条例によって上乗せされる場合がある。全国展開するコンビニのシステムは、最も厳格な自治体の要件に対応できる設計が必要。

```
【都道府県差異の例】
  - 営業許可の種類と要件: 惣菜製造・飲食店営業等の区分が都道府県ごとに異なる
  - 温度管理記録の保存期間: 都道府県条例で2年以上を義務付けるケースあり
  - 衛生教育の頻度: 年1回以上の食品衛生教育を義務付ける条例あり

【PM としての対応方針】
  1. 移行前に法令チェックリストを作成（47都道府県の条例を確認）
  2. 最も厳格な要件（最長保存期間・最高頻度の記録）をシステム要件の基準とする
  3. 法令変更のモニタリング体制（年1回の定期確認）を運用設計に組み込む
```

## 11. 用語集

| 用語 | 読み | 説明 |
|------|------|------|
| **HACCP** | ハサップ | Hazard Analysis and Critical Control Points。食品製造・販売における衛生管理の国際標準手法。2021年6月からコンビニを含む食品事業者に義務化 |
| **消費期限** | しょうひきげん | 食品衛生法・食品表示法に基づく安全に消費できる期限。おおむね5日以内の食品（弁当等）に使用。期限後の販売は法律違反 |
| **賞味期限** | しょうみきげん | 品質が保たれる期限。期限後も即座に危険ではないが、品質保証外となる。スナック・乳製品等 |
| **3分の1ルール** | さんぶんのいちルール | 製造から賞味期限までを3等分し、最初の1/3を納品期限・次の1/3を販売期限とする業界慣行。フードロス削減のため緩和が進む |
| **品温確認** | ひんおんかくにん | 入荷時・販売時に食品の温度を実測する作業。HACCP の重要管理点（CCP）の一つ |
| **CCP** | シーシーピー | Critical Control Point（重要管理点）。HACCP においてハザードの管理が必須な工程点。温度管理はコンビニの主要CCPの一つ |
| **CL** | シーエル | Critical Limit（管理基準値）。CPPで管理すべき上限・下限の数値。チルドなら10℃以下等 |
| **マークダウン** | まーくだうん | 廃棄ロス削減のための販売期限前値引き。ダイナミックプライシングの形で自動化が進む |
| **FIFO** | ファイフォ | First In First Out（先入れ先出し）。古い商品から先に販売する陳列・在庫管理の原則 |
| **BAS** | ビーエーエス | Building Automation System（ビル管理システム）。店舗の空調・照明・冷蔵設備を統合管理するシステム。温度センサーを集約する場合がある |
| **WORM** | ワーム | Write Once Read Many。データを書き込んだ後に変更・削除不可にするデータ保存方式。監査ログの改ざん防止に使用（S3 Object Lock等） |
| **食品リコール** | しょくひんりこーる | 食品の製造・販売段階での問題発覚時に、市場に出回った商品を回収すること。ストコンがリコール対象品の在庫照合・販売禁止に直接関与する |
| **品質管理基準（QC）** | ひんしつかんりきじゅん | Quality Control。各チェーン本部が独自に設定する品質基準。法定基準より厳格な場合が多い |
| **食中毒予防3原則** | しょくちゅうどくよぼうさんげんそく | 「つけない（清潔）・増やさない（冷却・迅速）・やっつける（加熱）」。コンビニの食品管理の基本哲学 |
| **逸脱（Deviation）** | いつだつ | 管理基準値（CL）から外れた状態。温度逸脱はHACCPにおける記録・対応・報告が義務付けられる事象 |
