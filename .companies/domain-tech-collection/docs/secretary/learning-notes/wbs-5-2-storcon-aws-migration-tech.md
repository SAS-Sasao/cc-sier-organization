## 1. システム概要と技術的特性

### ストアコンピューターとは

コンビニエンスストアの各店舗に設置され、以下の業務を担うシステム。

| 機能カテゴリ | 具体的な処理内容 |
|------------|--------------|
| 商品発注 | 自動発注・手動発注の管理、仕入先への発注データ送信 |
| 廃棄管理 | 賞味期限切れ商品の廃棄登録・集計 |
| 売上精算 | POSデータの集計、日次・月次精算 |
| 在庫管理 | 商品在庫の入出庫管理、棚卸 |
| スタッフ管理 | シフト管理、勤怠データ連携 |
| 本部連携 | 本部への売上・在庫データ送信 |

### 技術的制約・要件

- **可用性**: 24時間365日稼働（コンビニは深夜も営業）
- **規模**: 全国数千〜数万店舗からの同時接続
- **リアルタイム性**: 発注データ・在庫データの即時反映
- **オフライン耐性**: 店舗のネットワーク障害時も店舗単体で業務継続できること
- **データ量**: 全店舗の日次トランザクションデータ（大量バッチ）
- **セキュリティ**: 売上・顧客データを扱うため厳格なアクセス制御が必要

## 2. AWS基本技術スタック

### 2-1. コンピュート

| サービス | 特徴 | ストコンでのユースケース |
|---------|------|----------------------|
| **EC2** | 仮想サーバー。フルコントロール可能 | 既存オンプレアプリのLift&Shift先、特殊な設定が必要なミドルウェア |
| **ECS / Fargate** | コンテナオーケストレーション（Fargate はサーバーレス） | マイクロサービス化後のアプリケーション実行基盤、バッチ処理コンテナ |
| **Lambda** | イベント駆動のサーバーレス関数 | 発注トリガー処理、データ変換、軽量API、Webhook受信 |
| **Elastic Beanstalk** | PaaS型アプリデプロイ | 既存Webアプリの簡易移行（リプラットフォーム） |

**ユースケース別推奨**

- 常時起動・高スループットのAPI → ECS/Fargate
- 低頻度・イベント駆動の処理 → Lambda
- Lift&Shift の移行初期 → EC2
- 日次バッチ処理 → ECS on Fargate / AWS Batch

### 2-2. データベース

| サービス | 特徴 | ストコンでのユースケース |
|---------|------|----------------------|
| **RDS (Aurora MySQL/PostgreSQL)** | マネージドRDB。Multi-AZ対応 | 発注マスタ、商品マスタ、トランザクション履歴 |
| **Aurora Serverless v2** | 負荷に応じた自動スケール | アクセス量が不規則な管理系API |
| **DynamoDB** | フルマネージドNoSQL。ミリ秒レイテンシ | セッション管理、在庫キャッシュ、高頻度アクセステーブル |
| **ElastiCache (Redis)** | インメモリキャッシュ | 商品マスタキャッシュ、セッション管理、ランキングデータ |
| **Redshift** | データウェアハウス | 売上分析、在庫分析、本部向けBIデータ基盤 |
| **Timestream** | 時系列データベース | センサーデータ、温度管理、店舗機器モニタリング |

**ストコンデータの特性と推奨DB**

| データ種別 | 特性 | 推奨DB |
|----------|------|--------|
| 商品マスタ | 読み取り多、更新少、参照高速性重要 | RDS Aurora + ElastiCache |
| 発注トランザクション | ACID必須、高整合性 | RDS Aurora (Multi-AZ) |
| 在庫カウンタ | 高頻度更新、ピーク集中 | DynamoDB (アトミックカウンタ) |
| 売上集計 | バッチ集計、分析クエリ | Redshift |
| セッション | 揮発性、低レイテンシ | ElastiCache Redis |

### 2-3. ストレージ

| サービス | 特徴 | ストコンでのユースケース |
|---------|------|----------------------|
| **S3** | オブジェクトストレージ、高耐久性(99.999999999%) | 日次バッチファイル、売上レポート、マスタデータCSV、ログアーカイブ |
| **EBS** | EC2にアタッチするブロックストレージ | EC2インスタンスのOSボリューム、DB用ストレージ |
| **EFS** | 共有ファイルシステム（NFS互換） | 複数コンテナから参照する共有設定ファイル |
| **S3 Glacier** | 低コスト長期アーカイブ | 数年分の売上データアーカイブ |

### 2-4. ネットワーク

| サービス | 概要 | ストコンでの役割 |
|---------|------|----------------|
| **VPC** | 仮想プライベートクラウド | AWS上の論理ネットワーク分離（本番/開発/管理の分離） |
| **Direct Connect** | オンプレ（本部DC等）とAWSの専用線接続 | 本部データセンターとAWS間の高信頼接続、移行期のハイブリッド構成 |
| **Transit Gateway** | 複数VPC・オンプレをハブで接続 | 大規模展開時の複数VPC間ルーティング集約 |
| **CloudFront** | グローバルCDN | 管理コンソールの静的コンテンツ配信、店舗向けAPIのエッジキャッシュ |
| **API Gateway** | マネージドAPIゲートウェイ | 店舗→クラウド間のRESTful/WebSocket API公開 |
| **Route 53** | DNSサービス + ヘルスチェック | マルチリージョンフェイルオーバー、レイテンシルーティング |

### 2-5. セキュリティ

| サービス | 概要 | ストコンでの役割 |
|---------|------|----------------|
| **IAM** | 認証・認可の管理 | 最小権限の原則の実装、サービスロール管理 |
| **KMS** | 鍵管理サービス | DBデータの暗号化、S3バケットの暗号化 |
| **WAF** | Webアプリケーションファイアウォール | 店舗API・管理コンソールへの不正アクセス防止 |
| **Security Hub** | セキュリティ状態の一元管理 | コンプライアンスチェック、セキュリティスコア可視化 |
| **GuardDuty** | 脅威検知（ML活用） | 異常な発注パターン・不正アクセスの自動検知 |
| **Secrets Manager** | シークレット管理 | DB接続情報、APIキーの安全な管理 |
| **CloudTrail** | AWS操作ログ | 監査ログ、不正操作の追跡 |

## 3. 小売・店舗システム特有の技術要件

### 3-1. 大規模同時接続の処理

全国数万店舗から業務ピーク時（発注締め切り時間帯等）に同時リクエストが集中する。

**対応パターン**

```
店舗 → API Gateway → SQS（キューで均等化） → Lambda/ECS → Aurora
                          ↓
                    バックプレッシャー制御
```

- **API Gateway + SQS**: リクエストをキューに溜め、後続処理のスループットを制御
- **Auto Scaling**: ECS/EC2のオートスケーリングでピーク対応
- **ElastiCache**: 頻繁に参照されるマスタデータをキャッシュして DB 負荷を軽減
- **DynamoDB On-Demand**: 予測不能なトラフィックスパイクに対応

### 3-2. リアルタイムデータ処理

| サービス | 特徴 | ユースケース |
|---------|------|------------|
| **Kinesis Data Streams** | リアルタイムデータストリーミング（順序保証） | 全店舗からの売上データリアルタイム収集 |
| **Kinesis Data Firehose** | ストリームデータのS3/Redshift自動ロード | ログデータ・イベントデータの自動永続化 |
| **SQS (Standard/FIFO)** | マネージドメッセージキュー | 非同期発注処理、サービス間デカップリング |
| **SNS** | Pub/Sub通知 | 発注完了通知、アラート配信 |
| **EventBridge** | イベントバスとルーティング | マイクロサービス間のイベント連携、スケジューラ |

**SQS Standard vs FIFO の選択基準**

| 観点 | Standard | FIFO |
|------|----------|------|
| 順序保証 | なし | あり（厳密な順序が必要な場合） |
| スループット | 無制限 | 3,000 msg/sec（バッチ）|
| 重複 | 可能性あり（at-least-once） | なし（exactly-once） |
| ストコンでの用途 | ログ転送、通知 | 発注処理（二重発注防止） |

### 3-3. バッチ処理

コンビニは日次精算・棚卸集計・マスタ更新等の大規模バッチが必須。

| サービス | 特徴 | ユースケース |
|---------|------|------------|
| **AWS Batch** | フルマネージドバッチコンピューティング | 日次売上集計、大量在庫照合、マスタ一括更新 |
| **Step Functions** | ワークフローオーケストレーション | 複数ステップのバッチ処理フロー管理、エラーハンドリング |
| **Lambda (スケジュール実行)** | EventBridgeでのcron実行 | 軽量な定期処理（データ検証、通知送信） |
| **Glue** | ETLサービス | DWHへのデータ変換・ロード |

**推奨バッチアーキテクチャ**

```
EventBridge (cron) → Step Functions → AWS Batch (EC2/Fargate)
                         ↓
                    エラー時: SNS通知 + CloudWatch Alarm
```

### 3-4. 店舗-クラウド間の通信設計

```
[店舗ストコン端末]
      ↓ HTTPS / WebSocket
[API Gateway]
      ↓
[WAF] ← 不正アクセス遮断
      ↓
[ALB (Application Load Balancer)]
      ↓
[ECS/Lambda] ← アプリケーション層
      ↓
[Aurora / DynamoDB] ← データ層
```

- **REST API**: 発注データ送信、マスタデータ取得（API Gateway + Lambda/ECS）
- **WebSocket API**: リアルタイム在庫更新の双方向通信（API Gateway WebSocket）
- **IoT Core**: 店舗内センサー・機器（冷蔵庫温度、電力計等）との接続

### 3-5. エッジコンピューティングとオフライン耐性

**AWS Outposts**
- AWSのラックを店舗/地域DCに設置し、AWS APIをオンプレで利用
- 大手チェーンの地域DC統合に有効（全店舗への設置はコスト面から非現実的）

**オフライン耐性パターン（重要）**

コンビニでは店舗のインターネット障害時も業務継続が必須。

```
[通常時]          店舗端末 ←→ クラウドAPI（リアルタイム同期）
[障害時]          店舗端末 ←→ ローカルDBキャッシュ（オフライン動作）
[復旧時]          店舗端末 → クラウドAPI（差分データの同期）
```

実装パターン:
- ローカルにSQLite/組み込みDBでキャッシュを持つ
- 店舗端末にマスタデータを事前同期しておく
- 障害期間のトランザクションをローカルキューに蓄積し、復旧後にクラウドへ送信

## 4. オンプレ→クラウド移行パターン

### 4-1. 移行戦略の比較（7R）

| 戦略 | 概要 | リスク | コスト | スピード | ストコンでの適用場面 |
|-----|------|--------|--------|---------|-------------------|
| **Retire** | 廃止・削除 | 低 | 削減 | - | 使われていないレガシー機能 |
| **Retain** | 現状維持 | 低 | 変化なし | - | 移行対象外のシステム |
| **Rehost (Lift & Shift)** | 変更なしにAWSへ移行 | 低 | 中 | 速い | オンプレアプリをEC2へ移行する初期段階 |
| **Replatform** | 最小限の改修でクラウド最適化 | 低〜中 | 中 | 中 | DBをRDS管理化、アプリをコンテナ化 |
| **Repurchase** | SaaS等への乗り換え | 中 | 変動 | 中 | 在庫管理SaaSへの切り替え |
| **Refactor / Re-architect** | クラウドネイティブに再設計 | 高 | 高 | 遅い | マイクロサービス化、イベント駆動化 |
| **Relocate** | VM単位でVMware on AWSへ移行 | 低 | 高 | 速い | VMware環境の一括移行 |

**ストコン移行の推奨ステップ**

```
Phase 1: Rehost（Lift & Shift）
  → オンプレのアプリをEC2へ移行、DBはRDSへ
  → ダウンタイム最小化、リスク低減

Phase 2: Replatform
  → アプリをコンテナ化（ECS/Fargate）
  → マネージドサービス活用（RDS, ElastiCache等）

Phase 3: Refactor（段階的）
  → ストラングラーフィグパターンで機能単位にマイクロサービス化
  → イベント駆動アーキテクチャへの移行
```

### 4-2. ストラングラーフィグパターン

モノリスを一気に置き換えず、機能ごとに段階的に新システムへ移行する手法。

```
[旧システム(オンプレ/EC2)]
        ↓
[API Gateway / ALB（ルーティング層）]
      ↙         ↘
[新機能(マイクロサービス)]  [旧機能(そのまま)]
      ↓
[段階的に旧機能を移行・廃止]
```

メリット: 移行リスクの分散、ロールバックが容易、段階的なテスト

### 4-3. AWS移行ツール

| ツール | 用途 |
|-------|------|
| **AWS Migration Hub** | 移行プロジェクトの一元管理・進捗トラッキング |
| **Application Discovery Service** | オンプレサーバーの自動探索（構成情報・依存関係の把握） |
| **Database Migration Service (DMS)** | 異種DB間のデータ移行（Oracle → Aurora 等） |
| **Schema Conversion Tool (SCT)** | DBスキーマの自動変換（Oracle SQL → Aurora MySQL等） |
| **AWS DataSync** | オンプレ→S3/EFS のデータ同期 |
| **Snow Family (Snowball等)** | 大容量データの物理輸送 |
| **Application Migration Service (MGN)** | サーバーのリアルタイムレプリケーションによる移行 |

### 4-4. データ移行戦略

**マスタデータ（商品マスタ等）**
- SCTでスキーマ変換後、DMSで初期ロード
- 移行後の差分はCDC（Change Data Capture）で同期

**トランザクションデータ**
- 大量データはS3経由でバルクロード（Redshiftの場合はCOPYコマンド）
- 移行期間中はデュアルライト（旧DBと新DBに同時書き込み）で整合性確保

**カットオーバー戦略**
- ブルー/グリーンデプロイ: 旧環境(Blue)と新環境(Green)を並走させ、切り替え
- ストコンのような24/365システムでは深夜のメンテナンスウィンドウを活用

## 5. 運用・監視

### 5-1. 監視サービス

| サービス | 用途 | ストコンでの活用 |
|---------|------|----------------|
| **CloudWatch Metrics** | AWSリソースのメトリクス収集・アラート | EC2/RDS/Lambdaの CPU・メモリ・レイテンシ監視 |
| **CloudWatch Logs** | ログ収集・検索 | アプリケーションログの一元管理 |
| **CloudWatch Logs Insights** | ログのSQLライクな分析 | エラーパターン分析、パフォーマンス調査 |
| **X-Ray** | 分散トレーシング | マイクロサービス間のリクエスト追跡、ボトルネック特定 |
| **CloudTrail** | AWS API操作ログ | 不正操作検知、監査証跡 |
| **AWS Config** | リソース設定変更の追跡 | コンプライアンス監視、設定ドリフト検知 |
| **Health Dashboard** | AWSサービス障害情報 | リージョン障害の早期把握 |

### 5-2. 高可用性設計

**Multi-AZ構成（基本）**

```
[AZ-a]          [AZ-c]
EC2/ECS  ←→    EC2/ECS    (ALBで負荷分散)
RDS Primary → RDS Standby (自動フェイルオーバー)
```

**マルチリージョン（最高可用性）**

- Active-Active: 東京/大阪リージョンに両方トラフィックを分散
- Active-Passive: 東京がプライマリ、大阪がDR（災害対策）
- Route 53 フェイルオーバールーティングで自動切り替え

**Auto Scaling**

```yaml
# ECSサービスのオートスケーリング例（概念）
ScaleOut条件: CPU使用率 > 70% が5分継続
ScaleIn条件:  CPU使用率 < 30% が15分継続
最小タスク数: 2 (AZ冗長)
最大タスク数: 20 (ピーク対応)
```

### 5-3. CI/CDパイプライン

```
[コードプッシュ (CodeCommit / GitHub)]
          ↓
[CodeBuild] ← ユニットテスト、Dockerイメージビルド
          ↓
[ECR] ← コンテナイメージプッシュ
          ↓
[CodeDeploy / ECS Blue/Green]
          ↓
[ステージング環境] → [承認] → [本番環境]
```

| サービス | 役割 |
|---------|------|
| **CodeCommit** | Gitリポジトリ（または GitHub/GitLab連携） |
| **CodeBuild** | ビルド・テスト実行環境 |
| **CodeDeploy** | EC2/ECS/Lambdaへのデプロイ自動化 |
| **CodePipeline** | 上記の一連のパイプラインオーケストレーション |
| **ECR** | Dockerコンテナイメージのプライベートレジストリ |

### 5-4. IaC（Infrastructure as Code）

| ツール | 特徴 | 推奨用途 |
|-------|------|---------|
| **CloudFormation** | AWS公式IaC。YAMLで記述 | AWSリソース管理の標準（AWS公式サポート） |
| **CDK (Cloud Development Kit)** | TypeScript/Python等でAWSインフラを記述 | 開発者がコードでインフラ管理、再利用性高 |
| **Terraform** | マルチクラウド対応IaC | AWS以外も含むハイブリッド環境、チームの標準がある場合 |
| **AWS SAM** | サーバーレスアプリ専用CloudFormation拡張 | Lambda/API Gateway構成の管理 |

**ストコン案件での推奨**: AWS CDK（TypeScript）
- 理由: AWSとの親和性が高く、型安全性がある。大規模インフラの再利用性も高い

## 8. 技術選定サマリー（比較表）

### コンピュート選定

| 要件 | 推奨サービス | 理由 |
|-----|------------|------|
| 既存オンプレアプリの移行初期 | EC2 | 変更最小で移行可能 |
| 常時起動のAPIサーバー | ECS/Fargate | コンテナで管理コスト削減 |
| イベント駆動の軽量処理 | Lambda | サーバーレスでコスト最適 |
| 大規模バッチ処理 | AWS Batch | 並列ジョブ管理が容易 |
| バッチのオーケストレーション | Step Functions | 複雑なフロー制御 |

### データベース選定

| 要件 | 推奨サービス | 代替 |
|-----|------------|------|
| 発注トランザクション（ACID必須） | Aurora MySQL/PostgreSQL | RDS MySQL |
| 高頻度アクセス・スケール必要 | DynamoDB | - |
| セッション・キャッシュ | ElastiCache Redis | DynamoDB |
| 売上分析・DWH | Redshift | - |
| 時系列データ（機器監視） | Timestream | - |

### メッセージング選定

| 要件 | 推奨サービス | 理由 |
|-----|------------|------|
| 非同期発注処理（順序保証） | SQS FIFO | 二重発注防止 |
| ログ・通知の非同期転送 | SQS Standard | 高スループット |
| リアルタイム売上データ収集 | Kinesis Data Streams | 順序保証・リプレイ可能 |
| サービス間イベント連携 | EventBridge | ルーティング・スケジュール |
| プッシュ通知 | SNS | Fan-out配信 |

## 付記: よく使われるAWS用語（ストコン移行文脈）

| 用語 | 意味 |
|-----|------|
| RTO (Recovery Time Objective) | 障害から復旧するまでの目標時間 |
| RPO (Recovery Point Objective) | 障害時に許容できるデータ損失時間 |
| Multi-AZ | 複数のアベイラビリティゾーンに冗長配置 |
| Blue/Green Deployment | 旧環境と新環境を並走させて切り替えるデプロイ手法 |
| Canary Release | トラフィックの一部のみ新バージョンに流す段階的リリース |
| CDC (Change Data Capture) | DB変更を検知してリアルタイムに連携する技術 |
| Idempotency (冪等性) | 同じ処理を複数回実行しても同じ結果になること（発注二重処理防止に重要） |
| Backpressure (背圧制御) | 処理能力を超えたリクエストをキューで調整する仕組み |
