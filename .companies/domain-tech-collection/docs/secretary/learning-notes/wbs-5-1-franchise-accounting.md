## 1. フランチャイズ契約と会計構造の全体像

### 1-1. コンビニFCの基本構造

コンビニのフランチャイズ（FC）は「**本部がシステム・商品・物流を提供し、加盟店がオペレーションを担う**」分業構造を基本とする。会計面では、本部が売上を一元管理した上でロイヤリティ（チャージ）を控除し、残額を加盟店に分配するという「**売上代理受領→精算型**」の仕組みが採用されている。

```
[コンビニFC 会計構造の全体概念図]

顧客 → 支払い → POS/レジ
                    |
                    v
            店舗の売上（現金・電子マネー・クレジット・QR）
                    |
                    v
            ストコン: 日次精算・EOB送信
                    |
                    v
            本部 オープンアカウント（OA）
            +-------------------------------+
            | 売上         : +   日販合計  |
            | 仕入原価     : -   仕入合計  |
            |                  ─────────  |
            | 粗利益       :              |
            | チャージ     : -   粗利×率  |
            |                  ─────────  |
            | 加盟店収入   :   加盟店へ   |
            +-------------------------------+
```

### 1-2. 本部 vs. 加盟店オーナーの役割分担

| 項目 | 本部（チェーン本部） | 加盟店オーナー |
|------|-----------------|-------------|
| 売上の受領 | 現金・電子マネー等を代理受領（OA管理） | 実際にレジで受け取るが資金は本部口座へ |
| 仕入原価の支払い | 共配センター・ベンダーへ先払い | OA経由で後払い（本部が立替） |
| チャージ（ロイヤリティ） | 粗利の一定率を受取 | 粗利から控除される形で負担 |
| 廃棄ロス | 一部補填（チェーンにより異なる） | 原則として加盟店が全額負担 |
| 人件費 | 負担なし | 全額加盟店負担 |
| 光熱費 | タイプCは本部負担（建物込み契約の場合） | タイプAは全額加盟店負担 |
| システム利用料 | 提供（チャージに含まれる場合が多い） | ストコン・POS等の利用料を負担 |
| 物流費 | 共配センター運営費（加盟店からの配送費で回収） | 配送費（仕入原価に含まれることが多い） |

### 1-3. 加盟店タイプと会計への影響

コンビニのFC契約には複数のタイプがあり、チャージ率や費用負担が異なる。

| タイプ | 土地・建物 | 設備投資 | チャージ率目安 | 加盟店負担 |
|-------|---------|--------|------------|---------|
| タイプA（独立型） | オーナー所有 | オーナー負担 | 低め（36〜43%程度） | 大（土地建物・設備・光熱費全て） |
| タイプC（借用型） | 本部が借り上げ | 本部負担 | 高め（60〜76%程度） | 小（人件費・廃棄ロスが主要コスト） |
| Cタイプ亜種 | 多様（契約による） | 一部負担 | 中間 | 契約条件次第 |

**会計上のポイント**: タイプCは本部への初期投資が不要な分、チャージ率が高い。タイプAはチャージ率が低いが、土地建物・設備の減価償却費を加盟店が単独で負担する。

## 2. ロイヤリティ（チャージ）計算方式

### 2-1. 粗利分配方式（コンビニの主流方式）

日本のコンビニ大手3社はいずれも「**粗利分配方式**」を採用している。これは「売上高」ではなく「粗利益（売上 - 仕入原価）」に対してチャージ率を掛ける仕組みである。

```
粗利益 = 売上高 - 売上原価（商品仕入原価）

チャージ額 = 粗利益 × チャージ率

加盟店収入 = 粗利益 - チャージ額 - 廃棄ロス相当額（粗利減少分として計上済み）
```

**粗利分配方式の特徴**

| 観点 | 内容 |
|------|------|
| 本部の収入源 | チャージは「粗利」に連動するため、売上が高くても原価が高ければ本部収入も下がる |
| 加盟店のインセンティブ | 粗利を高めることが本部・加盟店双方の利益向上につながる（目標が一致しやすい） |
| 廃棄ロスの影響 | 廃棄ロスは仕入原価が回収できない（売上ゼロ・原価あり）ため、粗利を直接引き下げる → 本部のチャージも下がり、本部も損失を間接的に被る |
| 値引き販売の影響 | 値引き分だけ売上が減少し粗利も減少。ただし廃棄ロス全額よりは粗利への影響が小さい |

### 2-2. 各チェーンのチャージ率（参考）

公開情報をベースとした概算値（実際の契約では立地・タイプ・年数等で変動する）。

| チェーン | 契約タイプ | チャージ率（粗利基準） | 特記事項 |
|---------|---------|-----------------|---------|
| セブン-イレブン | Aタイプ | 約43% | 土地建物オーナー負担 |
| セブン-イレブン | Cタイプ（建物本部） | 約57% | 建物費用は本部負担 |
| セブン-イレブン | C2タイプ（土地建物本部） | 約71〜76% | 土地建物ともに本部負担 |
| ファミリーマート | Aタイプ | 約36〜40% | 開業時の支援金で調整 |
| ファミリーマート | Cタイプ | 約55〜64% | 建物費用は本部負担 |
| ローソン | Aタイプ | 約40〜45% | 地域・立地によって幅あり |
| ローソン | Cタイプ | 約55〜72% | 建物費用は本部負担 |

**比較ポイント（SIer視点）**:
- セブン-イレブンは高チャージ率だが、それを上回る高日販（日次売上）を誇るため加盟店の実収入はむしろ高い
- ロイヤリティ率の違いはストコン上の精算ロジックに直結。チェーン固有の精算プログラムが存在する

### 2-3. 粗利分配方式 vs. 売上歩合方式

日本のコンビニ大手は粗利分配方式が主流だが、小規模FC・海外チェーンでは売上歩合方式も存在する。比較を整理する。

| 項目 | 粗利分配方式（日本コンビニ主流） | 売上歩合方式（一般的なFC） |
|------|---------------------------|----------------------|
| チャージの計算基準 | 売上高 - 仕入原価 = 粗利益 | 売上高（仕入原価を差し引かない） |
| チャージ率の水準 | 高い（粗利ベースのため36〜76%） | 低い（売上ベースのため3〜8%程度） |
| 廃棄ロスの影響 | 廃棄で粗利が下がり本部チャージも下がる | 廃棄に関わらず売上ベースで計算される |
| 本部にとっての利点 | 加盟店との目標一致・廃棄ロス削減へのインセンティブ | 計算が単純・原価管理のリスクを本部が負わない |
| 加盟店にとっての利点 | 廃棄が多い時期も本部チャージが緩和される | チャージが予測しやすい（売上の一定%） |
| ストコン処理の複雑さ | 仕入原価・廃棄ロスも正確に計上する必要あり（複雑） | 売上集計のみ（単純） |

### 2-4. チャージ計算のデータフロー（ストコン中心）

```
[チャージ計算に必要なデータの流れ]

ストコン（店舗側）
  ├── POS売上データ      → 売上高（決済種別・商品別）
  ├── 仕入入荷データ     → 仕入原価（検品・入荷確認）
  ├── 廃棄登録データ     → 廃棄ロス額
  └── 値引き登録データ  → 値引きロス額
         |
         | EOB送信（日次締め）
         v
本部 会計・精算システム
  ├── 粗利計算: 売上高 - 仕入原価 - 廃棄ロス（売価）= 粗利
  ├── チャージ計算: 粗利 × チャージ率 = チャージ額
  ├── 加盟店収入: 粗利 - チャージ = 加盟店取り分
  └── OA口座更新: 月次で加盟店口座に振込
```

## 3. オープンアカウント（OA）方式の仕組み

### 3-1. オープンアカウントとは

「**オープンアカウント（OA）**」は、コンビニFC特有の資金決済方式である。加盟店の売上を本部が一元的に受領・管理し、仕入原価・チャージを控除した残額を加盟店に分配する仕組みを指す。加盟店は自ら仕入れ代金をベンダーに支払う必要がなく、本部が立替えてOA上で一括処理する。

この仕組みにより、以下が実現される。

1. **加盟店の資金繰り安定化**: 仕入代金の日々の支払いが不要
2. **本部による売上データの一元管理**: 精算データの完全性を本部が担保
3. **チャージの自動控除**: 本部が精算時に自動的にチャージを差し引く

### 3-2. OA口座の構造（月次サイクル）

```
[オープンアカウント 月次サイクル]

月初
 ↓
日次: 売上入金
  - 現金売上は集金業者が店舗から回収 → 本部OA口座へ入金
  - 電子マネー・クレジット: 各事業者 → 本部 → OA口座へ入金（数日後）
  - QR決済: 各QR事業者 → 本部 → OA口座へ（数日〜1週間後）

日次〜週次: 仕入代金の立替
  - 共配センターが店舗へ配送した商品の仕入原価を本部が立替
  - OA口座の「借方（仕入控除枠）」に計上

月末
  - 月間粗利益 = 月間売上高 - 月間仕入原価 - 廃棄ロス額
  - チャージ = 月間粗利益 × チャージ率
  - 加盟店収入 = 月間粗利益 - チャージ
  ↓
月次精算
  - 本部 → 加盟店口座へ「加盟店収入」を振込
  - 各種手数料（POSリース料・システム利用料等）も精算
  ↓
翌月繰越
```

### 3-3. OA口座の構成要素（詳細）

| 区分 | 内容 | プラス（収入側） | マイナス（費用側） |
|------|------|-------------|---------------|
| 売上入金 | POS取引の売上全額 | + 日販合計 | — |
| 仕入原価 | 共配センターからの商品代 | — | - 仕入合計 |
| 廃棄ロス | 廃棄商品の仕入原価 | — | - 廃棄額（原価分） |
| チャージ | 粗利 × チャージ率 | — | - チャージ額 |
| 電気代 | タイプCは本部が一部補助 | + 補助額（タイプCのみ） | — |
| POSリース料 | 機器のリース費用 | — | - リース月額 |
| システム利用料 | ストコン等の利用料 | — | - 月額料 |
| 加盟店収入（振込） | 最終的な加盟店への支払い | — | - 振込額 |

### 3-4. OA方式と金融規制

OA方式は「**売上金の代理受領**」の形式をとるため、加盟店と本部の間に「預り金」的な関係が生じる。この仕組みについては、公正取引委員会の調査対象になることもある（加盟店側が不利な契約条件への是正要求の根拠として用いられることがある）。

SIer視点での重要点：

- OA口座の残高・精算データは法的証拠として扱われることがあるため、**データの完全性・変更不可性**の確保が求められる
- 精算データの保持期間（税務上7年、法的には最大10年）に対応するアーカイブ設計が必要

## 4. 日次精算の詳細フローと決済種別処理

### 4-1. 日次精算フロー全体

日次精算（日計処理）は、1営業日分の取引を締め、ストコンが集計データをEOBとして本部へ送信するプロセスである。多くの店舗で早朝（6〜7時）に実施される。

```
[日次精算 全体フロー]

Step 1: POSレジ締め
  ├── 全レジの営業終了操作（各担当者単位）
  ├── レジ内現金の集計（レジプリントアウト）
  └── 担当者別の売上・点数・釣銭確認

Step 2: ストコンによる売上データ収集
  ├── 全POSレジの取引データを自動収集
  ├── 取引種別ごと（現金・IC・クレジット・QR等）に分類集計
  └── 廃棄データ・発注データとの突合

Step 3: 現金実査
  ├── レジ内の現金を実際に数える（実査）
  ├── ストコン帳簿値との差異（オーバー/ショート）を確認
  └── 差異が一定額超の場合 → 原因調査（ログ確認）

Step 4: 精算確定・書類作成
  ├── 差異が許容範囲内 → 精算確定
  ├── 日計表・精算書の印刷
  └── 店長・オーナーの承認

Step 5: EOB送信（本部への締め送信）
  ├── ストコン → 本部へ精算データを送信
  ├── 送信完了確認（タイムスタンプ記録）
  └── 送信失敗時 → 蓄積・再送（通信回復後に自動リトライ）

Step 6: 翌日業務準備
  ├── ストコンの売上カウンタをリセット
  ├── 翌日の発注データ確認
  └── 新しい日付の業務開始
```

### 4-2. 決済手段別の精算処理

現代のコンビニでは多様な決済手段が混在しており、それぞれの精算タイミング・処理フローが異なる。ストコンはこれらを統合管理している。

| 決済手段 | 主な種類 | 店舗での処理 | 入金タイミング | OA口座への反映 |
|---------|--------|-----------|------------|-------------|
| **現金** | 紙幣・硬貨 | レジで受け取り、集金袋へ格納。集金業者が日次回収 | 翌営業日 | 即時反映（集金確認後） |
| **クレジットカード** | Visa/Mastercard/JCB等 | 端末認証後、POSに売上計上 | 3〜7日後（カード会社締め処理） | 数日後にOA口座へ入金 |
| **ICカード電子マネー** | Suica/PASMO/nanaco/WAON/Edy | 残高引き落とし、端末認証後POS計上 | 翌日〜3日後（清算センター経由） | 数日後にOA口座へ入金 |
| **QRコード決済** | PayPay/楽天Pay/au PAY/LINE Pay等 | QRスキャン後POS計上 | 翌日〜1週間後（QR事業者締め） | 数日〜1週間後にOA口座へ入金 |
| **収納代行** | 公共料金・税金・宅配便等 | バーコードスキャン、手数料収入を計上 | 収納代行機関への月次請求後 | 月次精算でOA反映 |
| **プリペイド系** | QUOカード、図書カード | カード読取後POS計上 | 発行会社との定期精算 | 月次精算でOA反映 |

### 4-3. 電子マネー・QR決済の精算処理詳細

電子マネーとQR決済は「**売上は発生したが現金は手元にない**」状態が続くため、タイムラグ管理が重要である。

```
[電子マネー・QR精算タイムライン]

Day 0（取引日）
  POS決済 → 電子マネー端末認証 → ストコン: 売上計上
                                          |
                                          v
Day 1〜3（清算センター処理）
  電子マネー事業者（Suicaの場合: ビューカード/JR東日本）
  → 加盟店チェーン本部 → OA口座への入金指示

Day 2〜7（QR決済の場合）
  QR決済事業者（PayPayの場合: ソフトバンク/PayPay株式会社）
  → 加盟店チェーン本部 → OA口座への入金指示

月次精算時
  全決済手段の入金が完了した状態でOA月次精算を確定
  → 加盟店への振込計算
```

**ストコンが管理する決済別データ構造**:

| データ項目 | 内容 |
|-----------|------|
| 決済種別コード | 現金/クレジット/IC電子マネー/QR/収納代行 等を区分するコード |
| 決済事業者コード | nanaco/WAON/Suica/PayPay等の事業者識別 |
| 取引金額 | 税込み取引金額 |
| 取引日時 | タイムスタンプ（POSの時計に依存） |
| 端末ID | どのレジ端末で処理されたか |
| 承認番号 | クレジット・電子マネーの認証番号 |

### 4-4. 現金管理・差異調整プロセス

```
[現金過不足発生時の対応フロー]

現金実査で差異発見
        |
        v
差異額・方向の確認（オーバー / ショート）
        |
        +------- 少額（許容範囲内: 例100円未満） ------→ 現金過不足として記録・精算確定
        |
        +------- 中程度（要調査: 例100〜1,000円） ------→ レジログで原因調査
        |                                                     ↓
        |                                              原因特定 → 修正登録
        |                                              特定不能 → 過不足記録
        |
        +------- 大額（要エスカレ: 例1,000円超） ------→ SVへ連絡
                                                           ↓
                                                   詳細調査・店舗指導
                                                   (不正・盗難の可能性も)
```

**差異原因の類型**:

| 原因分類 | 具体例 | ストコンでの対処 |
|---------|--------|--------------|
| 釣銭間違い | お釣り多く/少なく渡した | レジ操作ログ確認・過不足記録 |
| 入力ミス | 金額の二重入力・取消漏れ | 取消・返品記録の確認 |
| 偽造通貨 | 偽札を受け取った | 過不足記録・警察届出 |
| セルフレジ誤操作 | 客の操作ミス | セルフレジログ確認 |
| システム障害 | POSフリーズ中の取引 | 障害ログと手書き記録の突合 |

## 5. 会計データの流れ（店舗→本部→税務申告）

### 5-1. 会計データフロー全体

コンビニの会計データは、店舗レベルのPOS取引データから出発し、本部の会計システムを経由して、本部・加盟店それぞれの税務申告へとつながる多層構造をとる。

```
[会計データフロー 全体図]

┌──────────────────────────────────────────────────────────┐
│ 店舗レベル                                               │
│  POS端末 → ストコン(日次精算) → EOB送信                 │
│   └── データ: 売上明細・廃棄・発注・在庫・決済種別別集計  │
└────────────────────────┬─────────────────────────────────┘
                         │ 専用回線・VPN（EOB通信）
                         v
┌──────────────────────────────────────────────────────────┐
│ 本部システム（チェーン本部）                             │
│  ├── 精算システム（OA口座管理）                         │
│  │    └── 店舗別チャージ計算・加盟店収入計算              │
│  ├── 会計システム（本部財務会計）                        │
│  │    └── チャージ収入・物流収入・金融収入の計上          │
│  └── DWH（データウェアハウス）                          │
│       └── 全チェーン売上データ・単品管理・需要予測       │
└────────────────────────┬─────────────────────────────────┘
                         │
         ┌───────────────┴───────────────┐
         v                               v
┌────────────────┐              ┌─────────────────────┐
│ 本部 税務申告  │              │ 加盟店 税務申告      │
│ ・法人税       │              │ ・個人事業 or 法人   │
│ ・消費税       │              │ ・加盟店収入が売上   │
│ ・決算書       │              │ ・人件費・廃棄が費用 │
└────────────────┘              └─────────────────────┘
```

### 5-2. 仕入れ原価の計算方式

コンビニの仕入原価計算には、主に「**売価還元法**」と「**原価法**」の2つが存在する。日本の大手コンビニは商品数が膨大（1店舗あたり約3,000〜5,000 SKU）なため、売価還元法が採用されることが多い。

#### 売価還元法（コンビニ主流）

```
[売価還元法 計算ロジック]

原価率 = 期首商品原価 + 当期仕入原価
         ──────────────────────────────
         期首商品売価 + 当期仕入売価

期末商品原価 = 期末商品売価（棚卸） × 原価率

売上原価 = 期首商品原価 + 当期仕入原価 - 期末商品原価
```

**売価還元法のメリット（コンビニ向き）**:

| 観点 | 内容 |
|------|------|
| 管理コスト低減 | 品目ごとの原価管理が不要。棚卸は売価で行えばよい |
| 多品種対応 | 数千SKUを個別に原価管理せずに済む |
| POS連携 | 売価データはPOSで常時管理されているため入力コスト小 |

**売価還元法の限界**:
- 廃棄・値引き・盗難が多い場合、原価率の歪みが生じやすい
- 商品カテゴリをまたがった計算精度の低下（高原価率商品と低原価率商品の混在）

#### 原価法（一部精緻な店舗・高価格帯商品で採用）

```
[原価法 計算ロジック]

SKUごとに実際の仕入原価を管理
  ↓
売上原価 = 販売数量 × 商品ごとの仕入単価
  ↓
期末在庫 = 期首在庫 + 仕入数量 - 販売数量 - 廃棄数量
```

**原価法が向くケース**: タバコ（在庫管理が厳格）・酒類・高単価商品

### 5-3. 加盟店の税務申告における会計処理

加盟店（個人事業主の場合）の確定申告における主要科目を整理する。

| 科目 | 内容 | 金額の出所 |
|------|------|----------|
| 売上高 | 本部から通知される「売上高」（OA精算書） | 本部OA精算書（月次） |
| 売上原価 | 仕入原価（廃棄分含む） | 本部OA精算書（月次） |
| 粗利益 | 売上高 - 売上原価 | 計算（本部通知と一致確認） |
| チャージ（加盟店費用） | 粗利益 × チャージ率 | 本部OA精算書 |
| 人件費 | アルバイト・パートの賃金 | 加盟店の賃金台帳 |
| 光熱費 | 電気・ガス・水道 | 請求書（タイプAは全額加盟店） |
| 廃棄ロス | 廃棄商品の原価（すでに売上原価に含まれている） | ストコン廃棄レポート |
| 減価償却費 | タイプAの場合、建物・設備の償却 | 固定資産台帳 |

### 5-4. 消費税の扱い（軽減税率への対応）

2019年10月の消費税増税・軽減税率導入以降、コンビニの会計処理は複雑化した。

```
[軽減税率対応の例]

軽減税率（8%）適用:
  - 食品（酒類を除く）
  - 新聞（定期購読）

標準税率（10%）適用:
  - 酒類・タバコ
  - 日用品・文具
  - サービス（収納代行・宅配）
  - 外食（イートイン使用時）
  ※ テイクアウト・持ち帰りは8%（同一商品でも食べ方で税率が変わる）
```

**ストコンの対応**: 商品マスタに軽減税率フラグを設定。POS取引時に自動で税率を判定し、日次精算・EOBデータに8%売上・10%売上を分離して記録する。

## 6. 廃棄・値引きの会計処理

### 6-1. 廃棄ロスの会計処理（誰が負担するか）

廃棄ロスの負担は「**加盟店の粗利が減少する形**」で実質的に加盟店が負担することになる。

```
[廃棄ロスの会計処理メカニズム]

廃棄が発生すると:
  売上 = 0（売れなかった）
  仕入原価 = 発生済み（仕入れた費用）

  粗利への影響:
  粗利 = 売上（0） - 仕入原価（発生済み）= マイナス
  ↓
  月間粗利が廃棄ロス分だけ減少
  ↓
  チャージ計算: 粗利 × チャージ率
  粗利が減少 → チャージも減少（本部も間接的に影響を受ける）
  ↓
  加盟店収入: 粗利 - チャージ
  チャージ減少分より粗利減少の方が大きい → 加盟店の取り分が実質減少
```

**チェーン別廃棄補填制度**

| チェーン | 廃棄補填の仕組み | 補填率目安 |
|---------|---------------|---------|
| セブン-イレブン | 廃棄ロスの一定率を「見切り販売奨励金」等で補填 | 加盟店タイプ・実績による |
| ファミリーマート | 廃棄ロス一定額を超える部分を本部が補填 | 超過分の15〜25%程度 |
| ローソン | 廃棄補填制度あり（金額・条件は非公開が多い） | 契約内容による |

### 6-2. 廃棄のストコン会計処理フロー

```
[廃棄 → 会計処理フロー]

1. 廃棄入力（ストコン・ハンディ）
   - 商品コード・数量・廃棄区分（期限廃棄/破損等）を入力
   - 廃棄時刻のタイムスタンプ自動記録

2. ストコン内部処理
   - 理論在庫から廃棄数を減算
   - 廃棄金額を計算（廃棄数 × 仕入単価 または 廃棄数 × 売価）
   - 廃棄ロス累計を更新

3. 日次精算時（EOB送信）
   - 廃棄明細データを本部へ送信
   - 本部側で廃棄ロス額を粗利計算に反映

4. 月次精算
   - 廃棄ロス合計が粗利計算に算入済み
   - OA口座の月次確定時に廃棄の影響が加盟店収入に反映
```

### 6-3. 値引き（マークダウン）の会計処理

値引き販売（消費期限前の値引き）の会計処理は廃棄とは異なる扱いになる。

```
[値引き販売の会計処理]

定価 500円 の弁当を 30% 値引き（350円）で販売した場合:

通常販売:
  売上高   = 500円
  仕入原価 = 350円（例）
  粗利     = 150円

値引き販売（350円で販売）:
  売上高   = 350円
  仕入原価 = 350円
  粗利     = 0円
  値引きロス = 150円（定価売上時と比べた粗利の減少分）

廃棄した場合:
  売上高   = 0円
  仕入原価 = 350円
  粗利     = -350円（廃棄ロス）

比較:
  値引きロス（150円損失）< 廃棄ロス（350円損失）
  → 値引き販売の方が経済的損失が小さい
```

**ストコンでの値引き処理**:

| 値引き方式 | ストコン・POS処理 | 会計上の記録 |
|----------|--------------|-----------|
| シール値引き | POSで値引き後価格を手入力または割引コードスキャン | 値引き前・後の売上差額を「値引き額」として記録 |
| 電子値引き（システム自動） | POSが自動で時間帯・残数に応じた値引き額を計算 | 値引き額・値引き率・適用時刻を自動記録 |
| ポイント値引き | ポイント付与分を実質値引きとして処理 | ポイント費用は本部が一部負担（チェーンにより異なる） |

## 7. ストコン移行での考慮事項（PM視点）

ストコンをオンプレからクラウドへ移行する際、フランチャイズ会計構造に起因する固有リスクと対応策を以下にまとめる。

### PM視点 1: 精算データの移行・整合性確保

**リスク**: 移行前後の精算データの連続性が断絶すると、OA口座の計算がずれてチャージ計算・加盟店収入計算に誤りが生じる。

| 確認事項 | 内容 | 対応アクション |
|---------|------|-------------|
| 精算データ移行範囲 | ストコン内の日次精算データ・EOB送信履歴の全期間分を移行対象に含めるか確認 | 移行スコープ定義書に「精算データ X年分」を明記 |
| 移行前後の残高突合 | 旧ストコン最終EOBデータと新ストコン開始時データの残高一致確認 | 移行テスト項目に「精算データ突合」を必須化 |
| OA口座との同期確認 | 移行日前後で本部OA口座の計算に齟齬が生じないか | 本部会計システムチームとのインターフェース確認会議を設定 |
| データ形式変換 | 旧ストコンのデータ形式（独自仕様が多い）を新システムへ変換する際の精度 | 変換テスト（サンプルデータ100件以上で突合） |

### PM視点 2: 日次締め処理のダウンタイム影響

**リスク**: 移行作業中にストコンが停止すると、その日の精算（EOB送信）ができなくなり、本部OA口座の計算が停止する。現金管理・チャージ計算のタイムラグが発生する。

| 対応策 | 内容 |
|-------|------|
| 移行タイミングの選定 | 日次精算（EOB送信）が完了した直後（翌朝7〜8時台）を移行ウィンドウに設定。精算処理中のダウンタイムを回避 |
| 締め処理の冗長化 | 旧ストコンと新ストコンを並行稼働させ、二重で精算データを生成・確認してから切り替え |
| 手動精算手順書の準備 | 万一の障害時に紙ベースで日次精算を実施できる手順書を店舗に配布 |
| 切り戻し手順の確立 | 移行後に精算データ不整合が発覚した場合の旧システムへの切り戻し手順を事前に定義 |

### PM視点 3: 会計監査への対応（データ保持期間・追跡可能性）

**リスク**: ストコン移行後、旧システムのデータがアクセス不能になると、税務調査や公正取引委員会の調査対応で証拠提出ができなくなる。

| 保持対象データ | 法的根拠 | 保持期間 | 移行時の対応 |
|-------------|---------|---------|-----------|
| 売上データ（POSレシート相当） | 所得税法・法人税法 | 7年 | 旧システムデータを移行先またはアーカイブストレージに保持 |
| 仕入・廃棄データ | 法人税法施行規則 | 7年 | 同上 |
| OA精算書（月次） | 商法・会社法 | 10年 | 本部会計システムに保存（加盟店も写しを保存） |
| 発注・入荷データ | 商法 | 5〜7年 | 旧システムデータの移行またはアーカイブ化 |
| 電子マネー・QR決済明細 | 割賦販売法・資金決済法 | 5年 | 各決済事業者との連携ログも含めて保持 |

**追跡可能性（トレーサビリティ）の確保**:

```
[移行後に必要な追跡パス]

特定の取引（例: 2026年X月X日 Y時Z分 A商品 B円 Suica決済）
  ↓
旧ストコン → 新ストコン のどちらに記録されているかを特定できる
  ↓
EOBデータ（本部受信ログ）と突合
  ↓
OA口座の精算記録に該当月の計上を確認
  ↓
税務申告書の売上高との整合性確認
```

### PM視点 4: 決済手段多様化への継続対応

**リスク**: ストコン移行後も新たな決済手段（新QR規格・デジタル通貨・後払いサービス等）が追加される可能性があり、新ストコンの決済インターフェースが拡張可能かどうかを事前確認しておく必要がある。

| 確認事項 | 質問例 |
|---------|--------|
| 新決済手段の追加コスト | 新たなQR決済（例: 新サービス）の追加は設定変更で対応できるか、プログラム改修が必要か |
| 精算処理への影響 | 新決済手段を追加した場合、日次精算・EOBデータ形式が変わるか |
| 本部側インターフェース | 新決済データを本部OA口座に取り込む処理の変更コスト |

### PM視点 5: 加盟店向けの移行コミュニケーション

**リスク**: ストコンの日次精算手順が変わることで、加盟店オーナー・店長が混乱し、精算ミス・EOB送信漏れが増加する。

| 対応策 | 内容 |
|-------|------|
| 操作手順書の更新・配布 | 新ストコンの日次精算画面・手順を旧システムとの差分を明示した形で作成 |
| SV向け先行トレーニング | SVが加盟店指導の第一線となるため、移行前にSVへの集合研修を実施 |
| 移行直後のホットライン設置 | 精算時刻（早朝6〜7時）に対応できるサポート窓口を設置（移行後1〜2週間） |
| OA精算書の変更告知 | 移行後に本部から送付されるOA精算書のフォーマットが変わる場合、加盟店に事前告知 |

## 8. 用語集

| 用語 | 読み | 定義 |
|------|------|------|
| フランチャイズ（FC） | フランチャイズ | 本部が加盟店にブランド・システム・商品を提供し、加盟店はロイヤリティを支払う事業モデル |
| チャージ | チャージ | コンビニFCにおけるロイヤリティ。粗利益に対するチャージ率で計算される |
| チャージ率 | チャージりつ | 粗利益に対するチャージ（ロイヤリティ）の割合。タイプ・チェーンにより異なる |
| 粗利益 | そりえき | 売上高 - 仕入原価。コンビニのチャージ計算の基準となる利益 |
| 廃棄ロス | はいきロス | 廃棄した商品の仕入原価分の損失。粗利益を直接減少させる |
| 値引きロス | ねびきロス | 定価との差額の損失。廃棄ロスよりも損失が小さいことが多い |
| オープンアカウント（OA） | オープンアカウント | 本部が管理する加盟店の精算口座。売上・仕入・チャージを集中管理し月次精算する |
| EOB | イーオービー | End of Business。ストコンから本部へ送る日次の締め送信データ |
| 日計表 | にっけいひょう | 1日の売上・廃棄・客数等をまとめた精算レポート |
| 現金実査 | げんきんじっさ | レジ内の現金を実際に数えて帳簿と照合する作業 |
| オーバー（現金過剰） | オーバー | 実際の現金が帳簿より多い状態 |
| ショート（現金不足） | ショート | 実際の現金が帳簿より少ない状態 |
| 売価還元法 | ばいかかんげんほう | 棚卸在庫を売価で把握し、原価率を使って期末在庫原価を算出する在庫評価方法 |
| 原価法 | げんかほう | 商品ごとの実際仕入原価を使って売上原価・期末在庫を算出する方法 |
| 軽減税率 | けいげんぜいりつ | 食品（酒類除く）・定期購読新聞に適用される消費税率8%（標準10%と区別） |
| スーパーバイザー（SV） | スーパーバイザー | 本部から加盟店へ派遣され、経営指導・データ活用支援を行う担当者。1人で7〜12店舗を担当 |
| 日販 | にっぱん | 1日あたりの売上高。店舗パフォーマンスの代表指標 |
| タイプA | タイプエー | 加盟店が土地・建物を所有するFC契約タイプ。チャージ率は低め |
| タイプC | タイプシー | 本部が建物（または土地建物）を用意するFC契約タイプ。チャージ率は高め |
