## 1. エグゼクティブサマリー

2024〜2026年のコンビニ業界では、長年オンプレ専用機として運用されてきたストアコンピューターの**クラウド移行**が本格化フェーズに入った。セブン・ファミマ・ローソンの三大チェーンは異なるクラウドベンダー（AWS/Azure/マルチ）を採用しながらも、共通して「分析・AI機能をクラウドへ先行移行し、決済・販売処理はエッジで維持するハイブリッド構成」を選択している。AI需要予測は3社とも本稼働段階に達しており、廃棄削減15〜20%などの成果が出始めている。エッジAI・電子棚札・エッジデバイス管理（MDM）が次の主戦場であり、MACHアーキテクチャへの段階的移行（〜2030年）が中長期ロードマップの標準形となっている。PM視点では、PCI DSS v4.0対応・マスタ同期の原子性・EOBの時間分散設計・オフライン継続動作の要件定義が、「見落とされがちだが設計の根幹を左右する」要件として特に重要である。

## 2. ストアコンピューターの最新技術動向（2024〜2026年）

### 2-1. 業界全体のベクトル

2024〜2026年に起きている変化を一言で表すと「**オンプレ専用機からの脱却と、エッジ＋クラウドのハイブリッド化**」である。

| トレンド | 2023年以前 | 2024〜2026年 |
|---------|-----------|-------------|
| 処理場所 | 店舗内ストコンで完結 | エッジ（販売・決済）＋クラウド（分析・AI）の分担 |
| ハードウェア | Windows専用機（5〜7年周期更新） | Thin Client化・コンテナ化・クラウドAPI化 |
| AI活用 | ルールベース発注が中心 | 需要予測AI本稼働、エッジAI推論開始 |
| データ基盤 | 本部への夜間バッチ転送 | リアルタイムストリーミング（Kafka/Kinesis） |
| セキュリティ | 境界型防御 | ゼロトラスト適用、PCI DSS v4.0対応 |

### 2-2. アーキテクチャの標準形：「ファスト/スロー」2層構造

現在の業界標準アーキテクチャは、処理特性によってエッジとクラウドを明確に分ける**2層構造**である。

```
[エッジ層 ─ ファスト処理]          [クラウド層 ─ スロー処理]
-------------------------------    ---------------------------------
決済処理（〜500ms）                AI需要予測・発注最適化
商品マスタキャッシュ参照（〜200ms） 全社分析・DWH
廃棄管理・温度監視                 マスタ管理・MDM
エッジAI推論（棚カメラ等）          MLOpsサイクル・モデル再学習
回線断時のローカル継続動作          本部→店舗マスタ配信
```

この分割が意味するのは、「クラウドが落ちても店舗は販売を継続できる」という**オフラインファースト設計**が大前提であることだ。

### 2-3. データフロー全体像

```
[POS端末（2〜8台）]
   │ (1) 売上トランザクション（リアルタイム or バッチ）
   ▼
[ストアコンピューター]
   │ (2) 内部処理: 理論在庫更新、売上累計、廃棄候補検知、AI発注フィード
   │ <── 商品マスタ・価格マスタをPOSへ配信（下り）
   ▼
[本部システム / クラウド]
   │ (3) リアルタイム売上、発注データ、EOBデータ（上り）
   │ ──> 商品マスタ・棚割マスタ・キャンペーン情報（下り）
   ▼
[DWH / AI基盤]
```

**データ量概算（1店舗/日）**: 売上トランザクション1〜5MB、商品マスタ更新数百KB〜1MB、EOB数十MB。全国5.5万店規模では本部側のEOBだけで日次数TB規模になる。

### 2-4. リアルタイム性要件

| 機能 | 許容レイテンシ | 処理場所 |
|------|-------------|---------|
| 決済処理（現金） | 〜500ms | POSローカル |
| 電子マネー決済 | 〜1秒 | POS＋決済センター |
| 商品マスタ参照（スキャン時） | 〜200ms | POSキャッシュ |
| 在庫更新（販売後） | 〜数秒 | ストコンローカル |
| 売上の本部送信 | 〜数分許容 | ストコン→本部 |
| AI発注推奨更新 | 発注締め前（数時間） | クラウド |

### 2-5. 障害時フォールバック設計

| 障害シナリオ | フォールバック設計 |
|------------|----------------|
| 回線断 | ストコンがローカル処理継続。回線復旧後に蓄積データ一括送信 |
| ストコン停止 | POSはマスタキャッシュで独立動作継続。翌日の代替機交換が標準 |
| 商品マスタ配信障害 | ストコンのローカルマスタで継続。差分の遅延配信で追いつく |
| 決済センター通信断 | 現金決済への誘導。一部チェーンはオフライン承認蓄積方式 |
| クラウド障害 | ルールベース自動発注（従来方式）に自動フォールバック |

**設計原則**: 必須機能（販売・現金決済・廃棄管理・温度監視）はローカル完結。通信依存機能（収納代行・チケット発券・電子マネー）は停止許容。

## 4. クラウド移行・エッジコンピューティングのトレンド

### 4-1. MACHアーキテクチャへの移行パス

業界の中長期ロードマップとして、**MACHアーキテクチャ**（Microservices / API-first / Cloud-native / Headless）への段階的移行が標準指針になりつつある。

```
フェーズ1（〜2026年）: 分析・AI系のクラウドリフト
  └── 既存ストコンをそのまま残し、分析・AI機能のみクラウドへ移行

フェーズ2（2026〜2028年）: 店舗端末のシン化・API化
  └── ストコン機能をAPIとして切り出し、店舗端末をThin Client化

フェーズ3（2028〜2030年）: フルクラウドネイティブ（専用HW廃止）
  └── 店舗固有ハードウェアを廃止し、汎用端末＋クラウドAPIで完結
```

### 4-2. エッジコンピューティングの実装パターン

**エッジデバイスの分類と主要製品**:

| エッジデバイス種別 | 代表製品 | 用途 |
|----------------|---------|------|
| マネージドエッジ（クラウド延伸） | AWS Outposts、Azure Stack Edge | クラウドAPIをオンプレで動作させる |
| IoT統合基盤 | AWS IoT Greengrass | センサーデータ収集・エッジ推論 |
| エッジAIチップ | NVIDIA Jetson Orin、Intel OpenVINO | 棚カメラリアルタイム画像認識 |
| 軽量Kubernetes | K3s（RAM 512MB〜1GBで動作） | エッジ側コンテナオーケストレーション |

**エッジAIの標準実装パターン**: 棚カメラ + NVIDIA Jetson Orinでリアルタイム欠品検知→ストコン画面にアラート。モデル更新はクラウドからOTA配信。

### 4-3. エッジデバイス管理（MDM）の重要性

5万店×5台=**25万台規模**のデバイス管理は、SIer視点で「最も見落とされがちな要件」とされる。

- **AWS IoT Fleet Hub** / **Azure Device Update for IoT Hub** によるOTA更新・遠隔監視
- ソフトウェア更新・障害対応が運用コストの大部分を占める
- 中央管理アーキテクチャの設計が不可欠

### 4-4. Eventual Consistency設計

クラウド化に伴い、エッジ側のデータがクラウドに同期されるまでのラグ（数秒〜数分）の間に生じる不整合への対処が必要になる。「エッジとクラウドの整合性担保」をシステム設計の初期段階から議論する必要がある。

### 4-5. 店舗内通信の刷新

- **Wi-Fi 6（802.11ax）**: IoTデバイス接続の高密度化対応
- **ローカル5G**: 冷蔵ケース温度センサー・自動搬送ロボット等との連携強化
- **電子棚札（ESL: Electronic Shelf Label）**: ストコンからの価格マスタ更新をトリガーに自動更新

## 5. AI・データ活用の方向性

### 5-1. AI活用の成熟度マップ（2025年時点）

| AI活用領域 | 成熟度 | 代表事例 |
|-----------|--------|---------|
| 需要予測・AI発注 | 実用段階（3社とも本稼働） | セブンの廃棄削減15〜20%、ローソンAI.CO全店 |
| 欠品・在庫可視化 | 一部店舗で実用段階 | 棚カメラ＋エッジAI |
| 廃棄削減・ダイナミックプライシング | 試験〜実用段階 | 消費期限連動自動値引き |
| シフト最適化 | 試験段階 | — |
| LLMによる発注支援・Q&A | 実証実験段階 | ファミマAzure AI Foundry（GPT-4o）実証 |
| 映像AI（作業効率化） | 実証段階 | NEC×ローソン 映像認識AI（EVA）実証（2025年10月〜） |

### 5-2. AI需要予測の技術詳細

- **多変量モデル**: 気象API・地域イベント・ポイント利用履歴を組み合わせ
- **再学習サイクル**: 月次→週次に短縮（SageMaker Pipelines活用）
- **XAI（説明可能AI）の重要性**: 店長・オーナーが発注根拠を理解できないと現場の信頼が得られない。発注根拠の可視化UIが必要

### 5-3. MLOpsサイクルの設計

AI需要予測モデルの「精度維持」には継続的なMLOpsサイクルが必須であり、SIerの差別化領域として有望である。

```
データ収集（店舗センサー・POS）
    → 特徴量エンジニアリング
    → モデル再学習（週次サイクル）
    → A/Bテスト・精度評価
    → 本番デプロイ（SageMaker Pipelines等）
    → 監視・ドリフト検知
    → （繰り返し）
```

### 5-4. データ品質問題

クラウド移行でデータを統合する際、長年の運用で発生した**商品マスタの重複・表記ゆれ**等のデータ品質問題が表面化する。移行前のデータクレンジング工数が大幅に増加するリスクを予算に織り込む必要がある。

### 5-5. リアルタイムデータ基盤

Apache Kafka（Amazon MSK）・AWS Kinesis・Azure Event Hubsを使ったストリーミング処理で、店舗売上・在庫データのリアルタイム分析基盤を構築する案件が2024〜2025年に増加している。

## 6. ハードウェア進化とシステムアーキテクチャ

### 6-1. Thin Client化の意味

従来のストコンはWindows専用機（各社専用設計）だったが、クラウド移行に伴い**Thin Client化**が進む。

| 方式 | 説明 | メリット | デメリット |
|-----|------|---------|----------|
| 従来型（オンプレ専用機） | 業務ロジックを店舗内で完結 | 高可用性、低レイテンシ | 更新コスト高、専用ハード依存 |
| Thin Client化 | ローカルはUI表示のみ、業務ロジックをクラウドへ | ハードコスト削減、一括アップデート | 回線依存度増大 |
| ハイブリッド | 販売・決済はエッジ、分析・AIはクラウド | バランス型 | 設計複雑度増大 |

**現実的な移行タイムライン**: セブンイレブンの全国2.1万店Thin Client化は2027年以降が現実的。段階的移行（Strangler Fig）が主流。

### 6-2. コンテナ化・Kubernetes活用

- AWS EKS、Azure AKS上でのストコン後継システム構築事例が国内でも出始めている
- エッジ側では**K3s**（軽量Kubernetes、RAM 512MB〜1GBで動作）が注目される
- コンテナ化により、可搬性向上とデプロイ自動化が実現される

### 6-3. 省人化・マルチスクリーン対応

- セルフレジ・セミセルフレジの全店展開が加速（セブン：2025年末までに全2.1万店計画、ローソン：4,500店以上）
- 完全無人化は法規制（酒類・タバコ販売の年齢確認義務）の壁で限定的
- セルフレジUI設計では多言語対応・バリアフリー設計が必要

### 6-4. 決済基盤のモダナイズ

- キャッシュレス決済比率は2024年時点で60〜70%（業界推定）
- PayPay・楽天Pay・交通系ICカードの乱立から、**決済プラグイン型アーキテクチャ**（新決済手段をプラグインとして追加可能）への移行がコスト削減の鍵

### 6-5. セキュリティ（PCI DSS v4.0）

**2025年3月31日完全移行義務化**の主要変更点：

| 変更点 | 内容 |
|-------|------|
| カスタマイズドアプローチ導入 | リスクベースで独自コントロール定義が可能に。証跡整備・文書化工数が増大 |
| Req.6（ソフトウェアセキュリティ）強化 | WAF必須化、SRI（Subresource Integrity）義務化。決済画面・モバイルアプリ対象 |
| Req.8（認証強化） | MFAの適用範囲がCDE全非コンソールアクセスに拡大。ストコン管理画面・本部システムへのリモートアクセスにMFA必須 |
| 将来日付要件 | 2025年3月31日対応分と2026年3月31日猶予分が混在。工程管理で区別が必要 |

**コンビニ特有のスコープ広範化リスク**: ATM設置・マルチメディア端末・収納代行等、一般小売より決済タッチポイントが多い。P2PE（Point-to-Point Encryption）やトークナイゼーションでCDE最小化が鍵。

## 7. ストコン移行案件との関連

NECがPMとして参画するローソン向けコンビニストコンのオンプレ→AWSクラウド移行案件において、本ノートの知識は以下の観点で直接活用できる。

### 7-1. 移行スコープの設計判断（最重要）

ソースレポートが繰り返し強調する「**決済・EOB等のミッションクリティカル機能と、発注支援・分析機能を分離して段階移行する**」という原則は、提案フェーズからアーキテクチャ設計まで一貫して適用すべき判断軸である。

- 「全部まとめて移行」は失敗リスクが高い
- セブンが採用した**Strangler Figパターン**が参考になる（機能ドメインごとの段階切り出し）
- 「どの処理をどこで実行するか」の原則（エッジ：販売・決済・廃棄、クラウド：AI・分析・マスタ）を要件定義で明示する

### 7-2. 見落とされがちな設計要件の先取り

ソースレポートが「SIer視点で見落とされがちな要件」として明示している4項目は、参画初期から議題に上げるべきである：

| 要件 | なぜ重要か | 対応タイミング |
|------|-----------|-------------|
| EOBの時間分散設計 | 全店一斉のEOBがクラウド輻輳を引き起こす | アーキテクチャ設計 |
| マスタ同期の原子性（POS複数台への一斉配信） | 切り替えタイミングのズレが販売エラーにつながる | 詳細設計 |
| オフライン継続動作の要件定義 | 回線断時の店舗運営継続が保証されないと加盟店クレームになる | 要件定義 |
| エッジデバイスの5〜7年TCO算出 | ハード選定時にOTA更新・MDM運用コストを忘れると後から大きく膨らむ | 提案 |

### 7-3. ローソン固有の文脈への対応

WBS 2.5.1（NEC×ローソンDX戦略）で得た知識と組み合わせると、移行案件は単なるインフラ更新ではなく**Real×Tech Lawsonの土台作り**として位置付けられる。

- 移行後のAWS基盤が**AI.CO（全店導入済みAI発注システム）**のデータ連携強化につながることを語れる
- **LDI（ローソンデジタルイノベーション）**との役割分担を契約・スコープで明確化する（内製化領域との競合回避）
- KDDIがマルチクラウドを好む傾向があるため、「なぜAWSか」の技術的根拠を早期に準備する

### 7-4. PCI DSS v4.0対応の組み込み

2025年3月31日の完全移行義務化を踏まえ、AWS移行設計にPCI DSS v4.0要件を最初から組み込む必要がある。

- MFA必須化（Req.8）：ストコン管理画面へのリモートアクセスへの対応
- CDE最小化設計：P2PEまたはトークナイゼーションによるスコープ縮小
- 「将来日付要件」（2026年3月31日猶予分）はプロジェクトスケジュールと重なる可能性があり、工程管理で区別が必要

### 7-5. MLOpsサイクルとAI需要予測の連携設計

ローソンはAI.COを全店導入済みであり、AWS移行後の**AI需要予測モデルの再学習・改善サイクル（MLOps）**の設計がプロジェクトの付加価値提案になる可能性がある。SageMaker Pipelinesを活用した週次再学習サイクルの設計は、セブンイレブンの先行事例が参考になる。

## 10. 理解度セルフチェック

以下の問いに自分の言葉で答えられるか確認する。

**Q1. ストコンのクラウド移行においてなぜ「全部まとめて移行」は失敗リスクが高いのか？**

<details>
<summary>解答例を確認する</summary>

決済・POSコア処理はミッションクリティカルであり、数百ms〜1秒以内のレイテンシが要求される。クラウドへの全移行は回線依存度を増大させ、障害時に店舗運営が止まるリスクがある。また商品マスタのデータ品質問題が全システム停止と同時に表面化すると対処困難になる。Strangler Figパターン（機能ドメインごとの段階切り出し）が標準的なアプローチ。

</details>

---

**Q2. EOBの「時間分散設計」が必要な理由を説明できるか？**

<details>
<summary>解答例を確認する</summary>

全国5.5万店が同じ閉店時刻（例：深夜0時）にEOBデータを一斉送信すると、クラウド側に瞬間的な大量トラフィックが集中し輻輳（ネットワーク混雑・処理キュー溢れ）が発生する。店舗ごとに閉店時刻をずらしたり、送信タイミングをランダムにオフセットしたりすることで分散させる設計が必要。全国規模では日次数TBのデータが集中するため、この設計を怠るとクラウド側のコストやSLA違反リスクが高まる。

</details>

---

**Q3. ローソンがセブン・ファミマと異なりマルチクラウド戦略を採る背景は何か？**

<details>
<summary>解答例を確認する</summary>

2024年にKDDIと三菱商事の完全子会社化が完了したことで、KDDI独自の通信・クラウド基盤（MECや専用線）を積極活用するインセンティブが生まれた。IoTデータ収集はAWS、一部MLはGCP、通信系はKDDIクラウドという役割分担になっており、KDDI回線集約とReal×Tech Lawsonの実現に必要な技術を各クラウドから選択するマルチクラウド戦略を採っている。

</details>

---

**Q4. PCI DSS v4.0でストコン管理画面に何が求められるようになったか？**

<details>
<summary>解答例を確認する</summary>

Req.8（認証強化）により、MFA（多要素認証）の適用範囲がCDE（カードデータ環境）へのすべての非コンソールアクセスに拡大された。ストコン管理画面・本部システムへのリモートアクセスにMFAが必須となった。また「将来日付要件」として2026年3月31日まで猶予される要件も混在しており、AWS移行プロジェクトのスケジュールと重なる可能性があるため工程管理での区別が必要。

</details>

---

**Q5. SIerとして差別化できる領域として、ソースレポートが挙げているものを3つ以上述べよ。**

<details>
<summary>解答例を確認する</summary>

1. **MLOps基盤構築と運用**: AI需要予測モデルの継続的改善サイクル（週次再学習・精度監視）
2. **エッジデバイス管理基盤**: MDM・OTA更新・遠隔監視の統合管理（25万台規模）
3. **データガバナンス・マスタ管理**: マルチシステム商品マスタのゴールデンレコード化
4. **クラウド移行PMO**: 段階移行計画の策定・ベンダー調整（Strangler Fig実行支援）
5. **セキュリティ（ゼロトラスト化）**: エッジデバイス増加に伴うPCI DSS v4.0対応

</details>
