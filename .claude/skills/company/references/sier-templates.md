# SIer業務特化テンプレート集

このファイルは各種ドキュメント生成時に参照するテンプレート集です。
ADR、ポストモーテム、テスト戦略、DWH設計など、SIer業務に特化したテンプレートを収録しています。

---

## 1. ADR（Architecture Decision Record）テンプレート

```markdown
---
adr_id: "ADR-{{NUMBER}}"
created: "{{YYYY-MM-DD}}"
status: proposed  <!-- proposed / accepted / deprecated / superseded -->
deciders: []
---

# ADR-{{NUMBER}}: {{TITLE}}

## ステータス
{{STATUS}}

## コンテキスト
何が問題か？なぜ意思決定が必要か？

## 決定
何を決めたか？

## 選択肢の比較

| 観点 | 選択肢A | 選択肢B | 選択肢C |
|------|---------|---------|---------|
| メリット | | | |
| デメリット | | | |
| コスト | | | |
| リスク | | | |

## 結果
この決定によってどうなるか？トレードオフは？
```

### ADR 運用ルール
- 番号は `.company/architecture/adrs/` 配下の連番
- ステータス遷移: proposed → accepted → (deprecated | superseded)
- superseded の場合は後続ADRのIDを記載する
- 過去のADRは変更不可。新しいADRで上書きする

## 2. ポストモーテムテンプレート

```markdown
---
postmortem_id: "PM-{{NUMBER}}"
created: "{{YYYY-MM-DD}}"
type: {{TYPE}}  <!-- incident / project / process -->
severity: {{SEVERITY}}  <!-- critical / major / minor -->
tags: []
---

# ポストモーテム: {{TITLE}}

## サマリー（3行以内）

## タイムライン
| 時刻 | イベント |
|------|---------|

## 根本原因（5 Whys）
1. Why:
2. Why:
3. Why:
4. Why:
5. Why:

## 影響範囲

## 対応内容

## 再発防止策
| アクション | 担当 | 期限 | ステータス |
|-----------|------|------|----------|

## 学び・教訓
```

### ポストモーテム運用ルール
- type に応じた使い分け:
  - `incident`: 障害・インシデント対応後
  - `project`: プロジェクト完了後の振り返り
  - `process`: プロセス改善の記録
- severity は影響範囲で判定:
  - `critical`: 本番障害、データロス
  - `major`: 大幅な遅延、品質問題
  - `minor`: 小規模な問題、改善事項
- ポストモーテムは blame-free（非難しない）の原則で記述する

- リスクベースドテストで優先順位をつける
- 非機能テストは初期段階から計画する

```markdown
|--------|------|---------|------|
```

```
|---------|--------|---------|

| パイプライン | ツール | スケジュール | SLA |
|------------|--------|-----------|-----|

## 非機能要件
| 項目 | 要件 |
|------|------|
| データ量 | |
| 更新頻度 | |
| レイテンシ | |
| 可用性 | |
| リテンション | |
```

### DWH設計運用ルール
- メダリオンアーキテクチャ（Bronze/Silver/Gold）を基本とする
- データリネージは必ず記録する
- データ品質ルールは各レイヤーで定義する
- パイプラインのSLAを明確にする
