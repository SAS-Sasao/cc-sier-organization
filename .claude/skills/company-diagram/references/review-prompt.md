# L2 独立レビュープロンプト — AWS構成図（LLM-as-Judge）

`/company-diagram` Phase 6 で、**秘書とは別の fresh な `general-purpose` agent** に渡して AWS構成図を採点させるためのプロンプトテンプレート。

秘書は以下のテンプレートの `{...}` を埋めて `Agent` ツールで起動する。
**重要**: agent には Read ツールで PNG 画像を実際に読み込ませ、視認評価をさせること。

---

- PNG画像パス: {png-path}

## レビュー手順（必須）

1. **Read ツールで PNG 画像を読み込む** — 実際にアーキテクチャ図を視認する
2. **Read ツールで詳細HTMLを読み込む** — 5セクション構成を確認
3. **Read ツールで IaC YAMLを読み込む** — CFn/CDK テンプレートの内容を確認
4. Pythonコードと画像・HTML・YAMLの整合性を検証する
5. 6軸で採点する

## 構造仕様（これに準拠しているかを判定）

### 詳細HTML 必須5セクション（順序固定）
1. **凡例** — 構成図画像の直下、Edge色ごとのフロー種別を日本語で説明
2. **概要** — 目的・背景・対象ユースケース
3. **データフロー** — flow ステップ表示、複数パターン時はバッジ分類
4. **レイヤー構成** — テーブル形式（レイヤー / AWSサービス / 用途）
5. **設計のポイント** — 重要判断・トレードオフ・BP（2〜5項目）

### IaC 必須ファイル
- `{filename}.yaml` — CFn または CDK YAML
- `{filename}-iac.html` — IaCビューア（YAMLを埋め込み表示）

### Python コード規約
- `with Diagram(...)` 開始、`show=False` 指定
- **ラベルは必ず英語**（diagrams パッケージは日本語フォント非対応）
- `Cluster()` でレイヤーグルーピング
- `Edge()` でフロー種別可視化

## 採点基準

各軸を 0.00〜1.00 で採点し、根拠を 1〜2 行で述べてください。

### s1. 構造準拠
- HTML 5セクションの順序と存在
- 1.00: 全5セクション順序正 / 0.70: 全5セクションあるが順序ずれ / 0.50: 1セクション欠落 / 0.30: 2+欠落 / 0.00: 章構造崩壊

### s2. IaC生成【致命軸】
- `{filename}.yaml` と `{filename}-iac.html` の両方が存在し、内容が図と整合
- 1.00: 両ファイル存在・整合 / 0.80: 存在するがノードと要素が一部不一致 / 0.50: 片方のみ存在 / 0.30: 内容が空・スタブ / 0.00: 両方欠落

### s3. PNG整合性（画像Read精度最大化）
- Read した PNG 画像に描画されたノード・エッジ・Cluster構造と、Pythonコード/HTMLレイヤー表が一致
- 1.00: 完全一致 / 0.80: 軽微な差異（ラベル表記揺れ等） / 0.50: ノード1-2件の差異 / 0.30: 重大な不整合 / 0.00: 画像とコードが別物

### s4. 凡例完全性
- 画像に含まれる全Edge色が凡例に記載され、凡例記述が画像と矛盾しない
- 1.00: 完全網羅 / 0.70: 1色欠落 / 0.50: 2色欠落 / 0.30: 凡例セクション存在するが空 / 0.00: 凡例欠落

### s5. 一覧ページ更新
- `docs/diagrams/index.html` にカード追記済み、件数（右側数字）更新済み
- 1.00: カード+件数更新 / 0.70: カードあり件数未更新 / 0.50: 件数のみ / 0.00: 更新なし

### s6. 英語ラベル準拠【致命軸】
- Python コード内の `with Diagram(...)` 下の label 文字列に日本語（ひらがな/カタカナ/漢字）が含まれない
- 1.00: 全ラベル英語 / 0.80: 1-2ラベルに全角記号 / 0.50: 日本語ラベル1件混入 / 0.30: 日本語ラベル複数 / 0.00: ほとんどが日本語

## 判定ルール

- 致命軸 (s2, s6) のいずれかが `< 0.5` → composite を **強制的に 0.00** とし `fail`
- それ以外: `composite = (s1 + s2 + s3 + s4 + s5 + s6) / 6`
  - `composite >= 0.85` → `pass`
  - `composite < 0.85` → `fail`

## 秘書側（呼び出し元）での後処理

1. レビュアー応答を JSON パース（失敗時は再試行1回、それでも失敗なら L2 fail）
2. `verdict == "pass"` かつ `critical_triggered == false` → Phase 7 へ進む
3. `verdict == "fail"` → `findings` と `fix_suggestions` を秘書にフィードバック
   - 致命軸 fail → 該当ファイルを直接修正（IaC再生成、Pythonコードのラベル英訳）
   - 非致命軸 fail → HTML/index.html を修正
   - 1回だけ修正 → 同じレビュアー条件で再採点
4. 再採点結果も fail → Phase 7 以降を中断、現在のブランチを維持してユーザーに JSON 結果を報告

---

## しきい値調整の運用メモ

- 初期しきい値: **composite ≥ 0.85**
- diagram は視覚成果物のため L2 の精度に限界がある。2-4 週間運用して fail 率が 30% を超える場合は 0.80 に緩和を検討
- s3 (PNG整合性) のスコア分布を特に注視する。画像Readの判定精度を実データで評価する
- 致命軸（s2, s6）のしきい値 0.5 は下げない（構造的な品質保証の最後の砦）
