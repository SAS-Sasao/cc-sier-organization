# 日次ダイジェスト統合実行 Skill

日次ダイジェストの巡回 〜 HTML 公開までの全工程を単一コマンドで自動化する。
従来は `wf-daily-digest`（ブランチ+PR）と `/company-digest-html`（main直コミット）を
ユーザーが 2 回に分けて呼んでいたが、本 Skill で 1 発呼び出しに統合する。

---

## 1. 適用条件

- `.companies/.active` に org-slug が存在する
- その組織の `masters/workflows.md` に `wf-daily-digest` が定義されている
- `.claude/hooks/generate-daily-digest-html.sh` が存在する

未定義組織で呼ばれた場合は「`wf-daily-digest` が未定義です」と報告し終了する。

---

## 2. 8フェーズ概要

| Phase | 名称 | 中断条件 |
|---|---|---|
| 1 | 前処理（org 特定・既存チェック・ブランチ作成） | git dirty, 本日分既存（--force未指定） |
| 2 | Web巡回（Agent Teams 並列） | 両teammateが0件返却 |
| 3 | MD集約 | - |
| 4 | L1 セルフ品質ゲート（構造） | 1回自動修正後もfail |
| 5 | L2 独立レビュー（内容） | composite < 0.85 が 1回リトライ後も継続 |
| 6 | PR作成 & 自動マージ | gh pr merge失敗 |
| 7 | HTML再生成 | スクリプト失敗 |
| 8 | task-log & Issue & 報告 | - |

各中断時はその時点までの成果物・task-log・部分 PR を保持したうえで、ユーザーに原因と復旧手順を報告する。

---

## 3. Phase 1: 前処理

```
1. .companies/.active から {org-slug} を読む
2. .companies/{org-slug}/masters/workflows.md から wf-daily-digest を確認
3. {date} = 今日（YYYY-MM-DD）
4. {task-id} = YYYYMMDD-HHMMSS-daily-digest
5. 本日分 MD 存在チェック:
     .companies/{org-slug}/docs/daily-digest/{date}.md
     存在かつ --force 未指定 → 中断して報告
6. git status --porcelain が空でなければ中断して報告
7. {branch} = {org-slug}/feat/{date}-daily-digest
8. git checkout -b {branch}
9. .task-log/{task-id}-daily-digest.md を YAML フロントマターで作成
    （subagents: [general-purpose-tech, general-purpose-retail, general-purpose-reviewer]）
```

task-log の **subagents 欄には必ず英字** で記録する（Case Bank 検出のため）。

---

## 4. Phase 2: Web巡回（Agent Teams 並列）

以下の 2 agent を `Agent` ツールで **同一メッセージ内に並列起動**する。

| team | subagent_type | 担当 |
|---|---|---|
| tech | `general-purpose` | `info-source-master.md` 技術系 優先度「高」 |
| retail | `general-purpose` | `info-source-master.md` 小売系 優先度「高」 |

**注意**: `tech-researcher` / `retail-domain-researcher` 型は WebFetch を持たないため使用不可。
必ず `general-purpose` を指定する。

### 各 agent に渡すプロンプト要件

- `info-source-master.md` の該当カテゴリ優先度「高」を全件巡回
- 各記事について **タイトル・URL（https://〜 絶対パス必須）・1行要約** を返却
- タイトル中に半角 `[...]` が含まれる場合は **全角 `【...】`** に置換して返却
- 取得失敗ソースは理由を併記
- 応答形式はテーマ別に章分けしたテーブル（`| # | 記事 | ソース | 要約 |`）

---

## 5. Phase 3: MD集約

秘書が Phase2 の 2 結果を統合し、`.companies/{org-slug}/docs/daily-digest/{date}.md` を生成する。

フォーマットは **`masters/quality-gates/by-type/daily-digest.md` のフォーマットテンプレートセクションに従う**。要点のみ再掲:

- 章順序固定: ハイライト → A章(技術) → B章(小売) → C章(クロスドメイン分析) → D章(巡回メタデータ)
- A章/B章はサブセクション（A1〜A6, B1〜B6）のテーマ別分類
- 記事はテーブル形式（リスト形式 `- [タイトル](URL) — 要約` は禁止）
- C章のみパラグラフ形式で 3〜5 トピック、SIer 示唆を含める
- MDリンクタイトル内の半角 `[...]` は全角 `【...】` に置換（HTML変換でリンク消失防止）

---

## 6. Phase 4: L1 セルフ品質ゲート（構造）

秘書自身が `masters/quality-gates/by-type/daily-digest.md` の **必須項目** を全件セルフチェックする。

**チェック観点（構造）**:
- 全記事タイトルが `[タイトル](URL)` 形式
- URL が `https://` で始まる
- ヘッダー・ハイライト・A章・B章・D章の存在
- 優先度「高」ソースが全て巡回されている
- 半角 `[...]` 残存なし

**リトライ**: fail項目があれば 1 回だけ自動修正 → 再チェック。それでも fail なら中断して報告。

---

## 7. Phase 5: L2 独立レビュー（内容）

**別の fresh な `general-purpose` agent** を `Agent` ツールで起動し、生成した MD を渡してレビューさせる。
秘書自身にレビューさせない（authoring bias 回避）。

### 起動方法

```
Agent(
  description: "Daily digest L2 review",
  subagent_type: "general-purpose",
  prompt: <references/review-prompt.md の内容 + MD フルテキスト + 評価対象日付>
)
```

### 採点

レビュアーは `references/review-prompt.md` に従い **6軸スコア（各0-1）** と **composite** を返す:

1. 章構成準拠
2. ソースリンク完全性 **【致命軸】**
3. 要約品質
4. クロスドメイン分析の質
5. 重複処理・テーマ分類の妥当性
6. 禁則違反（全角括弧・テーブル形式・絵文字Mermaid等） **【致命軸】**

**判定ルール**:
- 致命軸 (2, 6) のいずれかが `< 0.5` → composite を強制的に 0 とし fail
- それ以外は `composite = 等重み平均`、`≥ 0.85 で pass`

### リトライ

fail 時は指摘事項を秘書にフィードバックして MD を 1 回修正 → 再レビュー。
再度 fail なら **auto-merge を中止**し、現状ブランチを維持したままユーザーに報告する。

---

## 8. Phase 6: PR作成 & 自動マージ

```bash
git add .companies/{org-slug}/docs/daily-digest/{date}.md \
        .companies/{org-slug}/.task-log/{task-id}-daily-digest.md
git commit -m "feat: 日次ダイジェスト {date} を生成（技術{N}件+小売{M}件={合計}件） [{org-slug}] by {operator}"
git push origin {branch}
gh pr create --title "feat: 日次ダイジェスト {date} [{org-slug}]" \
  --body "$(PR本文)"
gh pr merge --auto --squash --delete-branch
```

### PR 本文テンプレート

```markdown
## 概要
- 日次ダイジェスト {date}（{org-slug}）
- 技術 {N} 件 + 小売 {M} 件 = 合計 {合計} 件

## L1 セルフ品質ゲート
- status: PASS
- retries: {0|1}

## L2 独立レビュー
- reviewer: general-purpose (fresh)
- composite: {score}
- breakdown: 章構成 {s1} / リンク {s2} / 要約 {s3} / クロスドメイン {s4} / 重複 {s5} / 禁則 {s6}
- retries: {0|1}

## 成果物
- `.companies/{org-slug}/docs/daily-digest/{date}.md`

🤖 Generated by /company-daily-digest
```

---

## 9. Phase 7: HTML再生成（main直コミット）

auto-merge 完了後、main を取り直して HTML を再生成する:

```bash
git checkout main
git pull origin main
bash .claude/hooks/generate-daily-digest-html.sh {org-slug}
git add docs/daily-digest/ docs/index.html
git commit -m "chore: ダッシュボード更新 [company-daily-digest]"
git push origin main
```

**注意**: GitHub Actions などで auto-merge に遅延がある場合は `gh pr view {pr} --json state --jq .state` が `MERGED` になるまで待つ（最大 60 秒ポーリング、timeout で中断報告）。

---

## 10. Phase 8: task-log & Issue & 報告

### task-log 更新（YAML フロントマター必須）

```yaml
---
task_id: "{task-id}"
org: "{org-slug}"
operator: "{operator}"
status: completed
mode: "agent-teams"
started: "{YYYY-MM-DDTHH:MM:SS}"
completed: "{YYYY-MM-DDTHH:MM:SS}"
request: "{ユーザー依頼原文}"
issue_number: {n}
pr_number: {n}
subagents: [general-purpose-tech, general-purpose-retail, general-purpose-reviewer]
l1_gate: {pass|fail}
l1_retries: {0|1}
l2_composite: {0.00}
l2_retries: {0|1}
---
```

### `## judge` セクション追記（dashboard 統合のため必須）

task-log YAML フロントマターに加えて、`## judge` セクションを以下の YAML 形式で**必ず**追記する。これは `rebuild-case-bank.sh` が読み取り、`/company-dashboard` の「judge スコア推移」グラフに反映される。

6 軸 → 3 軸マッピング:
- `completeness` = `(s1_structure + s5_dedup) / 2`
- `accuracy` = `(s2_links + s3_summary) / 2`
- `clarity` = `(s4_cross_domain + s6_violations) / 2`
- `total` = `l2_composite`

```markdown
## judge

\`\`\`yaml
completeness: {0.00}
accuracy: {0.00}
clarity: {0.00}
total: {l2_composite}
failure_reason: ""
judge_comment: "/company-daily-digest l2_scores から自動マッピング: completeness=avg(s1_structure,s5_dedup), accuracy=avg(s2_links,s3_summary), clarity=avg(s4_cross_domain,s6_violations)"
judged_at: "{ISO8601 with TZ}"
\`\`\`
```

### Issue 作成

```bash
gh issue create \
  --title "[{org-slug}] 日次ダイジェスト {date}" \
  --label "org:{org-slug},mode:agent-teams,type:daily-digest" \
  --body "$(Issue本文 / references/task-log-template.md に従う)"
```

### 最終報告フォーマット

```
✅ 日次ダイジェスト {date} を公開しました！

技術 {N} 件 + 小売 {M} 件 = 合計 {合計} 件

L1品質ゲート: PASS（retry {0|1}）
L2独立レビュー: composite {score}（retry {0|1}）

PR:     {pr_url} (merged)
Issue:  {issue_url}
HTML:   https://sas-sasao.github.io/cc-sier-organization/daily-digest/#{date}
```

## 11. オプションフラグ

| フラグ | 既定 | 効果 |
|---|---|---|
| `--force` | off | 本日分 MD 既存でも上書き |
| `--no-merge` | off | Phase 6 で PR 作成まで実行、`gh pr merge` はスキップ（手動マージを要求）。Phase 7 以降は実行しない |
| `--dry-run` | off | Phase 1〜5 まで実行、Phase 6 以降はスキップ（MD 生成とレビュースコアの確認のみ） |

---

## 12. エラー時の中断ポリシー

- 各 Phase で fail した時点で task-log を `status: blocked` で保存し、ユーザーに **現在の Phase・原因・手動復旧コマンド** を報告する
- 部分的に作成済みのブランチ / PR は削除しない（ユーザーが手動で引き継げるように）
- HTML 再生成（Phase 7）のみ失敗時は PR は既に merged なので、`/company-digest-html` の手動再実行を案内する
