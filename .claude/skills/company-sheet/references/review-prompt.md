# L2 独立レビュープロンプト — Excel 試算表（LLM-as-Judge）

`/company-sheet` Phase 7 で、**秘書とは別の fresh な `general-purpose` agent** に渡して
生成された xlsx を採点させるためのプロンプトテンプレート。

秘書は以下のテンプレートの `{...}` を埋めて `Agent` ツールで起動する。
**重要**: 値の正誤（数式の計算結果が正しいか）は採点しない。Phase 2 の壁打ちで人間が承認済み。
L2 は **YAML→xlsx の写像と数表の構造妥当性**を採点する。

---

## 起動時に渡すプロンプト全文

```
あなたは独立レビュアーです。以下の Excel 試算表（xlsx）を 6 軸で採点してください。
あなたは作成者ではないため、バイアスなく厳密に評価してください。
重要: 数式の計算結果の正誤は採点しません（人間が承認済み）。構造と写像を評価します。

## 評価対象
- 組織: {org-slug}
- シート名: {name}
- xlsx パス: docs/office/{name}.xlsx
- xlsx テキスト抽出結果（シート/列/数式の一覧）: {xlsx-text}
- サムネイル PNG パス: {thumbnail-path}
- YAML 設計図パス: {yaml-path}
- task-log request 原文: {request}

## レビュー手順（必須）

1. **Read ツールで YAML 設計図を読み込む** — 確定した assumptions / sheets / 列分類を把握
2. **xlsx テキスト抽出結果を確認** — 全シート・全列・数式が存在するか
3. **Read ツールでサムネイル PNG を読み込む** — 入力セルと計算セルの書式区別を目視確認
4. YAML 設計図と xlsx の整合性（写像が壊れていないか）を検証する
5. request 原文に対し、数表の列構成・粒度が依頼意図を満たすか評価する
6. 6軸で採点する

## 構造仕様（これに準拠しているかを判定）

### YAML→xlsx 写像
- `assumptions[]` → `_assumptions` シート（または先頭シート上部）に key/value/note の表
- `sheets[].columns` → 各シートのヘッダー行
- `sheets[].input_cols` → 手入力セル（塗りつぶし色で区別）
- `sheets[].formula_cols` → 数式セル（`=` 始まりの式が入る、openpyxl は計算しない）
- `sheets[].rows` → データ行
- `images[]`（任意）→ 指定セルに画像挿入

### 列分類の原則
- `input_cols ∩ formula_cols = ∅`（入力と計算は排他）
- 全列が input か formula のどちらかに分類される

### 書式規約
- input セル: 塗りつぶし色（例: 薄黄）で手入力箇所を明示
- formula セル: 塗りつぶしなし or 淡色（入力と区別）

## 採点基準

各軸を 0.00〜1.00 で採点し、根拠を 1〜2 行で述べてください。

### s1. 構造準拠【致命軸】
- YAML の全シート・全列が xlsx に存在するか
- 1.00: 全シート全列一致 / 0.70: 列順ずれ / 0.50: 1列欠落 / 0.30: 1シート or 複数列欠落 / 0.00: 構造崩壊

### s2. 写像完全性【致命軸】
- input_cols / formula_cols が漏れなく反映されているか
- 1.00: 全 input/formula 反映 / 0.70: 軽微な漏れ1件 / 0.50: formula 1件欠落 / 0.30: 複数欠落 / 0.00: ほぼ未反映

### s3. 数式整合性
- 数式列に `=` 始まりの式が入っている / 参照先が範囲内 / 循環参照なし
- 1.00: 全数式健全 / 0.70: 軽微な参照ずれ / 0.50: 数式列の一部が値になっている / 0.30: 参照範囲外あり / 0.00: 数式が機能しない

### s4. 入力↔計算の書式区別
- input セルと formula セルが書式（塗りつぶし色等）で区別されているか（サムネイル PNG で確認）
- 1.00: 明確に区別 / 0.70: 区別あるが弱い / 0.50: 一部のみ / 0.30: ほぼ区別なし / 0.00: 全く区別なし

### s5. request 充足
- 列構成・粒度・シート分割が task-log の request 意図を満たすか
- 1.00: 依頼意図を的確に充足 / 0.70: おおむね充足 / 0.50: 一部不足 / 0.30: 重要な観点欠落 / 0.00: 依頼と乖離

### s6. 禁則違反【致命軸】
- xlsx 破損 / 空シート / assumptions 未反映 / input と formula の重複
- 1.00: 違反なし / 0.70: 軽微な1件 / 0.50: assumptions 一部未反映 / 0.30: 空シート or 重複 / 0.00: 破損

## 判定ルール

- 致命軸 (s1, s2, s6) のいずれかが `< 0.5` → composite を **強制的に 0.00** とし `fail`
- それ以外: `composite = (s1 + s2 + s3 + s4 + s5 + s6) / 6`
  - `composite >= 0.85` → `pass`
  - `composite < 0.85` → `fail`

## 出力形式（JSON のみ、コードブロック不要、他の前置き一切なし）

{
```

## 秘書側（呼び出し元）での後処理

1. レビュアー応答を JSON パース（失敗時は再試行1回、それでも失敗なら L2 fail）
2. `verdict == "pass"` かつ `critical_triggered == false` → Phase 8 へ進む
3. `verdict == "fail"` → `findings` と `fix_suggestions` を秘書にフィードバック
   - s1 / s2 fail → YAML と xlsx の写像を見直し、欠落シート/列/数式を補完
   - s4 fail → input セルの塗りつぶし書式を付与
   - s6 fail → 破損/空シート/assumptions 未反映を修正
   - **値の正誤に関する findings は対象外**（人間承認済みのため、構造の指摘のみ反映）
   - 1回だけ修正 → 同じレビュアー条件で再採点
4. 再採点結果も fail → Phase 8 以降を中断、現在のブランチを維持してユーザーに JSON 結果を報告

---

## しきい値調整の運用メモ

- 初期しきい値: **composite ≥ 0.85**
- 写像の健全性は L0/L1 で決定論的に担保されるため、L2 は構造妥当性（特に s5 request 充足）が中心
- s5 は主観が入りやすいため、過剰な fail が s5 に偏る場合はしきい値を 0.80 に緩和を検討
- 致命軸 (s1, s2, s6) は写像の壊れを捕捉するため、しきい値ではなく Phase 4 の生成ロジック側で改善する
